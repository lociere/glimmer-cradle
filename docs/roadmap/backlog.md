# Backlog

> 范围：记录有价值但尚未承诺的候选能力；不作为实施计划。
> 事实依据：蓝图差距、历史阶段材料、真实运行反馈和当前里程碑外的增强需求。
> 维护触发：候选项被拒绝、晋升为里程碑、合并到现有里程碑或失去价值。

候选项只有在补齐成果、范围、非范围、依赖、风险、owner 和验收门后，才能进入 `milestones/`。没有触发条件、没有 owner、没有验收方式的愿望不保留。

## 拒绝或移出规则

- 与蓝图方向冲突且无 ADR 支持：拒绝并记录原因。
- 已被当前里程碑吸收：移出 backlog，写入对应 milestone。
- 价值依赖尚未出现：保留触发条件，不提前实现。
- 已完成：事实进入 Architecture/Reference/Guide，过程进入 History。

## 晋升为里程碑前必须补齐

每个候选项晋升前必须写明：目标用户或系统成果、范围、非范围、依赖、风险、owner、验收门、完成后文档归属。若无法补齐这些内容，说明它仍是想法，不应进入当前推进面。

Backlog 审阅时应主动删除失去价值的条目。Roadmap 的价值在于让当前承诺更清晰，而不是保存所有曾经想到的功能。
