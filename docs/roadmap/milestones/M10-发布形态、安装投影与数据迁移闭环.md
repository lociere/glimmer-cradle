# M10：发布形态、安装投影与数据迁移闭环

- 状态：completed
- 关联架构：[当前架构](../../architecture/current/README.md)
- 关联参考：[Packaging Layout Reference](../../reference/packaging-layout.md)、[Data Layout Reference](../../reference/data-layout.md)、[Configuration Reference](../../reference/configuration.md)
- 关联指南：[客户端打包](../../guides/release/客户端打包.md)、[数据迁移与恢复](../../guides/operations/数据迁移与恢复.md)、[日志、Trace 与 DLQ 排障](../../guides/operations/日志、Trace与DLQ排障.md)

## 目标成果

Glimmer Cradle Personal Server 从开发期可运行闭环进入可安装、可升级、可诊断、可恢复的正式产品形态。发布物不依赖源码树、不写安装目录、不携带私人资产或密钥；Desktop 继续共享配置、数据、Extension、可观测性和生命周期边界，但其 Windows、Avatar 与本机 Audio 打包矩阵独立验收，不与服务器发行形成耦合发布门。

## 范围

- 固化开发期路径、安装目录、用户数据域、组件域、缓存域、日志域和备份域的 resolver 与打包映射。
- 验证 Personal Server 的公开 Release、OCI 镜像、一键安装器、只读版本目录、宿主配置域、数据卷、健康检查和浏览器登录。
- 建立新装、升级、回滚、卸载保留、坏数据恢复和重复迁移的发布期验证路径。
- 让 GitHub Release、GHCR 和离线完整包共享同一版本、摘要与 OCI digest；安装器保留可信 HTTPS 来源参数，网络差异不得形成第二套安装逻辑。
- 保留不含镜像的 `glimmer-cradle-personal-server-v<version>-linux-amd64.tar.gz` 轻量部署包作为标准在线安装载体；新增 `glimmer-cradle-personal-server-v<version>-linux-amd64-full.tar.gz` 完整安装包，携带同一次发布构建导出的镜像归档，供区域弱网、Registry 不可达和离线安装使用。
- 让标准在线、可信 HTTP(S) 完整包和本地离线包统一进入同一安装器、候选版本、健康检查、原子切换与回滚链路；项目方区域副本在出现稳定用户规模和网络需求后再独立验收。
- 检查包内容不包含真实 `configs/secrets/secrets.yaml`、provider key、私人模型、个人声线、开发缓存、测试输出或本机临时投影；仅允许交付空值 `secrets.example.yaml` 模板。
- 让打包版启动、ready/degraded、process log、DLQ、退出回收和缺组件诊断可解释。
- 补齐发布验证手册、打包路径 Reference 和必要的操作排障入口。

## 非范围

- 不新增核心认知功能、真流式回复、LLM 记忆重排或多平台大集成。
- 不补做 M09 的跨场景记忆、真实 Skill handler、聊天输出修整或 UI 可用性收口；若这些验收门未满足，本阶段不得进入 in-progress。
- 不把发布便利作为绕过 Cognition / Kernel / Renderer / Extension 边界的理由。
- 不把官方 Audio、Avatar、Cognition 或 Kernel 生命周期伪装为 Extension。
- 不承诺所有第三方 Extension 生态分发治理；该内容属于后续 Extension 内容生态阶段。
- 不在本阶段实现完整 Personal Server 网页配置、远程日志工作台、Extension 商店或 NapCat Linux 场景闭环；这些成果由 [M11](./M11-Personal%20Server控制面、区域分发与跨产品Extension闭环.md) 承担。
- 不把尚未完成的 Desktop Windows、Avatar 和本机 Audio 打包矩阵写成 Personal Server 的发布门；该产品组合另行验收。
- 不在尚未取得 Debian 实机记录时声明 Debian 正式支持；首个服务器支持基线为 Ubuntu 24.04 LTS、linux/amd64。
- 不在缺少确认 UI 或权限审计时放行高风险工具。

## 依赖

- M09 已完成的主体可用性、跨场景记忆、真实 Skill handler、UI 确认/诊断体验和开发期闭环；普通聊天自动 tool-use 闭环若被确认进入近期实施，应先作为 M09 后修整项或独立里程碑完成，不混入发布打包。
- `packaging-layout`、`data-layout`、`configuration`、`observability` 等 Reference 能表达当前路径与状态事实。
- 打包脚本、Electron builder、Unity Avatar 投影、Audio 资源目录和 Native 构建产物可在 Windows 开发环境定位。
- Personal Server Release workflow、OCI build、Compose、安装器和宿主运维命令能使用同一版本事实与健康门。
- Kernel runtime/readiness 能区分进程存在、协议握手、资源准备、首帧 present、业务 ready、degraded 和 failed。

## 风险

| 风险 | 应对 |
|---|---|
| 打包版依赖源码相对路径而开发期未暴露 | 搜索硬编码路径，优先通过 resolver、catalog 或配置 owner 修正 |
| 用户数据升级覆盖或不可恢复 | 迁移前备份，legacy 只读，迁移可重复，恢复步骤写入 Guide |
| 私人资产、secret 或 provider key 进入包 | 包内容检查作为发布验收门，发现后删除产物并修打包规则 |
| Avatar/Audio 开发期可用但安装投影缺组件 | 把组件定位、缺失诊断和 degraded/failed 投影纳入打包版验证 |
| 安装目录被运行时写入 | 修路径 owner 和 resolver，不通过提升权限或复制目录规避 |
| 退出后 sidecar、stdio、WebSocket 或子进程残留 | 发布验证覆盖退出回收、process log 和端口释放 |
| GitHub/GHCR 在部分网络不可稳定访问 | 当前使用完整包和显式可信来源完成降级；真实规模触发后再由同一流水线复制到项目方区域端点，安装器只选择传输源并强制校验 |
| 区域副本漂移成第二套发行物 | 版本、SHA-256、OCI digest、SBOM 与 provenance 不一致时阻断发布 |
| 完整安装包与 GHCR 镜像独立构建而产生内容漂移 | 镜像只构建一次；完整包从该发布结果导出，禁止为对象存储重新构建 |
| 过早同时维护多个 Registry、镜像站和安装脚本 | 保留一套安装协议、完整包与显式来源参数；达到真实规模和层复用触发条件后再评估区域 HTTPS 端点或 Registry |

## 第一验收门

- 盘点现有打包脚本、resolver、配置、数据目录、组件目录和发布指南，列出开发路径与安装投影差距。
- 建立 M10 发布验证清单，覆盖新装、升级、回滚、坏数据、缺组件、无 secret、日志路径和退出回收。
- 对当前代码做硬编码源码路径和安装目录写入风险搜索，形成需要修正的 owner 列表。
- 同步 `packaging-layout`、`data-layout`、`configuration`、`客户端打包` 中与真实代码不一致的路径事实。
- 在受支持 Linux 上验证 Personal Server 全新安装，记录区域网络下载与镜像拉取阻塞，并验证权威在线源、完整包和显式可信来源遵守同一摘要与镜像身份。

## 最终验收门

- `pnpm typecheck`、`pnpm build` 通过；受影响 Python 子项目 `uv run pytest -q` 通过。
- 打包产物可构建，并用打包版而非开发服务器完成启动验证。
- Personal Server 可从公开 Release 一条命令安装；完整包可在不访问 OCI Registry 的情况下安装同一内容，可信 HTTPS 或本地目录来源复用同一校验和事务链路。
- Release 明确区分轻量部署包与完整安装包；完整包能够在不访问 OCI Registry 的前提下导入同一镜像并完成安装，两个包的版本、镜像身份和校验信息一致。
- 新装、升级、回滚、卸载保留、坏数据恢复和重复迁移都有验证证据或明确未覆盖原因。
- 用户数据、日志、缓存、模型和备份写入用户数据域；安装目录保持只读语义。
- 包内容检查确认无 secret、私人模型、个人声线、开发缓存和测试产物。
- 无新增非测试 DLQ；主日志、trace 和 process log 能定位发布期关键失败。
- `stop`、更新失败和回滚后无残留容器、宿主端口、子进程或临时安装状态。
- Architecture、Reference、Guide 与实际代码、配置、脚本和打包产物同步。

## 当前验收记录

截至 2026-07-18，发布与安装主链已完成以下可重复验证：

- tag workflow 从同一次 BuildKit 镜像结果生成轻量包与完整包，两个包、稳定安装器进入同一 `SHA256SUMS`；完整包不触发第二次镜像构建。
- 安装器以 `GLIMMER_CRADLE_RELEASE_SOURCE` 和 `GLIMMER_CRADLE_PACKAGE_VARIANT` 统一默认 GitHub Release、可信 HTTPS 目录和本地目录；远程明文 HTTP、摘要篡改、越界路径与链接归档在写入安装目录前停止。
- 外层摘要负责传输完整性，两个包共享的 `RELEASE-MANIFEST.sha256` 负责只读部署内容身份，因而在线与完整包重复安装同版本不会形成第二版本事实；镜像归档不持久化到版本目录。
- 已在 Ubuntu 24.04.4 LTS、linux/amd64 的本机 WSL/Docker 环境中，用 v0.1.1 digest 固定正式镜像完成完整包离线加载、`/readyz`、重复安装、主动一致性备份、坏数据标记恢复、数据连续性和 `stop` 回收验证；另以必定退出的 v0.1.2 故障候选完成更新故障注入，旧镜像、数据和 `current` 指针均自动恢复。验证结束后两个容器、Compose 网络与宿主 `8080` 监听均释放。该环境不是全新裸机，因此不替代 Ubuntu 全新宿主验收记录。
- `scripts/test-personal-server-release.sh` 固化了双包内容、敏感路径、摘要篡改、明文 HTTP、链接归档、TERM 中断回滚和 `current`/CLI 入口准备失败负向测试；`scripts/verify-personal-server-full-install.sh` 固化真实镜像安装验证。安装器只用 EXIT 执行一次清理，INT/TERM 明确退出为 130/143；部署脚本在健康门、状态持久化、历史清理和结果标记全部成功前保持事务可回滚。
- 在阿里云 Ubuntu 24.04.2 LTS、linux/amd64 轻量服务器上完成当前安装器的轻量包重复安装、版本不可变拒绝、健康恢复、访问令牌保持、备份、恢复和正式 `v0.1.1` 入口复原。实机验收发现并修复了轻量包在 `set -e` 下计算文件名时静默退出的问题，并新增自动化成功路径防回归。

M10 至此完成。项目方区域副本、Debian 支持和 Desktop/Avatar/Audio 产品打包矩阵均保留为独立后续成果；未完成真实验收前不得写入当前支持矩阵。
