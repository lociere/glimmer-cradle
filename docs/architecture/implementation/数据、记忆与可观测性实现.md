# 数据、记忆与可观测性实现

> 范围：`data/` 目录、Cognition / Kernel 数据 owner、记忆与 Experience 持久化、observability、诊断索引、bundle 与 cleanup 的代码入口。  
> 源码依据：Cognition `memory/`、`memory/storage/`、`experience/`、`observability/`，Kernel `foundation/storage/`、`foundation/logger/`、`foundation/observability/`，Desktop `src/main/observability-query.ts`。  
> 维护触发：目录、owner、Schema、索引、bundle、cleanup、DLQ 或 process log 实现变化。

## 数据 owner 地图

| 数据 | 代码 owner | 物理位置 |
|---|---|---|
| Experience Ledger | `experience/ledger.py`、`recorder.py` | `data/state/cognition/experience/catalog.db` 与 `packs/YYYY/YYYY-MM.experience.db` |
| Conversation 投影 | `conversation/store.py`、`controller.py` | `data/state/cognition/conversations/conversations.db` |
| Episode 投影 | `experience/episodes.py` | `data/state/cognition/projections/episodes.db` |
| Cognition Memory 事实库 | `memory/storage/database.py` 与各 repository | `data/state/cognition/memory/memory.db` |
| Kernel 基础设施库 | `core/kernel/src/foundation/storage/db-manager.ts` | `data/state/kernel/` |
| Extension state | Kernel extension storage repository 与扩展 owner | `data/state/extensions/` |
| Desktop 本地状态 | Desktop main 持久化入口 | `data/state/desktop/` |
| 模型 / 包 / 缓存 / 工作材料 | 对应 engine 或 service owner | `data/models/`、`data/packages/`、`data/cache/`、`data/work/` |
| 可观测数据 | 对应进程 logger / tracer / ledger / query service | `data/observability/` |

业务代码不得直接拼源代码相对路径写状态；应通过 resolver 或 owner 暴露的受控入口。

## Cognition 记忆实现

| 实现 | 职责 |
|---|---|
| `conversation/store.py` | 从 Ledger 派生消息、Chapter、Segment 与 Conversation State |
| `conversation/controller.py` | 投影 checkpoint、Working Set 恢复与作用域受控 Prompt 查询 |
| `memory/substrate.py` | 版本化 Memory 的加载、写入与有预算检索 |
| `memory/consolidation.py` | Episode 筛选、结构化推理、证据校验与幂等 run |
| `memory/storage/consolidation_job_repo.py` | 持久任务、claim/lease、退避重试与崩溃重领 |
| `memory/relationship_projection.py` | 按 checkpoint 消费 Moment，以 `moment_id` 幂等派生互动计数 |
| `memory/knowledge_base.py` | 角色知识 |
| `memory/storage/memory_repo.py` | Memory item/revision/evidence 持久化 |
| `memory/storage/relationship_repo.py` | 互动计数、关系 revision/evidence 持久化 |
| `memory/storage/knowledge_repo.py` | 知识条目持久化 |
| `memory/storage/vector_repo.py` | embedding 向量 |

Kernel 不直接写 Cognition Memory，只消费投影、动作结果与受控回执。所有 Memory revision 必须绑定 Moment evidence；没有 evidence 的写入会被拒绝。

## Experience 实现

```text
experience/events.py
  -> Moment / MomentKind / SourceDescriptor / retention ceiling
experience/recorder.py
  -> 单写者记录门面与缓冲刷新
experience/ledger.py
  -> catalog + 月度 SQLite pack + position/causation 索引 + 完整性校验
experience/episodes.py
  -> 可重建 Episode 投影、边界、封口与待巩固队列
```

Experience 是角色经历，不是普通运行日志。provider stderr、stack trace、性能采样进入 observability，不进入 Experience。Episode 和叙事可以删除重建；Ledger 不可由 projection 或日志反向改写。

Global Workspace 的候选、竞争结果、周期广播与 thought Intent 不写入 Ledger，只由认知循环 span、metrics 和 Presentation thought frame 观察。`MomentKind` 不包含 `thought`。Ledger 记录角色实际经历与具有因果意义的状态事实；运行节拍、注意力候选和重复采样属于可观测性。

`activity/` 的 `engaged / ambient / quiescent` 迁移同样属于调度可观测性，只写 `cognition.activity.state` gauge、`cognition.activity.transition` counter、结构化日志和 `state_sync` 快照，不写 Ledger。`maintenance/` 在终结 Moment 后立即处理 sealed Episode，并按独立节拍补偿扫描，写维护 span/metric、Episode projection、consolidation run 与有证据 Memory；维护调度本身不产生 Moment，停机也不运行模型巩固。

## Kernel storage 实现

Kernel storage 承载 Extension Host 基础设施状态、宿主配置投影与 Kernel DLQ。它不保存角色会话或 Cognition Memory，也不能把 renderer store 写成系统事实源。

## 已落地的真实接入

- `SkillInvocationGateway` 写 `events/` 与 `audit/`
- Desktop main 对 open URL、扩展动作等高风险 IPC 写 `audit/`
- Cognition `ReasoningService`、`AgentPlanUseCase`、`AgentSynthesisUseCase` 进入 模型调用观测
- Kernel DLQ 已带恢复字段，`scripts/dlq.py` 可读
- Kernel / Cognition metrics 已收敛 label 白名单

## Desktop 诊断查询与维护

P1/P2 的主实现集中在 `products/desktop/src/main/observability-query.ts`：

- 读取 `events/`、`audit/`、`model-invocations/records/`、`traces/` 与 Kernel DLQ
- 在 `data/observability/index/observability.db` 维护 SQLite 查询索引
- SQLite 不可用时 fallback 到 JSONL / DLQ scan
- 导出 diagnostic bundle
- 导出 process log tail 截断片段
- 按保留期 cleanup
- cleanup 后重置 index，等待下次查询重建

对应 IPC 位于 `src/main/ipc-handlers.ts`：
- `ui:get-observability-recent-errors`
- `ui:get-observability-trace`
- `ui:get-observability-maintenance`
- `ui:export-observability-bundle`
- `ui:cleanup-observability`

Preload 与 Renderer 类型入口：
- `src/preload/index.ts`
- `src/renderer/global.d.ts`

Control Center 消费入口：
- `src/renderer/components/control-center/pages/logs/LogsPage.tsx`

Renderer 只消费 projection、maintenance status、bundle export result 与 cleanup result，不直接触碰 `data/observability/` 或 `observability.db`。
其中模型调用观测的 full capture 引用保留在 Desktop main；Renderer 只能看到摘要字段，不能通过公开 IPC 直接拿到 prompt、provider payload 或 raw response 的 capture 引用。

## 应用 console 日志引用

受管进程 stdout/stderr 与第一方应用文件日志统一写入 `data/observability/logs/application/`。Desktop 查询层的 `process_log_ref` 只返回规范化 `path`、`owner`、`source` 与 `exists`，不保留旧目录 alias。

## Bundle 与 Cleanup

bundle 由 Desktop main 导出到 `data/observability/bundles/`。默认只包含摘要、trace、audit、LLM summary、DLQ summary、process refs、runtime summary 与 process tail，不默认包含完整 prompt 或 raw response。

cleanup 入口当前只处理可再生观测数据：
- `events`
- `traces`
- `metrics`
- `audit`
- `llm summary`
- `llm full captures`
- `logs/application`
- `bundles`
- 已 `resolved` / `replayed` 且超过保留期的 DLQ

不会清理：
- Cognition state
- Experience
- 用户导入模型 / 资源
- 扩展私有状态
- 托管包
