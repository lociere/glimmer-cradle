# 当前架构阅读说明

> 范围：说明 `architecture/current/` 的阅读顺序、信息边界和与蓝图/实现文档的关系；不保存字段表、命令步骤或未来计划。
> 事实依据：`docs/README.md`、[文档维护规范](../../文档维护规范.md)、当前仓库目录与 `architecture/blueprint/`。
> 维护触发：文档分类、架构目录结构、主要系统边界或维护规范变化。

`architecture/current/` 回答“Glimmer Cradle 当前系统是什么形状”。它采用接近 arc42 的组织方式：先说明目标和上下文，再说明运行拓扑、构件、关键链路、跨切面机制、子系统当前视图、部署形态和术语。这样读者可以从整体结构进入，而不是被某个历史阶段、某个包或某个实现细节牵着走。

## 与其他文档的分工

| 文档域 | 回答的问题 | 本目录如何使用 |
|---|---|---|
| [blueprint](../blueprint/README.md) | Glimmer Cradle 为什么这样设计、长期审美与不变量是什么 | Current 必须与蓝图一致；若不一致，应写明当前偏差和收口方向 |
| current | 当前系统由哪些进程、构件、边界和链路组成 | 本目录的唯一职责 |
| [implementation](../implementation/README.md) | 这些架构概念在真实代码里从哪里进入、如何组装、如何调试 | Current 只给系统视角，代码入口链接到 Implementation |
| [reference](../../reference/README.md) | 字段、配置键、目录、SDK、日志与打包映射的精确事实 | Current 不复制字段表，只链接权威页 |
| [guides](../../guides/README.md) | 开发、迁移、排障和发布的执行步骤 | Current 不写操作流程，只给边界依据 |
| [roadmap](../../roadmap/README.md) | 未完成工作的成果、风险与验收门 | Current 只描述已落地事实，不写愿望清单 |

## 阅读顺序

1. [01-目标、约束与质量属性](./01-目标、约束与质量属性.md)：确认 Glimmer Cradle 不是聊天窗口，而是承载当前角色连续人格和桌面身体的本地优先多进程系统。
2. [02-系统上下文](./02-系统上下文.md)：确认用户、桌面、平台扩展、模型服务、数据域与 Glimmer Cradle 的边界。
3. [03-运行拓扑与进程边界](./03-运行拓扑与进程边界.md)：确认 Kernel、Cognition、Desktop、Avatar、Engines、Extension/MCP 的故障域和 readiness。
4. [04-构件、分层与依赖](./04-构件、分层与依赖.md)：确认仓库包和允许依赖方向。
5. [05-关键运行时链路](./05-关键运行时链路.md)：确认启动、输入、认知、输出、语音、身体与停机链路。
6. [06-跨切面机制](./06-跨切面机制.md)：确认协议、配置、数据、权限、日志、trace、metrics 与 DLQ 的统一规则。
7. [07-子系统当前视图](./07-子系统当前视图/README.md)：确认 Kernel、Cognition、Desktop/Avatar、Engine、Extension、Data/Observability 的当前子系统事实。
8. [08-部署与交付形态](./08-部署与交付形态.md)：确认源码树、构建投影、安装目录、用户数据和外部运行时之间的关系。
9. [09-术语表](./09-术语表.md)：统一跨文档术语。

## 维护规则

- Current 只写当前已落地或已成为运行时约束的结构。仍在候选或未验收状态的内容进入 Roadmap。
- 当蓝图长期不变量与当前实现之间存在距离，Current 应描述“当前事实”，并链接 Roadmap 或 ADR 描述收口计划；不得把未来形态伪装为已落地事实。
- 当一个陈述需要具体字段、目录、配置值、命令或 API，Current 只写概念位置，并链接 Reference。
- 当一个陈述需要源码入口、组合根、状态机、调用链或调试点，Current 只写架构关系，并链接 Implementation。
- 历史阶段材料已归档到 `docs/history/`。历史材料可以作为证据来源，但不能覆盖 Current 的事实。
