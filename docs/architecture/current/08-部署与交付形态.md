# 部署与交付形态

> 范围：源码树、开发投影、安装包、只读资产、外部运行时、用户数据和更新边界；不写逐步打包命令。
> 事实依据：Desktop 打包配置、路径 resolver、`scripts/`、`assets/`、`data/`、Avatar/Audio 构建脚本与发行约束。
> 维护触发：构建输入、安装目录、部署单元、资源投影、更新策略、用户数据布局或平台目标变化。

Glimmer Cradle 的部署形态不是源码树的机械复制。源码目录表达开发所有权；构建投影表达组件如何被打包；安装目录表达只读发布形态；用户数据目录表达本机连续性。任何运行时都必须通过语义 resolver 找路径，不能依赖“当前源码相对路径刚好存在”。

## 四类形态

| 形态 | 例子 | 职责 | 禁止 |
|---|---|---|---|
| 源码事实源 | `products/`、`core/`、`protocol/`、`engines/`、`native/`、`assets/` | 主产品开发、构建输入、默认资产 | 写入用户状态、加载扩展源码、保存运行缓存 |
| 开发投影 | Unity `Assets/Resources/AvatarModels/`、本地 DLL、生成 catalog | 让开发工具消费源码事实 | 手工修改投影并当成事实源 |
| 安装投影 | Electron `resources/app.asar`、`resources/components/*`、只读默认资产 | 发布给用户的应用与组件 | 保存用户状态、写日志、下载第三方模型 |
| 用户数据域 | `data/state`、`data/models`、`data/cache`、`data/observability` | 本机连续性、模型、日志、缓存、备份 | 进入 Git 或被安装更新覆盖 |

## 部署单元

| 单元 | 来源 | 发布语义 |
|---|---|---|
| Desktop 产品组合 | `products/desktop/` | `Glimmer Cradle Desktop`，组装 Kernel、Avatar、Audio、Extension Host 与 Electron 表面 |
| Server 产品组合 | `products/personal-server/` | `Glimmer Cradle Personal Server`，组装无 Avatar 的 Kernel 与受认证 HTTP/WebSocket 服务边界 |
| Kernel | `core/kernel/` | TypeScript runtime，作为生命周期与能力中枢 |
| Cognition | `core/cognition/` | Python 独立进程与项目环境，由 Kernel 监督 |
| Audio Engine | `engines/audio/` | 官方音频 engine，TTS/ASR lane 可独立 warmup 和诊断 |
| UnityAvatarHost | `core/avatar/unity-host/` + native composition | Avatar Host 构建包和平台合成边界 |
| Native 组件 | `native/` | DirectComposition/D3D/FFI 等平台原语，作为构建产物投影 |
| Extension | `data/packages/extensions/<id>/<version>/` | 独立仓库发布的可选生态能力，按 manifest/config 激活；主产品不从源码仓库加载 |
| MCP Server | `configs/system/skills.yaml` 指向的外部 server | 外部工具 provider，按配置托管或连接 |

每个部署单元都需要声明构建输入、版本、许可证、运行依赖、资源检测和缺失诊断。未启用的可选增强属于正常基础运行形态；只有用户显式启用后仍无法启动或服务时，才进入对应能力的 degraded/failed，不能生成假可用投影。

## Personal Server 容器边界

Personal Server 以 Compose 应用栈交付：一个非 root、只读根文件系统的应用容器持有 Kernel、Cognition、Personal Server Host、Extension Host 和可选云端 TTS；Caddy 作为独立入口容器承担 TLS 与反向代理。Kernel 继续拥有内部子进程生命周期，Compose 只监督产品级部署单元，二者不重复争夺同一进程所有权。

镜像内部应用根固定为 `/opt/glimmer-cradle/app`。Python 环境在 builder 和最终镜像中保持同一绝对路径，不搬移既有虚拟环境；宿主机上的仓库、Compose 目录和持久化盘位置可以变化。配置与数据分别挂载到 `/var/lib/glimmer-cradle/config`、`/var/lib/glimmer-cradle/data`，短期协调状态使用 `/run/glimmer-cradle` tmpfs。应用端口 `3210` 仅存在于 Compose 内部网络，宿主机只暴露 Caddy。

## Avatar 与 Native 发布边界

Avatar 当前由 UnityAvatarHost 和 Native Composition Host 呈现到桌面 Surface。开发期 `pnpm avatar:sync`、`pnpm avatar:doctor`、`pnpm avatar:build` 会同步模型资产、SDK 和 native 适配器。Unity 模型投影、第三方 SDK 导入目录、构建 DLL 和 Host executable 是构建产物，不进入源码事实源。复杂身体 readiness 必须以 `host_ready` 和首帧 present 为准。

Windows DirectComposition、D3D11 surface、透明命中和 DPI 由 native/composition 边界处理。禁止用颜色键、普通透明 WebView、动态 `setShape()` 或隐藏 Unity 普通窗口替代正式发布路径。

## 用户数据与更新

安装更新不得覆盖 `data/state/`、`data/models/`、`data/backups/` 或用户导入资产。可重建缓存可以清理，但不能把缓存清理伪装为状态迁移。旧路径迁移必须先备份不可再生状态，再执行显式 migration，并在 Reference/Guide 中说明删除条件。

## 验收要求

- 打包前检测 required 资产、native 组件、Avatar 构建产物、音频资源 catalog 和路径 resolver。
- 打包后用真实安装投影验证：启动、ready/degraded、Control Center、Avatar、音频、日志、DLQ、退出回收。
- 发行包不得包含真实 `configs/secrets/secrets.yaml`、个人模型、私有 Avatar 资产、开发缓存、测试输出或本地 provider key；空值 `secrets.example.yaml` 作为首次配置模板可随默认配置交付。
- 任何路径差异必须由 resolver 吸收，业务代码不判断“开发/生产目录结构”。

具体映射见 [Packaging Layout Reference](../../reference/packaging-layout.md)，操作步骤见 [客户端打包](../../guides/release/客户端打包.md)。
