# 跨切面机制

> 范围：协议、配置、数据、权限、安全、可观测性、错误与兼容策略这些横跨多个子系统的规则；不维护具体字段全表。
> 事实依据：`protocol/`、`configs/`、`data/`、Kernel/Cognition/Desktop/Engine/Extension 实现、Reference 文档。
> 维护触发：跨边界契约、配置来源、数据布局、权限模型、日志/trace/metrics/DLQ、错误码或兼容策略变化。

## 事实源、契约、投影与缓存

| 名称 | 含义 | 可以写入者 | 禁止行为 |
|---|---|---|---|
| 事实源 | 某个状态或结构的权威存储 | 唯一 owner | 多进程共同写同一原始状态 |
| 契约 | 跨边界稳定语义，如 Schema、SDK Port、Engine 协议 | 契约 owner | 从消费者倒推字段并长期双轨 |
| 投影 | 面向 UI、日志、诊断、catalog 的只读视图 | owner 或映射层 | 让投影反向成为事实源 |
| 缓存 | 可删除、可重建的性能材料 | 缓存 owner | 把缓存当用户连续性或迁移源 |

这个四分法是 Glimmer Cradle 跨切面的底线。常见错误是：Renderer 用本地 store 推断系统 ready；Extension 把 handler 当权限事实；模型缓存被当作用户记忆；旧字段在 producer 和 consumer 间无限保留。

## Protocol

- 两个及以上进程/语言共享的结构以 `protocol/src/schemas/` 为权威。
- 生成物位于 TS/Python 消费端，禁止手改。
- 事件必须携带稳定 `kind/type`、错误 code、trace 上下文和足够诊断信息。
- 破坏性语义变化应设计迁移或删除旧路径；不要通过堆叠 optional 字段制造隐式版本。

精确规则见 [Protocol Reference](../../reference/protocol.md)。

## 配置

- 系统默认配置位于 `configs/system/`；角色相关 Cognition 配置位于 `configs/characters/<character-id>/`，由 `configs/system/identity.yaml` 的 `character.active_id` 选择；Extension 配置位于 `configs/extensions/`。
- Kernel 启动只加载和校验配置，不生成系统配置或角色包，也不对缺失角色选择做默认角色 fallback；安装器、仓库模板和受控配置命令负责创建配置。
- 一个配置项只有一个写入 owner；运行时消费经过 Schema/normalizer 校验的投影。
- 密钥不进入普通 YAML、日志、文档、Skill 或示例；只进 `configs/secrets/` 或环境变量。
- Renderer 不能直接读取原始配置文件；必须消费 Electron main/Kernel 提供的安全投影。

精确入口见 [Configuration Reference](../../reference/configuration.md)。

## 数据布局

- `assets/` 是随源码管理的只读默认资产源。
- `data/state/` 保存用户连续性状态，例如 Cognition 数据库、renderer 持久化外观、扩展状态。
- `data/models/` 保存本机模型和用户导入模型，不进 Git。
- `data/cache/` 保存可重建缓存，例如 TTS 合成缓存。
- `data/work/` 保存运行产物，例如 ASR 临时输入。
- `data/observability/` 保存日志、trace、metrics、DLQ。
- `data/packages/` 保存本机托管第三方运行时，例如 GPT-SoVITS 或 Avatar 投影。
- `data/backups/` 保存迁移或用户主动备份；旧路径迁移完成后必须从运行主线删除。

精确目录见 [Data Layout Reference](../../reference/data-layout.md)。

## 权限与安全

- Extension 通过 manifest 声明 `contributes`、`requires`、`permissions`、`activationEvents`；`requires` 是 Port 需求，不等同授权。
- Skill Plane 在调用前执行 Policy；风险、确认、副作用和审计必须能解释。
- MCP 远端工具按保守策略处理：未声明只读的工具不能默认当安全能力。
- Desktop preload 只暴露白名单 API；诊断动作只能打开预定义路径或受控投影。
- 日志与 DLQ 不记录密钥、token、完整 provider key 或用户隐私大 payload。

## 可观测性

- 日志解释离散事件和失败摘要。
- trace/span 解释一次跨进程操作的关键路径。
- metrics 解释趋势和容量。
- DLQ 保存无法安全处理但需要追查的结构化失败。
- process log 保存子进程 stdout/stderr；主日志只保留摘要和路径。

所有入口应创建或继承 trace，跨 IPC、WebSocket、stdio、Engine、Provider 和工具调用延续它。高频身体帧和逐帧参数默认不进主日志，只在显式 debug trace 中记录。

精确字段见 [Observability Reference](../../reference/observability.md)。

## 兼容与迁移

Glimmer Cradle 偏向“明确迁移并删除旧路径”，而不是长期兼容壳。旧配置、旧字段、旧目录、旧 bridge 和旧协议只允许在迁移窗口内存在，必须有 owner、删除条件和验证方式。完成迁移后，历史材料进入 `docs/history/`，当前事实收口到 Architecture/Reference/Guide。
