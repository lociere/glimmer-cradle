# ADR-0010 产品组合与扩展仓库边界

- 状态：accepted
- 日期：2026-07-15（2026-07-17 修订扩展生态仓库布局）

## 背景

Desktop 原本位于 `core/desktop/`，使产品组装层与本体能力层混在同一物理层级；Personal Server 尚无独立组合入口。与此同时，NapCat 扩展既是主 workspace 源码又是运行期发现目录，导致主产品构建、扩展开发和用户安装态互相耦合。

## 决策

1. 主仓库采用一个产品 monorepo，`products/desktop/` 与 `products/personal-server/` 是平级 Product Composition。
2. 用户名称分别为 `Glimmer Cradle Desktop` 与 `Glimmer Cradle Personal Server`；稳定技术 ID 为 `desktop` 与 `personal-server`。
3. `products/<id>/product.json` 是组合实例，统一遵循 Protocol 的 `ProductComposition` 契约；Kernel 只按清单启停 Control Surface Gateway、Avatar、Audio 与 Extension Host。
4. `core/` 只保存本体能力和中枢，不保存产品 Shell；Desktop 内部仍保留 Electron 的 main/preload/renderer 正式进程术语。
5. 第一方扩展源码与默认审核目录共同进入独立 `glimmer-cradle-extensions` 生态仓库，分别位于 `extensions/` 与 `registry/`；两者共享审核和 CI 入口，但不共享 Manifest、制品或版本事实。第三方 Adapter 使用发布者自己的仓库，只通过 `registry/catalog.json` 参与默认目录。主仓库只保留 Protocol、公开 SDK、基础模板、宿主和安装能力。
6. Extension Host 只发现 `data/packages/extensions/<id>/<version>/` 的已安装发布物，不扫描扩展源码仓库；多个版本可并存，激活配置精确选择版本。
7. 扩展 Registry 是可审计目录，发布者仓库的 Release artifact 是不可变发布物，用户安装目录是本地投影；三者不得混为一个目录或事实源。
8. 开发期删除 `core/desktop/`、主仓库 `extensions/`、`scripts/launch-local.mjs` 和旧端点命名，不保留兼容别名。

## 结果

- Desktop 与 Personal Server 共享同一 Kernel/Cognition/Protocol 主线，又能按产品组合拥有不同宿主、网络和 Avatar 策略。
- 扩展可以独立发布、升级和回滚，不再拖慢主产品启动准备，也不再获得 workspace 内部依赖特权。
- 主仓库与扩展仓库拥有清晰的发布节奏；公开 SDK 使用语义化版本作为唯一跨仓库代码依赖。
- 第一方扩展与默认 Registry 的变更可以在同一 PR 中原子审核，避免双仓库同步；扩展仍按各自 SemVer 和不可变 Release 独立发布。

## 验证

- `pnpm-workspace.yaml` 包含 `products/**`，不包含主仓库 `extensions/**`。
- `pnpm dev:desktop` 与 `pnpm dev:personal-server` 均由 `scripts/launch-product.mjs` 建立并回收完整进程树。
- `products/personal-server` 在非回环绑定时没有 token 会拒绝启动。
- Extension Host 默认根目录是 `data/packages/extensions`，NapCat 可从独立仓库安装后被发现。
- 架构检查阻止旧 Desktop、Extension 源码和启动入口重新出现。
- `glimmer-cradle-extensions` 根校验同时检查 `extensions/` 和 `registry/catalog.json`；旧 `glimmer-cradle-extension-registry` 只保留迁移说明，不再形成第二事实源。
