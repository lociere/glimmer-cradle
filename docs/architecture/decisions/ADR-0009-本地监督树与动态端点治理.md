# ADR-0009：本地监督树与动态端点治理

- 状态：accepted
- 日期：2026-07-15

## Context

Kernel、Desktop、Cognition、Audio、Avatar 和 Extension 同时运行时，固定端口会与用户机器上的其他服务冲突，也会让测试实例互相干扰。仅依赖子进程自行退出还会在 Windows 留下 QQ、Unity 或脚本外壳，在 Linux 留下脱离父进程的进程组。Extension 若在 Kernel 进程内加载，还会把第三方依赖、阻塞和崩溃传播到本体中枢。

## Decision

1. 仓库开发入口由 `scripts/launch-product.mjs` 按 Desktop 或 Server 产品组合监督 Kernel 与产品 Host；任一主进程退出都会回收其余本地进程树。
2. Kernel 是内部通信端点的唯一 owner。Cognition RPC、Desktop Shell 和 Avatar Host 均绑定 `127.0.0.1` 的动态端口，并原子发布到 `data/run/host/endpoints.json`。消费者校验 catalog owner PID、purpose 和回环地址后连接，不从普通配置读取固定端口。
3. Kernel 继续拥有 Cognition、Audio、Avatar 与 Extension 子树的协议停机、超时和强制回收。Windows 回收根 PID 进程树；POSIX 使用独立进程组；生产 Linux 再由 systemd/cgroup/容器提供最终期限。
4. 每个启用 Extension 在独立 Extension Host 进程中加载。Kernel 只读取 manifest 和静态声明，通过 SDK RPC 暴露受权限约束的 Port；停用或 Host 退出时撤销 handler、catalog 与订阅。
5. 本体对话只等待 Cognition 与必要传输。Audio warmup、Avatar ready 和 Extension activation 属于后台可降级能力，不阻塞文本入口；它们必须持续发布真实 readiness。
6. 第三方协议要求 TCP/WS 时由所属 Adapter 管理端点。NapCat OneBot 默认使用自动选择的回环端口；外部可访问端点必须显式配置和授权，不进入 Kernel 内部端点 catalog。

## Consequences

- 多实例、测试和用户已有服务不再争抢摇篮内部固定端口。
- Extension 故障被限制在独立进程，代价是所有 Host Port 必须可序列化并承担 RPC 延迟。
- 动态 catalog 是可再生产物，不是长期状态；最后一个端点撤销或 Kernel 停机时立即删除，消费者不得缓存跨代端点。
- 自动选择第三方端口仍受第三方启动间隙限制，因此 readiness 与重试必须保留，不能把“已分配”当成“已监听”。
- 开发监督器不替代生产监督。云端部署仍必须配置 cgroup、停止宽限期、持久卷和单实例数据库约束。

## Alternatives considered

- 固定端口加冲突重试：仍把实现细节泄露到配置，且多实例行为不确定。
- Kernel 内加载 Extension：调用快，但无法隔离第三方崩溃、依赖污染和遗留进程。
- 所有能力启动完成后才开放输入：冷模型或可选集成会无意义地拉长文本可服务时间。
- 只在停机时杀直接子进程：无法覆盖脚本外壳、Unity CrashHandler、QQ 或 POSIX 进程组。
