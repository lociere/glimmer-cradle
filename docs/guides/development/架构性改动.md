# 架构性改动

> 适用场景：改变进程边界、owner、依赖方向、生命周期、协议主线、数据域、权限模型或长期不变量。
> 前置条件：已读 [架构蓝图](../../architecture/blueprint/README.md)、[当前架构](../../architecture/current/README.md)、相关 Implementation 和现有 ADR。

## 步骤

1. 写清用户/系统可观察目标，以及为什么普通功能修复不足以解决。
2. 描述现状：当前 owner、事实源、进程、依赖、链路、失败语义和验证缺口。
3. 描述目标：新 owner、唯一事实源、边界契约、生命周期、ready/degraded、删除对象。
4. 对照蓝图不变量：Cognition/Kernel/Renderer/Extension/Engine 是否仍各守其位。
5. 如改变长期取舍，先写 ADR；如改变目标形态或架构审美，先更新 Blueprint。
6. 按“契约/Port -> producer -> mapping -> consumer -> projection -> deletion”顺序落地。
7. 删除旧主线、旧 bridge、旧字段、旧配置和无期限 fallback。
8. 更新 Current、Implementation、Reference、Guide 和 Roadmap/History。

开发阶段以概念完整性为优先：如果没有已发布版本或明确迁移窗口，架构升级必须一次切换到目标物理形态。不得用旧目录套新解释、双配置读取、旧字段 alias、双写或无期限 fallback 降低迁移成本；Git 历史负责追溯。

## 设计检查

| 检查项 | 必须回答 |
|---|---|
| owner | 谁唯一拥有事实源和生命周期？ |
| boundary | 哪些数据跨进程/语言/信任边界？ |
| contract | 契约由 Schema、SDK Port 还是 Engine protocol 定义？ |
| readiness | 什么条件代表真实可用？什么是 degraded？ |
| failure | 缺依赖、超时、崩溃、重启、取消如何表现？ |
| observability | trace、日志、metrics、DLQ 如何定位？ |
| deletion | 哪些旧路径必须删除？如果暂留，何时删除？ |

## 失败与恢复

- 如果新设计需要两个 owner 同时写事实源，回到设计。
- 如果 UI 或 Extension 需要 import 内部对象才能工作，边界错误。
- 如果迁移无法一次完成，拆成带验收门的里程碑，不保留永久兼容。
- 如果发现蓝图与当前实现长期冲突，不用 roadmap 掩盖；应改蓝图/ADR 或明确实现收口。

## 验证

架构性改动至少运行类型/构建、受影响测试和一条真实启动/降级/停机链路。涉及窗口、Avatar、音频或外部 provider 时必须做实机或模拟失败验证。
