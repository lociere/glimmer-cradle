# 文档维护

> 适用场景：代码、Schema、配置、脚本、目录、流程、架构边界或路线图变化后同步文档。
> 前置条件：已读 [文档维护规范](../../文档维护规范.md)，并确认本次变化的唯一事实源。

## 步骤

1. 判断信息类型：
   - 设计审美/长期不变量：Blueprint。
   - 当前结构/边界：Current。
   - 代码入口/链路：Implementation。
   - 字段/配置/路径/API：Reference。
   - 操作步骤：Guide。
   - 未完成承诺：Roadmap。
   - 长期取舍：ADR。
   - 过期材料：History。
2. 先更新唯一权威页，其他页面只更新链接或一句摘要。
3. 核对代码、Schema、配置、脚本和测试，不凭记忆写“当前事实”。
4. 用 `rg` 搜索旧术语、旧路径、旧字段、旧命令、冲突陈述和断链。
5. 如果迁移旧文档，把原始材料放入 `docs/history/`，在活跃页写清新入口。
6. 运行文档自查：非索引页是否有范围、事实依据、维护触发；Guide 是否有步骤、失败、验证、文档同步。

## 迁移旧文档的方法

- 不把旧文档原样搬入新结构。先按信息类型拆分：架构思想进 Blueprint/Current，代码入口进 Implementation，字段进 Reference，步骤进 Guide，计划进 Roadmap。
- 旧文档中仍有效的具体约束要迁移到权威页，例如日志字段、音频超时、Avatar 透明命中、Extension 权限。
- 旧文档中已过期的阶段描述留在 History，并在页首说明当前入口。
- 迁移完成后搜索旧路径，避免新文档继续指向 `docs/extensions/`、旧 `architecture/current/07-Cognition认知核.md` 或旧 roadmap 文件。

## 验证

- 相对链接存在。
- 活跃文档没有只剩链接列表的空页。
- 文档说“已落地”的内容能在代码、Schema、配置或测试中找到证据。
- 文档没有密钥、token、个人 provider key、私人模型路径。

## 需要同步的文档

文档维护本身改变分类或规则时，同步 `docs/README.md`、`AGENTS.md`、`.codex/skills/glimmer-cradle/SKILL.md` 和相关 skill references。

## 完成定义

一次文档维护完成时，活跃入口应不存在旧路径残留，非索引页不应只是链接列表，Reference/Guide/Roadmap/Skill reference 均满足各自质量门。若保留历史材料，必须在页首声明当前入口，并确保它位于 `docs/history/` 或兼容目录，不再作为事实源。

完成后建议至少运行一次断链检查和旧路径搜索。旧路径包括历史 `docs/extensions/`、旧 `architecture/current/07-Cognition认知核.md`、旧平铺 `docs/guides/*.md`、旧 roadmap 文件名和旧 `.agents` reference 中的活跃事实。
