# 命名规范

> 适用场景：新增或重命名代码标识符、目录、包名、命令、配置键、事件、Git 分支、文档术语。
> 前置条件：已确认该名称的 owner、边界和使用层级；涉及角色身份时已确认它属于 Character Profile，而不是平台层。

## 原则

1. 名称表达职责，不表达当下实现细节。避免用 `runtime`、`manager`、`helper`、`util` 作为默认后缀。
2. 平台层不写死默认角色。`Selrena` 只出现在角色配置、角色资产、默认 profile、测试 fixture 或明确面向该角色的内容里。
3. 产品或企划名只用于对外品牌、发布物、仓库包 scope、文档标题和跨模块 namespace；内部模块优先用职责名。
4. 同一个概念只保留一个英文名；跨代码、Schema、配置和文档时保持大小写与拼写一致。
5. 中文文档和注释可以使用正式中文名“微光摇篮”，在上下文已明确时可以使用项目内简称“摇篮”；代码标识符、包名、命令、环境变量、Schema、IPC 和配置键仍使用稳定英文名 `Glimmer Cradle` / `glimmer-cradle` / `GLIMMER_CRADLE`。

## Git 分支命名

- `main` 是主线分支；已有历史分支不因本规则重写。
- 工作分支使用 `<type>/<scope>`，其中 scope 使用 lowercase kebab-case，并表达工作性质与边界。
- 稳定类型为 `feature/`、`fix/`、`refactor/`、`docs/`、`release/`、`chore/`；发布分支固定使用 `release/v<semver>`。
- 禁止使用 `codex/`、`claude/`、`copilot/` 或其他 agent/tool 身份前缀。分支名描述交付内容，不描述由哪个工具或 agent 完成。

示例：`feature/extension-catalog`、`fix/avatar-readiness`、`docs/development-workflow`、`release/v0.1.8`。

## 企划名称用法

| 场景 | 推荐写法 | 说明 |
|---|---|---|
| 文档标题、README、发布说明首次出现 | `Glimmer Cradle（微光摇篮）` | 同时给出英文正式名和中文正式名 |
| 中文正文、架构叙述、注释 | `微光摇篮` | 适合强调企划/平台语境 |
| 同一段落或同一文档已明确对象后 | `摇篮` | 仅用于自然语言，不用于机器接口 |
| 包名、scope、目录、命令、环境变量 | `@glimmer-cradle/*`、`glimmer-cradle`、`GLIMMER_CRADLE_*` | 保持英文稳定，避免跨平台和工具链问题 |
| Unity / TS / Python namespace | `GlimmerCradle.*`、`glimmer_cradle` 形态 | 按语言惯例拼写，不使用中文简称 |

## 推荐词表

| 场景 | 推荐命名 | 不推荐 |
|---|---|---|
| 进程或承载边界 | `Host`, `Shell`, `Worker`, `Service` | `Runtime` 泛用后缀 |
| 第三方开发包或导入目录 | `SDK`, `Package`, `Plugin` | `Runtime` |
| 生命周期监督单元 | `Runtime`, `RuntimeModule`, `Readiness` | `Service` 混用生命周期 |
| 状态与缓存投影 | `State`, `Snapshot`, `Projection`, `Store` | `Runtime` |
| 行为协调入口 | `Controller`, `Coordinator`, `Scheduler` | `Manager` 兜底 |
| 协议/平台转换 | `Adapter`, `Bridge`, `Port` | `Util` |
| 领域核心包 | `Core`, `Domain`, `Application` | 产品名或角色名硬编码 |
| 配置事实源 | `Config`, `Profile`, `Catalog`, `Manifest` | `Settings` 泛用 |

## Avatar 命名边界

| 概念 | 正式名称 | 禁止混用 |
|---|---|---|
| 人物完整本体 | `Character` | `Avatar`、Unity 进程 |
| 人物形象与身体领域 | `Avatar` | `Surface`、`Shell`、外显模块 |
| Avatar 实现容器 | `AvatarHost` | `Avatar` 本体 |
| 当前 Unity 实现 | `UnityAvatarHost` | 通用领域名 |
| 呈现和交互位置 | `Surface` | Avatar、Host |
| 桌面合成原语 | `NativeCompositionHost` | Avatar 行为或状态 owner |

UI 与中文文档把 Avatar 页面称为“形象”。“外显”只可作为“情绪外显为表情”等自然语言动词，不作为模块、页面、配置域、协议或进程名称。Avatar 配置归 `avatar.yaml`，Surface 配置归 `surfaces.yaml`。

## `Runtime` 的使用边界

允许：

- Kernel lifecycle 监督树和 readiness 模型，例如 `RuntimeModule`、`RuntimeReadinessSnapshot`。
- 语言、平台或第三方固定名称，例如 `.NET System.Runtime`、`RuntimeError`、Unity 官方 assembly 名。
- 文档中明确指“运行期生命周期对象”的 architecture term。

不允许：

- 把 SDK、插件、导入包、缓存目录、状态对象、UI hook、控制器命名为 `Runtime`。
- 用产品名加 `Runtime` 形成内部模块名。
- 用角色名命名平台层模块、进程、协议字段或通用测试 fixture。

## 改名检查

1. 搜索旧名、大小写变体、环境变量、脚本命令、Git 分支约定、文档链接和注释。
2. 生成物先改生成器，再运行生成命令，不手改生成物作为唯一修复。
3. 保留兼容旧数据时，在注释里写清楚 legacy 来源，例如 `legacy selrena.db`。
4. 改完运行相关 typecheck、测试和 `pnpm check:encoding`。
5. 架构改名必须同步目录、文件名、Schema、配置键、生成物、数据路径、构建产物、测试和文档；开发阶段不为旧名保留 alias。

## 需要同步的文档

命名规则变化时，同步 `docs/guides/README.md`、`docs/文档维护规范.md`、`AGENTS.md` 和 `.codex/skills/glimmer-cradle/references/common/文档、注释与编码.md`。

## 完成定义

新名称能从职责推出，不依赖作者记忆；Git 工作分支表达工作性质与 scope，不含 agent/tool 身份前缀；平台层没有误写默认角色；`Runtime` 只保留在生命周期或固定术语中；旧名称搜索结果要么为历史材料，要么有明确兼容理由。
