# 功能开发与缺陷修复

> 适用场景：在既有架构边界内新增功能、修复 bug、调整 UI/配置/服务行为。
> 前置条件：已读 [开发手册](../开发手册.md)，确认任务不需要先做架构性改动或 Schema 变更。

## 步骤

1. 执行 `git status --short`，确认哪些改动不是自己产生的。
2. 用 `rg` 找入口、owner、生产者、消费者、测试、配置和文档引用。
3. 写清本次可观察行为：用户会看到什么、日志/状态如何变化、失败时如何表现。
4. 确认唯一事实源：字段看 Protocol，配置看 Configuration，路径看 Data Layout，UI token 看 Reference。
5. 在既有抽象内实现：领域逻辑放 owner 层，平台 IO 放 adapter/main/native，UI 只消费投影。
6. 覆盖成功、空态、错误、取消、降级、重试和资源释放。
7. 删除被替代的旧入口、旧字段、旧事件、旧 bridge 和无期限兼容壳。
8. 运行与风险匹配的验证，并更新唯一权威文档。

## 判断是否升级为其他流程

| 迹象 | 转向 |
|---|---|
| 两个进程或语言需要同一新字段 | [Schema 与跨进程契约变更](./Schema与跨进程契约变更.md) |
| owner、进程边界、依赖方向或长期不变量变化 | [架构性改动](./架构性改动.md) |
| 涉及用户状态、旧目录、备份或不可再生数据 | [数据迁移与恢复](../operations/数据迁移与恢复.md) |
| 涉及发布投影、安装目录或外部 runtime | [客户端打包](../release/客户端打包.md) |

## 失败与恢复

- 如果实现需要读取私有对象或跨层 import，停止并重设边界。
- 如果只能靠 UI 本地状态“猜测”系统状态，改为由 owner 提供投影。
- 如果 fallback 会让用户以为能力已 ready，改成 degraded/failed 状态。
- 如果修复依赖保留旧双轨，写明删除条件；无删除条件则不应合入。

## 验证

最低验证通常是 `pnpm typecheck` 和受影响包的 build/test。涉及 Cognition 执行 `cd core/cognition; uv run pytest -q`；涉及 Audio 执行 `cd engines/audio; uv run pytest -q`；涉及 UI 运行对应 Playwright/desktop 验证；涉及 runtime 则验证启动、ready、degraded、崩溃和停机。

## 需要同步的文档

- 改字段/事件：`reference/protocol.md` 和对应 Implementation。
- 改配置/路径：`reference/configuration.md` 或 `reference/data-layout.md`。
- 改子系统链路：对应 `architecture/implementation/`。
- 改开发步骤：对应 Guide。
- 改长期边界或不变量：Blueprint/Current/ADR。
