# 测试与验收

> 适用场景：为改动选择验证范围、运行测试、记录交付证据、说明未验证风险。
> 前置条件：已识别改动 owner、用户可见行为、跨边界契约、失败/降级路径和不可再生数据风险。

## 基线命令

```powershell
pnpm check:architecture
pnpm typecheck
pnpm build
pnpm test
```

根测试会执行架构适配度检查、Cognition、Audio 与 Kernel 测试；涉及 UI、Avatar、打包或外部 provider 时仍需追加对应实机验证。

## 风险到验证

| 改动 | 必须补充 |
|---|---|
| Protocol/Schema | `pnpm sync:contracts`、合法/非法 payload、旧字段搜索、TS/Python 消费端 |
| Kernel runtime | 启动、ready、degraded、failed、restart、stop、子进程回收 |
| Cognition | `cd core/cognition; uv run pytest -q`、provider 错误、outbound |
| Audio Engine | `cd engines/audio; uv run pytest -q`、health、warmup、synthesize/recognize、timeout |
| Desktop UI | desktop typecheck/build、Electron 实机、preload、窗口/托盘 |
| Avatar | `pnpm avatar:doctor`、`pnpm avatar:build`、Unity/Native/打包版实机 |
| Extension/Skill Plane | 注册、拒权、确认缺失、调用、失败、断连、dispose |
| Data migration | 备份、旧版本、损坏样本、重复迁移、回滚、旧路径搜索 |
| Observability | trace、日志、metrics、DLQ、process log、脱敏 |
| Packaging | 安装投影启动、ready/degraded、路径 resolver、退出回收 |

## 验收维度

- 成功路径：用户目标可观察完成。
- 空态：无配置、无数据、无模型、无扩展时不崩。
- 错误：非法输入、缺依赖、provider 失败、权限拒绝有清晰反馈。
- 降级：可选能力失败时不伪装 ready。
- 并发/重复：重复点击、重连、重启、重复事件幂等。
- 资源释放：订阅、计时器、文件句柄、子进程、WebSocket、stdio 回收。
- 可观测性：trace、日志、metrics、DLQ 能定位。
- 安全：密钥、token、provider key 和隐私 payload 不进日志/文档。

## 交付记录模板

```md
完成：
- ...

影响边界：
- ...

验证：
- `pnpm typecheck` 通过
- ...

未运行：
- ...（原因）

风险：
- ...

文档：
- ...
```

模板可以调整，但这些信息不能缺。涉及用户数据、密钥、外部平台、打包版或桌面身体时，必须明确是否做过真实环境验证。

## 失败处理

验证失败先定位 owner：契约、生产者、映射、消费者、投影、资源或环境。不要通过放宽测试、吞错、禁用类型检查、改成 warn、保留假 fallback 或跳过 ready 检查来制造“通过”。

如果环境缺失导致无法验证，写清：

- 缺什么；
- 如何复现验证；
- 已替代运行了什么；
- 剩余风险是什么。

## 常见错误

- 只跑 `pnpm typecheck` 就声称功能完成。
- 改 Cognition 不跑 `uv run pytest -q` 也不说明原因。
- 改 Avatar 只看 renderer，不验证 Unity/Native。
- 改 Schema 忘记 `pnpm sync:contracts`。
- 日志里有错误或 DLQ 但交付说明不提。
