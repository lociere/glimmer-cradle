# Glimmer Cradle 开发手册

> 适用场景：任何代码、文档、配置、Schema、Extension、Engine、Avatar 或发布改动的入口。
> 前置条件：已阅读根目录 `AGENTS.md`，确认当前工作树状态，不覆盖用户已有改动。

本手册是开发入口，不替代 Architecture、Implementation 或 Reference。开始前先判断“这次工作改变了哪个事实源、哪个 owner、哪个跨边界契约、用户会看到什么变化”。

## 1. 开始前检查

```powershell

```

然后按任务读取：

| 任务 | 必读 |
|---|---|
| 理解设计方向 | `docs/architecture/blueprint/` |
| 判断当前边界 | `docs/architecture/current/` |

## 3. 标准实施流程

1. 复现问题或明确目标行为。
2. 找 owner：Protocol、Kernel、Cognition、Desktop、Avatar、Engine、Extension、Data。
3. 找入口：读对应 Implementation 页，用 `rg` 搜生产者、消费者、配置、测试和旧术语。
4. 先改事实源：Schema、config、owner service、repository、runtime module 或 SDK contract。
5. 再改生产者、映射层、消费者、投影和 UI。
6. 删除被替代的旧字段、旧事件、旧 bridge、旧 fallback。
7. 同步物理形态：目录结构、文件名、配置键、默认模板、生成物和测试必须匹配新架构。
8. 跑风险匹配验证。
9. 同步唯一权威文档。

## 4. 常见任务完成定义

### Schema / 协议

- 改 `protocol/src/schemas/`。
- 运行 `pnpm sync:contracts`。
- TS/Python 生成物更新且未手改。
- producer、consumer、validator、错误 code、Reference 同步。
- 搜索旧字段无运行时残留；若暂留有删除条件。

### Runtime / 子进程

- runtime id、依赖、blocking/degradable、ready、failed、restart、stop 明确。
- 进程存在、协议连接、资源 warmup、业务 ready 分开。
- UI/log/snapshot 可解释 waiting/degraded/failed。
- 验证正常启动、缺资源、超时、崩溃、重启、停机和回收。

### Cognition

- 感知只走唯一认知主线。
- 平台 payload、窗口状态、Extension handler 不进入心智域。
- context、memory、reasoning、outbound 都带 trace。
- `cd core/cognition; uv run pytest -q` 已运行或明确说明未运行原因。

### Renderer / Avatar

- Renderer 只消费 preload 白名单和 Kernel 投影。
- Avatar ready 以 `host_ready` 和首帧为准。
- Control Center、Presence、Unity、Native 的失败状态不互相伪装。
- 涉及窗口/身体/音频时必须做实机或说明缺失条件。

### 数据迁移

- 识别不可再生状态并备份。
- migration 可重复、可失败恢复；迁移完成后删除旧入口读取代码。
- 新代码不再写旧路径。
- Reference、Guide、ignore、打包规则同步。

## 5. 验证矩阵

| 改动 | 最低验证 |
|---|---|
| 全仓 TS | `pnpm typecheck`、`pnpm build` |
| Protocol | `pnpm sync:contracts`、protocol/kernel typecheck、Cognition 测试 |
| Kernel | `pnpm --filter @glimmer-cradle/kernel typecheck` |
| Cognition | `cd core/cognition; uv run pytest -q` |
| Audio | `cd engines/audio; uv run pytest -q`，必要时 health/warmup/synthesize/recognize |
| Desktop | desktop typecheck/build、Electron 实机 |
| Avatar | `pnpm avatar:doctor`、`pnpm avatar:build`、Unity/打包版实机 |
| Extension/Skill | SDK/kernel typecheck，注册/拒权/调用/断连/dispose 验证 |
| Data migration | 备份、旧样本、坏样本、重复迁移、回滚、旧路径搜索 |

## 6. 交付说明

交付时必须说明：

- 完成了什么用户可见行为或工程边界；
- 改了哪些事实源；
- 运行了哪些验证及结果；
- 哪些验证未运行及原因；
- 剩余风险；
- 同步了哪些文档。

不要用“类型通过”代替功能完成；不要把未实机验证的 Avatar/窗口/音频写成“已验证体验”。
