# 数据迁移与恢复

> 适用场景：移动目录、修改持久化格式、迁移旧数据库、清理缓存、恢复用户状态或调整模型/资产布局。
> 前置条件：已读 [Data Layout Reference](../../reference/data-layout.md)，确认哪些数据不可再生。

## 步骤

1. 分类数据：用户状态、模型、缓存、运行产物、日志、备份、legacy。
2. 识别 owner：Cognition、Desktop、Extension、Engine、Avatar、Kernel。
3. 对不可再生状态先备份到 `data/backups/` 或用户指定目录。
4. 写迁移脚本或显式迁移流程，禁止运行时到处散落一次性 if。
5. 迁移只读旧 owner 路径，新写入必须进入当前 owner 的正式位置；迁移完成后删除旧入口读取代码。
6. 更新 resolver、ignore、打包配置、Reference 和 Guide。
7. 用旧数据样本验证迁移、回滚和重复执行。
8. 搜索旧路径，删除旧写入入口。

## 数据分类

| 数据 | 是否可直接删除 | 处理 |
|---|---|---|
| Cognition DB | 否 | 备份、迁移、校验 |
| 用户导入模型/声线 | 否 | 备份或保留路径，更新 catalog |
| 扩展状态 | 视扩展语义 | 由扩展 owner 提供迁移 |
| Avatar 外观偏好 | 否 | 迁移或 reset 到可解释默认 |
| TTS cache | 是 | 可删除重建 |
| ASR 工作输入 | 通常是 | 根据是否用户导出判断 |
| process logs | 可轮转 | 不作为记忆 |
| legacy 目录 | 只读 | 迁移完成后不再写入 |

## 恢复

恢复流程应能回答：恢复哪个 owner 的状态、从哪个备份、覆盖哪些文件、如何停止运行时、如何验证成功、失败如何回滚。恢复 Cognition 状态前必须停止相关 runtime；恢复模型或 Avatar 资产后必须重新跑资源检查或 avatar doctor。

## 验证

- 迁移前后用户可见状态一致或差异有说明。
- 重复运行迁移不会破坏数据。
- 缺文件、坏文件、旧版本、权限不足有清晰错误。
- 新代码不再写旧路径。
- 打包/安装形态不会覆盖用户数据。

## 需要同步的文档

目录和 owner 更新 [Data Layout Reference](../../reference/data-layout.md)；发布投影更新 [Packaging Layout Reference](../../reference/packaging-layout.md)；流程变化更新本页；长期迁移取舍写 ADR。

## 迁移完成定义

迁移完成不是“新代码能跑”。还必须确认：旧路径不再写入，备份可用于恢复，重复迁移安全，失败时不会破坏原数据，文档和 ignore/打包规则同步，用户能看到必要的错误或恢复提示。
