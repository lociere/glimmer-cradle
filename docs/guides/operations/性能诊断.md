# 性能诊断

> 适用场景：启动慢、回复慢、TTS/ASR 延迟高、Avatar 掉帧、UI 卡顿、MCP/Extension 调用超时。
> 前置条件：已读 [Observability Reference](../../reference/observability.md)，能取得 trace、metrics 或 process log。

P1 起优先通过 Control Center 诊断页或 Desktop main 查询服务取得 `trace_id` 对应的 projection；metrics 继续只保留低基数字段 label，`trace_id` 仅作为查询键，不进入 metrics labels。

## 步骤

1. 先定义“慢”的用户感知：首次启动、首次回复、连续回复、TTS 合成、ASR 转写、Avatar 首帧、UI 点击响应。
2. 取得基线：同一输入、同一配置、同一模型、冷/热启动分开记录。
3. 用 trace 拆分阶段，不先猜优化点。
4. 按 owner 定位：Kernel 路由、Cognition 上下文/LLM/记忆、Audio 模型、Avatar 首帧、Renderer 渲染、MCP 网络；必要时联查 `events/`、`llm/` 和 `audit/`。
5. 确认是否资源门前移：模型下载、warmup、runtime 导入不能藏在第一次业务请求里。
6. 修复后重复同一基线，记录改善和副作用。

## 常见瓶颈

| 症状 | 可能位置 | 处理 |
|---|---|---|
| 首次回复慢 | Cognition context、LLM provider、记忆检索 | 分 span，区分网络、上下文组装、模型推理 |
| 每次短句 TTS 慢 | Audio TTS lane、云端热连接、cache miss | 检查 CosyVoice provider、`data/cache/audio/tts/`、timeout |
| ASR 首次慢 | FunASR pipeline 或模型下载 | 确认 warmup 在 readiness 阶段完成 |
| Avatar 首帧慢 | Unity SDK、模型导入、Composition Host | 区分 `host_hello` 与 `host_ready` |
| UI 卡顿 | renderer 大组件、过多同步 IPC、日志倾倒 | 用受控投影和分页/折叠，不在 UI dump 原始日志 |
| MCP 调用慢 | 外部 server、网络、timeout | 设置 timeout、重连退避、provider degraded |

## 禁止做法

- 为了“快”跳过权限、Schema 校验、ready 检查或日志。
- 把模型下载放到第一次用户请求。
- 用隐藏 fallback 掩盖 provider 慢或失败。
- 在高频 Avatar/鼠标流中写主日志。
- 未测基线就替换技术栈。

## 验证

性能修复必须有前后数据：trace span、metrics、启动时间、首帧时间、合成时间或 UI 操作耗时。涉及用户体验的性能变化应同时验证错误和降级路径，避免只优化成功路径。
