# 音频引擎开发

> 适用场景：修改 TTS、ASR、Audio Engine、角色声线、云端路由、本地模型、录音转写、播放队列或音频状态投影。
> 前置条件：已读 [Engines 与 Native 当前视图](../../architecture/current/07-子系统当前视图/Engines与Native.md) 与 [Engines 与 Native 实现](../../architecture/implementation/Engines与Native实现.md)。

## 职责边界

| 层 | 负责 | 不负责 |
|---|---|---|
| Character Package `voice.yaml` | 角色声音身份、语言、稳定表达和 provider 声线绑定 | 系统路由、超时、密钥 |
| `configs/system/audio.yaml` | TTS 路由、provider 执行参数、熔断、缓存和 ASR 启停 | 角色身份、API Key |
| Kernel AudioService | Engine 监督、能力门、缓存、状态投影和业务调用 | provider 枚举、重试、模型调用 |
| Audio Engine | provider adapter、云端连接、路由、fallback、熔断、模型 warmup | Cognition 回复语义、UI |
| Desktop recorder/player | 录音、按 trace 排队播放、响度包络和 UI 状态 | TTS/ASR 路由 |

ASR 是感知入口，TTS 是表达出口。两者失败都不能伪装成 Cognition 失败或删除文本回复。

## 当前正式路线

```text
TTS: dashscope-cosyvoice
ASR: funasr
```

- `dashscope-cosyvoice` 使用 CosyVoice WebSocket 双向协议，连接在 TTS Engine 进程内复用；每个任务使用独立 `task_id`。
- 云端缺 key、缺角色 `voice_id`、连接失败或熔断时，TTS 明确投影为 `unavailable`；文本回复和 Cognition 状态不受影响。
- TTS 默认不启用，文本对话是完整的最小运行形态。显式启用 TTS 后，CosyVoice 是当前可用 provider；未来微调 provider 应通过同一 adapter/route 契约接入并成为 primary，CosyVoice 可配置为 fallback。
- FunASR 是当前唯一 ASR provider；UI 不提供无效的 provider 选择。
- 旧 Edge-TTS、Windows SAPI、Whisper、GPT-SoVITS sidecar 和 CosyVoice stub 已退出运行主线。

CosyVoice 的事件顺序以阿里云官方文档为准：[WebSocket API](https://help.aliyun.com/zh/model-studio/cosyvoice-websocket-api)、[客户端事件](https://help.aliyun.com/zh/model-studio/cosyvoice-client-events)、[服务端事件](https://help.aliyun.com/zh/model-studio/cosyvoice-server-events)。

## 配置

1. 在 `configs/characters/<id>/voice.yaml` 填写已创建或复刻的 CosyVoice `voice_id`。
2. 通过 `DASHSCOPE_API_KEY` 注入密钥；无法使用环境变量时才写 `configs/secrets/secrets.yaml` 的 `audio.dashscope.api_key`。
3. 在 `configs/system/audio.yaml` 调整云端执行、熔断与缓存参数；当前不提供虚假的本地 TTS 路线。
4. 配置变更后重启摇篮；Audio Engine 启动后配置被冻结。

密钥不得写入普通配置、日志、请求快照或测试 fixture。日志只能记录 credential 是否存在。

## Provider 开发规则

1. 先更新 `AudioConfig`、`VoiceConfig` 或 Audio Engine 协议 Schema，再运行 `pnpm sync:contracts`。
2. provider 必须实现 `available`、`warmup`、`synthesize_to_file`、`config_snapshot` 和 `close`。
3. 路由只调用明确声明且启用的 provider；不可执行实现不得进入 health 或资源目录。
4. 云端重试只覆盖瞬态失败，次数有界并使用指数退避；非瞬态配置错误立即失败。
5. 连续失败触发熔断；恢复窗口后才能再次尝试，避免每个角色短句都重复打向故障服务。
6. 输出先写随机 `.partial` 文件，成功后原子替换，失败不能留下可被缓存命中的半文件。
7. 子进程 stdout/stderr 进入 `data/observability/logs/application/audio-*.log`，主日志只记录 provider、fallback、耗时和结果摘要。
8. Engine Host 必须接受协议契约中的 `host.shutdown`，ACK 后关闭连接与模型资源并自然退出；Kernel 只在停机超时后回收进程树。
