# 扩展开发

> 适用场景：开发 Extension、平台 Adapter、Extension SDK、Skill Provider、MCP 接入、权限、确认、工具调用或扩展状态。
> 前置条件：已读 [Extension 与 Skill Plane 当前视图](../../architecture/current/07-子系统当前视图/Extension与SkillPlane.md)、[Extension 与 Skill Plane 实现](../../architecture/implementation/Extension与SkillPlane实现.md) 和 [Extension SDK Reference](../../reference/extension-sdk.md)。

## 改动路径

| 任务 | 主要文件/目录 |
|---|---|
| SDK API | `packages/extension-sdk/src/` |
| 官方扩展源码 | 独立 `glimmer-cradle-extensions/extensions/<extension-id>/` |
| 第三方扩展源码 | 发布者自己的独立仓库；不进入第一方仓库 |
| 默认审核目录 | `glimmer-cradle-extensions/registry/catalog.json`；与第一方源码共仓，只保存审核信息与作者侧发布来源指针 |
| 本机安装态 | 主产品 `data/packages/extensions/<extension-id>/<version>/` |
| 基础模板 | `templates/extension-basic/` |
| manifest/permissions | `packages/extension-sdk/src/manifest/`、`permissions/` |
| Extension Host | `core/kernel/src/host/extension-manager.ts` |
| Host service | `core/kernel/src/application/services/extension-host-app.service.ts` |
| Runtime projection | `core/kernel/src/application/services/extension-runtime-registry.ts`、`core/kernel/src/host/managed-resource-supervisor.ts` |
| Skill provider | `core/kernel/src/application/skill-plane/providers/extension/` |
| MCP provider | `skill-plane/providers/mcp-server/`、`configs/system/skills.yaml` |
| Core tools | `skill-plane/providers/core/` |

## 标准步骤

1. 明确扩展是生态能力，不是本体核心器官。
2. manifest 声明 identity、entry、`contributionPoints`、按 point id 分组的 `contributes`、requires、permissions、activationEvents。
3. 每个可执行或可投影 contribution 明确 `audience`：人物可用才写 `character`，管理 UI 写 `user`，Host lifecycle 写 `host`，协议/平台收发写 `adapter`，展示投影写 `renderer`，扩展内部写 `extension`。
4. requires 只表示需要 Port，不等于授权。
5. handler 只能通过 SDK/Port 调用宿主能力，不能 import Kernel 内部路径。
6. skill/tool 必须进入 Registry、Policy、Gateway，不绕过执行。
7. 停用、失败、升级时释放 disposable，并撤销 catalog。
8. 平台 Adapter 必须清洗 payload，再注入统一感知。
9. 需要认知接入时构造平台中立 `ConversationAddress`：普通输入走 `perception`，证据候选走带 `sourceEventId` 和 `schemaRef` 的 `evidenceProposal`；canonical ID、scope、Conversation、Experience 和 Memory 都由 Host/Cognition 拥有。
10. 需要第三方包、外部进程、本地服务或协议桥时，在 `contributes.glimmer.managedResource` / `contributes.glimmer.protocolBridge` 声明；扩展本体不静默下载依赖，也不把第三方包提交进源码树。
11. 第三方包安装目录优先写 `data/...` 相对路径；Host 会解析到数据根，打包后可通过 `GLIMMER_CRADLE_DATA_ROOT` 迁移到用户数据目录。
12. 需要用户管理的开关、配置和启动状态必须能投影到 Control Center 扩展页，不要求每个扩展自己做 UI 窗口。
13. 运行事实通过 Host `ExtensionRuntimeProjection` 暴露；扩展只能通过 `runtime` Port 上报 Capability Graph 增量和 diagnostics，不能写 Renderer view model 或让 Desktop 读取日志/DB/端点推断 ready。
14. ID 必须是 `publisher.extension`，并在 `gcex.package.yaml` 显式列出 payload；不要把源码树、测试、密钥、第三方程序或本机状态打入包。
15. 使用 SDK `distribution` 的 `buildGcexPackage()` 构建唯一必需发布物 `.gcex`。包必须自带摘要与 SPDX SBOM；普通扩展不生成或提交包外 JSON。
16. 只有多平台、channel 或下载前外层摘要绑定等需求才显式调用 `buildExtensionReleaseManifest()`，并把生成的 `release-manifest.json` 与可选签名/构建证明放在作者自己的 Release，不提交到官方 Registry。
17. Release 必须绑定精确 tag。社区扩展可以申请 Registry 收录，也可以只发布一个规范命名的 `.gcex`；两条路径使用相同包规范，Registry 不接管扩展本体。

## 发布与安装验收

```powershell
pnpm validate
pnpm release:prepare
```

发布前解包验证以下内容：

- 只有 `extension/` 与 `META-INF/` 两个顶层目录；
- manifest 的 ID、publisher、version 与规范制品名称一致；存在 Release Manifest 时还必须与其一致；
- `checksums.json` 覆盖所有文件且验证通过；
- `sbom.spdx.json` 是 SPDX 2.3，并列出扩展包及构建时依赖；
- 可选 Release Manifest 使用相对制品路径或 HTTPS URL，并绑定精确 SHA-256；
- Registry PR 只提交审核元数据与作者侧发布来源指针，不复制 manifest、Release Manifest、SBOM、签名或制品。

## Skill/Tool 规则

| 情况 | 处理 |
|---|---|
| `contract_only` | 不可执行 |
| 需要确认但确认通道缺失 | 拒绝，不偷跑 |
| 权限未声明 | 拒绝并记录 policy decision |
| 非 `character` audience | 不进入人物 Skill catalog；Gateway 拒绝人物调用 |
| MCP 断连 | catalog 标记不可用或撤销 |
| handler 超时/抛错 | Gateway 归一化错误，保留 trace |
| 扩展停用 | dispose，旧 handler 不可再调用 |

## 平台 Adapter 规则

- 平台 payload 不进 Cognition。
- account/space/thread/endpoint 必须映射为 `ConversationAddress`，不得自行伪造 canonical scene/conversation/actor id。
- 平台限流、撤回、失败要在 Adapter/Kernel 边界处理。
- 输出动作必须通过受控 Channel/Port。
- 不把平台账号 token 写日志或文档。
- Extension 私有业务缓存、ambient perception、evidence proposal 和长期记忆必须分层；只有后两者进入认知入口，长期记忆只能由 Cognition 沉淀。

## 验证

```powershell
pnpm --filter @glimmer-cradle/kernel exec vitest run tests/extension-host tests/skill-plane
pnpm --filter @glimmer-cradle/kernel exec vitest run src/application/services/extension-runtime-registry.test.ts
pnpm --filter @glimmer-cradle/extension-sdk typecheck
pnpm --filter @glimmer-cradle/kernel typecheck
pnpm typecheck
```

手动/集成验证：

- 扩展启用；
- 扩展禁用；
- Control Center 能看到扩展、保存配置、启动和关闭；
- manifest 非法；
- 权限拒绝；
- 确认缺失；
- 调用成功；
- 调用超时；
- handler 抛错；
- MCP 断连；
- dispose 后不可调用。

Runtime projection / degraded 矩阵：

- package 缺失：resource 为 `missing` 或 `failed`，命令保持 disabled，并给恢复动作；
- 进程 live 但协议未 ready：process resource 可为 `live`，核心 capability 仍不可用；
- 协议 ready 但管理面板不可用：核心 capability 可用，management capability 降级；
- command precondition 不满足：Host 给出 `disabled_reason`，Renderer 只展示；
- 扩展停用：Capability Graph 节点进入 stopped/disabled，旧 handler 不可调用；
- 重启：Host 重新注册 contribution point definition、graph node、edge 和 action intent，不复用旧状态。
