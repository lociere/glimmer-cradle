# 文档维护规范

## 选择正确的文档

| 信息类型 | 唯一归属 |
|---|---|
| Glimmer Cradle 的解释、架构审美、长期不变量与目标形态 | `architecture/blueprint/` |
| 当前结构、边界、运行时关系与质量要求 | `architecture/current/` |
| 源码入口、组装、映射、状态与调试链路 | `architecture/implementation/` |
| Schema、配置键、路径、日志字段、SDK API | `reference/` |
| 开发、排障、发布操作 | `guides/` |
| 未完成工作的顺序与验收 | `roadmap/` |
| 长期架构取舍 | `architecture/decisions/` |
| 已结束阶段或被替代设计 | `history/` |

## 文档头部

除索引页外，文档开头必须说明：

- **范围**：它回答什么，不回答什么；
- **事实依据**：对应代码、Schema、配置或 ADR；
- **维护触发**：改动哪些边界时必须同步更新。

不要维护人工“最后验证日期”；Git 历史才是更新时间。路线图的 `now.md` 例外，它必须标记本次审阅日期。

## 写作与校验

1. 先改唯一权威页，再改索引和链接。
2. 结构图使用 Mermaid，并附上可读文字；图不能成为唯一信息源。
3. 活跃长文档若超过约 80 行或包含 6 个以上二级章节，应在头部契约之后提供 `## 目录`；索引页、短任务页和 `history/` 归档不强制。
4. 示例只保留最小可执行或可验证片段；精确字段链接到 Reference。
5. 合并前检查相对链接、旧路径、旧术语和重复事实。
6. 代码、Schema、配置或脚本改动后，在同一工作内更新对应文档。
7. 新建 Git 分支或新增、重命名标识符时检查 [命名规范](./guides/development/命名规范.md)，确保分支表达工作性质与 scope，并避免把角色名、产品名或 `Runtime` 用作泛用后缀。

## 活跃文档质量门

合并前逐页检查：

| 类型 | 最低内容 |
|---|---|
| Reference | 范围、权威来源、事实模型/所有权、变更与兼容规则、验证 |
| Guide | 适用场景、前置条件、步骤、决策/失败分支、验证、文档同步 |
| Roadmap | 成果、范围/非范围、依赖/风险、验收门、完成后的归档位置 |
| Skill reference | 适用任务、判断方法、实施顺序、禁止项、验证或交付检查 |

只有目录、原则摘要或链接列表的页面不是完成文档；应扩写为权威页，或并入更完整的权威页后删除。

## 文档模板

### Architecture

```md
# 标题

> 范围：
> 事实依据：
> 维护触发：

## 结构与边界
## 不变量
## 运行时关系
## 相关 Reference、Guide 与 ADR
```

### Guide

```md
# 任务名称

> 适用场景：
> 前置条件：

## 步骤
## 失败与恢复
## 验证标准
## 需要同步的文档
```

### ADR

```md
# ADR-XXXX：决策名称

- 状态：proposed | accepted | superseded | deprecated
- 日期：YYYY-MM-DD

## Context
## Decision
## Consequences
## Alternatives considered
## Links
```
