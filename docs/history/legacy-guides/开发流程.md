# 开发流程

本文是月见项目的端到端开发流程。它回答“开始改之前看哪里、改动时守什么边界、改完后如何验证”。具体子系统细节仍看同目录下的 Kernel / Cognition / 桌面渲染指南。

## 1. 开始前

每次开发先确认三件事：

1. **当前阶段**：读 `docs/roadmap/当前推进.md`，确认这次工作是否落在当前承重点上。
2. **事实源**：读 `docs/architecture/current/` 对应层，确认当前代码事实。
3. **目标形状**：若改的是架构方向，先读并更新 `docs/architecture/blueprint/月见架构蓝图.md`。
4. **阶段方法**：若要决定“下一步怎么推进到理想形态”，读 `docs/roadmap/蓝图落地流程.md`。

不要从历史阶段文档直接推断当前代码。`docs/history/architecture-decisions/` 只保存当时的取舍证据。

## 2. 判断改动归属

| 改动类型 | 首看文档 | 主要代码位置 |
|---|---|---|
| 跨语言协议 / schema | `architecture/current/01-协议层.md` | `protocol/src/schemas/` |
| Kernel 编排 / 生命周期 / 扩展宿主 | `architecture/current/02`~`04` | `core/kernel/` |
| 扩展贡献点 / SDK / requires / ports | `architecture/current/05-插件层.md`、`docs/extensions/` | `extensions/`、`core/kernel/` |
| Cognition 认知核 / 情绪 / 记忆 / 推理 | `architecture/current/07-Cognition认知核.md` | `core/cognition/` |
| data / observability / trace / memory layout | `architecture/current/08-记忆与日志架构.md` | `data/`、`core/*/observability` |
| Desktop Shell / Control Center / Presence Surface | `guides/桌面渲染开发指南.md`、`guides/UI设计指南.md` | `core/renderer/desktop/` |
| 配置结构 | `configs/README.md`、相关 current 文档 | `configs/` |
| 当前计划或验收门 | `roadmap/当前推进.md` | 文档为主 |

如果一个改动横跨多层，先写清边界：谁是事实源、谁只是投影、谁负责路由。

## 3. 标准工作流

### 3.1 普通功能 / bugfix

1. 复现或读日志，确认断点。
2. 定位所属层，查对应 `current/` 文档。
3. 改代码，优先修底层边界，不在调用端堆补丁。
4. 补最小回归测试。
5. 跑该层验证。
6. 若行为或边界变化，同步对应文档。

### 3.2 架构性改动

1. 先更新 `docs/architecture/blueprint/月见架构蓝图.md`，明确新形状。
2. 再更新 `docs/architecture/current/` 对应文档，描述本次实际落地范围。
3. 改代码；删除已经被替代的旧入口、旧命名和旧 fallback。
4. 跑跨层验证。
5. 更新 `docs/roadmap/当前推进.md` 的阶段状态或下一步。

架构性改动不要同时翻太多层。一次选择一个承重点做透。

### 3.3 Protocol / schema 改动

1. 改 `protocol/src/schemas/`。
2. 运行：

```powershell
pnpm sync:contracts
```

这等价于：

```powershell
pnpm --filter @glimmer-cradle/protocol gen:all
```

3. 检查生成产物：
   - TypeScript：`protocol/src/generated/`
   - Python：`core/cognition/src/selrena/protocol/generated/`
4. 更新消费端映射和校验逻辑。
5. 运行：

```powershell
pnpm --filter @glimmer-cradle/protocol typecheck
pnpm --filter @glimmer-cradle/kernel typecheck
cd core/cognition
uv run pytest -q
```

两个及以上语言层消费的类型必须以 schema 为单一事实源。不要手写镜像模型。

### 3.4 Cognition 改动

1. 读 `docs/architecture/current/07-Cognition认知核.md`。
2. 确认是否涉及经历日志、记忆、trace；若涉及，同步读 `08-记忆与日志架构.md`。
3. 在 `core/cognition` 包内运行 Python 验证：

```powershell
cd core/cognition
uv run pytest -q
```

不要从仓库根目录直接 `uv run pytest core/cognition`，根目录不是 Python 包根。

### 3.5 Kernel / Extension 改动

最小验证：

```powershell
pnpm --filter @glimmer-cradle/kernel typecheck
pnpm --filter @glimmer-cradle/extension-sdk typecheck
```

改扩展包时再跑对应包：

```powershell
pnpm --filter @glimmer-cradle/extension-napcat-adapter typecheck
```

扩展是能力贡献点，不是月见本体器官。需要本体能力时，优先补 Kernel / Cognition / Renderer 的正式接口。

### 3.6 Renderer / Desktop Surface 改动

1. 读 `docs/guides/桌面渲染开发指南.md`；涉及 Control Center 结构、样式或交互时同时读 `docs/guides/UI设计指南.md`。
2. 区分三个表面：
   - **Presence Surface**：月见的存在体。
   - **Control Center**：普通管理窗口。
   - **Avatar Shell**：可替换身体壳，Live2D / Unity 都应走统一契约。
3. 运行：

```powershell
pnpm --filter @glimmer-cradle/desktop-surfaces typecheck
pnpm test:ui
pnpm dev
```

人工确认：托盘入口、Control Center 呼出、窗口行为、连接状态、基础对话。Electron 原生窗口、托盘和 Unity 集成不由浏览器 mock 覆盖，仍需实际运行验证。

### 3.7 启动编排 / 性能改动

改 `core/kernel/src/app.ts`、`core/lifecycle/runtime/`、Cognition 启动、音频/模型资源门、Electron dev 启动脚本时，必须同时检查启动体验。

业内成熟桌面项目的一般体感标准：

| 阶段 | 目标 | 说明 |
|---|---:|---|
| 可见壳体出现 | 1-3 秒 | 用户应能看到窗口、托盘或清晰连接状态。 |
| 基础可交互 | 3-5 秒 | 超过 5 秒没有明确进度，体感就是“卡住”。 |
| 重资源准备 | 可更久 | AI 模型、ASR/TTS、embedding、vision 可耗时更长，但必须显示阶段、进度或明确的“准备中/不可用”状态。 |
| 首次下载 | 不设硬上限 | 网络与模型大小不可控，但不得让 UI 空白等待；下载日志进 process log，主时间线只保留摘要。 |

月见自己的启动验收目标：

- `transport + presentation` 应尽早完成，Control Center / Presence 状态通道优先出现。
- `Cognition` 与默认 `Audio` 资源门并行准备；不要把 FunASR / Piper / embedding 串行塞在 UI 可见前。
- `IngressGateManager.openIngress()` 只能在核心资源门、扩展和 Organism 就绪后执行；UI 可早出现，但不能在半启动态注入真实输入。
- `runtime.pretty.log` 默认只展示阶段、模块、耗时、readiness 与摘要告警；路径、完整 JSON、trace 细节留给 `runtime.log` 或 debug。
- 启动模块必须用统一 `RuntimeReadinessSnapshot` 表达 ready / degraded / failed；pretty 日志只读摘要，模型下载、路径和第三方输出进入 process log。
- 任一 required runtime 超过 5 秒，应在 UI 或日志中有可读状态；超过 15 秒，应能从 `runtime.pretty.log` 看出慢在哪个 phase。

启动改动后至少检查：

```powershell
pnpm --filter @glimmer-cradle/kernel typecheck
pnpm --filter @glimmer-cradle/desktop-surfaces typecheck
pnpm dev
```

然后看：

- `data/observability/logs/runtime.pretty.log` 是否能一眼读出阶段顺序和慢点。
- Control Center 是否早于模型资源完全 ready 出现，并清楚显示“准备中/不可用”。
- 首次下载或模型缺失时，UI 不空白、不误报在线、不让用户输入静默丢失。

## 4. 调试流程

### 4.1 对话无回复

按链路从外到内查：

1. Renderer 是否发出 `chat_input`。
2. Kernel `desktop-ui-engine` 是否收到 UI 请求。
3. `perception-gateway` 是否记录感知输入。
4. `attention.ipc.perception_message` 是否发送到 Cognition。
5. Python span 中 `appraise.perceptions` 是否大于 0。
6. `cognitive_cycle.broadcast_source` 是否为 `perception`。
7. `deliberate.generated_reply` 是否为 true。
8. `act.actions_emitted` 是否大于 0。
9. Kernel 是否收到 `ACTION_COMMAND` 并发布 `ChannelReplyEvent`。

### 4.2 慢响应

优先看 span，不凭体感猜：

```powershell
python scripts/telemetry.py spans latest
python scripts/telemetry.py spans slowest 20
python scripts/telemetry.py spans tree <trace_id>
```

常见慢点：

- `attention.flush`：输入聚合 / 防抖。
- `attention.ipc.perception_message`：Kernel ↔ Cognition 交付。
- `llm.generate`：云端模型或网络。
- 未来 ASR / TTS / vision：音频与多模态链路。

### 4.3 DLQ

```powershell
python scripts/dlq.py list
python scripts/dlq.py show <trace_id>
python scripts/dlq.py replay <source:id>
```

DLQ 有新增非测试记录时，不要只清空；先找事件类型、trace_id 和失败原因。

## 5. 文档同步规则

| 改动 | 必查文档 |
|---|---|
| `protocol/` | `architecture/current/01-协议层.md` |
| `core/kernel/` | `architecture/current/02`~`04` |
| `extensions/` 或 extension SDK | `architecture/current/05-插件层.md`、`docs/extensions/` |
| `core/cognition/` | `architecture/current/07-Cognition认知核.md` |
| 日志、记忆、data layout、trace | `architecture/current/08-记忆与日志架构.md` |
| 进程拓扑、启动顺序、跨层数据流 | `architecture/current/00-架构总览.md` |
| 架构目标变化 | `architecture/blueprint/月见架构蓝图.md` |
| 当前阶段或下一步变化 | `roadmap/当前推进.md` |
| Codex / AI 开发事务变化 | `.codex/skills/glimmer-cradle/`、`AGENTS.md` |

文档与代码不一致就是 bug。不要留下“先这样以后再说”的旧叙事。

## 6. 提交前验收门

最低线：

```powershell
pnpm typecheck
pnpm build
```

按改动追加：

```powershell
cd core/cognition
uv run pytest -q
```

```powershell
pnpm sync:contracts
```

```powershell
pnpm dev
```

最后人工确认：

- UI 显示 connecting → online。
- Control Center 能呼出并能对话。
- Presence Surface 不抢占管理窗口语义。
- DLQ 无新增非测试记录。
- `docs/architecture/current/` 与代码一致。
