# 迁移材料：桌面渲染开发指南

> 新的操作步骤见 `docs/guides/subsystems/桌面与Avatar开发.md`，代码实现见 `docs/architecture/implementation/Desktop与AvatarShell实现.md`，稳定视觉规则见 `docs/reference/ui-design-tokens.md`。本页不再更新。

桌面渲染层位于 `core/renderer/desktop/`，负责月见在桌面上的可见身体与管理界面。它是 Native Shell，不是 Extension。

## 技术栈

- Electron：桌面主进程、托盘、窗口生命周期。
- Vite：renderer dev server 与构建。
- React：Control Center / Presence Surface UI。
- Zustand：轻量状态管理。
- AvatarSurface：Electron Presence 的形象编排器，只负责选择后端与轻量占位，不直接把复杂 Live2D 当默认身体。
- Sprite Avatar：`assets/avatar/models.json` 支持 `kind: "sprite"` 的透明 PNG 身体；是否作为降级身体由 catalog 的 `fallback` 显式声明。
- Unity Avatar Shell：位于 `core/renderer/avatar-unity/`，是复杂 Live2D / 未来 3D 身体的正式目标后端。

## 当前界面模型

```text
Electron Main
├── Tray Resident              # 系统托盘常驻入口
├── Presence Surface           # 月见存在体，默认只显示人物
├── Control Center             # 管理面板，按需创建 / 唤起 / 聚焦
└── External Avatar Shell      # Unity 身体渲染器，通过 Kernel 8082 接入
```

原则：

- Presence Surface 展示“月见在这里”；静态 sprite 可做像素级透明穿透，复杂 Live2D 默认由 Unity 接管窗口、透明区域和高频交互。
- Presence Surface 在模型清单加载期间保持透明空 surface；目标 backend 是 Unity 时，Electron 只标记等待外部外壳并进入透明穿透态，不显示 Q版临时身体。
- Control Center 是正常软件窗口，不长期置顶，不像调试 overlay 一样压住其他应用。
- Control Center 使用 frameless `BrowserWindow` + renderer 自绘中文标题栏；renderer 只通过 preload 白名单调用最小化、最大化、关闭。
- 关闭 Control Center 默认隐藏，不等于退出月见。
- 托盘菜单提供显示面板、隐藏、退出等明确入口；「退出月见」是应用级停机动作，开发期会让 `pnpm dev` 进程组一起收束。
- 开发期 Vite 端口 `5173` 是固定进程契约，端口被占用时必须快速失败，不允许自动漂移到 `5174` 等临时端口。

## Avatar 渲染架构

月见的正式身体渲染不是 Electron 页面或普通 Unity 窗口，而是由 Kernel 监督的 Avatar Shell。Composition Host 是唯一桌面窗口所有者，Unity 是无可见窗口的 Render Worker；Control Center 与 Electron Presence 不参与正式身体合成。

```text
AvatarShellFrame
  → Avatar Behavior Runtime       # 意图、动作、视线、口型和本地反应仲裁
  → Unity Render Worker           # Cubism 模型与带 alpha 的 GPU surface
  → Composition Host              # 原生组合、透明窗口、命中、拖动与 DPI
  → Desktop                       # 只有人物身体可见
```

当前实现：

- **Unity Render Worker**：复杂身体的正式渲染路线，位于 `core/renderer/avatar-unity/`。它通过 Avatar Shell 协作链路消费身体意图，负责 catalog 选择的 Live2D / 未来 3D 模型、动作状态机和透明帧输出，但不拥有正式桌面窗口。
- **Composition Host**：正式桌面身体的窗口与合成所有者。它消费 Unity 的共享 GPU surface，负责 per-pixel alpha、透明区穿透、身体命中、拖动、DPI、显示器与停机收束。
- **Unity 启动策略**：Kernel 已接入 Avatar Shell sidecar。当前默认 `avatar_shell.sidecar.launch_mode: managed`，
  标准构建产物是 `data/packages/avatar-shell/windows/SelrenaAvatarShell.exe`，由 `pnpm avatar:build` 生成。
  该路径是当前迁移事实，不是目标安装边界；目标是构建到 `out/dev/avatar-shell/`，安装时投影到
  `resources/components/avatar-shell/`，见 [客户端安装形态](./客户端安装形态.md)。
  `manual` 只作为 Unity Editor 调试方式保留。sidecar 进程输出写入
  `data/observability/logs/processes/avatar-shell.log`，主启动日志只展示 ready / degraded 摘要。
- **Unity 当前实现**：`AvatarShellBootstrap` 可在空场景自动创建 Shell；`AvatarShellClient` 负责配置、连接、重连和错误上报；`SelrenaLive2DController` 负责把协议帧交给 `CubismAvatarModelDriver`。正式配置只在 catalog 的 `resourceKey` 对应 Cubism prefab 完成初始化后发送 `shell_ready`；缺 runtime 或 prefab 时必须报告 degraded，不能回退为静态贴图。
- **Sprite backend**：使用透明 PNG 作为显式轻量身体或明确的故障降级；Unity 首选模型不再自动显示 Q版占位。

长期实现：

- **Unity Avatar Shell** 继续作为高性能与复杂身体表现的主路线，适合更复杂的 3D / 2.5D 场景、动作状态机、物理、后处理和更稳定的原生渲染性能。
- Unity 不嵌入 Control Center，不替代 Electron / React 管理面板；Control Center 是配置、状态与诊断工作台，继续由 Desktop Surfaces 承载。Unity 只作为外部 Avatar Shell / 身体渲染器接入 Kernel，消费同一组 `AvatarShellFrame`。
- 一旦模型首选 Unity，Electron Presence 应进入透明等待 / 让位，不再与 Unity 同时显示身体。切换模型不应影响 Cognition、Kernel、配置页或扩展生态。

成熟边界：

- Composition Host 不懂模型细节，只管共享 surface、透明窗口、命中、拖动、DPI 和呈现生命周期。
- Unity Worker 不移动平台窗口、不拥有 Control Center，也不读取管理配置事实源。
- Behavior Runtime 的语义层不依赖 Unity、Cubism 或平台窗口；只有 Runtime Adapter 接触具体模型运行时。
- Render Worker 可以被替换。当前复杂身体的唯一实现是 Unity；协议和 UI 不随渲染器变化。

`pnpm sync:unity-assets` 读取根目录 `assets/avatar/models.json`，只选择 `preferredBackend: unity` 的模型，
校验其路径后生成精简 `avatar-registry.json` 和对应资产目录。`pnpm sync:unity-runtime` 按
`avatar-runtimes.json` 校验第三方 runtime 供应包；复制型 runtime 生成 Unity 投影，`.unitypackage` 则由
`avatar:build` 通过 Unity 原生导入流程生成 `Assets/Live2D/Cubism/`。根目录 `assets/` 是内置源资产事实源；
Unity 模型副本与第三方 runtime 导入目录都是运行 / 构建投影，不入 Git、不手工修改。
新增模型或改变 manifest 文件名只改 catalog 与资产，不改 C#、场景或同步脚本。
Desktop 发布投影同样由 catalog 收口：只有 sprite 降级后端实际消费的模型资产会进入 renderer public；Unity 首选模型只投影到 Unity，不在 Desktop 产物中重复携带。未注册的历史 Avatar 资产和未被模型引用的运行时不进入客户端产物，源文件是否保留不再影响支持矩阵与包体积。

### Unity Avatar Shell

开发期有两种运行方式，正式路径优先使用受管构建产物：

1. `managed`：运行 `pnpm avatar:build`，脚本会投影 catalog 中的 Unity 模型与 runtime、调用 Unity batchmode，
   并把构建产物输出到 `data/packages/avatar-shell/windows/SelrenaAvatarShell.exe`。
  `pnpm dev` 启动后，Kernel 会在 `SurfaceRuntime` 阶段拉起该 sidecar。退出月见时，Desktop Shell 先请求 Kernel
  有序停机，Kernel 向 Unity 发送 `shutdown` 帧并等待退出；超时才回收该 sidecar 的进程树，随后 Electron 才关闭。
2. `manual`：只用于调试 Unity Editor。把 `configs/system/surfaces.yaml` 的
   `avatar_shell.sidecar.launch_mode` 临时改为 `manual`，再手动播放 `core/renderer/avatar-unity` 工程。

Unity Shell 的在线状态不靠 Electron 猜测，而由 Kernel `AvatarEngineController` 根据
`shell_hello / shell_ready / heartbeat` 生成 `AvatarShellStatusChangedEvent` 与
`RuntimeReadinessSnapshot(runtime_id='avatar.shell')`。Control Center 只能展示这些状态，
不直接探测 Unity 进程。

### Composition Host

正式身体窗口不是 Unity 普通游戏窗口，也不是 Electron 的透明 WebView。`AvatarCompositionHost` 是
跨平台的外显边界：Unity / Cubism 只渲染到带 alpha 的 GPU surface，平台适配器负责原生窗口、桌面合成、
透明区穿透、拖动和 DPI。当前 Windows 适配器位于 `native/src/composition/composition_host_windows.cpp`，
以 DirectComposition + D3D11 实现；`native/include/selrena_native.h` 暴露的 API 只使用
`selrena_composition_host_*` 通用命名，未来 macOS、Linux 适配器不需要改变 Unity、Kernel 或协议。

`pnpm avatar:build` 会先运行 `pnpm avatar:composition:build`，把本机构建出的适配器投影到
Unity 插件目录。该 DLL 是构建投影，不进入 Git。不要回退到颜色键、`SetLayeredWindowAttributes`、
`setShape()` 或由 renderer 累加拖动 delta；这些路径会破坏 per-pixel alpha、拖动稳定性或透明区命中。
Windows DirectComposition Host 必须使用 `WS_EX_NOREDIRECTIONBITMAP`，且不处理普通背景绘制；DWM 重定向
位图与 composition visual 同时存在时，窗口 resize 间隙会暴露黑色矩形。尺寸变化只能更新 Host bounds 与
visual transform，不得重建 Unity 模型、RenderTexture 或 composition swap chain。

Unity Worker 采用显式帧生产：主相机不依赖普通 Player 窗口刷新，而由 `AvatarCompositionHost` 主动渲染到
固定 RenderTexture；随后在独立导出纹理上完成 Unity / DXGI 纵轴归一化，并通过 Unity render event 在渲染线程
提交给 Host。Worker 容器窗口会被完全隐藏，不能作为黑底的第二个桌面窗口出现；手动 `Camera.Render()` 仍持续输出
给 Composition Host。项目必须保留 framebuffer alpha，并以 Linear 色彩空间渲染，避免透明通道被丢失或半透明边缘错误。
禁止用“隐藏窗口后等待 Unity 自己继续画”的隐式时序，因为 Windows/DWM 可能抑制不可见窗口的相机更新。

`shell_ready` 是事件链的终点，不是某个 `Update` 轮询条件：`Composition Host 首帧 present` 触发
`AvatarPresentationController.PresentationReady`，再由 `AvatarShellClient` 在 `shell_hello` 已发送且 WebSocket 仍打开时
单次发布 `shell_ready`。重连只重发协议握手与 ready，不重建模型、RenderTexture 或 Host。这样 Kernel 能区分“已连接”
与“实际已有一帧身体在桌面可见”；若缺失 ready，应从 Host 创建、surface attach、提交帧数和首帧状态诊断，而不是猜测。

为保持桌宠尺寸下的 Live2D 边缘质量，`AvatarShellConfig.renderScale` 允许 Worker 以高于 Composition Host
窗口的分辨率输出透明纹理；Host 使用 DirectComposition 的二维变换缩放到实际窗口。该参数是输出采样倍率，不是
模型显示大小，不得与 `displayScale` 或桌面驻留预设混为一谈。默认 `1.5`，范围限定为 `1.0` 到 `2.0`；
超过该范围先评估显存与帧时间，而不是无上限提高纹理尺寸。

默认驻留位置由 Composition Host 根据目标 monitor 的 work area 计算。Unity 只传 `visibleRatio`、右边距和底边距；
不得用 `Screen.currentResolution` 推导 Win32 窗口坐标。拖动后的本机位置使用带版本的状态记录，状态格式变化时直接
回到 catalog 默认停靠，不继续解释旧坐标。

目标源码边界中，应用级 Host 归入 `core/renderer/composition-host/`，而共享纹理、D3D11/DirectComposition
封装和平台原语留在 `native/`。当前 Windows Host 仍位于 `native/src/composition/`，属于待迁移的实现事实；
迁移不得改变协议、共享 surface 契约或 Kernel 的 Avatar Shell 监督关系。

开发期可以运行 `pnpm avatar:sync` 同步角色资产与 runtime；构建前先运行 `pnpm avatar:doctor`。脚本根据角色 catalog 收集所需 runtime，并按
`core/renderer/avatar-unity/avatar-runtimes.json` 解析本机包来源与环境变量覆盖。
构建脚本只处理选中 runtime：先验证本机供应包，再按 `installMode` 复制或调用 Unity 原生导入。第三方 runtime
不进入源码事实源和 Git 历史。Cubism SDK for Unity 5 R5 使用 URP，Avatar Shell 工程声明 Unity
`6000.0.77f1` 与 URP `17.0.3`；Unity 首次解析 package 后由 `AvatarShellProjectSetup` 创建并绑定唯一的 URP pipeline asset。`assets/avatar/models.json` 经 `sync-unity-assets` 投影到 `Assets/Resources/AvatarModels/`，Cubism SDK 在该投影中生成模型 prefab；只有 `CubismAvatarModelDriver` 成功加载 catalog 的 `resourceKey` 后才能宣告 Live2D runtime ready。

`shell_hello` 只表示外壳进程与协议连接建立；`shell_ready` 表示 catalog 投影元数据、模型 prefab 和真实 `CubismAvatarModelDriver` 都已就绪。Kernel 必须把“已连接但模型未就绪”展示为 degraded，不能把进程存活等同于 Live2D 可用。

## 边界规则

- Renderer 不直接访问 Node API，必须走 preload 暴露的 `window.selrenaAPI`。
- Electron main 可以持有 BrowserWindow、Tray、nativeImage 等原生对象，但不能暴露给 renderer。
- UI 只展示状态，不决定 Cognition 语义。
- Live2D / Unity 都是 Avatar Shell 的实现选择，不要把它们写成普通扩展。
- Unity 端的具体模型 runtime 只能藏在 `IAvatarModelDriver` 后面。当前正式实现是 Cubism4/5 的 `CubismAvatarModelDriver`，不允许把 Cubism 对象、角色文件名或资源目录泄漏到 Kernel 协议或 Desktop Surfaces。
- 月见只正式支持现代 Cubism4/5 资产。`live2dVersion` 表示模型最低兼容代际，必须使用 `.moc3/model3.json`；Cubism2/3 不进入 catalog、runtime catalog 或 Web 调试后端。Cubism4/5 共用官方 Cubism SDK for Unity 供应边界，通过 `assets/avatar/models.json` + `avatar-runtimes.json` + `IAvatarModelDriver` 三层收口。
- Renderer public 资产必须通过 `resolvePublicAssetUrl()` 解析，避免 Vite dev 与 built `file://` 的 public 目录差异泄漏到组件里。
- Q版 sprite 只用于显式选择的轻量 Presence 或诊断占位，不是 Unity 正式身体失败时的静默兜底。正式身体未就绪时应显示 degraded 状态，并保持 Electron Presence 透明让位。
- Live2D 模型必须经 `assets/avatar/models.json` 注册，不在组件里写死角色名、路径、动作或 runtime 脚本；`preferredBackend` 是后端选择事实源，具体 manifest、情绪映射、交互动作与 Web runtime 也由条目声明。
- 全局 `body` 背景只能由具体 surface opt-in；Presence 必须保持透明背景，只显示人物本体。
- 形象模型、显示大小与桌面驻留属于 `AvatarAppearanceSettings`，由 Electron main 持久化到 `data/state/renderer/avatar-appearance.json` 并广播给 Control Center / Presence；不要只改某个 renderer 窗口的本地 store。
- Control Center 保存形象外观后，Electron main 通过 `AvatarShellUpstreamFrame.avatar_presentation` 请求 Kernel；Kernel 验证并转发 `AvatarShellDownstreamFrame.presentation` 给 Unity。Unity 模型驱动只在加载时把完整自然边界 `contain` 到相机；Composition Host 负责完整表面的显示大小与停靠；拖动只写位置状态，三者不得互相回写。
- `placement_id` 描述半身驻留、大半身驻留或完整身体等 catalog 预设；预设只声明完整透明表面留在工作区内的比例与停靠边距。模型始终以 `contain` 方式完整装入 Unity 相机，半身效果只来自透明窗口的一部分位于屏幕工作区之外。不得恢复相机聚焦、模型倍率或垂直焦点字段。`reset_placement` 清除用户拖动位置并回到该预设。
- 模型动作只由该模型自己的 `actionsPath` 清单声明，口型、视线与空闲等可选反应由 `behaviorPath` 声明。Control Center 通过受控 IPC 读取净化后的动作投影，并以 `avatar_intent.action_id` 请求 Kernel 转发给 Avatar Shell；行为运行时在 Shell 内解析为 expression、motion 或参数组合。UI、Kernel 和协议不写模型作者的文件名、按键、参数名或资源路径。`toggle: true` 的 expression 动作必须由分阶段 Behavior Blend Graph 按 Add / Multiply / Override 语义组合；常规参数贡献在 Physics 之前写入，不能继续使用通用 post-Physics overlay。非 toggle 表情优先复用 SDK 的瞬时表情与淡入淡出。
- Avatar 行为遵循“模型能力内最优表现”原则：优先组合模型已声明的 Parameter、Motion、Expression 与 Physics，使空闲、注意力和动作过渡自然连续。若模型没有独立手臂或姿态控制，不得猜测参数或写入未知值；身体参数变化仍可通过模型层级和 Physics 带动手臂、袖子与装饰轻微从动，但不能把这种从动冒充模型并不具备的新姿势。
- 鼠标视线属于 Avatar Shell 的有时效本地注意力反应：窗口组合层只提供归一化全局指针位置，Behavior Runtime 只在检测到有效移动时取得注意力通道；短暂保持后平滑释放给身体基线。具体眼球、头部与少量身体参数只能由模型 `behaviorPath.gaze` 映射，并由该通道的唯一 mixer 合成。它不需要每帧穿过 Kernel，也不应由 Control Center 直接修改 Cubism 对象。
- 身体基线由模型 `behaviorPath.baseline` 声明，负责眨眼、呼吸、静态姿态、低频 idle 与物理输入；它是永久身体层，不是 Cognition 未接入时的占位。姿态、idle 与注意力必须在 Cubism Physics 之前统一混合，禁止再并列使用 LookController 写相同参数。模型没有声明的手臂姿态或作者 motion 属于不可用能力，不能通过猜测物理输出参数伪造。用户、Cognition、场景和扩展后续都通过同一个 Behavior Runtime 以带通道、优先级和有效期的请求覆盖基线，租约结束后自然恢复。
- 身体行为采用“单一权威、多职责组件”，不得建立万能 `AvatarManager`。持久且互斥的外显模式使用 Presentation Statechart；口型、视线、表情、姿态、动作和配件按并行通道使用有时效租约仲裁。优先级只在发生冲突的通道内比较，安全停机除外。
- 已落地的连续行为主线使用不可变 `AvatarBehaviorFrame`：`AvatarBehaviorRuntime` 每帧只采样一次本地指针，`AvatarBehaviorEvaluator` 合成口型与短时注意力，`IAvatarModelDriver.ApplyBehaviorFrame()` 只应用解析结果。作者 idle motion 由 `AvatarIdleMotionScheduler` 独占管理，离散动作由 `AvatarActionScheduler` 管理；它们不再和连续行为共同直接写 Cubism 参数。
- 当后续需要让 Cognition、用户、场景或扩展并发覆盖同一身体通道时，再把内部输入提升为不可变 `BehaviorRequest`、按通道租约仲裁并解析为更通用的 `ResolvedAvatarFrame`。只有跨进程稳定语义进入 `protocol/`，Unity 内部求值类型不得反向污染协议层；不要为了尚不存在的来源先制造空队列或第二套状态机。
- Behavior Inbox 必须区分三类时序：离散 action 使用可取消队列，持续 state 使用版本替换，鼠标目标与 lip-sync 等连续 stream 使用 latest-wins 有界采样。Unity 每帧只读取一次不可变快照，禁止高频 producer 直接写 Cubism 参数或积压历史输入。
- Cubism 自定义控制器必须实现 `ICubismUpdatable` 并由 `CubismUpdateController` 排序。参数贡献按 Motion Fade、Pose、Expression、Blink、Mouth、Harmonic、Look、Physics、Render 的依赖分阶段接入；同一参数只能有一个最终 writer，常规行为参数不得在 Physics 后补写。
- Idle Scheduler 优先播放模型作者提供的 idle motion；无 authored motion 时使用带冷却、静默期、幅度上限和可复现随机源的有界微行为，禁止用一组永久正弦波作为最终待机架构。
- 模型同步或构建阶段应把 action、参数范围、通道、冲突组、混合模式和更新阶段编译进 Capability Catalog，并检查缺失参数、重复 writer 与非法 post-Physics 写入。运行时缺失能力只降级跳过，不猜参数。
- 行为日志只记录请求接受/拒绝、租约切换、动作开始/结束/取消和 capability 缺口；逐帧参数只允许进入显式 debug trace。测试至少覆盖通道隔离、淡入淡出、抢占、缺失能力和更新顺序。

上述规则对齐官方运行时能力，而不是自建另一套动画引擎：

- [Cubism UpdateController](https://docs.live2d.com/en/cubism-sdk-manual/updatecontroller/) 规定自定义组件通过 `ICubismUpdatable` 进入统一更新顺序。
- [Cubism Motion for Unity](https://docs.live2d.com/en/cubism-sdk-manual/motion-unity/) 提供 motion layer、priority、播放与停止能力；[Motion Fade](https://docs.live2d.com/en/cubism-sdk-manual/motionfade/) 提供作者 motion 的淡入淡出数据。
- [Unity Animation Layers](https://docs.unity3d.com/6000.0/Documentation/Manual/AnimationLayers.html) 使用 Override / Additive 与层级隔离组合并行动画；[State Machine Basics](https://docs.unity3d.com/6000.0/Documentation/Manual/StateMachineBasics.html) 适合表达互斥状态与转换。因此月见只把持久外显模式放入 Statechart，把可并发身体维度留在通道租约和 Blend Graph。
- Windows 上“透明矩形窗口”与“真实 alpha 合成身体”不是同一能力。颜色键只可作为显式开发降级，不能承担最终的黑底消除、像素级穿透或发布 readiness；正式路径必须由原生 Composition Host 消费带 alpha 的 Unity 输出并管理命中策略。
- `assets/avatar/` 中除月见官方资源外的角色源文件，以及 Unity `Assets/Resources/AvatarModels/` 下由同步导入产生的 Cubism 投影，都只保留在本机并被 Git 忽略。注册表与 Schema 可以入库，私人模型内容、贴图、moc3、prefab 和 import 产物不得进入提交。
- Live2D 的模型加载生命周期只由模型 ID / manifest 驱动；情绪、口型和点击动作只能驱动 expression / motion / parameter，不允许重建 Avatar Shell 或重载模型。
- Presence 交互策略保留两类：`alpha-shape` 只用于静态透明 sprite；`transparent` 用于 Unity Avatar Shell 接管时，让 Electron Presence 完全穿透，不吃掉人物周围空白点击。
- `BrowserWindow.setShape()` 在 Windows 上会裁剪视觉窗口，不是纯命中测试 API。动态 Live2D 模型、motion、resize 期间的 bounds 不能回灌给 Electron 视觉窗口；像素级透明交互属于 Unity / 原生身体层的职责。
- 拖动期间 Electron main 只移动单一 Presence 窗口；Live2D renderer 不暂停、不恢复、不重算 layout。避免 Windows 透明窗口、WebGL canvas、DPI 缩放和任务栏 hover 互相追赶。
- 形象大小滑块采用“本地预览、节流提交、手势结束强制提交”：UI store 可以逐帧更新，但 `avatar_presentation` 不得按每个 DOM input 事件穿过 Kernel。拖动中的控制帧应合并到有限频率，鼠标释放、失焦与键盘操作后必须提交最终值；高频 presentation 帧不写入主运行日志。
- 显示倍率的产品公共区间为 `50%~250%`，首次默认 `120%`。滑块刻度、Desktop 持久化、协议校验、Kernel 转发和 Unity Presentation 必须使用同一范围；禁止任一层私自截断上限。
- `actionsPath` 必须符合 `assets/avatar/avatar-actions.schema.json`；同步脚本在投影 Unity 资产前校验版本、稳定 ID、目标类型、toggle 规则、重复 ID、依赖引用和无环依赖图。`toggle: true` 的动作在 Control Center 使用按类别分组的开关；一次性 motion / expression 使用命令按钮。UI 必须消费 Shell 上报的 `avatar_action_state.active_action_ids`，不能乐观翻转本地布尔值，并明确显示开启、关闭、依赖与互斥原因。`requires` 与 `exclusiveGroup` 只用于提前解释和禁用明显无效操作，Shell 的 `AvatarActionScheduler` 仍要再次校验依赖、互斥与关闭顺序；Cubism driver 无法解析目标资源时必须返回拒绝，不能把失败写入活动集合。
- 桌面捕获默认允许：Presence 与 Control Center 不启用内容保护。正式录屏验收使用打包后的 `Selrena.exe`；`pnpm dev` 使用 Electron 自带的 `electron.exe`，第三方录屏软件可能按 Chrome/Electron 分类，不能为绕过该识别而关闭 sandbox、webSecurity 或其他 Chromium 安全能力。
- Presence 拖动属于高频输入，走 fire-and-forget IPC；不要用 request/response 式 `invoke` 造成 Promise 排队。
- Live2D 缩放只能使用未缩放的模型自然尺寸（`internalModel` / local bounds），不能用受 scale 影响的 `targetModel.width/height` 反复回算。
- Presence 命中区域只在 `alpha-shape` 策略下由 sprite backend 上报，Electron main 去重后 `setShape()`；默认不打印命中区细节，排障时设置 `SELRENA_SURFACE_DEBUG=1`。

## Control Center UI

Control Center 采用自然桌面主题基底，可在系统浅色/深色之间适配；Mica / Acrylic 风格材质只用于窗口、菜单和浮层层级。青、蓝、金是可选强调来源而非强制人设皮肤，MD3 动态取色也只能替换受控 accent token。详细色彩规范见 [UI色彩参考](./UI色彩参考.md)。

Control Center 的产品结构、视觉系统、交互状态和 Playwright 验收以 [UI设计指南](./UI设计指南.md) 为事实源。本节只说明 renderer 的实现边界，不再复制一份独立设计规范。

Codex 修改 Control Center 前应读取 `.codex/skills/glimmer-cradle/references/subsystems/Desktop与Avatar.md`。当前视觉规则以 `docs/reference/ui-design-tokens.md` 为准；本段旧色彩约束仅作为历史材料，不再更新。

形象页的 Avatar Shell 状态来自 Electron main 提供的 `AvatarShellDiagnosticsSnapshot`：它汇总
`configs/system/surfaces.yaml`、Unity 资产投影、构建产物和 Avatar 日志入口。Renderer 只能消费这个投影，
不直接读 `configs/`、`data/` 或 Unity 项目文件。

当前 Control Center 的参考系是 Discord / Codex / VSCode 的交集：

- Discord 提供“活动栏 + 频道式导航 + 常驻状态”的组织经验。
- Codex 提供“主工作区留白 + 输入焦点 + 任务上下文”的体验参考。
- VSCode 提供“Workbench 分区 + 可调宽侧栏 + 稳定状态入口”的桌面软件骨架。

落地时不复制任何一个产品的皮肤，而是采用成熟桌面客户端的结构：自绘标题栏、窄活动栏、可调宽导航栏、主工作区、必要时的右侧检查器。所有入口必须有 hover / pressed / focus-visible / disabled / loading / selected 等交互状态，避免像静态 PPT。

核心任务优先级：

1. **配置月见**：Provider、推理参数、人格锚点、注意力节奏、语音、形象、扩展、数据与诊断路径都应通过结构化表单编辑。
2. **查看状态**：Kernel、Cognition、音频引擎、Presence、Live2D、扩展、模型资源、启动阶段和最近错误要能一眼判断健康度。
3. **轻量对话**：提供基础文字对话、语音入口和快速试探，不把 Control Center 变成完整聊天产品。

- 顶部为自绘中文标题栏“月见控制中心”，按钮通过 preload IPC 调用窗口动作。
- 左侧采用两层：窄活动栏负责快速切换，导航栏负责页面分组与说明。导航栏桌面宽屏下允许用户拖动分隔条微调宽度；窄屏下折叠导航，保留活动栏。
- 主区为当前页面工作区：对话页承载日常交流；配置页承载受控配置投影；状态页展示健康摘要与最近错误；语音页展示 TTS / FunASR / Piper；诊断页提供日志入口。
- 默认窗口尺寸面向桌面管理工作流：导航保持稳定，主区根据页面内容自适应。
- 所有按钮、导航项、表单控件必须有 hover / active / focus-visible 反馈，保证桌面软件的点击感和键盘可达性。
- 交互状态必须完整：禁用、加载、保存中、成功、失败、需要重启都要有明确反馈。不要把静态 `div` 伪装成按钮。
- 配置页必须支持脏状态提示、保存、重置、恢复默认、校验错误和需要重启标记；密钥只展示配置状态，不显示明文。
- 状态页展示摘要，不倾倒原始日志；详细路径、完整错误和进程输出放进折叠详情或“打开日志”动作。
- 语音错误只显示短摘要，长路径和完整诊断放进详情折叠区，避免状态栏被单条错误撑散。
- 语音能力只展示 Kernel 下发的 `audio_status`，不在 renderer 里直接探测 Piper、GSV、FunASR 或 native。
- 当前默认 TTS 是 Piper；GPT-SoVITS（停云）是显式启用的身份声线，ASR 目标路线是 FunASR。备用 provider 不得伪装成主能力。
- 诊断入口通过 preload 白名单请求 Electron main 打开预定义目录 / 日志文件，不允许 renderer 传任意本机路径。
- 配置页通过 preload 白名单请求 Electron main 读写受控配置投影，覆盖推理参数、注意力节奏、embedding、provider、人格锚点和桌面端口；不提供任意 YAML 文本编辑器，也不触碰 `configs/secrets/`。

实现分层：

- `components/ControlCenter.tsx` 只做页面路由和 Zustand / emotion store 投影，不承载具体页面布局。
- `components/control-center/model.ts` 是 Control Center 的前端页面模型事实源：页面清单、状态标签、provider 显示名、健康摘要、配置投影类型都在这里收口。
- `components/control-center/ui.tsx` 放 Shell、中文标题栏、可调宽导航和通用 UI primitive。这里不能写某个具体 provider 或日志路径的业务逻辑。
- `components/control-center/pages.tsx` 放具体页面。配置页采用“分组表单 + 右侧检查器”结构：保存、重置、脏状态、打开配置目录集中在检查器里，避免每个配置分组各自做一套保存逻辑。
- 样式统一在 `styles.css` 复用 token 和结构类。新增页面优先组合 `.page-grid`、`.surface-card`、`.settings-section`、`.feature-list` 等现有结构；不要回到巨型组件 + 临时 CSS 的写法。

## AvatarShellFrame 消费链路

Desktop Surfaces 只消费 Kernel 下发的统一 `AvatarShellDownstreamFrame`，不在 renderer 里自造 WebSocket 协议：

```text
Kernel DesktopUIController
  ├── thought      # 思考态开始 / 结束
  ├── reply        # 文本回复
  ├── emotion      # 情绪快照
  ├── shell_status # Unity 外壳在线 / 离线
  ├── audio_status # ASR/TTS 能力状态
  └── audio_play   # TTS / 声音播放
    → Electron main ipc-handlers
    → preload window.selrenaAPI
    → renderer runtime/useSelrenaRuntime.ts
    → Zustand stores → Control Center / Presence Surface / AvatarSurface
```

约束：

- `thought` 是状态，不是聊天消息；Control Center 只能显示“正在思考”指示，不写入历史。
- `reply` 才能追加 assistant 气泡，并按 `trace_id + message.sequence` 做幂等。
- `emotion` 独立驱动 `emotionStore`，不要重新塞回 reply。
- 回复正文必须按用户内容原样呈现；代码块、配置、Markdown 等可复制内容由 Control Center 保留换行和 fenced code block。

```powershell
pnpm --filter @glimmer-cradle/desktop-surfaces typecheck
pnpm --filter @glimmer-cradle/desktop-surfaces build
pnpm test:ui
pnpm dev
```

人工确认：

- 托盘能呼出 Control Center。
- Control Center 可拖动、可聚焦、可最小化、不会始终置顶。
