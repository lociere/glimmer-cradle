# 客户端打包指南

本文说明如何把月见源码构建为桌面客户端。安装完成后的规范目录、组件职责和用户数据位置以 [客户端安装形态](./客户端安装形态.md) 为准。

## 打包流水线

```text
产品源码 + .deps 外部依赖
        ↓ build / codegen / import
out/dev | out/test | out/release
        ↓ assemble / sign / package
dist/installers | dist/portable
        ↓ install
客户端安装目录 + 独立用户数据目录
```

各阶段只有单向依赖：

- `.deps/` 是可重建的供应商 SDK、工具链与构建输入，不是月见源码。
- `out/` 是可删除重建的中间产物，不作为权威输入提交或发布。
- `dist/` 只放签名后的安装器、更新包和便携发行物。
- `data/` 不参与产品代码装配；开发期 `repo/data/` 只投影发布期用户数据域。

## 构建输入

| 输入 | 打包方式 |
|---|---|
| `core/renderer/desktop/` | 构建为 Electron main、preload、Control Center 与静态资源 |
| `core/kernel/` | 构建为受监督的 Kernel 组件 |
| `core/cognition/` | 按目标平台构建或装配为 Cognition 组件 |
| `core/renderer/composition-host/` | 构建为 Avatar Shell Host 平台二进制 |
| `core/renderer/avatar-unity/` | 构建为无可见窗口的 Unity Render Worker |
| `engines/` | 按启用能力装配为独立 Engine 组件 |
| `native/` | 只分发目标平台的编译后二进制，不分发源码 |
| `assets/` | 经过许可和完整性检查后投影为只读产品资产 |
| `configs/` | 只投影无密钥的默认模板 |

第三方 Cubism SDK、编译工具链和仅用于导入的供应商文件不应原样进入安装包。确需随运行时分发的文件必须经过许可审计，并由组件清单显式声明。

## Avatar Shell 装配

Avatar Shell 不是一个 Unity 窗口，而是两个部署单元协作形成的身体表面：

1. `composition-host` 拥有唯一原生透明窗口、DirectComposition 视觉树、DPI、位置、输入穿透、拖动和停机收束。
2. `unity-worker` 以无可见窗口模式加载 Cubism 模型，输出带预乘 alpha 的共享 GPU surface；它不创建正式桌面表面。

二者通过稳定的 surface / lifecycle 契约协作。打包验收必须拒绝黑底、颜色键、整矩形鼠标阻挡、普通 Unity 游戏窗口闪现，以及退出后 Unity Worker 残留。

## 组件清单

安装根的 `resources/component-manifest.json` 是部署事实源，至少声明：

- 组件 ID、版本、入口、平台与架构；
- 启动角色、依赖关系、readiness 与停机超时；
- 资源目录、原生库和可选模型包；
- 完整性摘要、更新策略与许可元数据。

业务代码通过清单和路径解析器定位组件，禁止硬编码仓库绝对路径或假设开发目录与安装目录相同。

## 用户数据映射

Windows 发布版使用 Electron `userData` 根目录，默认位于 `%LOCALAPPDATA%/GlimmerCradle/`。开发期 `repo/data/` 与发布期 `userData/data/` 语义一致，但物理位置不同。

- 安装目录只读，升级时可整体替换。
- 用户配置、记忆、日志、导入模型和缓存只写入用户数据目录。
- 真实密钥进入系统凭据、环境变量或用户配置密钥域，不进入 `resources/defaults/`。
- 大模型默认按需下载或导入，不强塞进主安装器。

## 发布产物

```text
dist/
├── installers/              # 安装器、增量更新包、签名与校验文件
└── portable/                # 明确支持时生成的便携发行物
```

便携版也必须保持“只读应用资源”和“可写用户数据”逻辑隔离；便携数据根可位于发行目录中的独立 `user-data/`，但不能混进 `resources/`。

## Windows 应用身份与桌面捕获

- Electron Builder 固定 `appId=com.glimmer.cradle.desktop`、`productName=Selrena` 与 `executableName=Selrena`。
- Desktop 启动后设置相同的 Windows AppUserModelID；任务栏、通知、录屏与窗口归组不得继续暴露为通用 Electron 宿主。
- Control Center、Presence 与 Composition Host 默认允许系统截图和桌面录制，不设置内容保护或捕获排除。
- NVIDIA 等工具的正式验收必须针对打包后的 `Selrena.exe`。开发模式由 `electron.exe` 承载，可能被归类为 Chrome/Electron；该差异不能通过放宽 Chromium sandbox 或 `webSecurity` 伪装修复。

## 最低验收

- 安装目录不包含源码、`.deps/`、构建缓存、测试数据或真实 secrets。
- `component-manifest.json` 能解析全部必需组件，缺失时给出明确安装修复提示。
- Kernel 能按依赖顺序监督 Cognition、Engines 与 Avatar Shell。
- Avatar Shell 只有 Composition Host 的透明窗口可见，Unity Worker 不闪现普通窗口。
- 关闭月见时，Kernel、Cognition、Engine、Composition Host 与 Unity Worker 均按契约退出。
- 升级或修复安装不会覆盖用户状态、模型和日志。
- 安装器、卸载器、首次启动和升级路径均在干净 Windows 用户环境验证。
- NVIDIA / Windows 桌面捕获能识别 `Selrena.exe`，Control Center 与桌面身体均可按用户选择录制。
