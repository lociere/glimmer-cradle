# 开发总则

本文是月见项目的开发边界总则。端到端流程见 [开发流程.md](./开发流程.md)，具体子系统指南见同目录下的 Kernel / Cognition / 桌面渲染指南。

## 1. 文档事实源

- 当前代码事实：`docs/architecture/current/`
- 长期架构目标：`docs/architecture/blueprint/`
- 当前推进计划：`docs/roadmap/当前推进.md`
- 历史决策记录：`docs/history/architecture-decisions/`

不要把旧阶段记录当成当前事实。当前事实与代码不一致时，二者必修其一。

## 2. 语言与编码

- 面向人的文字优先中文。
- 代码标识符、协议字段、配置键、事件类型保持英文。
- 源码注释优先中文，只解释 WHY、跨层契约、不变量和非显然决策。
- 所有 `.ts` / `.tsx` / `.py` / `.yaml` / `.json` / `.md` 使用 UTF-8。

## 3. 分层边界

```text
Protocol       # 跨语言契约
Kernel         # 系统编排、生命周期、扩展宿主、能力代理
Cognition      # 人格、情绪、记忆、推理、认知循环
Renderer UI    # 桌面壳、Presence Surface、Control Center
Extensions     # 平台接入、工具、贡献点
Data/Configs   # 运行时状态与配置
```

关键规则：

- 跨语言类型必须走 `protocol/src/schemas/`。
- Kernel 不写人格语义和 LLM prompt。
- Cognition 不接触平台私有载荷和窗口对象。
- Renderer 不直接访问 Node API。
- Desktop Shell / Avatar Shell 是原生身体，不是普通扩展。
- Extension 是能力贡献点，不是固定“插件类型”枚举。

## 4. 开发流程

完整流程见 [开发流程.md](./开发流程.md)。本总则只保留最小原则：

1. 先确认改动归属哪个层。
2. 先读当前事实，再改代码。
3. 架构方向变化先改蓝图。
4. 改完代码必须跑对应验证。
5. 行为、边界、命名变化必须同步文档和 skill reference。

## 5. 验证基线

```powershell
pnpm typecheck
pnpm build
```

改 Cognition：

```powershell
cd core/cognition
uv run pytest -q
```

改 schema：

```powershell
pnpm sync:contracts
```

改 Electron / UI：

```powershell
pnpm dev
```

并人工确认窗口、托盘、连接状态和 DLQ。
