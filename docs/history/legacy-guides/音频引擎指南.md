# 音频引擎指南

本文记录月见官方音频引擎的本地开发、运行时布局和接入方式。音频引擎位于 `engines/audio/`，独立于 `core/cognition` 的虚拟环境；它是月见的本体能力引擎，不属于 Extension。

## 职责边界

- `core/kernel/.../audio/`：统一调度 ASR/TTS provider，负责请求入口、输出路径、失败日志和能力状态上报。
- `engines/audio/`：官方语音引擎。当前 TTS 默认 Piper；ASR 默认 FunASR；GPT-SoVITS 保留为显式启用的身份声线，不占用默认实时 TTS 主线。
- `data/models/asr/funasr/`：FunASR / ModelScope 模型缓存。启动期自动下载进入这里，不散落到用户全局缓存。
- `engines/audio/src/audio_engine/resources.json`：官方音频资源 catalog，声明稳定 resource id、provider、模型仓库、artifact、落盘目录和声线元数据。
- `data/models/voice/`：catalog 选择的 TTS 模型与用户导入声线，不进入 Git；具体目录不是代码契约。
- `data/packages/gpt-sovits/`：本机托管的 GPT-SoVITS 上游包，不进入 Git；可放上游源码、独立虚拟环境和上游配置。
- `data/cache/audio/tts/`：TTS 合成缓存。月见会按文本与 provider 路线复用已生成短句，降低重复寒暄和常用语的体感延迟。
- `data/artifacts/asr/`：Control Center 上传的录音与 ASR 临时输入，属于可再生运行产物。
- `native/`：未来性能底座，只承担 C/C++ ABI / FFI / 热路径加速，不拥有模型编排。

默认 provider 路线为：

1. `official-audio:piper`：默认实时 TTS，由 `engines/audio` 运行 Piper ONNX voice，优先保证轻量、离线和低延迟。
2. `official-audio:funasr`：Kernel 通过 stdio 调用 `engines/audio`，由 FunASR 完成识别。
3. `official-audio:gpt-sovits-v2`：显式启用的身份声线。设置 `SELRENA_TTS_PROVIDER=gpt-sovits-v2` 时作为首选；设置 `SELRENA_ENABLE_GSV_TTS=1` 时作为可选 provider 加入。
4. `windows-sapi`：Windows 本地 TTS，仅作为工程兜底，不代表月见主声线。

Kernel 内部按能力拆分 TTS 与 ASR lane：TTS lane 默认使用 `uv run --extra tts` 启动 `engines/audio` 并预热 Piper；ASR lane 使用 `uv run --extra asr` 启动 `engines/audio` 并预热 FunASR。GSV 不在默认路线里启动 sidecar；只有显式启用 GSV 时，TTS lane 才会纳管 GPT-SoVITS sidecar。

模型准备是能力层资源门，不是第一次请求的副作用。Kernel 的 `CapabilityRuntime` 会等待 `AudioService.prepareRequiredAudioResources()` 完成默认 TTS / ASR 的准备尝试：

- 默认 TTS：`tts.warmup(provider=piper)` 根据 Audio resource catalog 检查依赖、下载所选 ONNX voice 与配置，并加载 voice。
- 显式 GSV TTS：`tts.warmup` 检查 GPT-SoVITS 上游包、配置、GPT 权重、SoVITS 权重、参考音频，并拉起 sidecar。
- ASR：`asr.warmup` 检查 FunASR / torch / torchaudio 依赖，并提前创建 FunASR pipeline；如果本地缓存缺模型，下载/缓存也发生在 warmup 阶段。
- `health`：返回 provider 状态与 `model_readiness` 快照，供 Control Center 展示诊断，不作为业务请求入口。

`engines/audio` 的 stdout 是严格 JSON 协议通道。FunASR / GPT-SoVITS / Piper 等第三方库的 banner、下载进度和 warning 会被隔离到 stderr，Kernel 只从 stdout 解析协议响应，并把上游日志写入 `data/observability/logs/processes/audio-tts.log` / `audio-asr.log`。

`native`、Edge-TTS、whisper.cpp 不是默认主线，只在显式设置环境变量时加入。Windows SAPI 是工程兜底；Piper 是当前默认实时 TTS 主线。后续若要追求更强身份音色，GSV / CosyVoice / 专门训练路线作为显式声线接入，不反向污染实时声带主线。

## 安装音频引擎

音频引擎使用独立 uv 环境：

```powershell
cd engines/audio
uv sync --extra dev
```

完整语音环境：

```powershell
cd engines/audio
uv sync --extra tts --extra asr --extra dev
```

只安装 TTS：

```powershell
cd engines/audio
uv sync --extra tts --extra dev
```

只安装 ASR：

```powershell
cd engines/audio
uv sync --extra asr --extra dev
```

## Piper TTS

Piper 是当前默认实时 TTS。它由 `engines/audio` 内的 `PiperEngine` 管理，Kernel 只通过 `official-audio:piper` provider 调用，不直接 import 第三方模型库。

默认配置由 Audio resource catalog 决定。环境变量只用于部署覆盖，不再充当内置模型事实源：

| 配置 | 默认值 | 说明 |
|---|---|---|
| `SELRENA_TTS_PROVIDER` | `piper` | 默认 TTS provider；设为 `gpt-sovits-v2` 可切到 GSV 身份声线 |
| `SELRENA_TTS_RESOURCE_ID` | catalog `defaults.tts` | 选择 TTS resource |
| `SELRENA_PIPER_VOICE_ID` | catalog `voiceId` | 覆盖声线标识 |
| `SELRENA_PIPER_MODEL_DIR` | catalog `modelDirectory` 在 `data/models/` 下的投影 | 覆盖 Piper 模型目录 |
| `SELRENA_PIPER_MODEL_PATH` | 自动推导 | Piper `.onnx` 模型路径 |
| `SELRENA_PIPER_CONFIG_PATH` | 自动推导 | Piper `.onnx.json` 配置路径 |
| `SELRENA_PIPER_MODEL_URL` | rhasspy piper voices | 模型下载 URL |
| `SELRENA_PIPER_CONFIG_URL` | rhasspy piper voices | 配置下载 URL |
| `SELRENA_PIPER_USE_CUDA` | 未启用 | Piper 默认 CPU；需要 CUDA provider 时显式启用 |

手动预热：

```powershell
cd engines/audio
'{"id":"tts-warmup","command":"tts.warmup","payload":{"provider":"piper"}}' |
  uv run --extra tts selrena-audio-engine
```

手动合成：

```powershell
cd engines/audio
'{"id":"tts-1","command":"tts.synthesize","payload":{"provider":"piper","text":"晚上好，我是月见。","output_path":"D:\\tmp\\selrena-piper.wav"}}' |
  uv run --extra tts selrena-audio-engine
```

Piper 的优势是轻量、离线、部署稳定，适合作为月见日常短句的实时声带。它不是最强身份克隆路线；如果需要“像停云”这样的强角色音色，仍应把 GSV 作为显式身份声线接入。

## GPT-SoVITS Sidecar

月见不把 GPT-SoVITS 嵌入 Kernel 进程，也不要求用户每次手动启动 GSV。正确形态是：

```text
Kernel
  -> engines/audio（stdio，月见官方音频引擎）
    -> GPT-SoVITS sidecar（HTTP，本机托管上游包）
```

`engines/audio` 会在显式启用 GSV 时，从 catalog 的 `providerDefaults.gpt-sovits` 解析声线模型、参考音频与 sidecar 配置，再确保 sidecar 就绪。当前 catalog 的上游包投影示例为：

```text
data/packages/gpt-sovits/api_v2.py
data/packages/gpt-sovits/<catalog-declared-config>.yaml
```

如果 `data/packages/gpt-sovits/.venv` 存在，sidecar 会使用该虚拟环境的 Python；否则使用当前 Python 启动。

可用环境变量覆盖：

| 变量 | 用途 |
|---|---|
| `SELRENA_GPT_SOVITS_HOME` | GPT-SoVITS 上游包目录，目录下应有 `api_v2.py` |
| `SELRENA_GPT_SOVITS_TTS_CONFIG` | `api_v2.py -c` 使用的 TTS 配置文件 |
| `SELRENA_GPT_SOVITS_COMMAND` | 完整启动命令；优先级高于 `HOME` |
| `SELRENA_GPT_SOVITS_CWD` | 启动命令工作目录 |
| `SELRENA_GPT_SOVITS_ENDPOINT` | GPT-SoVITS `/tts` endpoint，默认 `http://127.0.0.1:9880/tts` |
| `SELRENA_GPT_SOVITS_HEALTH_URL` | 健康检查 URL，默认从 endpoint 推导到 `http://127.0.0.1:9880/` |
| `SELRENA_GPT_SOVITS_STARTUP_TIMEOUT_SECONDS` | sidecar 启动等待时间，默认 `120` |
| `SELRENA_GPT_SOVITS_LOG_PATH` | sidecar stdout/stderr 日志，默认 `data/observability/logs/gpt-sovits-sidecar.log` |

如果上游包、依赖或底模未安装，Control Center 应显示明确原因，而不是静默降级到 Edge-TTS。

## wav + lab 数据集整理

如果后续训练月见身份声线，你的数据集可以继续使用同名 `wav` + `lab` 形式，例如：

```text
archive_tingyun_1.wav
archive_tingyun_1.lab
```

其中 `.lab` 保存该条 wav 的转写文本。训练前建议整理成统一 manifest：

```text
data/models/voice/datasets/tingyun/
├── wavs/
│   ├── archive_tingyun_1.wav
│   └── ...
├── metadata.csv
└── splits/
    ├── train.csv
    └── valid.csv
```

`metadata.csv` 推荐字段：

```csv
audio_path,text,speaker,language
wavs/archive_tingyun_1.wav,如此这般，停云实在不该重返罗浮。今夜风有点凉。,tingyun,zh
```

整理规则：

1. `wav` 与 `lab` 用文件名 stem 一一配对；缺任一方都丢到人工检查列表。
2. 音频统一为单声道、训练仓推荐采样率；响度不要过分压缩。
3. `.lab` 文本统一 UTF-8；去掉空白行、控制字符、不可朗读标记。
4. 单条音频优先 2~12 秒；太短不稳定，太长会拖慢训练并破坏对齐。
5. 划分训练 / 验证集时按语料分布抽样，验证集约 5%~10%。
6. 保留原始数据在外部或 `data/backup/`，训练目录只放可重建的整理副本。

Piper 本身更适合作为推理端轻量声带；角色音色训练可以走 Piper voice 训练仓、StyleTTS2 兼容训练仓、GSV 或 CosyVoice。月见当前工程选择是：默认实时声带用 Piper，强身份声线以显式 provider 形式接入。

## 运行时播放链路

桌面端文字回复和语音回复不是同一个消息类型：

```text
ChannelReplyEvent
  -> DesktopUIController
    -> AvatarShellDownstreamFrame{kind:'reply'}       # 文字内容，进入 ChatMessage UI 投影
    -> AudioService.synthesizeSpeech()
      -> 默认 engines/audio -> Piper；显式启用时可走 GPT-SoVITS sidecar
      -> data/cache/audio/tts/*.wav
    -> AvatarShellDownstreamFrame{kind:'audio_play'}  # 语音外显，不进入 ChatMessage
      -> Electron main IPC bridge
      -> renderer AudioPlaybackController
```

`audio_play` 与 `reply` 使用同一个 `trace_id` 关联。TTS 默认输出到 `data/cache/audio/tts/` 并按短句复用；显式指定 `output_path` 时按调用方路径写入。

## 运行时语音输入链路

语音输入也走 Avatar Shell 契约，不让 renderer 直接绑定 FunASR：

```text
Control Center
  -> VoiceRecorder 采集麦克风
     浏览器回声消除 / 噪声抑制 / 自动增益
     Web Audio 高通 + 低通 + 动态压缩
     PCM 噪声门 + 归一化
     编码 WAV(base64)
  -> preload 白名单 API: sendAudioInput()
  -> Electron main IPC bridge
  -> AvatarShellUpstreamFrame{kind:'audio_input'}
  -> DesktopUIController 写入 data/artifacts/asr/*.wav
  -> AudioService.recognizeSpeech()
  -> engines/audio -> FunASR
  -> AvatarShellDownstreamFrame{kind:'audio_transcript'}
  -> Control Center 更新语音输入状态
  -> DesktopUIController 将文本注入 PerceptionAppService
```

识别成功后，文本进入与键盘输入完全相同的 `PerceptionEvent` 链路；识别失败只更新 UI 状态，不伪造月见回复。

## 设计说明

Piper 作为默认实时声带，负责低延迟、低资源占用和离线稳定；GPT-SoVITS 作为显式身份声线，负责强角色音色和克隆表达；FunASR 负责本地 ASR。月见通过稳定的 stdio engine 协议调用 `engines/audio`，Kernel 只关心 TTS lane / ASR lane 的能力结果，不关心具体模型脚本；后续引入 sherpa-onnx Piper、更强降噪或替换 ASR 模型时，不会污染本体架构。
