## 一、目标与范围

阶段 3 把月见的**运营遥测**补成蓝图第六部分的三支柱形态。

**做：**
- **trace 四层**贯穿运营日志：`session / epoch / trace / span` 不再只活在经历日志里
- **metrics 从无到有**：counter / gauge / histogram + 本地 sink
- **span**：`tracer.py` 从空 stub 变成真的 span API，支持 span 树
- **Telemetry Facade**：统一 `log()` / `metric()` / `span()` 门面

**不做（留待后续）：**
- **第四支柱「内省」**——依赖经历日志的因果链成熟（蓝图 §6.3），留待阶段 5/6
- **OTLP 外部导出**（Jaeger / Grafana / Loki）——阶段 3 只落地**本地 sink**；
  metric / span 的事件结构对齐 OpenTelemetry 语义，未来接外部只是加一个 exporter
- **叙事日记**（阶段 6）
- 经历日志本身（阶段 1 已落地，不动）

## 三、核心决策

### 3.1 session/epoch 是认知核概念，不跨进程

`session`（一次"清醒"）/ `epoch`（一段心境）描述的是**月见这个认知主体**的生命，
是认知核（Python）的概念。Kernel 内核是基础设施，没有"清醒/心境"。

→ **内核日志只注 `trace_id` + `span_id`**（跨进程关联用），不需要 session/epoch。
认知核日志注全四层。

### 3.2 跨进程关联 = trace_id + span_id

一次交互（`trace_id`）跨内核↔认知核；span 嵌套（`span_id` / `parent_span_id`）
也要跨边界。→ IPC 信封（`KernelMessageEnvelope` / `IPCRequest`，均为手写契约，
非 schema 生成）增加可选的 `span_id` 字段，让子进程的 span 挂到父 span 下。
对齐 W3C TraceContext 的 `traceparent` 语义。

### 3.3 metrics 不建 TSDB —— 用事件流

月见是单机应用。**不引入时序数据库**。`metric()` 产出 metric 事件，append 到
`data/observability/metrics/*.jsonl`（JSON Lines，按大小轮转）。可被 CLI / 离线分析 / 未来
的 Grafana-via-exporter 消费。这与经历日志「append-only 文件」的取向一致，便宜。

### 3.4 每进程直接写本地，不绕 IPC

日志已经走「Python stdout → 内核汇总」。metric / span 事件**每进程直接写 `data/`**
（像经历日志、DLQ 那样），不绕 IPC —— 单机共享 `data/`，直接写更简单、不增 IPC 负担。
文件名带进程前缀（`metrics-python-*.jsonl` / `metrics-kernel-*.jsonl`）。

### 3.5 遥测不得拖慢认知热路径

`metric()` / `span()` 必须廉价：仅入内存缓冲，后台批量落盘（与经历日志写入器同构）。
高频场景按模块采样。

## 四、三支柱设计

### ① Logs（已有，仅补四层 trace）

把运营日志的 `trace_id` 注入升级为四层注入：

- Python：`trace_id_processor` → `trace_context_processor`，注入 `session_id` /
  `epoch_id` / `trace_id` / `span_id`（认知核四层都注）。
- TS：`trace-context.ts` 的 store 从 `{trace_id}` 扩为 `{trace_id, span_id}`；
  winston 注入这两层。
- 合成兜底：`synthetic-{module}-{n}` 升级为 `boot-{session_id}-{n}`，与启动周期绑定。

### ② Metrics（新增）

```
metric(name, kind, value, labels?)        kind ∈ counter | gauge | histogram
```

- 月见专属指标：情绪 valence/arousal、记忆数量/检索延迟、LLM 延迟/成本/token、
  认知循环频率（阶段 5 起）、意图产出 vs 实发等。
- sink：`data/observability/metrics/metrics-{proc}-{seq}.jsonl`，每条 = `{ts, name, kind,
  value, labels, trace_id}`。
- 异步缓冲写入器（复用经历日志写入器的同构实现）。

### ③ Traces（四层 + span）

```
session_id  一次"清醒"          认知核进程级
 └ epoch_id  一段心境            认知核进程级
    └ trace_id  一次认知闭环      跨进程，IPC 信封透传
       └ span_id   一个原子操作   跨进程，IPC 信封透传 parent
```

- `tracer.py` / TS tracer 提供 `span(name)` 上下文管理器：进入即新建 span（记
  `parent_span_id`），退出记时长 + 状态，落 span sink。
- span sink：`data/observability/traces/`（JSON Lines），结构对齐 OTel span。
- 跨进程：内核调 Python 前，把当前 `trace_id` + `span_id` 放进 IPC 信封；Python
  入站时以它们为父上下文建 trace/span。

### Telemetry Facade

统一门面，`log()` / `metric()` / `span()` 三个入口，trace 上下文自动注入：
- Python：`cognition_core.core/observability/`（现址）下补 `metrics.py`、充实 `tracer.py`，
  加 `telemetry.py` 门面。
- TS：`core/foundation/logger/` 旁补 `metrics.ts` / `tracer.ts`。

## 六、风险与缓解

| 风险 | 缓解 |
|---|---|
| 遥测拖慢认知热路径 | `metric()`/`span()` 仅入内存缓冲，后台批量落盘；高频按模块采样 |
| IPC 信封改动牵连协议 | 信封是**手写契约**（`kernel_ingress_contracts.py` / `ipc-types.ts`），非 schema 生成，不触发 `sync:contracts`；`span_id` 设为**可选**字段，旧消息兼容 |
| metrics / traces 文件无限增长 | 按大小轮转 + 保留期；与 `runtime.log` 轮转策略一致 |
| 字段命名分裂 | 所有遥测字段遵守 [日志字段表](../legacy-current-architecture/09-日志字段表.md)；新字段（`session_id`/`epoch_id`/`span_id`/`metric_*`）先入字段表再落代码 |
