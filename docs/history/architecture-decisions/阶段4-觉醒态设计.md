## 一、目标与范围

阶段 4 把蓝图 §3.4 的「觉醒态机」从架构愿景落成可观测、可影响认知热路径的子系统。

**做：**
- **契约权威落在 protocol**：`ArousalState` / `ArousalProfile` / `ArousalSnapshot`
  作为 schema 进 `protocol/src/schemas/`，由 `sync:contracts` 双向 codegen，
  两端 import 同一份生成产物 —— Python 与 TS 是对等两端，不是谁基于谁。
- **运行时权威落在认知核**（Python `selrena/arousal/`），Kernel 内核 LifeClock 转为"读状态、调心跳"的薄驱动 —— 不再有两套词汇并存。
- **四态模型完整建立**：`Dormant / Dreaming / Ambient / Awake`，状态名 + 转换规则 + 经历投影驱动衰减。
- **挂钩 ① 经历投影**（蓝图 §3.4 第 1 条挂钩）：`last_interaction_at` 从经历之流（Moment 流）投影，驱动空闲衰减。
- **挂钩 ② 最小可见效**：每态自带 `frequency_hint_ms`，TS LifeClock 从 state_sync 读到后据此设心跳间隔；
  推理后端档位、token / 上下文预算、深度 —— **接口预留 ArousalProfile，但行为整合留待阶段 5**。
- **可观测**：觉醒态进入 `state_sync` payload；状态转换写入经历之流为 `Moment(kind=arousal)`；
  Telemetry 加 `arousal.state` gauge + `arousal.transition` counter。

**不做（留待后续）：**
- **挂钩 ③ Volition 主动开口闸**：依赖 Volition 模块（阶段 5），本阶段只预留 `allows_proactive: bool` 字段。
- **Dreaming 态的真实记忆固化逻辑**：沿用 v4.5 既有 `consolidate_memory` IPC（由 TS 侧
  `ConsolidationTrigger` 触发），本阶段只引入态名与转换规则。
- **推理后端按觉醒态切档**：`reasoning_service` 尚未抽象成「按 tier 选后端」，整合留待阶段 5。
- **上下文预算按觉醒态缩放**：由 `ContextAssembly` / Reasoning prompt 预算层消费，按 arousal 动态缩放留待阶段 5。
- **觉醒态机的强情绪抵抗衰减**：蓝图说"她担心一件事就睡不着"——情绪强度 ≥ threshold 时 hold 当前态。
  本阶段实现最小版（emotion intensity > 0.8 时 hold），精细化留待阶段 5（情绪系统重写）。

### 2.1 既有"注意力"模型（Kernel 内核）

[`core/kernel/src/core/domain/organism/life-clock/life-clock-manager.ts`](../../../core/kernel/src/core/domain/organism/life-clock/life-clock-manager.ts)

| 蓝图 (§3.4) | TS 既有 | 含义对齐 |
|---|---|---|
| **Dormant 休眠** | `standby` | 大致对齐 |

| **Awake 清醒** | `focused` | 对齐 |

→ 阶段 4 收掉 TS 侧的 `AttentionMode` 决策权，统一改读认知核给的 `arousal.state`。

## 三、核心决策

### 3.1 权威态落认知核（不缝补）

蓝图 §3.4 明确："觉醒态是认知循环的运行档位"——它是月见**心智**的一部分，
不属于内核基础设施。

→ **新建 `core/cognition/src/selrena/arousal/`**，作为权威状态持有者。
TS LifeClock 通过 state_sync 读到当前 arousal，据此设心跳间隔；不再自维护 mode。

### 3.2 经历投影驱动衰减（挂钩 ①）

`last_interaction_at` = 经历之流中最近 `MomentKind.PERCEPTION` 或 `REPLY` 的
`occurred_at`。这是蓝图 §3.4 第 1 条挂钩的字面落地 ——"它是经历日志的一个投影"。

→ `arousal/projection.py` 提供 `compute_idle_seconds(now, recorder) -> float`，
从经历之流的最近事件计算 idle。

### 3.3 状态转换是同步纯函数

`ArousalStateMachine.tick(current, idle_seconds, state_elapsed_seconds, emotion_intensity, wake_signal) -> next`
是无副作用的纯函数。便于测试 / 离线重放。副作用（事件发布、Moment 记录、metric）
由 `ArousalManager` 统一负责。

`idle_seconds` 描述距离上次 perception/reply 的时间；`state_elapsed_seconds` 描述当前态已持续多久。
Dreaming 自然结束必须看 `state_elapsed_seconds`，否则长期空闲或冷启动时会在 Dormant/Dreaming 间震荡。

### 3.4 显式 wake 信号最优先

`wake_now(reason)` 直接强制转 Awake，不论当前态、不论 idle 时长。来源：
- 显式召唤（唤醒词 / @ / 私聊 / `address_mode=direct`）
- 内核传入的 perception（无论 direct/ambient，都计为"有人来"）
- 定时（如配置）

→ wake 信号在 tick 之前消费，避免与衰减打架。

### 3.5 情绪抵抗衰减（最小版）

蓝图："强情绪可抵抗衰减——她担心一件事就睡不着"。
本阶段最小实现：`emotion_intensity > 0.8` 时，禁止"向下衰减"（向上唤醒不受影响）。
精细情绪权重（valence × arousal 二维模型）留待阶段 5 情绪系统重写。

### 3.6 转换记入经历之流（不记入运营日志）

状态转换是"她经历的事件"——记为 `Moment(kind=arousal, content={from, to, reason})`，
进经历之流；同时 telemetry 侧记一个 `arousal.transition` counter。
**运营日志**（runtime.log）只打 info 级流水，**不替代** Moment。

> 这吻合阶段 3.6 已定的"经历 vs 遥测分离" —— 经历记她的事，遥测记机器的事。
> 觉醒态既是她经历的事（被困倦扑倒），又是机器的运行参数（心跳间隔），两边都要记。

### 3.7 ArousalState / ArousalProfile / ArousalSnapshot 是 protocol 契约

Python 认知核与 Kernel 内核是**对等两端**，不是谁基于谁 ——
凡两端都看见的概念，权威定义就在 `protocol/`，由 `sync:contracts` 双向 codegen，
**零漂移**。

| 类型 | 位置 | 谁看 | 谁写 |
|---|---|---|---|
| `ArousalState`（枚举） | `protocol/src/schemas/ArousalState.schema.json` | 认知核 + 内核 | codegen |
| `ArousalProfile`（结构） | `protocol/src/schemas/ArousalProfile.schema.json` | 认知核 + 内核 | codegen |
| `ArousalSnapshot`（state_sync 子结构） | `protocol/src/schemas/ArousalSnapshot.schema.json` | 认知核 + 内核 | codegen |
| `ArousalConfig`（衰减阈值等） | `selrena/arousal/states.py` | 仅认知核 | 手写 Pydantic |
| `MomentKind.AROUSAL` | `selrena/experience/events.py` | 仅认知核（经历之流是认知核独占） | 手写 |

落地节奏：**先 4.0 schema + sync:contracts**，然后 4.1 起两端代码 import 已生成的契约。

### 3.8 默认值与配置

| 转换 | 阈值（秒） | 配置键 |
|---|---|---|
| Awake → Ambient | 120 | `arousal.awake_to_ambient_idle_s` |
| Ambient → Dormant | 600 | `arousal.ambient_to_dormant_idle_s` |
| Dormant → Dreaming | 3600 | `arousal.dormant_to_dreaming_idle_s` |
| Dreaming → Dormant | 300 | `arousal.dreaming_duration_s` |
| 显式 wake | 立即 | n/a |

每态的 `ArousalProfile`：

| 态 | frequency_hint_ms | allows_proactive | model_tier | context_budget_factor |
|---|---|---|---|---|
| Dormant | 60000（仅维持心跳） | false | none | 0.0 |
| Dreaming | 30000 | false | local_only | 0.3 |
| Ambient | 45000 | true（受限） | local_only | 0.6 |
| Awake | 10000 | true | cloud_allowed | 1.0 |

本阶段只**消费** `frequency_hint_ms`；其他字段先就位、留待阶段 5 接 Reasoning / Volition / Context。

### 3.9 与既有 attention_mode 的过渡

`life_heartbeat.attention_mode` IPC 字段**暂保留**作向后兼容；TS 侧由
`arousal.state → attention_mode` 一对一映射（`awake→focused / ambient→ambient /
dormant→standby / dreaming→standby`），认知核侧不再依赖此字段做决策。

阶段 5 接 Reasoning 时再废弃 `attention_mode`。

## 四、模块设计（认知核侧）

```
core/cognition/src/selrena/arousal/
├─ __init__.py               # 公开 ArousalState / ArousalManager
├─ states.py                 # ArousalState 枚举 + ArousalProfile 映射表 + ArousalConfig
├─ state_machine.py          # ArousalStateMachine：纯函数 tick / signal_wake
├─ projection.py             # 从经历之流投影 last_interaction_at / idle_seconds
└─ manager.py                # ArousalManager：异步 tick 循环 + 副作用（事件 / Moment / metric）
```

### 4.1 ArousalState 枚举

```python
class ArousalState(str, Enum):
    DORMANT = "dormant"
    DREAMING = "dreaming"
    AMBIENT = "ambient"
    AWAKE = "awake"
```

### 4.2 ArousalProfile

冻结 dataclass：`frequency_hint_ms / allows_proactive / model_tier /
context_budget_factor`。本阶段只用 `frequency_hint_ms`；其他字段作为 placeholder
让阶段 5 直接读，不需要再改协议。

### 4.3 ArousalStateMachine（纯函数）

```python
@dataclass(frozen=True)
class ArousalTickResult:
    state: ArousalState
    changed: bool
    reason: str

def tick(
    current: ArousalState,
    *,
    idle_seconds: float,
    state_elapsed_seconds: float,
    emotion_intensity: float,
    wake_requested: bool,
    config: ArousalConfig,
) -> ArousalTickResult: ...
```

逻辑顺序：
1. `wake_requested` → Awake（reason="wake_signal"）
2. 否则按 idle 阈值 + emotion hold 决策。

### 4.4 ArousalManager（副作用层）

```python
class ArousalManager:
    def __init__(self, *, config, experience_recorder, self_entity, tick_interval_s=5.0): ...
    async def start(self) -> None: ...
    async def stop(self) -> None: ...
    def wake_now(self, reason: str) -> None: ...
    def get_state(self) -> dict: ...   # 给 SelfEntity.get_state 包入 state_sync
```

`get_state()` 返回 `{state, since_at, idle_seconds, profile}`，并入 SelfEntity 的
state_sync payload。

### 4.5 与 SelfEntity 集成

`SelfEntity.get_state()` 增加 `"arousal": arousal_manager.get_state()`。
`SelfEntity` 持有 `arousal_manager` 引用，由 DI 容器装配。

### 4.6 与感知集成（wake 信号源）

`container._on_perception` 在 perception 入站成功写入 `PerceptionEventQueue` 后调用
`arousal_manager.wake_now("perception")`，并通知 `CognitiveLoop.notify_external_input()`。
这保证直接输入先进入 Awake，再由认知循环的下一拍完成 Deliberate / Volition。
`active_thought_use_case`（生命心跳）不发 wake 信号 —— 心跳本身不算"被叫醒"。

## 五、协议与跨进程

### 5.1 schema 驱动的契约（首选路径）

新增三份 JSON Schema，由 `pnpm sync:contracts` 双向 codegen：

```
protocol/src/schemas/
├─ ArousalState.schema.json       # 枚举：dormant / dreaming / ambient / awake
├─ ArousalProfile.schema.json     # frequency_hint_ms / allows_proactive / model_tier / context_budget_factor
└─ ArousalSnapshot.schema.json    # { state, since_at, idle_seconds, profile }  ← state_sync 子结构
```

生成产物：

- `protocol/src/generated/ArousalState.ts` / `ArousalProfile.ts` / `ArousalSnapshot.ts`
- `core/cognition/src/selrena/ipc_server/contracts/generated/arousal_state.py` /
  `arousal_profile.py` / `arousal_snapshot.py`

**任一端要改契约必先改 schema → 跑 `sync:contracts` → 再改代码**；
忘了同步会被 `sync:contracts:check` CI 卡住。

### 5.2 state_sync payload 扩展

`SelfEntityStatePayload`（手写契约）的可选块从 schema 引用：

```typescript
arousal?: ArousalSnapshot;   // 来自 generated/ArousalSnapshot
```

向后兼容（可选字段；旧版本读取端忽略即可）。

### 5.3 TS LifeClock 改造

- 删除 `AttentionMode` 内部决策；改为订阅 state_sync，读 `state.arousal.profile.frequency_hint_ms` 设心跳间隔。
- `life_heartbeat.attention_mode` 字段按映射表填，向后兼容；不再驱动认知核分支。
- `_channelFocusStates` 保留，但**只**作为感知发射的"焦点信号"，不再决定 attention mode（焦点变更也会推 wake 信号给认知核，逻辑收归 arousal）。

## 七、风险与缓解

| 风险 | 缓解 |
|---|---|
| 认知核与内核两边状态短暂不一致 | state_sync 已经 1Hz 推送；态变化时 SelfEntity 提前 push 一次（事件驱动） |
| Dreaming 态名引入但功能未实现 → 误以为 Dreaming 自带固化 | 设计文档 §一 明确「沿用 v4.5 consolidate_memory，不新增固化逻辑」；MomentKind.AROUSAL 转换 reason 字段记真实原因 |
| projection 每次 tick 都扫整段经历日志 → I/O 浪费 | ArousalManager 在内存维护 `_last_interaction_at` 缓存；只有冷启动时从经历之流 fold 一次 |
| 情绪抵抗衰减的阈值 0.8 不准 | 阶段 4 视作占位；阶段 5 情绪系统重写时一并校准 |
| `attention_mode` 字段双重含义（兼容期） | 设计文档明确「阶段 5 废弃」；映射表写在 §3.8 与 §5.2 |
