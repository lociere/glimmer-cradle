## 一、目标与范围

阶段 2 把月见的数据持久化整理成蓝图第五部分的**五层存储模型**，并解决一个核心
结构问题：**持久化归属**。

**做：**
- 把长期记忆 / 短期记忆的持久化从 Kernel 内核**收归认知核**，与经历日志、DLQ 一致
- 知识库从「配置注入」升级为「curated 知识库存储」
- 引入向量层：embedding 持久化，省去每次启动全量重算
- 五层存储模型在物理目录上落地

**不做（留待后续阶段）：**
- 由经历日志驱动的状态投影（阶段 5）
- 记忆图谱 / 反思引擎（阶段 6）
- Layer 5 多模态 blob 持久化（暂无紧迫需求，单列后续小补丁）
- 跨进程 trace 四层传播（阶段 3）

## 二、现状盘点

### 2.1 持久化归属现状：分裂

| 数据 | 谁持久化 | 落点 | 方式 |
|---|---|---|---|
| 经历日志 | **认知核直接** | `data/experience/` | 阶段 1 已落地 |
| DLQ 失败事件 | **认知核直接** | `data/cognition/cognition.db`（DLQ 表组） | aiosqlite |
| 长期记忆 | **认知核直接** | `data/cognition/cognition.db`（memory 表组） | aiosqlite |
| 短期记忆 | **认知核内存态** | 重要片段进入长期记忆；用户可读转录在 `data/extensions/transcripts/` | 内存 + JSONL |
| 知识库 | 不持久化 | 内核从配置读，IPC 注入 | `knowledge_init` |

认知核已经直接持久化经历日志、DLQ、长期记忆、知识库和向量。旧的「认知核内存镜像 ↔
内核 SQLite」双向同步已经结束，Kernel 不再是记忆事实源。

### 2.2 记忆的同步链路（现状）

```
历史链路（已迁移完成）：

写：LongTermMemory.add()
      → 发 MemorySyncEvent（进程内事件总线）
      → MemorySyncUseCase.on_long_term_memory_sync
      → 出站 IPC → Kernel 内核
      → MemorySyncManager → MemoryRepository.upsertMemoryFromSync
      → 旧 data/cognition_core.db

读（仅启动时）：
   内核 MemoryRepository.getAllMemories() → memory_init IPC
      → LongTermMemory.init_from_kernel() → 认知核内存 dict
```

当前链路：认知核直接读写 `data/cognition/cognition.db`；Kernel 仅保留扩展宿主、
DLQ 基础设施和桌面 UI 等系统职责，对记忆不再持久化。

### 2.3 关键发现：内核对记忆「只存不用」

内核侧 `MemoryRepository` 有 `getRelevantMemories`（LIKE 关键词检索）等方法，
但**实际的记忆检索发生在认知核**（`LongTermMemory.retrieve_relevant`，向量 +
bigram）。内核对记忆**不做任何认知判断**，它纯粹是一个持久化代理。

> **这是阶段 2 可行性的基石**：把持久化收归认知核，是一次**无逻辑损失的纯粹搬迁** ——
> 不会动到任何业务规则，只是把"谁写盘"换了个进程。

### 2.4 现存数据资产（迁移必须考虑）

旧 `data/cognition_core.db` 的真实用户数据（`long_term_memory` / `short_term_memory`）已迁入
canonical layout：主记忆库在 `data/cognition/cognition.db`，历史来源归档到
`data/legacy/cognition_core.db`，迁移前快照在 `data/backup/`。

## 三、核心决策：持久化归属收归认知核

### 3.1 决策

**长期记忆、短期记忆、知识库、向量的持久化，全部收归认知核，直接写 `data/`。**
Kernel 内核不再承载任何记忆持久化职责。

### 3.2 理由

1. **消除双份事实源**：现状是"内存镜像 + 内核 DB"两份；收归后认知核的库即唯一真相。
2. **对齐已有形态**：经历日志、DLQ 已是认知核直接持久化；记忆收归后三者一致。
3. **无逻辑损失**：§2.3 —— 内核对记忆只存不用，搬迁不动业务规则。
4. **去掉一次纯 IPC 往返**：记忆读写不再跨进程，启动不依赖 `memory_init` 握手。
5. **符合三层解耦**：记忆是 Cognition 认知核 的认知器官；它的持久化本就该归 Cognition 认知核，
   而非中枢。现状是历史遗留，不是设计意图。

### 3.3 不违背的约束

- 当前 canonical layout 已收口到 [Data Layout Reference](../../reference/data-layout.md)；
  旧 `data/memory/` 与根部 `data/cognition_core.db` 不再作为运行路径。
- 引擎用 **aiosqlite**（认知核已依赖；异步，不阻塞事件循环）。
- 内核侧 `extension_storage` / `extension_short_term_memory` 表**不动** ——
  那是扩展宿主的数据，归内核合理，不在本阶段范围。

## 四、五层存储模型落地

蓝图第五部分的五层，在本项目的具体落点：

| 层 | 内容 | 落点 | 引擎 | 阶段 2 状态 |
|---|---|---|---|---|
| L1 真相 | 经历日志 | `data/experience/` | 分段 JSON Lines | ✅ 阶段 1 已落地 |
| L2 投影 | 长期记忆 | `data/cognition/cognition.db`（memory 表组） | aiosqlite | 🎯 本阶段收归 |
| L3 curated | 知识库 | `data/cognition/cognition.db`（knowledge 表组）+ `data/cognition/knowledge/sources/` | aiosqlite | 🎯 本阶段建立 |
| L4 向量 | embedding 持久化 | `data/cognition/cognition.db`（`embedding` 表，float32 BLOB） | aiosqlite | 🎯 本阶段引入 |
| L5 blob | 多模态文件 | `data/blobs/`（内容寻址） | 文件系统 | ⏸ 暂不做 |

> L2 命名为"投影"是蓝图的目标语义（记忆终将由经历日志投影重建，阶段 5）。
> 阶段 2 的 L2 仍是**独立权威存储**，不是投影 —— 这是过渡形态，doc 如实标注。
>
> **短期记忆不入 L2**：短期记忆是 working memory（蓝图 §4.3：易失、随循环刷新、
> 仅快照落盘）。它的对话内容已被经历日志 L1 完整记录，重要片段经固化进入 L2 长期
> 记忆 —— 单独为它建 DB 表既冗余又违背"working memory 易失"。故批次 2.3 移除其
> DB 持久化，退回纯内存。崩溃恢复由经历日志承担（阶段 5 起短期记忆可由 L1 投影重建）。

### 4.0 单库决策（L2 + L3 + L4 合一个 SQLite）

L2/L3/L4 合并到一个 `data/cognition/cognition.db`，**不是为了省事，是为了原子性**：

- 月见形成一条记忆时，要同时写「记忆行（L2）+ 它的 embedding（L4）」。同一个库 →
  一次事务 → 记忆与向量**要么都成、要么都不成**，不会留下孤儿向量或无向量记忆。
  这直接服务月见的命脉：数据完整、不失忆。
- 检索时上下文要同时查记忆 + 知识 + 向量，同库 JOIN 天然。
- 单机单进程单写者，单文件备份/连接管理更省心（符合「单人能扛」）。

**切分原则：按「访问模式 + 故障独立性」分库，不按「概念层级」分。**
- 经历日志（L1）—— append-only 流式，访问模式不同 → 独立（且非 SQLite）。
- DLQ —— 必须在主库损坏时仍可读 → 独立。
- 记忆/知识/向量 —— 一起原子写、一起查 → 合库 `cognition.db`。

### 4.1 认知核持久化包 `selrena/persistence/`

新增一个统一的持久化访问层，封装 aiosqlite，向记忆 / 知识 / 向量模块提供
repository。它是认知核内**唯一**接触 SQLite 的地方（横切基础设施）。

```
selrena/persistence/
├─ __init__.py
├─ database.py        aiosqlite 连接管理 + schema 初始化 + migration（cognition.db 唯一入口）
├─ memory_repo.py     长期记忆 CRUD
├─ knowledge_repo.py  知识库条目 CRUD
├─ vector_repo.py     embedding 读写（float32 BLOB，按 model 标识）
└─ migration.py       内核旧库 cognition_core.db → cognition.db 一次性迁移
```

`database.py` 持有 `cognition.db` 的唯一连接；三个 repo 共用它，跨表写入走同一事务。

### 4.2 物理目录

```
data/
├─ experience/        L1（阶段 1 已有，独立——append-only 流）
├─ cognition/
│  ├─ cognition.db       L2 记忆 + L3 知识 + L4 向量（单库，见 §4.0）
│  ├─ dead_letters.db    Cognition 侧 DLQ（独立——故障隔离）
│  └─ knowledge/sources/ L3 原始 lore / 文档（文件系统）
├─ kernel/kernel.db      Kernel 基础设施库（扩展宿主 / TS DLQ）
├─ observability/        日志、metrics、traces
├─ blobs/                L5 多模态内容寻址
├─ backup/               迁移 / 驱逐 / 删除前快照
└─ legacy/               只读历史迁移源
```

`data/kernel/kernel.db` 迁移后仅保留 `extension_*` / `dead_letters_ts` 等 Kernel 基础设施表。

## 五、知识库 / 记忆分离

### 5.1 现状

知识库当前是**配置派生**：内核读配置 → `knowledge_init` IPC → 认知核内存。
没有独立存储，没有"摄入"概念，条目数受配置规模限制。

### 5.2 目标

知识库升级为蓝图 §4.5 的 **curated 知识库**：

- 独立存储 `data/knowledge/kb.db`：条目 + **激活规则**（关键词 / 向量 / 常驻）+ 元数据
- 原始来源放 `data/knowledge/sources/`
- 摄入管线：文档 → 切块 → 入库 → 建向量
- 与记忆在**存储层彻底分开**（"被给予的" vs "活出来的"），但在检索层都进 L4 向量层、
  带 `source` 标签区分（统一检索是阶段 5「上下文装配」的事，本阶段只把存储分清）

### 5.3 阶段 2 的务实边界

阶段 2 **只建知识库的存储与摄入**；激活规则先落最简单的"常驻 / 关键词"，
向量激活与统一上下文装配留待阶段 5。`knowledge_init` IPC 可保留为"用配置预填
知识库"的一条摄入途径，不强行删除。

## 六、迁移分批清单

每批独立可提交、可回滚。**单批最多触及 内核 + 认知核 两层的一个数据类别** ——
不一次翻多层。

> **落地进度（截至 2026-05-16）**：批次 2.1~2.6 全部完成。阶段 2 已落地。
> 残留待办（不阻塞）：protocol 层 `MemorySyncEvent` / `ShortTermMemorySyncEvent` /
> `MEMORY_INIT` 等已无生产消费方的死类型尚未从 `protocol/src/` 删除（需走
> `sync:contracts`，单列小补丁）；旧 `cognition_core.db` 中已迁移的 `long_term_memory` /
> `short_term_memory` 表，新库不再建，存量库的残留表无害、未强制 DROP。

| 批次 | 内容 | 触及 | 风险 |
|---|---|---|---|
| **2.1** | 新增 `selrena/persistence/` 基座：`database.py`（`cognition.db` 连接 + schema 初始化 + migration）+ 空 repo 骨架。纯新增，不接线。 | 认知核 | 极低 |
| **2.2** | 长期记忆收归：`LongTermMemory` 启动改为从 `cognition.db` 加载、`add()` 写 `cognition.db`；删 `MemorySyncEvent` 链路；删内核 `long_term_memory` 表写入 + `MemorySyncManager` 对应路径。**带数据迁移**：首次启动把 `cognition_core.db.long_term_memory` 导入 `cognition.db`。 | 认知核 + 内核 | 中 |

## 七、风险与回滚

| 风险 | 缓解 |
|---|---|
| 迁移丢数据 | 旧 `cognition_core.db` 保留只读，不删（直到 2.6 且人工确认）；迁移做幂等 upsert；迁移前可手动备份 `data/`。 |
| 新库与旧库不一致 | 2.2 落地后提供一次性核对脚本（`scripts/` 下），比对条数与内容。 |
| aiosqlite 并发写 | 认知核单事件循环、单写者；记忆写入频率低，WAL 模式足够。 |
| `sqlite-vec` 新依赖 | 2.5 才引入；引入失败时向量层降级为"内存重算"（即现状），不阻断。 |
| 内核删码误伤 | 2.6 单独成批；删前 grep 确认 `MemoryRepository` 记忆方法无其它调用方。 |
| 跨进程启动握手变化 | `memory_init` 移除后，认知核启动不再等内核注入记忆 —— 需确认启动时序无依赖（认知核自给自足反而更简单）。 |

**回滚**：每批是独立提交。2.2~2.5 若出问题，`git revert` 该批即回到上一批的可用
状态；旧 `cognition_core.db` 一直在，回滚后内核侧同步链路恢复即可。

## 八、对其它阶段的影响

- **阶段 3（遥测）**：不冲突，可并行。
- **阶段 5（上下文装配 / 投影）**：阶段 2 的 L2/L3/L4 存储是其前置；届时 L2 从
  "独立权威存储"演进为"经历日志的投影"，存储接口尽量在阶段 2 就设计得便于该演进。
- **内核**：阶段 2 后内核彻底退出记忆领域，只保留扩展存储 —— 与蓝图"中枢不承载
  认知/认知数据"一致。

## 九、已定决策

落地 2.1 前的三个问题，已于 2026-05-15 拍板：

| # | 议题 | 决策 | 理由 |
|---|---|---|---|
| 1 | 单库 vs 多库 | **单库 `data/cognition/cognition.db`**（L2+L3+L4） | 原子写（记忆+向量同事务）、单人好维护；见 §4.0 |
| 2 | `knowledge_init` IPC 去留 | **保留**为知识库的预填摄入途径之一 | 不强行切断，配置仍可预填知识库 |
| 3 | 旧 `cognition_core.db` 记忆表清理时机 | 定在批次 **2.6** | 迁移验证一致后再清，足够保守 |
| 4 | 向量层用 `sqlite-vec` 还是 BLOB | **BLOB 列**（`embedding` 表，float32），相似度走 Python numpy | 月见单机数百条记忆，用不上 SQL 侧 KNN；`sqlite-vec` 要背一个 Windows 上有加载风险的扩展依赖。规模剧增再引入 |
