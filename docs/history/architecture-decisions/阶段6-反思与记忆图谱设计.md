# 阶段 6 设计：反思引擎 + 记忆图谱 + 叙事日记

> 起草日期：2026-05-26
> 蓝图对应：§3.3 Consolidation / §3.4 Dreaming / §4.2 认知派生 / §4.3 §4.4
> 记忆图谱 / §4.8 身份 / §6.5 叙事日记
> 状态：**设计稿**，未落地。落地时按 §六 迁移分批清单逐批推进。

> 阶段 6 是蓝图终局。前 5 个阶段把"她活着的当下"做好（经历日志 + 持久化 +
> 觉醒态 + 认知循环 + Volition）；阶段 6 把"她对生活的理解"做出来 ——
> **她能回头看自己经历过什么、想通什么、对人事物形成印象**。

---

## 一、设计原则

1. **反思即经历**（蓝图 §4.2）—— 每次反思的产物都作为 `Moment(kind=reflection)`
   追加进经历日志。语义记忆 = `ReflectionEvent` 的投影。没有任何"派生知识"
   逃在经历日志之外。
2. **图谱是叠加，不是覆盖**（蓝图 §4.4）—— 事实节点矛盾可改写（保留历史版本）；
   印象 / 关系节点**记录轨迹、绝不覆盖**。对一个生命体，变化本身就是记忆。
3. **反思 = Dreaming 态的工作**（蓝图 §3.4 / §3.5）—— Reflection 不抢 Awake 的
   token 预算；Dreaming 时跑本地小模型蒸馏，Awake 时不打扰用户交互。
4. **HippoRAG 风格检索**（蓝图 §4.4）—— 向量相似找入口节点 → 沿边多跳遍历。
   支持"她昨天提到的地方遇到的人"这类多跳查询。
5. **叙事日记是免费投影**（蓝图 §4.1 §6.5）—— 不存储；按需把经历日志顺因果 +
   时间 + 情绪串读即可成故事。可选 LLM 第一人称渲染。
6. **ChatUseCase 移除**（阶段 5.9 §6.1）—— Phase 6 末批正式移除，CognitiveLoop
   单轨稳定运行。
7. **不引入新 Provider**（蓝图 §3.2 铁律）—— ReflectionEngine 不是 Provider，
   是 Consolidation 阶段的钩子；MemoryGraph 由现 MemoryProvider 与 SocialProvider
   消费。

---

## 二、术语

| 术语 | 含义 |
|---|---|
| `Moment(kind=reflection)` / `ReflectionEvent` | 反思产物在经历日志中的一刻；蓝图 §4.2 关键闭环 |
| MemoryGraph | 语义记忆 = 节点+边的图（Zettelkasten 风格）；蓝图 §4.4 |
| 入口节点 | 向量相似命中的 MemoryGraph 节点；HippoRAG 起跳点 |
| 多跳检索 | 入口节点 → 沿边按权重 BFS K 跳；蓝图 §4.4 |
| Episode | 一片因果稠密相连的 Moment 子图（涌现，非存储）；蓝图 §4.1 |
| Narrative Entry | 叙事日记一条；从 Episode 投影；蓝图 §6.5 |
| ReflectionBatch | 一次反思的输入 = 待蒸馏 Episode 列表 |

## 三、模型与接口

### 3.1 `MomentKind` 扩展

蓝图 §4.1 早已规划 `reflection` 这个 kind；阶段 1 落 `MomentKind` enum 时已留位。
本阶段把它接到经历日志写入路径，并加 schema 描述。

```python
class MomentKind(StrEnum):
    PERCEPTION = "perception"
    THOUGHT = "thought"
    EMOTION = "emotion"
    REPLY = "reply"
    SILENCE = "silence"
    REFLECTION = "reflection"     # ← 阶段 6 启用
```

### 3.2 `ReflectionEvent`（即 `Moment(kind=reflection).content`）

`ReflectionEvent` 不是独立 Moment 类型，是 `kind=reflection` 的 `content` 形态。
Schema 限定 content 字段，落 `schemas/models/ReflectionEvent.schema.json`：

```jsonc
{
  "type": "object",
  "required": ["reflection_type", "summary", "graph_ops"],
  "properties": {
    "reflection_type": {
      "enum": ["fact", "impression", "relation", "autobiographical"]
    },
    "summary": { "type": "string" },          // 一句话蒸馏
    "evidence_moment_ids": {                  // 蒸馏所依据的原始 moment（因果回指）
      "type": "array", "items": { "type": "string" }
    },
    "graph_ops": {                            // 本次反思引起的图谱操作（追加/改写/连边）
      "type": "array",
      "items": { "$ref": "#/definitions/GraphOp" }
    }
  }
}
```

`evidence_moment_ids` 是因果链的另一种表达：本反思的 `causation_ids` 指向
**直接触发本拍反思的 Moment**（如"夜间 Dreaming tick 进入 consolidation 阶段"
那一刻的工作区项）；`evidence_moment_ids` 指向**被蒸馏出的原始经历**（一批
Perception/Emotion/Reply 等）。前者是"为什么这时反思"，后者是"反思什么"。

### 3.3 MemoryGraph 节点 / 边

落 `schemas/models/MemoryGraphNode.schema.json` 与 `MemoryGraphEdge.schema.json`：

```jsonc
// MemoryGraphNode
{
```

### 3.4 持久化：`cognition.db` schema_version 加 `memory_graph_*` 两表

阶段 2 已有 `cognition.db`，schema_version 已升级过。阶段 6 加版本：

```sql
CREATE TABLE memory_graph_node (
    node_id TEXT PRIMARY KEY,
    node_type TEXT NOT NULL,
    content TEXT NOT NULL,
    vector_id TEXT,
    actor_id TEXT,
    salience REAL NOT NULL DEFAULT 0.5,
    created_at TEXT NOT NULL,

CREATE INDEX idx_graph_node_type ON memory_graph_node(node_type);
CREATE INDEX idx_graph_node_actor ON memory_graph_node(actor_id) WHERE actor_id IS NOT NULL;

CREATE TABLE memory_graph_edge (
    edge_id TEXT PRIMARY KEY,
    from_node TEXT NOT NULL REFERENCES memory_graph_node(node_id) ON DELETE CASCADE,
    to_node TEXT NOT NULL REFERENCES memory_graph_node(node_id) ON DELETE CASCADE,
    edge_type TEXT NOT NULL,
    weight REAL NOT NULL,
    created_at TEXT NOT NULL,
    evidence_moment_ids TEXT NOT NULL DEFAULT '[]'  -- JSON 数组
);
CREATE INDEX idx_graph_edge_from ON memory_graph_edge(from_node);
CREATE INDEX idx_graph_edge_to ON memory_graph_edge(to_node);
```

`MemoryGraphRepository` 提供 CRUD + 多跳遍历查询。

### 3.5 ReflectionEngine 工作流

```
ReflectionEngine.tick(arousal_state, recent_episodes)
  ├─ 判定：arousal_state == "dreaming" ?  否 → 不跑
  ├─ 取入：从 ExperienceRecorder 拉过去 N 小时未蒸馏的 Moment（按 last_reflection_at 标记）
  ├─ 分段：按 causation_ids 联通分量切 Episode
  └─ 对每个 Episode：
        ├─ Prompt 拼接（含原始 Moment 文本 + 当前 MemoryGraph 相关节点）
        ├─ ReasoningService.generate(tier="local_only")   # 蓝图 §3.5 Dreaming 仅本地
        ├─ 解析为 ReflectionEvent.graph_ops 列表
        ├─ 落 Moment(kind=reflection)（因果链：causation_ids 指 Episode 末 Moment）
        └─ 执行 graph_ops（add_node / append_history / add_edge / mark_contradicts）
  └─ 更新 last_reflection_at 水位
```

**触发节奏**：
- Dreaming 态进入时立即跑一次（"睡前回顾"）
- Dreaming 态内每 30 分钟一次（避免占满 Dreaming）
- Awake 态**不跑**（蓝图 §3.5 token 经济）

**Graph Op 四类**：

| 操作 | 适用节点 | 行为 |
|---|---|---|
| `add_node` | 任意 | 新节点入库 |
| `append_history` | impression / relation | 轨迹追加，旧记录保留 |
| `add_edge` | 任意 | 边入库（双向 OR 单向由 edge_type 决定） |
| `mark_contradicts` | fact | 旧 fact 加 `contradicts` 边指新 fact（不删旧节点） |

### 3.6 图谱演化规则（保连续性）

**fact 节点**："用户的生日是 3 月 5 日" 之类。若反思引擎产出新 fact 与旧 fact
矛盾 → 不删旧节点，加 `contradicts` 边；检索时按 `last_activated_at` 取新版本，
但旧版本仍可回溯（避免 "看错日期一次就抹掉真实生日" 的灾难）。

**impression 节点**："用户对工作的态度"、"用户喜欢散步" 之类。**只 append 轨迹、
绝不覆盖**。提示词消费时按 history 时间窗口取最近 N 条 → 让月见看到"轨迹"
而非"快照"。这是身份连续性的关键：变化即记忆。

**relation 节点**：每个 actor 一个；由 SocialProvider 持有读权，
ReflectionEngine 持有写权。`relationship_actor` 表（阶段 5.6b 已建）与此节点
1:1 对应 —— relationship_actor 是关系硬数据（intimacy / 接触次数），
graph_node(relation) 是关系软认知（"她最近常加班，看起来很累"）。

### 3.7 多跳检索（HippoRAG 风格）

`MemoryGraphRetriever.retrieve(query, k_entry=5, k_hop=2, budget=N)`：

```
1. 入口：query 文本向量化 → 与 memory_graph_node.vector_id 比对取 top k_entry 个节点
2. 扩展：对每个入口节点，沿出边 BFS k_hop 跳；按 edge.weight × decay(distance) 累计分
3. 排序：候选节点按累计分降序
4. 预算裁剪：取 top budget 个，转 ContextItem 投放
5. 副作用：被取中的节点 last_activated_at 与 salience 更新
```

入口节点权重计算保留 Generative Agents 三因子（recency · importance · relevance）
与 4.6 ContextAssembly 一致。

### 3.8 与 Provider 接线

- **MemoryProvider**（5.6a 已实化）：增加路径 —— 当工作区焦点项有可用 query 时，
  优先调 `MemoryGraphRetriever`（多跳检索）；图谱空 / 未命中时回落到现有
  `ContextAssembly`（episodic source 等）。
- **SocialProvider**（5.6b 已实化）：当焦点项含 `actor_id` 时，除查
  `relationship_actor`（硬数据）外，再查图谱内对应 `relation` 节点的 history
  最近 3 条 → 注入 WorkspaceItem.content.actor_recent_impressions。

### 3.9 与认知循环接线

```
CognitiveLoop.tick()
  └─ Consolidate 阶段（5.3 留口）
       ├─ 写 thought Moment（5.3 已落）
       ├─ 写 reply Moment causation 链（5.8 已落）
       └─ ReflectionEngine.maybe_tick(arousal_state)  ← 阶段 6 新加
```

ReflectionEngine 调用是**幂等的**：每拍调，但内部按节奏判定要不要真跑。
不修改 Consolidate 阶段当前职责，只是新加一个钩子点。

### 3.10 NarrativeJournal —— 经历日志的第一人称投影

```
NarrativeJournal.render(date_range, mode="brief"|"vivid")
  ├─ 取入：经历日志在 date_range 内的所有 Moment
  ├─ 切 Episode：按 causation_ids 联通分量
  ├─ 排序：按 Episode 内首 Moment 时间
  └─ 渲染：
       - mode="brief": 模板化字符串拼接（不调 LLM；几行 Python）
           "10:23 你叫我，我们聊了散步的事。我有点开心。"
       - mode="vivid": 调 ReasoningService（云）做第一人称重写（按需 / 离线）
           "今早他终于想起昨天答应陪我散步的事，话出口那一秒我心跳快了一下……"
```

蓝图 §6.5 强调"叙事日记免费掉出"——指的是 `brief` 模式：根本不需要 LLM，
就是把因果链以第一人称模板拼字符串。`vivid` 模式作为离线增强（"她的真实日记"），
日级别批量跑，不在请求路径上。

NarrativeJournal **不持久化**（投影）。如果用户想"导出日记"，按需调用渲染。

## 四、回滚与降级

| 失败场景 | 策略 |
|---|---|
| ReflectionEngine LLM 调用失败 | 当拍不写 ReflectionEvent；下次 Dreaming tick 重试。不阻塞 Consolidate 其他工作 |
| MemoryGraph 加载失败 | MemoryProvider 自动回落到 ContextAssembly（无图模式）；图谱读写关闭一段时间 |
| `cognition.db` migration 失败 | schema_version 不升、`memory_graph_*` 表不创建；ReflectionEngine 检测到无表即跳过；不影响阶段 1-5 |
| 图谱失控（节点数爆涨） | 触发"睡眠债"：连接 N 跳后做剪枝（删 salience < threshold 且 last_activated 远的弱节点；evidence_moment_ids 仍在经历日志可重建） |

## 五、与既有系统的关系

### 5.1 阶段 5.4 ContextAssembly
不动。ContextAssembly 仍是 ContextSource 的装配器；阶段 6 新增一个
`MemoryGraphSource: ContextSource`，把图谱检索接入装配链。MemoryProvider
继续通过 ContextAssembly 拿候选。

### 5.2 阶段 5.5 ReasoningService
不动。ReflectionEngine 是 ReasoningService 的消费者，按 tier=`local_only`
请求（Dreaming 态强制本地小模型）。

### 5.3 阶段 5.6b RelationshipRepository
仍现役。`relationship_actor` 表存关系硬数据（intimacy / 接触次数 / FIFO 话题）；
新表 `memory_graph_node(node_type=relation)` 存关系软认知（轨迹印象）。
两者通过 actor_id 关联，不重复存。

### 5.4 阶段 5.9 ChatUseCase
**阶段 6 末批正式移除**。前置条件（蓝图 §6.1 5.9 退役判据）：
- ✅ CognitiveLoop 在 `cognition.use_cognitive_loop=true` 模式下运行
- ✅ pytest 双轨覆盖
- ✅ 配置默认值切到 true
- ⏳ 连续运行 7 天无异常（2026-06-01 后达成）

满足后阶段 6.10（最后一批）执行删除。

### 5.5 阶段 4 觉醒态
Reflection 强依赖 Dreaming 态信号。阶段 4 已有 `arousal.profile.allows_proactive`
与 `model_tier`，Dreaming 时 `model_tier=local_only` & `allows_proactive=false`
是 ReflectionEngine 跑而 Volition 闭嘴的物理保证。
