# 阶段 7 设计：自主输出通路（Autonomous Action Output）

> 起草日期：2026-05-28
> 蓝图对应：§3.3 Act / §4.7 Volition 与行动意愿 / §6.5 ActionStream（thought/reply/emotion 帧）
> 状态：**已落地**。CognitiveLoop Act → ActionCommand 是回复唯一出口。
> 解锁：阶段 6.9（ChatUseCase 移除）已完成。

> 阶段 5 把"她怎么想"做出来了（工作区 + 循环 + Volition + 仲裁），阶段 6 把
> "她怎么理解生活"做出来了（反思 + 图谱 + 日记）。但有一个致命接缝：
> **她想说的话从来没出过口** —— CognitiveLoop 的 Act 阶段是 no-op，
> 算出的 reply intent 存进 `last_arbitration` 就没了。阶段 7 接通 Act，
> 让循环的决定真正作用到用户。

---

## 一、现状缺口（为什么必须做这个阶段）

### 1.1 Act 是 no-op
`cognition/loop.py` 的九阶段里，Intend 算出 willingness + 仲裁出 accepted intents，
但 Act 阶段：
```python
with span("act"):
    pass   # ← 什么都不做
```
reply intent 走到这里就消失。

### 1.2 出站通道不支持主动推送
`ipc_server/outbound/` 只有 `send_state_sync` / `send_log` 两个 Python→kernel
单向推送。**没有"认知核主动发回复"的通道**。

### 1.3 历史缺口：同步 RPC 与循环范式不兼容
```
kernel: PerceptionAppService.processIngress
  → attentionMgr.ingest  →  AIProxy → IPC PERCEPTION_MESSAGE
  → Python 投递确认
```
旧版把回复文本塞进 IPC response 返回，是**请求-响应**；CognitiveLoop 是后台循环，
决定开口时没有 pending 请求可搭车。因此当前架构规定：
`PERCEPTION_MESSAGE` RPC 只表示投递成功，真实回复必须由 ActionCommand 异步 PUSH。

### 1.4 两块早已存在但断开的部件
- `schemas/models/ActionCommand.schema.json`（P.2 建）：`action_type` ∈
  reply/recall/react/noop + `target.scene_id` + `payload` + `emotion_state`。
  **至今无人 emit / consume** —— 为这一刻而设计。
- kernel `ChannelReplyEvent` + 适配器回传链路：作为 ActionCommand 下游继续复用。

阶段 7 = 用一条异步推送通道把这两块接起来。

---

## 二、设计原则

1. **复用既有契约**：用现成 `ActionCommand` 作 wire format，不新造 reply DTO。
2. **复用既有下游**：kernel 收到主动 ActionCommand → 映射成 `ChannelReplyEvent`
   → 现有适配器回传链路原样跑。不重写回传。
3. **异步推送，不是 RPC**：新增 `action_command` IPC 类型（Python→kernel 单向，
   与 `state_sync`/`log` 同形态）。内核不回 `success_response`，避免认知循环阻塞与响应帧回环。
4. **沉默是结构性默认**（蓝图 §4.7）：Volition 没产 reply intent / 产了 noop →
   什么都不推。"想说但忍住"也被记进经历日志（已有），只是不出口。
5. **单一路径**：回复不走 `PERCEPTION_MESSAGE` response，不保留 ChatUseCase 旧轨。

---

## 三、目标架构

```
CognitiveLoop.tick()  [Python 后台循环]
  └─ Intend：Volition + 仲裁 → accepted reply intent
  └─ Act（阶段 7 通电）：
       reply intent → 构造 ActionCommand(action_type=reply, target.scene_id,
                                          payload.text, emotion_state)
       → KernelEventOutboundAdapter.send_action_command(cmd)
       → kernel_bridge.send_message({type: 'action_command', payload: cmd})
                          │ ZMQ 异步推送
        ┌─────────────────▼─────────────────────────────────┐
        │ kernel: IPCServer 收 action_command                │
        │   → CognitionActionInboundHandler                  │
        │   → 校验 target.scene_id → ChannelReplyEvent         │
        │   → EventBus.publish（复用现有下游）                 │
        │   → 适配器捕获 → 回平台                              │
        └────────────────────────────────────────────────────┘
```

**关键**：Act 之后的一切（EventBus → 适配器 → 平台）复用既有下游。
阶段 7 新增并收口的是 "Python Act 产 ActionCommand" + "kernel 收 action_command 转
ChannelReplyEvent" 这一段。

## 四、模型与接口

### 4.1 IPCMessageType 加 `action_command`
`schemas/enums/IPCMessageType.schema.json` enum 加 `"action_command"`
（Python → kernel 单向推送）。codegen 双端。

### 4.2 ActionCommand（复用，不改）
已有 schema 足够。Act 阶段填：
- `trace_id`：取 reply intent 的 causation 源 perception 的 trace_id（关联回原请求）
- `action_type`：`reply`（payload 非空）/ `noop`（沉默，一般不推）
- `target.scene_id`：reply intent payload 里的 scene_id
- `payload.text`：reply intent payload 里的 text
- `emotion_state`：当前 EmotionSystem 快照（可选）

### 4.3 Python 出站：`send_action_command`
`KernelEventPort` + `KernelEventOutboundAdapter` 加：
```python
async def send_action_command(self, command: dict) -> None:
    await self.kernel_bridge.send_message({
        "type": IPCMessageType.ACTION_COMMAND,
        "trace_id": command.get("trace_id", ""),
        "payload": command,
    })
```

### 4.4 CognitiveLoop Act 阶段
```python
with span("act") as s_act:
    if self._action_sink is not None and self._last_arbitration:
        for intent in self._last_arbitration.accepted:
            if intent.type.value == "reply":
                cmd = self._intent_to_action_command(intent)
                await self._action_sink(cmd)   # 注入式：默认 None（测试 / 无出站时不推）
        s_act.set_attribute("actions_emitted", ...)
```
`_action_sink` 是注入的 `Callable[[dict], Awaitable]`（一般 = outbound_adapter.
send_action_command）。注入式 → 单测可用 spy，无 kernel 时 CognitiveLoop 仍能跑。

### 4.5 Kernel 入站：`action_command` handler
`cognition-manager` / IPCServer 注册 `action_command` handler：
```ts
onActionCommand(msg: ActionCommand) {
  if (msg.action_type === 'reply' && msg.payload.text?.trim()) {
    EventBus.instance.publish(new ChannelReplyEvent(
      {
        traceId: msg.trace_id,
        text: msg.payload.text,
        emotionState: msg.emotion_state,
        targetChannel: msg.target.scene_id,
      },
      createTraceContext({ trace_id: msg.trace_id }),
    ));
  }
  // noop / 空 text → 沉默，什么都不做
}
```
`target.scene_id` 是强制路由字段。缺失时内核丢弃该 reply，不回落 `traceId`
前缀推断；扩展只消费属于自己命名空间的 `targetChannel`。

## 五、顺带修：空 thought Moment 噪声

`_write_thought_moment` 当前每拍（含无广播）写 `{empty: True}` thought Moment ——
Awake 态 360 条/小时垃圾，污染经历日志 / 反思 / 叙事日记 / 多跳检索。

**修法**：
- 无广播（空拍）→ **不写 Moment**，改记 `gauge("cognition.tick_alive", 1)` 表 liveness
- 有广播 → 照常写 thought Moment（真实念头才入经历之流）

理由：liveness 是遥测的事（蓝图 §6 可观测性），不该塞进经历之流（蓝图 §4.1
"她活过的每一刻" —— 空拍不是"一刻经历"）。叙事日记 / 反思从此不必趟空念头。

## 七、风险与决策点

### 7.1 回复唯一出口
回复唯一出口是 CognitiveLoop 的 Act → ActionCommand。旧 ChatUseCase 请求-响应
路径不再保留为兜底，也不允许产生 `reply_content`。若出现双回复，优先检查是否有
旧 RPC reply 路径、重复 IPC listener 或重复 ActionCommand handler 残留。

### 7.2 trace_id 关联
CognitiveLoop 是后台循环，reply intent 源自 perception Moment 的 causation 链。
需把原 perception 的 trace_id 一路带到 ActionCommand，否则回复与原请求在
telemetry / 适配器侧关联不上。PerceptionEventQueue 已带 trace_id（5.6c），
确保它流进 WorkspaceItem → intent → ActionCommand。

### 7.3 时序：外部输入立即驱动下一拍
循环平时按 arousal tick 自转；外部 perception 入站时不等待自然 tick。
`container._on_perception` 在写队列后执行 `wake_now("perception")`，再调用
`CognitiveLoop.notify_external_input()` 打断长睡眠，让 dormant/ambient 状态下的用户消息
也能立即进入 Sense → Deliberate → Act。

### 7.4 ActionStream thought/emotion 帧
蓝图 §6.5 ActionStream 是 thought/reply/emotion **三类帧**。阶段 7 先只通 reply
帧（用户可见回复）。thought 帧（"思考中…"动效）/ emotion 帧（Live2D 表情）留
后续 —— 它们走渲染层，与 reply 出口正交，不阻塞本阶段。

## 九、关键缺口与架构决定（2026-05-29 调查）

### 9.1 缺口：Deliberate 也是 no-op，循环不生成回复
7.1 落地后排查发现 `cognition/loop.py` 的 Deliberate 阶段同样是 `pass`：
```python
# ── Deliberate（5.5 接 ReasoningService）──
with span("deliberate"):
    pass   # ← no-op
```
而 `_build_intents` 的 perception→reply intent 直接把 `broadcast.content["text"]`
（**用户输入原文**）当回复文本。即 **CognitiveLoop 现在的"回复"是回显用户输入，
没有任何 LLM 生成**。真正"组装上下文 + 人设 + 调 LLM + 边界校验"的能力全在
ChatUseCase（旧轨）里，循环里一步没有。

后果：若先做 kernel 切流（切到循环当唯一回复源），月见会把用户的话
原样回显 —— 严重回归。故必须**先补 Deliberate 生成（新 7.2），再切 kernel（7.4）**。

### 9.2 架构决定：CognitiveLoop **取代** ChatUseCase（不是分工共存）
曾考虑两条路：
- **取代**：Deliberate 接 ReasoningService 自己生成，ChatUseCase 在 6.9 删除
- **分工**：CognitiveLoop 决策 + ChatUseCase 执行生成，二者共存

**定为取代**，理由：
1. **蓝图已钉死**：§3.3 九阶段把 Deliberate 放在循环内，§3.6 Reasoning Service
   是"循环在 Deliberate 调的推理后端"。生成本就该在循环里发生；ChatUseCase 是
   循环出现前的旧请求-响应模型，本就是过渡脚手架。
2. **分工会造出"两个心智"**：ChatUseCase 是请求-响应，结构上**只能回应 incoming
   perception**；但循环的灵魂是自主/主动（Drive 发起"想找人聊"、Memory 发起
   "想起一件事"）—— 这些无 incoming request，ChatUseCase 生成不了。让它当生成
   执行器，一遇主动开口就哑火。
3. **避免双套 LLM 调用**：两处组装上下文 + 注入人设 + 边界校验 → 重复 + 漂移。
   月见会有"两个嘴、两套说话逻辑"。
4. **取代成本可控**：ChatUseCase 的零件（EmotionSystem / ContextAssembly /
   PersonaInjector / LLMEngine / validate_boundary）**全是现成独立服务**，
   ChatUseCase 只是编排。循环九阶段是更完整的编排（Appraise≈情绪、Recall≈记忆
   已在循环内）。Deliberate 要补的是**接线现成服务**，不是新写生成逻辑。

→ 月见回到"一个心智、一条循环、一个嘴"。6.9 删 ChatUseCase 是这个决定的收尾。
