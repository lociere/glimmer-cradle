## 一、目标与范围

阶段 5 把月见的"心智运转"从**当前流水线**（IPC 入站 → ChatUseCase → LLM → 输出）
换成**蓝图 §3.1 描述的活物**：并行专家模块 → 全局工作区竞争 → 广播 → 思考 →
行动 → 固化。

**做：**
- **全局工作区**（Global Workspace，脊柱③）：容量有限的"现在"，承载意识焦点。
- **认知循环九阶段**（脊柱②）：感知→评价→联想→竞争→广播→思考→决意→行动→固化，
  **持续运转，无输入也跑**。
- **专家模块**（Providers）：`Perception` / `Affect` / `Memory` / `Drive` / `Social`
  并行常驻，写入候选 → 工作区竞争。
- **上下文装配器**（Context Assembly，脊柱④）：统一"记忆 + 知识 + 关系"为可激活的
  上下文源；按觉醒态决定预算与深度。
- **Volition 与行动意愿**：连续意愿值（非二元 should_reply）+ 仲裁；
  挂钩阶段 4 的 `arousal.profile.allows_proactive` 与 token 预算。
- **Reasoning Service**：抽象推理后端 tier 选择（local_only / cloud_allowed），
  挂钩阶段 4 的 `arousal.profile.model_tier`。
- **观测**：每拍循环产出 trace span 树（感知/评价/.../固化），可在 `scripts/telemetry.py
  spans tree` 看到；每拍写一个 `Moment(kind=thought)` 或 reflection。

**不做（留待阶段 6）：**
- **记忆图谱 / 反思引擎**：阶段 6（A-MEM 风格的记忆图 + Self-Reflection）。
- **叙事日记**：阶段 6。
- **Dreaming 态的真实记忆固化逻辑**：阶段 4 留作沿用 v4.5 `consolidate_memory`；
  阶段 6 重写。
- **专家模块作为扩展点**：按蓝图 §3.2 "扩展不能注册新专家"，专家模块是认知核独占；
  扩展可贡献**上下文源**（见 §五.4），不能加新专家。

## 二、现状盘点

### 2.1 当前认知路径（流水线，非循环）

```
Kernel 内核 IPC 入站
  ↓ perception_message
ChatUseCase._execute
  ├ multimodal_router.route        视觉路由
  ├ emotion_system.update          情绪更新
  ├ _build_memory_digest           记忆/知识/偏好拼装（全部一次性、固定模板）
  ├ persona_injector.build_prompt  人设拼接
  ├ ContextAssembly                按激活规则装配上下文
  ├ llm_engine.generate            单次 LLM
  ├ validate_boundary              人设红线
  └ session.append_message + STM   会话沉淠
```

**特征**：
- 一次输入触发一次推理，**无输入不跑**。
- 记忆/知识检索是固定查询（向量 top_k + bigram），无"激活规则"统一表达。
- 情绪、记忆、动机、关系、社交 —— 各自独立子系统，无统一竞争。
- ActiveThought 用例（生命心跳触发的主动思维）独立于 ChatUseCase，不复用同套机制。

## 三、核心决策

### 3.1 一拍即一个 trace span 树

每次认知循环（一"拍"）开一个根 span `cognitive_cycle`，九阶段各为子 span：
`sense / appraise / recall / compete / broadcast / deliberate / intend / act /
consolidate`。已有遥测三支柱（阶段 3）天然承载 —— 不需要新基础设施，每拍的
trace tree 可在 `scripts/telemetry.py spans tree <trace_id>` 看到。

### 3.2 一拍即一个 Moment

每拍**至少**产生一个 `Moment(kind=thought)`（"她想了一下"），即使没有 reply。
若 Volition 决定不发声（沉默），写 `Moment(kind=silence)` 已有的机制保留。
**causation_ids 真正填充**：reply 的 causation_ids 指向触发它的 perception 与
中间 thought。这是阶段 1 留下的字段终于"通电"。

### 3.3 工作区是容量有限的纯数据结构

`GlobalWorkspace` 是一个 **小**（默认 7±2 个 item，对齐心理学 working memory 容量）
的优先队列。每个 item 携带：
- `source`：哪个专家投放（perception / affect / memory / drive / social）
- `content`：意义内容（dict）
- `salience`：显著度（0~1，竞争分数）
- `decay_at`：何时衰减出场

新 item 进入 → 若超容量按 salience 淘汰最低。
"广播"就是 `select_top_salient()` 取一份给 Deliberation。

### 3.4 专家模块是协程而非线程

并行 ≠ 多线程。Python `asyncio` 已经支持"并发 await"：每个专家是一个 async
方法，每拍由 `CognitiveLoop` 用 `asyncio.gather(*providers)` 并发推进。在 IO bound
场景（向量检索、记忆查询）这是更轻量的并行。CPU bound 重活（embedding、视觉
描述）该走 `asyncio.to_thread`（已是现有规范）。

### 3.5 循环驱动器与觉醒态联动

`CognitiveLoop.tick_interval_ms` 直接读 `arousal.profile.frequency_hint_ms`。
觉醒态变 → tick 间隔变。这把阶段 4 的挂钩 ② 从"仅 TS LifeClock 心跳"扩展为
"认知核内部循环也按它走"。

外部感知不等待自然 tick：Cognition 入站边界在 perception 入队后调用
`arousal_manager.wake_now("perception")`，再调用 `CognitiveLoop.notify_external_input()`。
循环内部用 `asyncio.Event` 打断当前睡眠，避免 dormant/ambient 的长间隔让用户消息滞留。

### 3.6 CognitiveLoop 是唯一对话编排

月见现在不再保留 ChatUseCase 双轨：perception 入站只写 `PerceptionEventQueue`，
回复由 CognitiveLoop 的 Appraise → Deliberate → Intend → Act 完整产生。
旧请求-响应 ChatUseCase 属于过渡脚手架，不再作为兜底、兼容路径或新增能力入口。

### 3.7 Reasoning Service 抽象推理后端

新增 `selrena/reasoning/` 抽象层：
- `ReasoningService.request(prompt, *, tier: ModelTier, ...) -> response`
- 实现：`CloudReasoning`（包装现 LLMEngine）+ `LocalReasoning`（本地小模型，
  阶段 5 留 stub，阶段 6 接 native FFI）
- `CognitiveLoop` 根据 `arousal.profile.model_tier` 选 tier 调 ReasoningService

挂钩阶段 4 的 `model_tier` 字段，使其真正影响推理路径。

### 3.8 Volition 是连续意愿值，不是二元闸

意愿计算（蓝图 §4.7）：

```
willingness = (
    address_weight  * address_score(perception.address_mode) +
    emotion_weight  * emotion_intensity +
    intimacy_weight * relationship_intimacy +
    drive_weight    * social_drive_budget +
    silence_weight  * f(seconds_since_last_reply) +
    persona_weight  * persona.extraversion
)
threshold = arousal_state_to_threshold(arousal.state)  # Ambient 时阈值高 → 易沉默
proactive_ok = arousal.profile.allows_proactive
```

`willingness > threshold` 且 `proactive_ok` → 触发 reply 意图。
意图进入仲裁器（同拍多个意图合并 / 延迟 / 抑制）→ ActionStream。

### 3.9 Drive 模型 —— 三种内在动机的累积/衰减/触发（5.6b 前置）

蓝图 §3.2 说 Drive 是"沉默时月见自己开始想事情的来源"，列了三种动机：好奇 / 陪伴欲 / 休息欲。但没给具体的数学。本节定数据/算法/参数 —— 是 DriveProvider 实化的设计依据。

**三种 drive，每种是一个 [0,1] 的连续值**，由 DriveProvider 内部维护：

| Drive | 上升来源 | 上升速率（每秒） | 满足条件（→ 立即衰减） | 满足后衰减量 |
|---|---|---|---|---|
| **curiosity 好奇** | 无新感知的时长；高觉醒态（awake）放大 | base `0.005/s` × `arousal_state_boost`（awake=1.5, ambient=1.0, dreaming=0.3, dormant=0）| 接收 perception_message（任何感知入站） / 触发 thought 意图中含问句 | `−0.4` |
| **companionship 陪伴欲** | 距离上次"被互动"的秒数 | base `0.003/s` × `arousal_state_boost`（同上）| perception_message 入站 / 月见自身 reply | `−0.5` |
| **rest 休息欲** | awake 状态持续时长 × 情绪强度 | base `0.002/s`；情绪强度 >0.7 时 ×2 | 进入 ambient / dormant / dreaming 任一态 | `−0.7`（充分休息） |

**投放策略**（每拍一次 propose）：

1. 先 update：`level += rise_per_second × tick_seconds × boost`
2. 检查满足事件：若本拍有对应满足条件，`level -= satisfaction_amount`，clip 到 0
3. 选当前 `level` 最高的 drive；若 `level > threshold`（默认 0.6）则投放 1 个 WorkspaceItem
4. 同拍最多投 1 个 drive 项（保持工作区简洁；其他 drive 留下次累积过阈再说）

```python
WorkspaceItem(source="drive", content={
  "drive": "curiosity" | "companionship" | "rest",
  "level": 0.7,
  "rise_since_satisfied_s": 134.2,
}, salience=level)
```

**持久化策略**：drive 状态**不持久化**（蓝图 §3.4 觉醒态保 boot_id 不保 drive）—— 每次启动从默认值起。理由：drive 是"此刻的状态"，跨次启动不连续；persistence 留作 5.6b 实化后视需要再加。

**阈值 / 速率均作 `DriveConfig` 字段可调**，与 ArousalConfig 同位置（`selrena/cognition/drive_config.py`）。

### 3.10 Social 模型 —— 关系记录的数据/算法（5.6b 前置）

蓝图 §3.2 说 SocialProvider 维护"对每个人的关系模型 —— 相处史、亲密度、内部梗"。本节定数据存储与亲密度算法 —— 是 SocialProvider 实化的设计依据。

**数据持久化在认知核独占的 cognition.db**（不跨语言，Python-only），新增表：

```sql
CREATE TABLE relationship_actor (
  actor_id TEXT PRIMARY KEY,         -- 跨 scene 唯一（如 napcat:user:U_123）
  display_name TEXT NOT NULL DEFAULT '',
  intimacy REAL NOT NULL DEFAULT 0.0,  -- [0, 1]
  first_seen_at TEXT NOT NULL,       -- UTC ISO ms
  last_seen_at TEXT NOT NULL,
  total_direct_interactions INTEGER NOT NULL DEFAULT 0,
  total_ambient_observations INTEGER NOT NULL DEFAULT 0,
  recent_topics TEXT NOT NULL DEFAULT '[]',  -- JSON list[str]（FIFO，cap 8）
  inside_jokes TEXT NOT NULL DEFAULT '[]',   -- JSON list[str]（手动 / 反思引擎产出，5.6b 留空）
  metadata TEXT NOT NULL DEFAULT '{}'        -- JSON 任意扩展
);
CREATE INDEX idx_relationship_last_seen ON relationship_actor(last_seen_at);
```

**亲密度算法**：

```
on perception_message(address_mode="direct"):    intimacy += 0.05  # 显式 @
on perception_message(address_mode="ambient"):   intimacy += 0.01  # 焦点窗口顺带
on reply to this actor:                          intimacy += 0.02
weekly natural decay (无任何接触一周):           intimacy -= 0.01
intimacy ∈ [0, 1]   clip 到边界
```

衰减由 cognition.db 启动时基于 `last_seen_at` 一次性折算，不需要后台 cron（懒计算）。

**投放策略**：

1. 当工作区焦点为 perception 且其 content 含 `actor_id` 时触发
2. SocialProvider 查 `relationship_actor` 表（无则新建 row）；同时累加 intimacy / interaction 计数 / 更新 `last_seen_at`
3. 投放 WorkspaceItem(source=social)：

```python
WorkspaceItem(source="social", content={
  "actor_id": "...",
  "display_name": "...",
  "intimacy": 0.42,
  "total_direct_interactions": 137,
  "recent_topics": ["猫", "天气", "工作"],
}, salience=min(1.0, 0.3 + intimacy * 0.5))
# 亲密度 0 → salience 0.3；亲密度 1 → salience 0.8
```

**`recent_topics`**：暂用 perception text 的前 16 字符存（实际 topic 提取留阶段 6 反思引擎）。FIFO cap 8。

**新仓库**：`selrena/persistence/relationship_repo.py`（CRUD + intimacy 折算函数）。

**与既有持久化的关系**：复用 `CognitionDatabase` 单库；新表 + DAO，不破现有 schema。版本化：`database.py` 加 `_migration_v2` 函数建该表（向前兼容 v1）。

## 六、与既有系统的关系

### 6.1 ChatUseCase 已退出对话主路径

ChatUseCase 是旧请求-响应架构的编排器，不再参与月见的运行时对话链路。
当前主路径是：

```
Kernel perception_message
  → Cognition container._on_perception
  → PerceptionEventQueue + wake_now + notify_external_input
  → CognitiveLoop 九阶段
  → ActionCommand
  → Kernel 路由到目标 scene
```

若仓库中仍出现新的 ChatUseCase 依赖、双回复兜底或同步 reply_content 路径，应视为旧架构残留并删除。

### 6.2 EmotionSystem / LongTermMemory / KnowledgeBase

不动。专家模块 `AffectProvider` / `MemoryProvider` 是这些子系统的**调用者**，
不是替代者。

### 6.3 觉醒态机

觉醒态机由认知核持有。`ArousalManager.wake_now` 只在入站边界调用，
`PerceptionProvider` 不承担唤醒副作用，保持专家模块的纯投放职责。

### 6.4 ActiveThoughtUseCase（生命心跳）

生命心跳不是对话编排器。CognitiveLoop 自驱动循环，LifeClock 只根据
`arousal.profile.frequency_hint_ms` 调整节律，并用于跨进程状态同步。
外部感知通过 `notify_external_input()` 打断睡眠，心跳本身不算"被叫醒"。

## 八、风险与缓解

| 风险 | 缓解 |
|---|---|
| 旧请求-响应残留造成双回复或沉默 | 禁止新增 ChatUseCase 路径；perception 入站只写队列并触发 Loop；测试覆盖队列→Deliberate→ActionCommand 主链路 |
| 专家模块并发竞争产 race | 工作区操作走单协程 lock；providers 只产候选不直写 |
| 每拍开 span 量大压垮 traces sink | 默认 tick 间隔走 arousal frequency_hint_ms（dormant 60s / awake 10s 等），单机日量可控；必要时 spans 按采样率落 |
| ContextAssembly 设计假大空、实际仍是固定模板 | 5.4 先做 IContextSource 抽象 + 3 个内置 source；MemoryProvider 必须通过它检索，不允许绕过 |
| Volition 意愿公式参数难调 | 5.7 各权重作配置项；默认值偏保守（沉默优先）；落地后按经历回放调权 |
| Reasoning Service 切档导致响应质量波动 | 5.5 default tier 锁 cloud_allowed；按 arousal.model_tier 切档作 opt-in 配置项；后期再默认开 |
| 内存工作区与 Moment 流的去重 | 工作区是"当前"，Moment 是"过去"；广播写 Moment 一次，工作区淘汰不写 Moment |
