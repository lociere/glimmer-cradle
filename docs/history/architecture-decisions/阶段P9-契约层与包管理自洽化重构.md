## 一、设计原则

1. **4 语言 polyglot，不是双生态**（蓝图 §3.6 早已规划 `core/native` C++ 后端；
   渲染层 §6.1 留 Avatar Shell 进程边界，未来可能换 Unity 实现）
2. **Schema 是唯一锚点**（铁律 1）—— TS / Python / C++ / C# 共享同一份
   `protocol/src/schemas/`；codegen emitter 按生态各一份
3. **每个生态用自己的最佳工具**（业内主流，对应 VS Code / Tauri / Deno / PyTorch
   / uv 等同形态项目的做法）—— 不引入 Bazel / Nx 这种 polyglot 总编排
4. **per-package 自治**：每个包声明它如何从 schema 派生自己的类型；不存在跨包
   的中央协调脚本
5. **commit 生成产物 + CI drift 守门**（Buf / OpenAPI / Apollo 业内主流）——
   保 IDE 跳转 / PR diff / 冷克隆体验
6. **进程边界即协议边界**：跨进程的通信全部经 schema-derived 协议；Avatar Shell
   从 Electron 换 Unity 等价于换实现，协议不变

## 二、月见的语言版图（终态）

```
┌────────────────────────────────────────────────────────────────┐
│  Layer 6.B  Avatar Shell（Live2D 渲染）   C# / Unity（可选）    │
│  Layer 6.A  Desktop Shell（Electron）     TypeScript            │
│  Layer 7    Extensions（适配器 / 插件）   TypeScript            │
│  Layer 1-5  Kernel（中枢）                 TypeScript            │
│  Layer X    Cognition（AI 认知核）         Python                │
│  Layer X    Native（推理 / 音频 / 向量）    C++                  │
│  Layer 0    Protocol（契约 + 运行时 helper） 语言无关 JSON Schema │
└────────────────────────────────────────────────────────────────┘
```

### 2.1 4 个跨语言触点

| 触点 | 连接 | 协议 / 工具 |
|---|---|---|
| **契约层** | 所有生态 | `protocol/src/schemas/*.schema.json` —— JSON Schema 单一事实源 |
| **TS ↔ Python IPC** | Kernel ↔ Cognition | ZMQ DEALER/ROUTER（已有，`KernelMessageEnvelope` 信封） |
| **Kernel ↔ Avatar Shell** | TS ↔ Electron-renderer 或 TS ↔ Unity | WebSocket port=8082（已在 `AvatarShellRuntimeConfig` 中定义） |
| **Python ↔ C++ FFI** | Cognition ↔ Native | pybind11（业内标准；SciPy / PyTorch 等都在用） |

**关键洞察**：所有触点都**只交换 schema-derived 数据结构**。没有任何"内部对象 / 框架对象"跨进程。这让 4 个生态独立演化，只共享一个契约层。

### 2.2 进程拓扑

```
┌─ Kernel (TS, Electron 主进程)
│   ├── ZMQ ──── Cognition (Python 子进程)
│   │             └── pybind11 ──── Native (C++ .pyd/.so 内嵌)
│   ├── WS  ──── Avatar Shell (Electron renderer 或 Unity 独立进程)
│   └── WS  ──── Desktop Shell (Electron renderer)
└── Extensions (Worker 沙箱内运行)
```

## 三、包管理决策

| 域 | 工具 | 状态 | 备注 |
|---|---|---|---|
| **TS workspace** | `pnpm` | ✅ 已用 | 业内最优 monorepo 工具；workspace 协议成熟 |
| **Python** | `uv`（Astral） | ⏳ 待迁（P.9a） | 替代 `pip`：装包速度 10-100×；PEP 621 兼容；workspace 支持；ruff 同源团队 |
| **C++** | `CMake` + `vcpkg` | ⏳ 阶段 3.5 引入 | vcpkg manifest 声明依赖；Windows / VS 集成顺；llama.cpp / Eigen 等都在 vcpkg 内 |
| **Python ↔ C++ 绑定** | `pybind11` + `scikit-build-core` | ⏳ 阶段 3.5 引入 | scikit-build-core 让 C++ 部分能作为 Python wheel 一部分通过 `uv pip install` 编译；终极一键 setup |
| **C# / Unity** | Unity Package Manager（UPM） | ⏳ 启用 Unity Avatar Shell 时 | Unity 自管依赖，不混进 monorepo 包管理体系 |
| **跨生态总编排** | **不引入** | 永久 | pnpm scripts + cargo / cmake / unity-cli 并存即可；4 生态规模未达 Bazel 阈值 |

**为什么不上 Bazel / Nx**：

| 项目 | 语言数 | 总编排工具 |
|---|---|---|
| VS Code | TS + Rust + C++ | 无（pnpm + cargo + 自写脚本） |
| Tauri | Rust + TS | 无（cargo + npm） |
| Deno | Rust + TS + V8 | 无（cargo + 各自工具） |
| uv 自身 | Rust + Python | 无（cargo + maturin） |
| PyTorch | Python + C++ + CUDA | 无（setup.py 调 CMake） |
| **月见** | TS + Python + C++ + C#? | **无** |

业内 polyglot 项目都不强求总编排 —— 总编排（Bazel）的真正价值在 100+ 包 / 100+ 人规模的远程缓存与 hermeticity。月见 solo / 小团队，**不上总编排是更优选择**，不是退而求其次。

## 五、契约层 codegen 自洽化

### 5.1 sync_contracts.py 的解构

**当前**：`scripts/sync_contracts.py` 一个文件做所有事 —— TS 生成 + Python 生成 +
frozen 注入 + 默认值字段处理 + 索引文件生成。中央外挂。

**目标**：拆解到各包内、各语言自己写：

| 现状（`scripts/sync_contracts.py`） | 目标位置 |
|---|---|
| TS 生成（npx json-schema-to-typescript） | `protocol/build/gen-ts.ts`（TS 写 TS）|
| `_drop_optional_for_defaulted_fields` 后处理 | 同上，TS 实现 |
| Python 生成（datamodel-code-generator） | `protocol/build/gen-py.py` |
| `_inject_frozen` 后处理 | 同上 |
| TS / Py index 生成 | 各自 emitter 内自管 |
| C++ 生成（未来） | `protocol/build/gen-cpp.py` |
| C# 生成（未来） | `protocol/build/gen-cs.py` |

### 5.2 触发方式

```
开发者改 schema.json
  ↓
本地 pnpm install / uv pip install / pnpm build 时自动跑 prebuild
  ↓
生成产物刷新（committed 文件覆盖）
  ↓
git diff 看到 schema + generated 同步变化
  ↓
PR 提交；CI 跑 drift check（再 generate 一次，diff 为空则绿灯）
```

不需要"`pnpm sync:contracts`"这种命令；它隐含在 `pnpm build` 的 prebuild 步骤里。

### 5.3 Runtime helper 上提

ajv-validator.ts / yaml-normalizers.ts 当前在 `core/kernel/.../config/`。问题：
未来 Renderer / Extension SDK 也可能需要 ajv 校验，重复造轮子。

**P.9 把它们上提到 `protocol/src/runtime/`**：
```ts
// 任何 TS 包都可以
import { validateAgainst, normalizeYamlNulls } from '@glimmer-cradle/protocol';
```

Python 端做同等抽离 —— 但 Python 用 Pydantic 直接 `model_validate()`，
所以 Python 端无需独立 runtime helper（Pydantic 本身就是）。

## 六、Unity Avatar Shell 集成方案

### 6.1 进程边界已经就位

你的 `system.yaml` 中早有：

```yaml
renderer:
  avatar_shell:
    enabled: true
    port: 8082
    heartbeat_interval_ms: 10000
    heartbeat_timeout_ms: 30000
    emotion_mapping: { ... }
```

—— **Avatar Shell 本来就是进程边界，不是代码边界**。Electron 实现是当前，Unity
实现是未来选项，两者**消费同一份协议**。

### 6.2 协议（C# codegen）

Unity Avatar Shell 只消费一个**子集** schemas（不需要 Persona / Memory / LLM 等）：
- `ActionCommand`（动画 / 表情触发）
- `AvatarEmotionMapping`（情绪映射，已在 RendererRuntimeConfig 内）
- `StateSync`（情绪状态广播）

C# emitter（`protocol/build/gen-cs.py`）只生成这个子集 —— 通过 schema 顶层 `$id`
路径前缀过滤（如 `schemas/models/Action*` + `schemas/config/Renderer*`）。

工具选择：**NJsonSchema 或 quicktype** —— 都能从 JSON Schema 出 C# class。
NJsonSchema 是 .NET 生态首选。

### 6.3 通信

Unity Avatar Shell 与 Kernel 之间：**WebSocket port 8082**（不变）—— Kernel 端
代码已经写好（`AvatarShellRuntimeConfig` + 心跳）。Unity 端实现 WebSocket client +
反序列化 ActionCommand → 触发 Cubism 动画。

### 6.4 资源管理

Cubism Live2D model（.moc3 / .motion3.json）放 Unity Avatar Shell 的 Assets/ 内 ——
不混进 monorepo 其他位置。Unity Avatar Shell 自成一界。

### 6.5 双实现并存

Phase X 期间（Unity 引入但 Electron 不删）：`configs/system/surfaces.yaml`
只声明 Avatar Shell / Desktop Shell 的端口、心跳和开关。Electron Presence 是内置
surface；Unity 作为外部 Avatar Shell 连接 8082，不通过一个 `implementation`
开关让 Kernel 硬切身体。

## 七、Native 集成方案

### 7.1 vcpkg manifest

`core/native/vcpkg.json`：

```json
{
  "name": "selrena-native",
  "dependencies": [
    "llama-cpp",
    "portaudio",
    "whisper-cpp",
    "eigen3"
  ]
}
```

vcpkg 会自动拉源码 / 二进制并配 CMake。Windows / macOS / Linux 跨平台都能跑。

### 7.2 CMake + scikit-build-core 集成

`core/native/CMakeLists.txt` 定义构建。**关键**：`core/cognition/pyproject.toml`
通过 `scikit-build-core` 把 `core/native/bindings/python/` 作为 Python wheel 的
一部分：

```toml
# core/cognition/pyproject.toml
[build-system]
requires = ["scikit-build-core", "pybind11"]
build-backend = "scikit_build_core.build"

[tool.scikit-build]
cmake.source-dir = "../native"
```

→ `uv pip install -e ./core/cognition` 时 scikit-build-core 自动调 CMake 编 native
代码并安装 .pyd / .so 到 cognition 包内。**一键 setup**，开发者不需要单独跑 CMake。

### 7.3 pybind11 模块边界

```cpp
// core/native/bindings/python/reasoning_module.cpp
#include <pybind11/pybind11.h>
PYBIND11_MODULE(_native_reasoning, m) {
    m.def("generate", &generate);
    // ...
}
```

Python 侧：
```python
from cognition_core._native_reasoning import generate  # 透明导入
```

### 7.4 Native 不直连 Kernel

**关键约束**：Native（C++）模块的所有调用都经过 Python wrapper。Kernel 不直接
调 C++。这让：
- Native 模块只需要懂一份"被 Python 调"的 API
- 调度逻辑、错误处理、降级策略全在 Python（蓝图 §3.6 推理服务的分层）
- IPC 边界保持简单（TS ↔ Python ZMQ；不引入 TS ↔ C++ 直连）

## 九、未来扩展可能（不进默认路径）

文档中保留，但不在默认架构中体现：

- **Rust**：如果某一天某个 native 模块（如音频 buffer 并发）发现 Rust 生态明显更
  优，可以单独引入 `core/native/rust/`，与 C++ 共存。**主路径仍是 C++**，因为
  llama.cpp / Cubism SDK / Whisper.cpp / Eigen 都是 C++ 原生。
- **WebAssembly**：如果月见做 Web 版（无 Electron），cognition 或 native 可能要
  WASM 编译。schema 层不需要改，emitter 需扩。
- **iOS / Android**：移动端 Swift / Kotlin 加入。同样 schema 不动，emitter 扩。

| 蓝图阶段 3.5（native） | P.9 预留入口；3.5 启动时 P.9e 接入 |
| 渲染层未来 Unity | P.9 预留入口；Unity 启动时 P.9f 接入 |

P.9 是**最后一块"把外挂内化"的清场动作**。完成后月见的物理架构永久锚定，新模块（无论 native / Unity / 未来语言）都按既定模式接入，不再回头改基础设施。
