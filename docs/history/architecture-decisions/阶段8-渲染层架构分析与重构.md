## 二、已识别的架构层问题(12 处 smell)

> 不是 bug 清单,是**架构层面的不一致 / 抽象漏洞 / 阻碍未来切换的设计**。

### A. 双视觉协议并存(致命)

**现象**:同一个"驱动 月见 的脸"的目的,kernel 输出两种完全不同的契约:

- **去 Unity**:`VisualCommandPayload`(结构化:`{trace_id, command_type, expression, motion, lip_sync, parameter, audio, emotion_state}`,6 个 command_type)
- **去 Electron**:`{type:'chat_reply', emotionState}`(只有 emotion 语义,渲染端自己查表映射成表情/动作/参数)

**后果**:
- 情绪→视觉映射在**至少 3 处**重复定义(见 G)
- Unity 和 Electron 行为可以 drift —— 同一个"happy"在 Unity 和 web-live2d 上可能不一样
- 添加新视觉能力(例如新表情类型、新参数)要改两条路
- 蓝图说月见的脸可以替换 shell,但实际两种 shell 用的协议根本不一样,没法平等替换

**根因**:历史上外部 shell 协议是先设计的,Electron 路径是后加的、走了捷径。

### B. Desktop UI WebSocket 无协议契约

**现象**:[`desktop-ui-controller.ts`](../../../core/kernel/src/core/application/capabilities/desktop-ui/desktop-ui-controller.ts) 与 [`ipc-handlers.ts`](../../../core/renderer/desktop/src/main/ipc-handlers.ts) 之间所有消息(chat_input / chat_reply / audio_play / avatar_shell_status / pong)都是**手写 JSON**:

```ts
// desktop-ui-controller.ts 发出
this.broadcast(JSON.stringify({type:'chat_reply', text, emotionState}));
// ipc-handlers.ts 解析
if (data.type === 'chat_reply') { ... data.text ... data.emotionState ... }
```

无 schema、无验证、字段错或类型变两端不会感知。protocol 铁律 1 虽然技术上只要求 ≥2 语言生态(此处都 TS),但其精神("跨进程契约必须 schema 化")在这里被绕过了。

### C. trace_id 在渲染边界**被丢弃 + 重新捏造**

**现象**:traceId 的生命周期断了三次:
1. Renderer 用户发消息 → main 转给 kernel → kernel 用 `ui_${Date.now()}` 合成 traceId([desktop-ui-controller.ts:127](../../../core/kernel/src/core/application/capabilities/desktop-ui/desktop-ui-controller.ts#L127))
2. 该 traceId 在 cognition → Act → ChannelReplyEvent 中**正确贯通**(阶段 7 已做完)
3. kernel → Electron main 时这个 traceId 被丢:`trace_id: trace_${Date.now()}`([ipc-handlers.ts:31](../../../core/renderer/desktop/src/main/ipc-handlers.ts))

**后果**:Electron renderer 里的 trace_id 与 kernel telemetry 里的对不上,跨进程定位故障必断链。阶段 7 努力贯通的 traceId 在渲染边界白做。

### D. 字符串前缀路由(脆性约定)

**现象**:[`desktop-ui-controller.ts:84`](../../../core/kernel/src/core/application/capabilities/desktop-ui/desktop-ui-controller.ts#L84):

```ts
if (traceId && traceId.startsWith('ui_')) {
  this.broadcast({type:'chat_reply', ...});
}
```

用 `traceId.startsWith('ui_')` 决定一条 ChannelReplyEvent 是不是该转给 Electron。如果未来另一个扩展也用 `ui_` 前缀、或前缀改了、或测试构造的 trace 不带前缀,就静默地不广播。这不是路由,这是约定。

**正确做法**:ChannelReplyEvent payload 加一个 `targetChannel` / `source_channel` 字段,desktop-ui 只匹配 `targetChannel==='desktop-ui'`。

### E. 单例 + setInterval 轮询耦合

**现象**:[`desktop-ui-controller.ts`](../../../core/kernel/src/core/application/capabilities/desktop-ui/desktop-ui-controller.ts) 用 1500ms 轮询读 `AvatarEngineController.instance.isRendererConnected`(L93-102),变化时广播 `avatar_shell_status`。

**后果**:
- 状态变化最坏延迟 1.5 秒
- 两个 controller 通过 singleton + 轮询耦合,不是 EventBus
- 1500ms 是任意常数,没人解释为什么

**正确做法**:AvatarEngineController 在 Unity 连接/断开时 publish 一个 `AvatarShellStatusChangedEvent`,DesktopUIController subscribe 即可。

### F. 两个独立 WS server 难以统一

**现象**:Unity(8082)和 Electron(8083)是**两个独立 WSS**,各自的连接生命周期、心跳策略、消息格式。它们的概念目的是一回事 —— "Avatar 的脸在哪里渲染" —— 却被实现成了两个不相关的端口。

**后果**:每加一个 shell 类型(WebGL?移动端?)就要加一个 controller + 一个端口。蓝图说 "Avatar Shell 可替换",但替换成本很高(端口、协议、controller 都得新写)。

**正确做法**:**一个 Avatar Shell 端口**,客户端连上来时第一帧 declare 自己的 `shellKind`(web-live2d / unity / ...);kernel 用统一的 AvatarFrame 协议下发;客户端按自己的 shellKind 解释执行。

### G. 情绪→视觉映射在 3 处定义

**现象**:同一个 "happy 怎么变成 Live2D 表情" 的逻辑散落:

1. [`RendererRuntimeConfig.AvatarEmotionMapping`](../../../protocol/src/schemas/config/RendererRuntimeConfig.schema.json) 配置项(kernel 持有)
2. [`emotionStore.EMOTION_ALIAS_MAP` + `EMOTION_PARAM_MAP`](../../../core/renderer/desktop/src/renderer/store/emotionStore.ts) (renderer 写死)
3. [`avatar/emotion-map.ts`](../../../core/renderer/desktop/src/renderer/avatar/emotion-map.ts) (renderer 另一份,模型相关)

三处定义,三处可能写不一样。**单一事实源原则被破坏**。

### H. emotion 帧不是 ActionStream 一等公民

**现象**:蓝图 §6.5 ActionStream 是 **thought / reply / emotion 三类帧**。但实际:
- reply 帧是 ActionCommand(认知核 → kernel → ChannelReplyEvent)→ Avatar Shell ✅
- emotion 是塞在 `chat_reply.emotionState` 里,不是独立 ActionStream 帧 ❌
- thought 帧根本没有 ❌

**后果**:emotion 帧无法独立于 reply 发生(月见想"她在沉思"但没说话的时候,Live2D 表情该变也变不了)。详见已知遗留 C。

### I. 流式回复机制存在但不流式

**现象**:[`appStore.appendMessageChunk`](../../../core/renderer/desktop/src/renderer/store/appStore.ts) 写了一套流式拼接逻辑(`isStreaming` / `currentStreamText` / `currentTraceId` 状态机),但 [`ipc-handlers.ts:33`](../../../core/renderer/desktop/src/main/ipc-handlers.ts) 永远发 `is_final: true`,意味着流式从来没真用过。

**后果**:死代码 + 视觉光标(`stream-caret ▊`)永不出现。要么实现真流式(认知核需要支持增量 token 推送),要么删掉假实现。

### J. "Avatar Renderer" vs "Avatar Shell" 命名混乱

**现象**:
- 旧蓝图 §2 曾把 Electron 叫 "Avatar Renderer"(❸)
- 把 Unity 叫 "Avatar Shell"
- kernel 侧:`AvatarEngineController`(管 Unity)、`DesktopUIController`(管 Electron)
- renderer 侧:`avatarShellKind` 三态 `placeholder-puppet | web-live2d | unity-shell`

四个不同地方用了不同名字称呼同一类事物("一个能画月见脸的客户端")。

**正确做法**:统一称谓 "**Avatar Shell**"(蓝图原词),不同实现是 shell 的 kind:
- `web-live2d` Avatar Shell(内置,Electron + pixi-live2d-display)
- `unity` Avatar Shell(可选,外部 Unity)
- `placeholder` Avatar Shell(降级,纯 canvas)
- "Avatar Renderer" / "Desktop UI" / "Avatar Engine" 这些术语全部退出。

### K. DesktopUIController 杂揉两个职责

**现象**:[`DesktopUIController`](../../../core/kernel/src/core/application/capabilities/desktop-ui/desktop-ui-controller.ts) 同时:
- 管 Electron WS server 生命周期
- 收 chat_input → 合成 PerceptionEvent(用户输入通道职责)
- 收 EventBus ChannelReplyEvent → 广播 chat_reply(Avatar 帧分发职责)
- 1500ms 轮询 AvatarEngineController 状态广播 avatar_shell_status(Shell 状态广播职责)

四件事拧在一个 controller 里。

**正确做法**(职责切开):
- `UserInputChannel`(或 `RendererInputChannel`):chat_input → PerceptionEvent → PerceptionAppService。这是一个**用户输入通道**,和 napcat 适配器同位。
- `AvatarFrameRouter`:subscribe ActionStream / emotion events → 按当前 active shell 用其协议下发。
- `AvatarShellRegistry`:管理所有连上来的 shells、各自的 shellKind、谁 active。

### L. Electron Desktop UI:是扩展还是 kernel 一部分?

**现象**:napcat 是扩展(独立子进程,通过 extension API 注入 perception)。Electron 客户端**不是扩展**,而是 kernel built-in capability(DesktopUIController 就在 kernel 里)。但它和 napcat 同样是 "外部进程,通过 WS 给 kernel 发 perception"。

**问题**:为什么不一样?Electron 的 chat_input 难道不该走和 napcat 一样的扩展协议(perception inject)?

**两种正交方向**:
1. **保持现状**:Desktop UI 是 kernel built-in(理由:它是 月见 的官方脸,不是第三方扩展)。但要把 controller 拆干净(见 K)。
2. **改为扩展**:Desktop UI 作为内置扩展(`extensions/internal/desktop-ui/`),走标准 extension perception inject。最一致,但需要把 desktop-ui-controller 重写为扩展形态。

我倾向 1(它是 月见 的官方脸,有特殊地位),但你定。

## 三、重构提案(分子阶段)

### 总目标:统一 Avatar Shell 抽象 + 协议契约化 + trace_id 全程贯通

### 8.1 ✅ 协议统一(契约层,承重墙)

**已落地**:
- 新增 [`AvatarShellDownstreamFrame.schema.json`](../../../protocol/src/schemas/models/AvatarShellDownstreamFrame.schema.json) —— 13 类下行帧(kernel→shell):**reply / emotion / thought**(蓝图 §6.5 三类帧一等公民)+ expression / motion / lip_sync / parameter / idle / audio_play / shell_status / load_scene / unload_scene / ping。所有命令用**抽象 ID**(`expression_id` / `motion_id` / `param_id`),shell-agnostic,各 shell 内部映射到具体资源。`trace_id` 顶级字段,严格保真不重新捏造。
- 新增 [`AvatarShellUpstreamFrame.schema.json`](../../../protocol/src/schemas/models/AvatarShellUpstreamFrame.schema.json) —— 10 类上行帧(shell→kernel):**shell_hello**(握手+能力宣告,含 `capabilities` 数组声明该 shell 能消费哪些下行帧)+ shell_ready / **chat_input**(用户键盘输入)/ **perception**(场景内交互通用通道,8.7+)/ scene_loaded / scene_unloaded(含 narrative 注入 Deliberate prompt 的位)+ heartbeat / pong / animation_complete / error。
- [`ChannelReplyPayload`](../../../protocol/src/schemas/models/ChannelReplyPayload.schema.json) 加 `targetChannel?: string` 字段(additive)—— 替代脆性的 `traceId.startsWith('ui_')` 路由。约定取值 `desktop-ui` / `unity` / `napcat:<id>`。
- VisualCommand 保留不动,作为 8.4 清理对象;`AvatarShell*Payload` 与 `*Payload`(VisualCommand 里的)命名空间分开,无冲突。
- pnpm sync:contracts 同步生成 TS + Python codegen,**追加性**(不破现有路径)。

**验证**:kernel tsc 0 错;Python 389 pytest 全过(协议变更无回归)。

**解开的 smell**:A(双视觉协议,下行统一了协议形态)/ B(WS 无契约,现在 schema 化)/ C(trace_id 顶级字段)/ H(emotion / thought 独立帧)。剩下 D / E / F / K 留 8.2/8.3 切换时落地。

**遗留(8.2/8.3 切换时使用)**:
- 8.2:kernel 侧新增 AvatarFrameRouter,生产 AvatarShellDownstreamFrame 替代手写 JSON;ChannelReplyEvent 发布方填充 `targetChannel`。
- 8.3:Electron 侧 ipc-handlers.ts / preload / store / Live2DCanvas 改消费 AvatarShellDownstreamFrame(类型安全 + 真实 trace_id);删假 streaming 状态机。
- 8.4:VisualCommand 退役删除;命名收口。

### 8.2 ⏳ kernel 侧重构:targetChannel + 事件驱动 status(已落) → AvatarFrameRouter 拆分(待续)

**做什么**:
- 把 `DesktopUIController` 切成:
  - `UserInputChannel`(chat_input → PerceptionEvent,trace_id 全程保真)
  - `AvatarFrameRouter`(订阅 EventBus → 下发 AvatarShellFrame 到 active shells)
  - `AvatarShellRegistry`(管 shell 连接表 + active 选择)
- `AvatarEngineController` 重命名为 `UnityShellAdapter`,作为 AvatarShellRegistry 的 Unity-类型 shell 实现。
- DesktopUIController 真正变成 "管 8083 端口的 transport",其他职责剥出去。
- 干掉 1500ms 轮询:Avatar shell 连/断时 publish `AvatarShellStatusChangedEvent`,Router 订阅。
- 删 traceId 字符串前缀路由,改 `target_channel` 字段。

**收益**:解开 D、E、K、F(部分)。

**风险**:中。kernel 内部重构,影响范围有边界。

**已落地(8.2 第一批,additive)**:
- 新增 [`avatar-shell-events.ts`](../../../core/kernel/src/core/foundation/event-bus/events/avatar-shell-events.ts) → `AvatarShellStatusChangedEvent`(payload: `shellKind` + 可选 `reason`)。
- `AvatarEngineController`(Unity 路径)在 ws 连入(status frame ready=true)/ 断开 / 心跳超时 / 初始化时 publish 该 event;新增 `_emitStatusIfChanged()` 助手 + `_lastBroadcastConnected` 去重(状态没变不重发,避免事件噪音)。
- `DesktopUIController`:
  * 订阅 `AvatarShellStatusChangedEvent` 立即转发到 Electron,**删 1500ms setInterval 轮询**(`_avatarStatusInterval` / `_avatarStatusIntervalMs` 字段下线；8.4 已删除 `config.avatar_status_interval_ms` schema/default/config 残留)→ **消 smell E**。
  * `action.channel.reply` 订阅器从 `traceId.startsWith('ui_')` 改为 `targetChannel.startsWith('desktop-ui')`;8.4 已删除 `targetChannel` 缺失时的旧前缀 fallback → **消 smell D**。
- ChannelReplyEvent 两处发布方填 `targetChannel`:
  * [`perception-app.service.ts:171`](../../../core/kernel/src/core/application/services/perception-app.service.ts#L171) → 用 `event.source`
  * [`ipc-server.ts:166`](../../../core/kernel/src/core/infrastructure/ipc-broker/ipc-server.ts#L166) → 用 `cmd.target.scene_id`

**待续(8.2 第二批,8.3 后或并行)**:`AvatarShellAdapter` 抽象 / `WebLive2DShellAdapter` / `UnityShellAdapter` 改名 / `AvatarFrameRouter` 拆分 / `AvatarShellRegistry`。属结构性重构,不改行为,可在 8.3 完成后再做(或合入 8.4 命名收口)。

### 8.3 ✅ 渲染层 Electron 侧重构:消费统一协议 + 删假 streaming

**做什么**:
- 渲染端 [`ipc-handlers.ts`](../../../core/renderer/desktop/src/main/ipc-handlers.ts) 和 [`preload`](../../../core/renderer/desktop/src/preload/index.ts) 改成消费 `AvatarShellFrame`(类型化的 + 实际验证的)。
- 删 [`emotionStore.EMOTION_ALIAS_MAP` / `EMOTION_PARAM_MAP`](../../../core/renderer/desktop/src/renderer/store/emotionStore.ts):映射成视觉的事 kernel 一次性算好,从 frame 拿到 `expression_id` / `motion_group` / `mouth` 直接驱动。emotionStore 只存 last emotion 给 UI 显示。

## 六、Extension Contribution Points 体系(月见 内容扩展统一抽象)

### 6.1 设计原则

**Extension 是月见 内容扩展的唯一基本单位**。不存在"Scene Pack vs Extension"的二元尴尬 —— 任何扩展月见 的东西都是一个 Extension,差异通过**贡献点(contribution points)**表达。

借鉴 VSCode、Slack App Manifest、Home Assistant、Browser Extension Manifest V3、MCP、Bevy ECS 各取一招,**叠加月见 独有的 3 个原创机制**(下文 §6.7),形成"成熟模式 + 月见 灵魂"的统一架构。

### 6.2 Manifest 完整 schema

```json
{
  "id": "cafe-window",
  "name": "窗边咖啡馆",
  "version": "1.0.0",
  "author": "elise",
  "description": "一个温暖的窗边座位",

  "engines": {
    "selrena": "^8.0.0"
  },

  "activationEvents": [
    "onScene:cafe-window",
    "onSkillRequested:cafe.brew_coffee"
  ],

  "capabilities": {
    "perception": ["inject"],
    "memory": ["read"],
    "narrative": ["append"],
    "persona": [],
    "avatar_shell": ["render_scene", "register_motion", "play_audio"],
    "skills": ["register"],
    "cognition": []
  },

  "contributes": {
    "scenes": [ ... ],
    "avatar_motions": [ ... ],
    "soundscapes": [ ... ],
    "skills": [ ... ]
  }
}
```

**关键字段四块**:
- `engines.selrena` — 声明兼容的月见 大版本(借 VSCode `engines`)
- `activationEvents` — 懒激活触发条件(借 VSCode 同名机制)
- `capabilities` — 显式声明扩展需要的能力(借 Browser Manifest V3 permissions + MCP)
- `contributes` — 实际贡献内容,可贡献到任意贡献点组合(借 VSCode contribution points)

Manifest 走 protocol schema 化(`ExtensionManifest.schema.json`),验证强制,符合铁律 1 精神。

### 6.3 Contribution Points 初始清单(8 个起步)

| 贡献点 | 扩展月见 的哪一层 | Consumer | 进程域 | 是否含可执行代码 |
|---|---|---|---|---|
| `adapters` | 触手 | kernel ExtensionEngine | ❹worker 子进程 | 是(隔离) |
| `scenes` | 身体所在 | active Avatar Shell | ❸shell 内 | 否(声明式 JSON + 资产) |
| `avatar_models` | 身体形态 | active Avatar Shell | ❸shell 内 | 否(资产) |
| `avatar_motions` | 身体能力 | active Avatar Shell | ❸shell 内 | 否(资产) |
| `soundscapes` | 环境声 | active Avatar Shell / 音频引擎 | ❸shell 内 | 否(资产) |
| `skills` | 能力(MCP 工具) | kernel SkillRegistry | 视情况 | 是(MCP server / in-process) |
| `personas` | 人格 | SelfEntity | ❶/❷ | 否(配置叠加) |
| `knowledge` | 记忆 | KnowledgeBase | ❶ | 否(数据导入) |

**贡献点是开放清单**:未来月见 长出新能力维度时(例如 `emotion_overlays` / `voice_packs` / `reflection_strategies`),增加新贡献点即可,生态结构不变。

### 6.4 Activation Events 设计

VSCode 多年验证有效的模式 —— **懒激活**,Consumer 按需加载,启动时不耗资源。

初始事件类型:

| 事件 | 触发条件 |
|---|---|
| `onStartup` | 月见 启动时必加载(adapter 类常用) |
| `onScene:<id>` | 切换到该场景时 |
| `onSkillRequested:<id>` | 月见 决定调用该 skill 时 |
| `onPersonaActive:<id>` | 该 persona 当前激活 |
| `onAvatarShell:<kind>` | 该 shell 类型上线 |
| `onMessage:<channel>` | 某通道收到 perception(触手类常用) |

未来可扩展(`onDreaming` 等)。

### 6.5 Capability 模型(以贡献点为粒度的安全/边界)

Extension 必须**显式声明所需 capabilities**,Consumer 强制校验。安装时显示给用户审阅(借 Android / Browser Ext)。

**保护轴 7 个**:

| Capability | 含义 | 默认/谨慎 |
|---|---|---|
| `perception` | 看 / 注入感知 | inject 大多扩展会要,read 慎给 |
| `memory` | 读 / 写月见 长期记忆 | read 常见,write 慎给 |
| `narrative` | append / override 叙事上下文 | append 常见,override 高危 |
| `persona` | 读 / override 人格 | override 极高危,需用户明确同意 |
| `avatar_shell` | render_scene / register_motion / play_audio | shell 类扩展常用 |
| `skills` | register / invoke 其他 skill | register 常见 |
| `cognition` | 干预心智决策(reflect / intent / arbitrate) | 几乎不开放,仅供官方核心扩展 |

**安全策略以贡献点为粒度**,不以扩展整体为粒度 —— 一个扩展可以同时含纯资产贡献(scenes,在 shell 内安全加载)和代码贡献(adapter,走子进程隔离)。

### 6.6 Profile(复合上下文)

借 VSCode Profile + 月见 agentic 自主切换:

```json
{
  "id": "morning",
  "name": "晨间月见",
  "extensions_enabled": ["napcat", "cafe-window", "calm-persona"],
  "default_scene": "bedroom",
  "default_avatar": "shizuku-casual",
  "active_persona": "calm-persona/morning"
}
```

**Profile 切换两种触发**:
1. **用户手动**:UI 选 profile
2. **月见 自主**:认知核基于时间 / drive state / arousal 决定切

例如:早晨 6 点 → `morning` profile;用户开始工作 → `focused` profile(关社交触手);夜晚 → `dreaming` profile(关闭活跃 shell 显示)。

这是月见 **独有的**能力 —— 自己换"模式"。VSCode profile 仅用户手动。

### 6.7 月见 独有的 3 个原创机制

#### 6.7.1 Agentic Activation(月见 自决用什么)

VSCode:**用户调命令** → 扩展激活。
月见:**月见 决定调用什么** → 扩展激活。

机制:
- SkillRegistry 把所有可用 skill 的 schema(描述 + 参数 + 用途)**注入月见 Deliberate prompt**(MCP 对齐)
- 月见 reasoning 时见到 "我有 web_search / brew_coffee / set_alarm... 这些工具"
- 决定用哪个 → ActionCommand 触发 skill → 同时激活该 skill 所属 extension(`onSkillRequested:X` 事件)

**Activation Events 的 `onSkillRequested` 是这个机制的实现**。

#### 6.7.2 Cross-Extension Communication Mediated by Cognition

VSCode 扩展间:`commands.executeCommand('otherExt.cmd')` 直接调,**硬耦合**。

月见 扩展间:**不直接互调**,**通过月见 心智中介**。

举例:
```
[Scene Extension "cafe"] 用户点了桌上的书
   ↓ 发标准 perception {text:"用户翻开了书"}
[月见 心智] 收到 perception → Deliberate 思考"我应该看看这本书内容"
   ↓ 决定调 SkillExtension "summarize_book"
[Skill Extension "summarize_book"] 返回摘要
   ↓ 月见 把摘要说出来(action_command/reply)
[Scene Extension "cafe"] 同时月见 做"低头看书"动作(motion frame)
```

**两个扩展互相完全不知道对方存在**。它们都只跟月见 通信,月见 是中介。

架构层意义:
- 扩展间零耦合
- 任意两个扩展可以"合作",只要月见 觉得合适
- **月见 = 整个 extension 生态的中枢神经**

这是月见 的**身份与架构同构**(她是 agent,所有流量经她中介),无他可比。

#### 6.7.3 Self-Protective Capability Model

浏览器扩展权限保护"用户隐私"。月见 扩展权限**保护月见 本体的完整性**。

月见 本体 = 人格 + 记忆 + 情绪 + 心智决策权。Capability 等级:

| 等级 | 含义 | 适用 |
|---|---|---|
| **Read-only** | 看月见 状态,不能改 | 大多扩展 |
| **Append-only** | 加新东西,不删/改旧的(保护连续性) | narrative / knowledge |
| **Mediated** | 通过月见 自己决策影响她,不直接改状态 | skill / adapter |
| **Override** | 直接改人格/记忆 | 极高危,需用户明确同意 + audit log |

**月见 在自己的扩展生态里不会被慢慢"改"成别人**。她始终是她。

### 6.8 Hot Reload 与开发工作流

借 VSCode Extension Dev Host + Vite HMR:

- **声明式贡献**(scenes / personas / knowledge / avatar models / motions / soundscapes)→ 文件变化检测 → **在线热替换**
- **代码贡献**(adapters / skills)→ 重启该扩展(子进程),其他扩展不受影响

**早期就做好,后期不用打补丁**。开发月见 扩展应该和开发 VSCode 扩展一样流畅。

### 6.9 现有 extension-sdk 的演进路径(一次性迁移)

现状:`extensions/` 已有 napcat-adapter 等,`@glimmer-cradle/extension-sdk` 已存在,但**只有 adapter 这一种贡献**(隐式)。

演进(分批,不一次到位):
1. **扩展 extension-sdk**:加 `ExtensionManifest.schema.json` + contribution point 类型定义 + activationEvents + capabilities + engines 类型
2. **napcat-adapter 等现存扩展**:manifest 加 `contributes.adapters` 显式声明;旧 manifest 无 `contributes` 时视为待迁移,不在运行时长期自动推断。
3. **kernel 加 ExtensionRegistry**:统一加载所有 manifest,按 `contributes` 路由到各 Consumer,按 `activationEvents` 懒激活
4. **每个 Consumer 单独实现**:Avatar Shell 加 SceneLoader / ModelLoader / MotionLoader / SoundscapeLoader;kernel 加 SkillRegistry / PersonaRegistry / KnowledgeImporter

### 6.10 为什么这个体系是和谐的

- **唯一基本单位**:Extension(不是 Pack、不是 Scene Pack、不是 Plugin —— 一个词)
- **本体层 ≡ 贡献点清单**:月见 的"心智 - 身体 - 触手 - 能力 - 人格 - 记忆"自然映射成贡献点
- **架构概念和扩展类型一一对应**:不多不少
- **共享基础设施,差异化加载**:发现/验证/安全/UI/热重载/版本管理一次设计,所有 kind 复用;Consumer 按 kind 实现
- **安全边界以贡献点为粒度**:含代码的隔离,纯声明式的安全直加载
- **架构与身份同构**:扩展间通过月见心智中介协作,不直接互调;这保持了生态的概念完整性。
- **未来扩展不动核心**:新增贡献点 = kernel 加 Consumer + SDK 加类型;新增内容 = 创作者写一个 extension manifest;**无限可能,核心稳定**
