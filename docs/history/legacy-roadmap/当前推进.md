# 迁移材料：当前推进

> 当前执行面已迁至 `docs/roadmap/now.md`，本页不再更新。

> 最后整理：2026-06-23

月见当前不是从零搭骨架的阶段。Protocol、Kernel、Cognition、Desktop Surfaces、Unity Avatar Shell、`data/` / `configs/` 结构、托盘驻留与 Control Center 已完成主线落地。渲染主链路已经形成可构建、可启动、可报告真实首帧就绪的闭环；下一段主工作转向 Skill Plane / Extension 能力平面，渲染质量继续按实机验收矩阵并行打磨。

## 当前阶段

**阶段 8：渲染层 + Skill Plane / Extension 体系收口。**

已落地：

- Electron 桌面壳采用 Presence Surface + Control Center 模式。
- Control Center 可与 Kernel 连接并完成基础对话。
- Desktop Surfaces 已接通 `thought/reply/emotion/shell_status/audio_status/audio_play/audio_transcript` 下行帧与 `audio_input` 上行帧；protocol 已用 `AvatarShellFrameClass` 归纳为表达流、身体控制、外壳生命周期三类。
- Kernel Application 层已落地 Skill Plane 最小骨架：`SkillProvider`、`SkillCatalogAppService`、`SkillRegistry`、`SkillCatalogSnapshot`、`SkillPolicyEngine`、`SkillInvocationGateway`、Extension Skill Provider 适配器、`contributes.skills`、Core Skill Provider 契约骨架、MCP Server Provider 配置目标读取，以及 User Provider 生命周期骨架。
- Kernel 启动编排已从 `app.ts` 长脚本收束为 `LifecycleOrchestrator + core/lifecycle/runtime/*` 阶段化 app-root delegates：Foundation、Transport、Application、Presentation、Core Readiness（Cognition + Audio 并行）、Extensions、Organism 分段启动；桌面壳与状态通道先亮起，IngressGate 只在 required runtime 全部就绪后放行。Kernel 主启动时间线只记录阶段、模块、readiness 与摘要告警，Cognition 等子进程内部日志进入 `data/observability/logs/processes/`。
- Avatar Shell 已纳入 Kernel presentation runtime：`AvatarEngineController` 负责 8082 协议网关与状态事件，`avatar_shell.sidecar` 默认以 managed 模式拉起 `data/packages/avatar-shell/windows/SelrenaAvatarShell.exe`，并通过 `RuntimeReadinessSnapshot(runtime_id='avatar.shell')` 投影到启动主线。`shell_hello` 与 `shell_ready` 已分离：只有真实模型驱动、Composition Host 和首帧呈现全部完成才进入 `unity-shell`；静态贴图预览不再冒充正式身体。`manual` 仅作为 Unity Editor 调试方式保留。
- Unity Avatar Shell 已完成 Unity `6000.0.77f1` + URP、Cubism SDK for Unity 5-r.5、catalog prefab 投影、真实动态模型驱动和 `AvatarBehaviorRuntime`。身体以完整自然边界进入相机，显示倍率与停靠只影响完整透明表面；动作命令与可保持动作分别使用命令/开关语义，状态由 Shell 权威快照回传。
- Control Center 已收束为单条分组侧栏与连续主工作区：页面切换使用独立滑动指示条，完整/图标模式通过品牌区双击切换；对话工具栏、正文和输入区使用轻边界形成同一会话表面。基础 Light / Dark、配置脏状态和 Avatar 动作状态均进入 UI 自动验收。
- Cognition 已由 CognitiveLoop 作为感知唯一编排者，旧 `ChatUseCase` 不再是运行时架构。
- `data/` 与 `configs/` 已按运行域整理。
- 官方音频引擎 `engines/audio` 已通过 Kernel provider 接入：实时 TTS 默认走 Piper，ASR 走 FunASR，GPT-SoVITS v2 保留为显式启用的身份声线。Kernel 内部已按能力 lane 拆分独立音频子进程，`CapabilityRuntime` 会等待默认音频资源完成本地检测、自动下载或明确不可用，避免 Piper / FunASR / GSV 冷启动落到第一次用户请求。Control Center 已具备麦克风录音入口，录音在 renderer 先做基础降噪和归一化，再经 `audio_input` 上行到 Kernel，落盘到 `data/artifacts/asr/` 后由 FunASR 识别，以 `audio_transcript` 回执并注入标准感知链路。TTS 已改为短句渐进播放并写入 `data/cache/audio/tts/` 复用缓存；native/ 作为热路径底座，Windows SAPI / Edge-TTS / whisper.cpp 作为显式排障兜底。
- 模型资产边界已收口：`assets/` 只放随应用发布的只读默认资产；embedding / voice / vision 等本机模型进入 `data/models/`。Cognition embedding 默认使用 `data/models/embedding/m3e-small`，缺失时可按 `model_id` 在启动期自动下载，下载或明确降级完成前不宣告 Cognition ready。
- 基础 telemetry 已能记录慢链路 span。

正在推进：

- Skill Plane + Extension Provider：MCP Server 已具备真实连接、枚举、调用与回收主线；planner 已通过 `SkillPlanningAppService` 消费 `SkillCatalogSnapshot` 的 ready 工具投影，并只经 `SkillInvocationGateway` 执行受目录约束建议；继续收束扩展贡献点、激活事件、权限与 requires/ports。
- Renderer 并行验收：日常 `desktop-companion` 是月见唯一正式人物表面；继续覆盖拖动、resize、DPI / 多显示器、作者 motion、口型、细粒度命中和打包版录屏身份。直播、场景、录制与视频输出留给后续扩展贡献。
- 音频与多模态延迟治理：为 ASR / TTS / 图片识别预留低延迟、可观测、可并行的链路。
- 启动观测收口：继续用 `runtime.pretty.log` 验证阶段顺序是否清晰，把音频 sidecar、模型预检与未来视觉引擎纳入同一轻量生命周期秩序。
- 文档体系收口：当前事实、蓝图、路线图、历史决策彻底分离。

## 下一步任务

1. **Skill Plane 与 Extension 体系基建**
   - 对齐 `docs/architecture/current/05-插件层.md`。
   - 明确 Skill / Tool / Resource / Prompt / Provider / Extension 的命名边界。
   - Core Skill Provider 只承载 Minimal Universal Skills，避免自带能力过细过杂。
   - 为 MCP Server Provider 补充 OAuth / 凭据投影、重连退避与 Control Center 诊断视图。
   - 接入 User Provider 配置、沙箱与审计策略。
   - 明确 contributes、activation events、requires、permissions、engine version。

2. **Renderer 实机质量验收（并行，不阻塞能力平面开发）**
   - Presence Surface 只承担“月见存在体”。
   - Control Center 是普通管理窗口，不置顶、不抢焦点、不伪装 overlay。
   - 托盘和任务栏图标作为稳定入口。
   - 继续实机验收 Unity Avatar Shell：透明窗口交互、显示器/缩放配置、模型表情、口型和 `load_scene` 真实场景切换。
   - `pnpm avatar:sync` / `pnpm avatar:doctor` 是 Unity 资产与现代 Cubism runtime 前置诊断；`avatar:build` 已由 Unity 6 导入官方 SDK、生成模型 prefab 和正式 Avatar Shell。
   - 继续打磨 Control Center 的可访问状态、错误提示、语音按钮反馈与后续设置入口。
   - 继续实机验收已打通的“Composition Host 唯一可见窗口 + Unity 隔离 Render Worker”共享 surface 主链路，重点覆盖拖动、DPI、多显示器、输入 hull、表情、动作与口型。

3. **音频与响应延迟观测**
   - 用 `scripts/telemetry.py` 检查最新 slow spans。
   - 区分模型延迟、网络延迟、Kernel/Cognition 编排延迟、ASR/TTS/vision 处理延迟。
   - 给 ASR / TTS / vision 增加阶段化 span 与并行策略。

4. **文档与 skill 对齐**
   - `AGENTS.md`、`.codex/skills/glimmer-cradle/` 与 `docs/` 的路径必须一致。
   - 新增或删除架构入口后，同步 skill 的 reference 索引。

## 验收门

进入下一阶段前至少满足：

- `pnpm typecheck` 通过。
- `pnpm build` 无新增构建错误。
- 改 Cognition 时，进入 `core/cognition` 跑 `uv run pytest -q` 并通过。
- 改音频引擎时，进入 `engines/audio` 跑 `uv run pytest -q` 并通过。
- `pnpm dev` 可启动：Kernel、Renderer、Control Center、Cognition 连接状态清晰。
- DLQ 无新增非测试记录。
- `docs/architecture/current/` 与实际代码一致。

本次渲染主线收尾已通过：`pnpm avatar:build`、`pnpm typecheck`、`pnpm build` 与 Desktop Playwright 7 项测试；受控 `pnpm dev` 启动中 Avatar Shell 已按 `connected → hello → first frame presented → shell_ready` 顺序就绪，Cognition、TTS 与 ASR 也完成 readiness。多显示器、不同 DPI、透明命中与发布包录屏仍属于实机质量矩阵，不因一次本机启动通过而视为全部完成。
