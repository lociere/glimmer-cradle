# 迁移材料：蓝图落地流程

> 新的架构改动操作见 `docs/guides/development/架构性改动.md`；当前承诺见 `docs/roadmap/now.md`。本页保留历史方法说明，不再更新。

本文指导如何把月见从当前实现推进到 `docs/architecture/blueprint/月见架构蓝图.md` 描述的理想形态。它不是日常编码步骤；日常开发看 `docs/guides/开发流程.md`。本文回答的是：**下一阶段做什么、为什么先做它、做到什么程度算真正落地**。

## 1. 文档分工

| 文档 | 作用 | 不应该承担 |
|---|---|---|
| `architecture/current/` | 当前代码事实 | 不写尚未落地的愿景 |
| `architecture/blueprint/月见架构蓝图.md` | 理想形态与最高架构约束 | 不记录临时任务清单 |
| `roadmap/蓝图落地流程.md` | 从现状到蓝图的推进方法 | 不替代具体阶段事实 |
| `roadmap/当前推进.md` | 眼下正在做什么 | 不保存长期历史 |
| `history/architecture-decisions/` | 已完成阶段的决策证据 | 不作为当前事实源 |

判断顺序：

1. 先读 `current/`：现在到底是什么。
2. 再读 `blueprint/`：理想形态要求什么。
3. 用本文决定：差距应该如何拆阶段。
4. 用 `当前推进.md` 固化：本阶段只做哪几个承重点。

## 2. 落地原则

### 2.1 一个阶段只收口一个承重点

月见是多进程、多语言、多表面的系统，但不要一次翻五层。每个阶段选择一个承重点，例如：

- 渲染表面收口。
- Skill Plane / Extension Provider 收口。
- Cognition 低延迟推理收口。
- 语音输入输出链路收口。
- Unity Avatar Shell 接入收口。

其他层只做该承重点需要的最小配套。

### 2.2 先契约，后编排，再体验

跨层能力按这个顺序推进：

1. **契约**：schema / capability / frame / config 先成为单一事实源。
2. **编排**：Kernel / Cognition / Renderer 各自只承担本层职责。
3. **体验**：窗口、托盘、Live2D、语音、视觉等表面最后收口。

如果先做体验，容易产生私有协议和临时桥；如果先做实现但没契约，容易产生双份事实源。

### 2.3 删除旧形状，不保留影子架构

当一个阶段确认新形状更优：

- 删除旧入口、旧 fallback、旧命名。
- 不保留“兼容旧架构”的空壳。
- 不把历史解释留在当前文档里。

历史进 `history/architecture-decisions/`，当前代码只保留当前形状。

### 2.4 每阶段必须有可运行验收

阶段完成不是“文档写完”，也不是“代码大概有了”。必须满足：

- `current/` 文档与代码一致。
- 最小自动验证通过。
- `pnpm dev` 或对应运行入口能证明链路活着。
- 关键日志 / trace / DLQ 能解释失败。

## 3. 阶段推进循环

### 3.1 差距审计

输入：

- `architecture/current/` 对应层。
- `architecture/blueprint/` 目标段落。
- 最近日志、测试、用户体验问题。

输出：

- 当前缺口列表。
- 旧架构残留列表。
- 不该在本阶段处理的事项。

缺口要写成结构差距，不要写成零碎 bug。例如“Control Center 窗口不像普通软件窗口”背后是 Desktop Surface 生命周期和窗口语义问题。

### 3.2 阶段切片

把缺口切成一个承重点：

```text
本阶段目标：一句话说明要让哪一块变完整
涉及层级：Protocol / Kernel / Cognition / Renderer / Extensions / Configs / Data
不做范围：明确哪些诱人的事暂不做
验收门：自动测试 + 运行验证 + 文档同步
```

切片太大时，优先拆成“契约阶段 → 编排阶段 → 体验阶段”。

### 3.3 蓝图更新

如果阶段会改变目标形状，先改：

```text
docs/architecture/blueprint/月见架构蓝图.md
```

只写长期成立的结构，不写临时任务。

### 3.4 当前投影更新

落代码后同步：

```text
docs/architecture/current/
```

`current/` 必须描述已经落地的现实。若某个蓝图概念只落了一半，就写清“已落地部分”和“尚未落地部分”，不要假装完成。

### 3.5 实现与删除

实现时优先处理以下问题：

- 双份事实源。
- 旧命名和新命名并存。
- 私有字符串协议。
- 运行时 fallback 掩盖架构错误。
- 文档说有但代码没有的入口。

能删除就删除；不能删除时，必须说明它当前仍承担什么职责。

### 3.6 验收与归档

阶段完成后：

1. 更新 `docs/roadmap/当前推进.md`。
2. 必要时把阶段设计归档到 `docs/history/architecture-decisions/`。
3. 更新 `.codex/skills/glimmer-cradle/` 中会影响 AI 协作的索引。
4. 跑验收命令。
5. 记录剩余开放问题到 `docs/roadmap/后续增强.md`，不要散落在各处 TODO。

```text
阶段 8：渲染层与 Skill Plane / Extension 体系收口
```

阶段 8 的理想结果不是“UI 能显示”，而是：

- Desktop Shell 的 Presence Surface / Control Center / Tray 生命周期自洽。
- Avatar Shell 契约能自然承载 Live2D 与未来 Unity。

- Renderer、Kernel、Cognition 不再靠临时事件名互相猜。
- 日志和 trace 能解释对话、表情、窗口、扩展激活的全链路。

## 5. 后续理想形态路线

### 8.x 渲染表面与 Skill Plane / Extension 收口

目标：月见有稳定桌面身体和能力生态入口。

承重点：

- `AvatarShellFrame` / surface frame 完整通电。
- `thought / reply / emotion / shell_status / audio_play` 已接入 Desktop Surfaces；后续继续补齐 expression / motion / parameter / load_scene。
- Control Center 成为正常管理窗口。
- Presence Surface 常驻但不侵入。
- Skill / Tool / Resource / Prompt / Provider 命名边界清楚。
- Core Skill Provider 只包含 Minimal Universal Skills；平台/服务/复杂流程走 Extension 或 MCP Server。
- Contribution Points / activation events / requires / permissions / engine version 成熟化。

验收：

- `pnpm dev` 可启动。
- 托盘、Presence、Control Center 可独立解释。
- 基础对话、情绪帧、回复帧有 trace。
- 扩展声明能力后由 Kernel 正式激活，不走私有 hack。

### 9.x 低延迟多模态交互

目标：为图片识别、ASR、TTS、未来视觉上下文建立低延迟链路。

承重点：

- 输入链路拆成 text / image / audio 的并行预处理。
- ASR / TTS / vision 都有独立 span。
- 模型延迟、网络延迟、编排延迟可区分。
- UI 有明确的 thinking / listening / speaking 状态。

验收：

- 慢响应可从 telemetry 定位。
- 文本对话不被语音 / 视觉链路阻塞。
- TTS 失败不影响文本回复。

### 10.x Native / 本地能力接入

目标：把性能敏感能力从脚本 fallback 推向正式 native 能力。

承重点：

- `engines/` 成为官方本体能力引擎边界，`native/` 成为 C++ FFI 性能底座。
- Cognition 通过抽象服务使用 native，不直接绑定实现。
- Kernel 只负责编排，不承载认知判断。

验收：

- native 缺失时有明确能力降级和日志，不伪装成功。
- ASR / TTS / embedding / local inference 的接口稳定。

### 11.x 外部 Avatar Shell / Unity

目标：Unity 成为 Avatar Shell 的一个实现，而不是另一个独立架构。

承重点：

- Unity 消费同一 `AvatarShellFrame`。
- Live2D 与 Unity 共用状态语义。
- 外部身体崩溃不影响 Cognition / Kernel。

验收：

- Live2D 与 Unity 可以替换而不改 Cognition。
- 表情、动作、回复、状态帧语义一致。

### 12.x 内容生态与自我演化

目标：Skill Plane / Extension / Lorebook / Evaluator 形成内容生态。

承重点：

- Extension 能作为 Skill Provider 贡献工具、上下文源、UI surface、agent skill。
- Core / MCP Server / User Script 也能作为 Skill Provider 接入同一 Skill Registry。
- Cognition-mediated cross-extension 由 Kernel 正式调度。
- Evaluator 能把行动结果回写经历之流。

验收：

- 扩展不能越权成为本体器官。
- 用户可解释“某能力来自哪个 extension”。
- 经历日志能追溯能力调用与结果。

## 6. 阶段完成定义

一个阶段完成必须同时满足：

- **形状完成**：该阶段承重点没有旧架构影子。
- **链路完成**：从入口到出口有真实运行路径。
- **观测完成**：失败时能从日志 / span / DLQ 定位。
- **文档完成**：blueprint、current、roadmap、skill 该更新的都更新。
- **验证完成**：自动测试和必要人工运行都通过。

缺任一项，只能算“阶段内可用”，不能算“阶段完成”。
