## C. ActionStream thought / emotion 帧(渲染向)

**现状**:阶段 7 只通了 reply 帧(ActionCommand → ChannelReplyEvent → 文字回复)。
蓝图 §6.5 ActionStream 是 thought / reply / emotion **三类帧**。后两类待:
- **thought 帧**:循环 Deliberate 期间推送"思考中…"动效给渲染层
- **emotion 帧**:emotion_system 变化时推送给 Live2D 表情驱动

**推**:与渲染层弧绑定 —— 没有渲染层这两类帧没消费者。所以排在 "**渲染层
弧**" 之后再做最自然。

**位置**:认知核加 thought / emotion ActionCommand 类型;kernel handler 路由
到对应 EventBus 事件;渲染层订阅。

**风险**:中。跨层(认知核 + kernel + 渲染层)。

## F. 真流式回复(从认知核到渲染层全链路 token 流)

**现状**:
- 认知核 Deliberate 一次性 LLM 调用,生成完整 reply 才推 ActionCommand
- kernel ipc-server ACTION_COMMAND handler 直接把整段 text 包成 ChannelReplyEvent
- 渲染层 [`appStore.appendMessageChunk`](../../../core/renderer/desktop/src/renderer/store/appStore.ts) 曾写过一套流式拼接(`isStreaming` / `currentStreamText` / `currentTraceId` 状态机)
- 但 [`ipc-handlers.ts`](../../../core/renderer/desktop/src/main/ipc-handlers.ts) 曾经永远发 `is_final: true` —— **流式机制是死代码,光标 `▊` 永不出现**

**含义**:用户看到的是"几秒后蹦完整段",体感比"边想边说"差很多。

**推**(真做时):
- ReasoningService.generate 改 streaming(LLMEngine 支持 stream 即可,云 API 一般原生支持)
- Deliberate 累积 chunks,边累积边推 partial ActionCommand
- ActionCommand 协议加 `is_final` / `chunk_index` 字段
- kernel ACTION_COMMAND handler 流式转发到 ChannelReplyEvent
- 渲染层去掉假实现,接真流式(现有 appStore 状态机直接可用)

**当前动作(阶段 8 已决策)**:
- 阶段 8.3 中**先删假实现**(`ipc-handlers.ts` 不再写无意义 `is_final:true`,`appStore` 流式机器删掉),让 UI 至少诚实
- 真流式作为本项 F 单独触发

**触发时机**:用户体感到延迟瓶颈、Awake 状态云大模型回复时长到注意阈值时。

**风险**:中。触及 LLM 推理层 + IPC 协议 + 三端流式语义。
