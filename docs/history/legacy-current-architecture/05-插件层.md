# 05 - Extension 宿主层

> 迁移材料：当前规则见 `docs/architecture/current/07-子系统当前视图/Extension与SkillPlane.md`、`docs/architecture/implementation/Extension与SkillPlane实现.md` 与 `docs/reference/extension-sdk.md`。本页不再更新。

> 文件编号保留“插件层”，但当前术语统一为 **Extension**。旧的 Plugin 命名只作为历史存在，不再作为架构事实源。

## 1. 定位

Extension 宿主层是月见向外部世界生长的边界。它允许第三方或项目内扩展为月见接入新的平台、技能、资源、界面入口和场景，但不暴露 Kernel、Renderer、Cognition 的内部对象。

核心原则：

- **无限可能来自组合**：扩展不是固定分类，而是 `contributes + requires + permissions + lifecycle` 的自由组合。
- **贡献点是声明面**：`contributes` 描述扩展给月见生态增加什么。
- **宿主端口是调用面**：`ctx.ports` 描述扩展激活后能调用哪些月见开放能力。
- **权限是授权面**：`permissions` 决定受保护 API 是否能执行。
- **工具不是类型**：WebSocket、HTTP、stdio、MCP 连接方式都是可组合工具，不定义扩展身份。

Extension 不是月见本体能力。人格、情绪、长期记忆、Cognition 推理、Presence Surface、Control Center、Avatar Shell、TTS、ASR、LLM provider 都不属于 Extension。

## 2. 物理位置

| 位置 | 职责 |
|---|---|
| `core/kernel/src/core/host/extension-manager.ts` | ExtensionManager：发现、加载、启动、停止扩展，创建沙箱上下文。 |
| `extensions/internal/sdk/` | `@glimmer-cradle/extension-sdk`：扩展作者公共 API 面。 |
| `extensions/internal/template/` | 新扩展模板。 |
| `extensions/adapters/` | 当前真实外部接入扩展所在目录，例如 NapCat。目录名是项目分组，不代表 SDK 类型系统。 |
| `configs/extensions/active.yaml` | 当前启用扩展列表。 |
| `configs/extensions/<extension-id>.yaml` | 单个扩展运行配置。 |

## 3. SDK 公共结构

`@glimmer-cradle/extension-sdk` 是 Extension Host 的公共边界，不是各内部层的导出口。

| 子入口 | 含义 |
|---|---|
| `@glimmer-cradle/extension-sdk/contracts` | 扩展可见协议投影；权威定义仍在 `@glimmer-cradle/protocol`。 |
| `@glimmer-cradle/extension-sdk/manifest` | Manifest、contribution points、`requires` 宿主端口声明。 |
| `@glimmer-cradle/extension-sdk/host` | `ExtensionContext`、`ExtensionHostPorts`、各类 Port 接口。 |
| `@glimmer-cradle/extension-sdk/events` | 系统事件 topic 与 payload map。 |
| `@glimmer-cradle/extension-sdk/lifecycle` | `defineExtension`、`BaseExtension`、生命周期契约。 |
| `@glimmer-cradle/extension-sdk/permissions` | 沙箱权限枚举与 helper。 |
| `@glimmer-cradle/extension-sdk/utilities` | 可选工具包，例如 `WebSocketBridge`。工具不定义扩展类型。 |

根入口仍导出全部公共 API，方便现有扩展使用；新代码可以按子入口导入，让协议、manifest、host、lifecycle 的概念边界更清楚。

## 4. 生命周期

```text
ExtensionManager.init()
  -> 读取 configs/extensions/active.yaml
  -> 扫描 extensions/ 下的 extension-manifest.yaml
  -> loadExtension(id)
       -> 解析 YAML manifest
       -> 校验 extension id
       -> require(main)
       -> 合并 defineExtension({ manifest }) 补充信息
       -> 校验最终 manifest 与 engines
       -> 读取并校验 configs/extensions/<id>.yaml
       -> createSandboxContext()
       -> 发布 ExtensionLoadedEvent
  -> startAllExtensions()
       -> extension.onActivate(context)
       -> 发布 ExtensionStartedEvent
```

停止时先调用 `onDeactivate()`，再统一 dispose `context.subscriptions` 中登记的资源。

## 5. Manifest

权威 schema 位于 `extensions/internal/sdk/src/manifest/index.ts`。

```yaml
id: my-extension
name: My Extension
version: 0.1.0
main: dist/index.js
minAppVersion: 0.1.0

engines:
  selrena: 0.1.0
  node: 20.0.0

permissions:
  - CONFIG_READ_SELF
  - COMMAND_REGISTER

requires:
  - commands

activationEvents:
  - onStartup
  - onCommand:my-extension.ping

contributes:
  commands:
    - command: my-extension.ping
      title: Ping My Extension
      category: My Extension
```

| 字段 | 含义 |
|---|---|
| `id` | 扩展唯一 ID，必须与启用列表一致。 |
| `main` | 构建产物入口，当前约定为 `dist/index.js`。 |
| `engines` | 扩展运行环境约束；Kernel 当前校验 `selrena` 与 `node`。 |
| `permissions` | 受保护 API 的实际授权边界。 |
| `requires` | 扩展需要宿主开放的端口，例如 `commands`、`perception`、`sceneAttention`。 |
| `activationEvents` | 扩展激活入口。 |
| `contributes` | 扩展贡献给月见生态的声明式事实。 |

## 6. Contribution Points

贡献点是静态事实，不是激活后的私有副作用。

| 贡献点 | 用途 | 当前状态 |
|---|---|---|
| `commands` | 可见命令。 | Kernel 命令系统已消费。 |
| `views` | UI 面板或视图入口。 | schema 已支持，UI 消费待接入。 |
| `statusBarItems` | 状态栏项目。 | schema 已支持，UI 消费待接入。 |
| `slashCommands` | 对话输入快捷命令。 | schema 已支持，Control Center 消费待接入。 |
| `settings` | 扩展设置项。 | schema 已支持，配置 UI 待接入。 |
| `themes` | 渲染主题资源。 | schema 已支持，渲染层消费待接入。 |
| `personaSkins` | 形象或人格外观切片。 | schema 已支持，Avatar Shell 消费待接入。 |
| `scenes` | 外部场景或身体入口。 | schema 已支持，场景调度消费待接入。 |
| `skills` | 声明式 Skill 身份、tools、resources、prompts 与 policy。 | schema 已支持；加载扩展时进入 `SkillCatalogSnapshot`，运行时 handler 仍由 `registerSubAgent` 绑定。 |

Skill Provider 已进入 Skill Plane。`contributes.skills` 是声明面，加载扩展时会注册为 `metadata.runtime_status = "contract_only"` 的目录项；`ctx.ports.agents.registerSubAgent(...)` 是当前运行时 handler 绑定入口，同 ID 注册后会覆盖为可执行的 ready Skill。`SkillPlanningAppService` 从 `SkillCatalogAppService` 的 `SkillCatalogSnapshot` 只投影 ready 工具到 Cognition；Cognition 返回稳定的 `skill_id + tool_name` 建议后，Kernel 会剔除目录外建议，并只能经 `SkillInvocationGateway` 执行。这样 planner 不直接依赖扩展内部对象、MCP client 或注册表单例。

当前 Skill Plane 已有统一 `SkillProvider` 生命周期接口。Core / MCP Server / User Provider 通过同一接口启动与停止；其中 MCP Server Provider 已读取 `configs/system/skills.yaml` 的启用 server 目标，并使用官方 MCP TypeScript SDK 建立 stdio / Streamable HTTP / WebSocket 会话。协议初始化与 tools/resources/prompts 枚举成功后，一个 Server 投影为一个 ready Skill；远端断连、超时或刷新失败会撤销目录项并投影为 Provider 降级，不能留下假 handler。User Provider 暂时只固定生命周期边界。Extension 的运行时注册仍由 `ExtensionHostAppService.registerAgent()` 适配进入能力目录。

## 7. Host Ports

扩展激活时获得 `ExtensionContext`：

```ts
interface ExtensionContext<TConfig = unknown> {
  readonly extensionId: string;
  readonly logger: ExtensionLogger;
  readonly config: TConfig;
  readonly subscriptions: Disposable[];
  readonly ports: ExtensionHostPorts;
}

interface ExtensionHostPorts {
  readonly storage: ExtensionKeyValueStore;
  readonly shortTermMemory: ShortTermMemoryPort;
  readonly perception: PerceptionPort;
  readonly sceneAttention: SceneAttentionPort;
  readonly events: ExtensionEventBus;
  readonly agents: AgentRegistryPort;
  readonly commands: CommandRegistryPort;
}
```

`logger`、`config`、`subscriptions` 是上下文基础设施；真正触碰月见能力面的调用统一进入 `ctx.ports`。这避免 Context 变成无限膨胀的大对象，也让“宿主开放能力”在结构上清晰可见。

## 8. 权限与事件边界

常用权限：

| 权限 | 允许行为 |
|---|---|
| `CONFIG_READ_SELF` / `CONFIG_WRITE_SELF` | 读取或写入自身配置。 |
| `PERCEPTION_WRITE` | 注入标准感知事件。 |
| `EVENT_SUBSCRIBE` / `EVENT_PUBLISH` | 订阅或发布事件。 |
| `AGENT_REGISTER` | 注册 SubAgent / MCP-style tool。 |
| `COMMAND_REGISTER` / `COMMAND_EXECUTE` | 注册或执行命令。 |
| `MEMORY_SHORT_TERM` | 读写扩展短期记忆。 |

自定义事件只能发布到 `extension.<extensionId>.*` 命名空间。订阅系统事件必须在 SDK 的 `ExtensionSystemEventTopics` 白名单内，或订阅自身扩展命名空间。

## 9. 可选工具

`utilities` 是工程辅助，不定义扩展类型。当前包含：

| 工具 | 用途 |
|---|---|
| `WebSocketBridge` | 给需要接入外部 WebSocket 进程的扩展复用连接管理、鉴权和 JSON 帧分发。 |

NapCat 是普通 Extension，它组合 `WebSocketBridge`，而不是继承某种 “Adapter Extension” 类型。

## 10. 写扩展的最小路径

1. 从 `extensions/internal/template/` 复制模板到合适目录。
2. 修改 `extension-manifest.yaml` 的 `id`、`name`、`engines`、`permissions`、`requires`、`contributes`。
3. 在 `index.ts` 只导出 `defineExtension(...)`。
4. 在 `src/` 下实现扩展类，优先继承 `BaseExtension`；需要外部 WebSocket 时组合 `WebSocketBridge`。
5. 构建扩展，并把 ID 加入 `configs/extensions/active.yaml`。

## 11. 禁止事项

- 不新增 `PluginManager`、`SystemPlugin`、`plugin-manifest.yaml` 或 `plugin.{id}.*`。
- 不允许扩展直接 import Kernel 内部 `foundation/`、`application/`、`capabilities/` 模块。
- 不允许把平台原始 payload 直接传入 Cognition；必须先清洗为标准感知事件。
- 不允许把 UI 贡献点做成扩展激活后的私有副作用；应先进入 manifest `contributes`。
