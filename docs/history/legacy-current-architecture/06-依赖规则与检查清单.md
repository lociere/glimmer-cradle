## 一、六层架构依赖白名单

```
┌──────────┐
│ Protocol │  ← 零依赖，纯类型定义
├──────────┤
│Foundation│  ← 仅依赖 Protocol
├──────────┤
│Infra­    │  ← Foundation + Protocol
│structure │
├──────────┤
│ Domain   │  ← Foundation + Protocol（通过端口注入使用 Application 能力）
├──────────┤
│Application│ ← Foundation + Infrastructure + Domain + Protocol
├──────────┤
│  Host    │  ← 仅通过 IExtensionHostService（Foundation port）
└──────────┘
```

### 详细依赖矩阵

| 层级 | Protocol | Foundation | Infrastructure | Domain | Application | Host |
|---|:---:|:---:|:---:|:---:|:---:|:---:|
| **Protocol** | — | ❌ | ❌ | ❌ | ❌ | ❌ |
| **Foundation** | ✅ | — | ❌ | ❌ | ❌ | ❌ |
| **Infrastructure** | ✅ | ✅ | — | ❌ | ❌ | ❌ |
| **Domain** | ✅ | ✅ | ❌ | — | ❌¹ | ❌ |
| **Application** | ✅ | ✅ | ✅ | ✅ | — | ❌ |
| **Host** | ✅² | ❌ | ❌ | ❌ | ❌ | — |

> ¹ Domain 通过 `@protocol/ports` 中定义的 `IAICapabilityPort` / `IActionStreamPort` 使用 Application 能力，不直接 import Application 模块  
> ² Host 仅通过 `IExtensionHostService` 接口（Foundation port），不直接 import Application 实现

## 三、端口注入模式

当下层需要上层能力时，**不得直接 import**，必须通过端口接口注入：

### 3.1 Domain → Application（通过 Protocol 端口）

```typescript
// ❌ 违规：Domain 直接 import Application
import { AIProxy } from "../../application/capabilities/inference/ai-proxy";

// ✅ 合规：通过端口接口
import { IAICapabilityPort } from "@glimmer-cradle/protocol";

class SomeManager {
  private _aiProxy!: IAICapabilityPort;
  init(aiProxy: IAICapabilityPort) {
    this._aiProxy = aiProxy;
  }
}
```

在 `app.ts` 中绑定实现：
```typescript
SomeManager.instance.init(AIProxy.instance);
```

### 3.2 Host → Application（通过宿主端口）

```typescript
// ❌ 违规
import { ExtensionHostAppService } from "../application/services/...";

// ✅ 合规
import { IExtensionHostService } from "../foundation/ports";

class ExtensionManager {
  constructor(private readonly hostService: IExtensionHostService) {}
}
```

## 四、已知例外（ARCH-EXCEPTION）

如果某个 import 必须违反依赖规则（极少数情况），需在代码中标注注释：

```typescript
// ARCH-EXCEPTION: <原因说明> — <负责人> <日期>
import { SomeModule } from "../../disallowed-layer/...";
```

当前已批准的例外：**无**。

## 五、PR 自检清单

每次提交代码前，请对照以下清单逐项检查：

### 新增/修改功能

- [ ] **功能归层**：新功能是否放在了正确的层级？（决策/记忆 → Domain，IO/适配 → Host/Plugin，编排 → Application）
- [ ] **依赖方向**：import 路径是否符合上方依赖矩阵？无越层 import？
- [ ] **端口注入**：如果 Domain 需要 Application 能力，是否通过 `@protocol/ports` 定义的接口注入？
- [ ] **数据模型**：跨层传递数据是否使用 Protocol 定义的类型？是否存在裸 `dict` / `any`？
- [ ] **事件通信**：模块间通信是否通过 EventBus？是否存在跨层直接方法调用？

### Cognition 认知核 / Adapter 隔离

- [ ] **平台字段**：Cognition 认知核（Python 认知层）是否完全不含平台私有字段（`user_id`, `group_id`, CQ 码等）？
- [ ] **通用路由**：是否使用 `adapter_id` / `source_type` / `source_id` 等通用路由字段？
- [ ] **Cortex 清洗**：插件内是否在 Adapter Cortex 层完成了协议脏数据的归一化？

### 代码质量

- [ ] **Type Hints**：所有函数参数和返回值是否有类型注解？
- [ ] **异步规范**：是否存在主线程阻塞操作（同步文件 IO、同步网络请求）？
- [ ] **错误处理**：异常是否被正确捕获并通过 logger 记录堆栈？
- [ ] **路径管理**：文件路径是否使用 `path.join()` / `path.resolve()`（TS）或 `pathlib.Path`（Python）？

### 编译验证

- [ ] `pnpm exec tsc --noEmit`（kernel）零错误
- [ ] `pnpm exec tsc --noEmit`（protocol）零错误

## 六、新功能接入指引

### 判断归属层

```
新功能涉及什么？
├── 纯数据类型/接口定义         → Protocol
├── 日志/配置/事件总线/数据库   → Foundation
├── IPC/外部通信中间件          → Infrastructure
├── 月见本体状态/行为逻辑       → Domain
├── 跨层编排/能力封装           → Application
└── 外部平台适配/用户交互       → Host (插件)
```

### 插件开发快速流程

1. 创建插件目录，编写 `plugin-manifest.yaml`
2. 实现 `SystemPlugin` 接口（`onActivate` + `onDeactivate`）
3. 在 `onActivate` 中通过 `ctx.ports.perception.inject()` 注入感知
4. 订阅 `action.channel.reply` 获取 AI 回复
5. 在配置中启用插件

### 新增 Domain 能力

1. 在 `@protocol/ports` 定义端口接口
2. 在 Application 层实现该接口
3. 在 Domain 层通过构造函数/init 接收接口
4. 在 `app.ts` 中注入实现

## 七、架构验证命令

```bash
# TypeScript 编译检查（推荐用 pnpm filter，避免 cd）
pnpm --filter @glimmer-cradle/protocol lint
cd core/kernel && pnpm exec tsc --noEmit

# 契约同步检查（阶段 P.9b 起；改 schema 后跑）
pnpm sync:contracts                       # 等价于 pnpm --filter @glimmer-cradle/protocol gen:all

# 依赖方向快速检查（Domain 不应 import Application）
grep -rn "from.*application" core/kernel/src/core/domain/
# 期望输出：空（无匹配）

# Host 不应直接 import Foundation/Application
grep -rn "from.*foundation\|from.*application\|from.*infrastructure" core/kernel/src/core/host/
# 期望输出：空（无匹配）

# Infrastructure 旧副本是否已清理
ls core/kernel/src/core/infrastructure/
# 期望输出：仅 ipc-broker/
```
