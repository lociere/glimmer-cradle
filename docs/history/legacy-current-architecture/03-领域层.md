# 03 - 领域层（Domain）

> 迁移材料：当前分层边界见 `docs/architecture/current/04-构件、分层与依赖.md`。本页不再更新。

> 物理位置：`core/kernel/src/core/domain/`  
> 定位：**月见本体的有状态业务逻辑**——"活着"的核心

---

## 一、设计原则

- Domain 层代表月见本体的核心状态与行为
- **只通过 EventBus 与外部通信**，永不被上层框架直接调用方法
- **不依赖 Application 层**：需要 AI 推理或动作流能力时，通过 `@protocol/ports` 定义的接口端口注入
- 所有外部副作用通过发布事件触发，由 Application 层订阅并执行

## 三、模块详解

### 3.1 LifeClockManager（生命时钟管理器）

**单例**，驱动月见的主动思维，实现"活着"的核心特性。

#### 核心概念

**注意力模式**（三态）：

| 模式 | 含义 | 心跳间隔 |
|---|---|---|
| `standby` | 月见休眠中，不主动思考 | 心跳停止 |
| `ambient` | 月见在后台活动，低频主动思维 | 45 秒 |
| `focused` | 月见被唤醒，高频主动思维 | 10 秒 |

**消息源聚焦策略**（6 种）：

| 策略 | 行为 | 典型场景 |
|---|---|---|
| `always_focused` | 始终聚焦，不退回 | 私聊（桌面端） |
| `wake_word_focus` | 需要唤醒词才聚焦，聚焦后不自动退回 | 特殊频道 |
| `wake_word_focus_with_timeout` | 唤醒词聚焦 + 超时自动退回 | QQ 群聊 |
| `chat_or_wake_focus_with_timeout` | 任意消息或唤醒词聚焦 + 超时退回 | 终端 |
| `ignore` | 完全忽略此来源 | 系统消息 |

**配置来源**：`configs/cognition/inference.yaml` 的 `life_clock` 节：

```yaml
life_clock:
  focused_interval_ms: 10000
  ambient_interval_ms: 45000
  default_mode: standby
  focus_duration_ms: 180000       # 聚焦超时后自动退回
  focus_on_any_chat: false
  summon_keywords: ["月见", "selrena"]
  active_thought_modes: ["ambient", "focused"]
  source_focus_policies:
    private: always_focused
    group: wake_word_focus_with_timeout
    channel: wake_word_focus_with_timeout
    terminal: chat_or_wake_focus_with_timeout
    system: ignore
```

#### 触发器系统

可扩展的触发器机制，用于评估用户消息是否应触发注意力状态变化：

```typescript
interface AttentionTrigger {
  id: string;
  evaluate(context: TriggerContext): AttentionTriggerResult;
}
```

内置触发器：
- `WakeKeywordTrigger` — 匹配 `summon_keywords` 配置中的唤醒词

可通过 `LifeClockManager.registerTrigger()` 注册自定义触发器。

#### 频道注意力聚合

LifeClockManager 订阅 `SceneAttentionChangedEvent`（来自插件上报的频道状态），维护 `_channelFocusStates` 映射表，综合计算后发布 `OrganismAttentionChangedEvent`：

```typescript
// 聚合逻辑
focusedChannels.length > 0  → ACTIVE
mode === 'ambient'          → PASSIVE
otherwise                   → IDLE
```

#### 心跳循环

在 `ambient`/`focused` 模式下，LifeClockManager 按配置间隔发送心跳给 Cognition 认知核：

```
startHeartbeatLoop()
  ↓ setTimeout(间隔)
  检查 AI 是否就绪 → 检查模式是否允许主动思维
  ↓
  aiProxy.sendLifeHeartbeat({ attention_mode })
  ↓ 递归继续
```

#### 依赖注入

```typescript
// init() 接收端口接口，不直接 import Application 层
async init(aiProxy: IAICapabilityPort): Promise<void>
```

`IAICapabilityPort` 由 `app.ts` 注入 `AIProxy.instance`。

### 3.2 AttentionSessionManager（注意力会话管理器）

**单例**，管理消息的防抖批处理和生成中断检测。

#### 核心机制

**每个 scene（消息来源）独立维护状态**：

```typescript
type SceneIngressState = {
  pending: PendingIngress[];      // 待处理消息队列
  timer: NodeJS.Timeout | null;   // 防抖计时器
  chain: Promise<void>;           // 串行化处理链
  inFlightTraceId: string | null; // 当前正在生成的 trace_id
  cancelRequested: boolean;       // 是否已请求中断
};
```

#### 防抖策略

| 模式 | 防抖间隔 | 含义 |
|---|---|---|
| `normal` | 1400ms | 默认间隔，等待用户连续输入 |
| `focused` | 700ms | 聚焦模式下更快响应 |

间隔由 `LifeClockManager.state.mode` 实时决定。

> 这两个值属于**交互节奏策略**，不是性能优化开关。桌面私聊、群聊、语音输入、
> 图像理解等入口对批处理等待的容忍度不同；调参前先看 telemetry span，
> 确认慢在输入聚合、IPC、云端 LLM、ASR/TTS 还是渲染反馈。

#### 批处理

- 最多合并 `maxBatchMessages`（4）条消息为一个请求
- 最多合并 `maxBatchItems`（24）个内容项
- 超出限制的消息直接返回 `null`（已被批量合并）

#### 生成中断

当新消息到达时，如果上一条消息的 AI 生成仍在进行中：

```
检测到 inFlightTraceId 存在
  ↓
发送 cancelStream() 到 ActionStreamManager（通过端口）
  ↓
发送 cancelPerception() 到 Cognition 认知核（通过端口）
  ↓
新消息进入防抖队列等待刷新
```

#### 处理流程

```
ingest(request)
  ↓
按 source 获取/创建 SceneIngressState
  ↓
tryInterruptInFlight()       // 如有在途生成则中断
  ↓
onIngressMessage()           // 通知 LifeClockManager 有消息
  ↓
push 到 pending 队列 → scheduleFlush(debounceMs)
  ↓
[防抖等待…]
  ↓
flushScene()
  合并消息 → startThinkingStream → sendPerceptionMessage
  → completeStream / cancelStream
```

#### 延迟观测

AttentionSessionManager 只负责入站聚合与认知核 RPC，不直接做模型或渲染决策。
为后续文本 / 图片 / 语音统一诊断延迟，Kernel 侧记录两类 span：

| span | 含义 |
|---|---|
| `attention.flush` | 一次 scene 批次刷新，属性包含 `queue_wait_ms`、`batch_size`、`attention_mode`、`address_mode`、`modality` |
| `attention.ipc.perception_message` | Kernel 向 Cognition 认知核发送 `PERCEPTION_MESSAGE` 并等待交付确认 |

同时写入 `attention.ingress_wait_ms` 与 `attention.batch_size` histogram。Cognition 侧已有
`cognitive_cycle` / `deliberate` / `llm.generate` 等 span；用
`python scripts/telemetry.py spans latest` 可以查看最近一次 trace 的跨进程树。

#### 依赖注入

```typescript
// init() 接收两个端口接口
init(aiProxy: IAICapabilityPort, actionStream: IActionStreamPort): void
```

`app.ts` 注入 `AIProxy.instance` 和 `ActionStreamManager.instance`。
