## 一、项目定位

Glimmer Cradle 是一个 **数字生命体** 项目——月见（Selrena）。她拥有独立人格、情绪系统、长期记忆，能通过多种平台（桌面端、QQ、直播等）与用户交互。项目核心理念是 **Cognition / Adapter / Kernel 分层协作**：

- **Cognition 认知核（认知核心）**：纯粹的思考核心，运行在 Cognition 认知核，只处理标准化数据结构
- **Adapter（适配器）**：IO 交互适配层，运行在扩展中，负责协议清洗与平台接入
- **Kernel（中枢）**：进程调度、事件编排、扩展宿主与 Skill Plane 能力汇聚层，负责搬运与路由，不承载复杂业务判断

`data/` 是统一的本地数据域。开发期它位于仓库根，打包后映射到 Electron `userData/data/`。
其中 `data/state/` 承载记忆、经历、数据库和扩展状态，丢失会损伤月见连续性；
`data/packages/` 存放可重装的外部运行包，`data/artifacts/` 与 `data/cache/` 只放可以重新生成、重新下载或安全清理的运行产物。
`assets/` 与 `data/` 不承担同一职责：`assets/` 是应用版本自带的只读默认资源，
`data/` 是本机可变数据域。模型权重、用户语音资产和启动期下载结果默认进入 `data/models/`，
不进入 `assets/`。

`assets/avatar/models.json` 是内置角色 catalog。它声明稳定 model id、默认/降级选择、资产类型、目标后端、
manifest、情绪映射和模型专属动作。`core/renderer/avatar-unity/Assets/StreamingAssets/avatar/` 与
`avatar-registry.json` 由 `pnpm sync:unity-assets` 从 catalog 生成，不作为第二份源资产提交；renderer public
资产与 Unity StreamingAssets 都是可删除、可重建的运行投影。Desktop 与 Unity 不再硬编码角色名或
`model.json` / `emotion-map.json` 文件约定。

当前 Windows Avatar Shell 合成实现仍位于 `native/src/composition/`，Unity 工程位于
`core/renderer/avatar-unity/`。正式链路以 Composition Host 为唯一透明窗口所有者，Unity 负责无窗口渲染与共享 surface；
当前代码仍在完成从 Unity 普通窗口/诊断降级到共享 GPU surface 主路径的迁移。目标源码边界将应用级 Host 收口到
`core/renderer/composition-host/`，`native/` 只保留可复用平台原语；此项列于 `roadmap/当前推进.md`，不得在 current 文档中冒充已完成。

模型 catalog 按资源所有者分域：内置角色归 `assets/avatar/models.json`；Audio Engine 的可下载模型、声线与
上游 artifact 归 `engines/audio/src/audio_engine/resources.json`；Cognition embedding 选择归受 Schema 校验的
`configs/cognition/inference.yaml`。代码只消费选择结果和 catalog 投影，不复制具体模型 ID、下载 URL或目录默认值。

### Avatar Shell Frame 归纳

`AvatarShellDownstreamFrame` 是 Kernel → 月见可见身体的统一协议脊柱。当前按 `AvatarShellFrameClass` 归纳为三类：`expression_flow`（表达流：`thought/reply/emotion`）、`embodiment_control`（身体控制：`audio_play/expression/motion/lip_sync/parameter/idle`）、`shell_lifecycle`（外壳生命周期：`shell_status/audio_status/audio_transcript/load_scene/unload_scene/ping`）。Desktop Surfaces 与未来 Unity Avatar Shell 都应消费同一组语义分类，不各自发明私有协议。`audio_status` 是 Control Center 展示 ASR/TTS 能力状态的正式帧，Renderer 不直接探测 Kernel 内部 provider。`audio_play` 是回复的语音外显帧，与 `reply` 通过同一个 `trace_id` 关联，但不进入聊天消息内容块。`audio_input` 是桌面端语音输入的上行帧，识别结果通过 `audio_transcript` 回执给 renderer，并在 Kernel 内部转成标准 `PerceptionEvent` 进入同一认知链路。

## 四、核心数据流

### 4.1 消息处理全链路（用户消息 → AI 回复）

```
用户发送消息（QQ / 桌面端 / 直播弹幕）
  ↓
[插件] Platform Adapter（如 napcat-adapter）Cortex 层
  清洗协议数据（去除平台私有字段）
  计算 address_mode：唤醒词/@/私聊 → "direct"；焦点窗口内普通消息 → "ambient"
  构造 PerceptionEvent（content 含 address_mode，无任何平台字段）
  ctx.ports.perception.inject(event)
  ↓
[Application] ExtensionHostAppService.injectPerception()
  ↓
[Application] PerceptionAppService.processIngress()
  入站防护（IngressGateManager.admit）→ 速率限制/熔断/就绪检查
  提取 address_mode → 组装 PerceptionMessageRequest（含 address_mode）
  ↓
[Domain] AttentionSessionManager.ingest()
  防抖批处理 + 生成中断检测
  mergeRequests：批次内任一 direct → 合并为 direct；保留被中断批次的 address_mode
  ActionStreamManager.startThinkingStream → ActionStreamStartedEvent
  → DesktopUIController emit AvatarShellDownstreamFrame{kind:'thought', thought:{active:true}}
  ↓
[Application] AIProxy → IPCServer → Cognition 认知核
  KernelIngressCortex → ParsedPerception（含 address_mode + model_input）
  ↓ container._on_perception 解析后入队（阶段 7.5b-7：感知唯一去向）
  PerceptionEventQueue → CognitiveLoop 九阶段循环
    Sense（PerceptionProvider drain）
    Appraise（multimodal_router 路由 → emotion.update_by_input → 写
      PERCEPTION/EMOTION Moment）
    Recall（MemoryProvider 图谱 + 向量检索）
    Compete（salience + 来源优先级 + 新鲜度；direct perception 同分压过 drive）→ Broadcast
    Deliberate（富 prompt：人设 + 摘要 + 短期/长期/知识 + 媒体描述 →
      ReasoningService 按 arousal tier 选云/本地 + vision 直发 →
      validate_boundary → _pending_reply）
    Intend（Volition + Arbiter）→ Act（ActionCommand → ZMQ 推内核）
    Consolidate（写 REPLY/SILENCE Moment + 因果链 + 会话写回）
  ↓ ZMQ action_command（Python → Kernel 单向推送，不回 success_response）
[Kernel] IPCServer ACTION_COMMAND handler → ChannelReplyEvent
  action_type=reply 且 text 非空 → normalizeReplyMessages(text, messages)
  → EventBus.publish(ChannelReplyEvent{
    trace_id, text, messages, emotion_state, target_channel
  })
  target_channel 必须来自 ActionCommand.target.scene_id；缺失即丢弃，不再按 trace_id 猜路由
  trace_id 全程贯通（追踪键，不承担路由语义）
  ↓
[Kernel] 按 target_channel 分发：
  ├─ 'desktop-ui:user' → DesktopUIController（端口 8083 WS → Electron Desktop Surfaces）
  │     先关闭思考态：AvatarShellDownstreamFrame{kind:'thought', thought:{active:false}}
  │     emit AvatarShellDownstreamFrame{kind:'reply', trace_id, reply:{text,messages}}
  │       + 独立 AvatarShellDownstreamFrame{kind:'emotion', emotion:{...}}（阶段 8.3）
  │     异步调用 AudioService → 默认 Piper TTS；显式启用时可走 GSV 身份声线
  │       → emit AvatarShellDownstreamFrame{kind:'audio_play', trace_id, audio_play:{audio_id,audio_uri/audio_data}}
  │     Electron main 按 kind 广播到已存在的 renderer surface（trace_id 真实保留）
  │     renderer preload 订阅必须返回 cleanup；Control Center 按 trace_id 做回复幂等，
  │       并以独立 thought 状态显示 / 隐藏“正在思考”；audio_play 由 renderer 的
  │       AudioPlaybackController 播放并向 Presence/Live2D 口型通道广播 envelope
  │     → ControlCenter: ChatPanel 渲染气泡 / PresenceSurface: AvatarSurface 更新表情与状态
  ├─ 'unity:...' → AvatarEngineController（端口 8082 WS → Unity，蓝图预留）
  └─ 'napcat:group:X' / 'napcat:private:X' → napcat-adapter 扩展（只消费 napcat:* 目标通道；
        按 messages 顺序转发，缺失时回落 text）
  ↓
[Application] ActionStreamManager
  发布流生命周期事件；DesktopUIController 将 started/cancelled 翻译为 thought 帧。
  reply / emotion 的真实出口仍是 Cognition Act → ActionCommand → ChannelReplyEvent。
  ↓
[Foundation] EventBus 发布 action.channel.reply
  ↓
[插件] Platform Adapter 订阅事件 → 转换为平台格式 → 发送回复
```

`text` 是一轮回复的完整规范正文，用于记忆、日志、fallback 与全文审计；`messages` 是可投递消息切片，
用于把多句回复、代码块或平台消息分开发送。二者权威类型均定义在 `@glimmer-cradle/protocol`，
renderer 的 `ChatMessage` 只是 UI 投影，extension-sdk 只向扩展作者 re-export 协议类型。

桌面语音输入走同一套 Avatar Shell 契约：

```
Control Center 麦克风录音
  → renderer VoiceRecorder 降噪 / 压缩 / 噪声门 / 归一化后编码 WAV(base64)
  → preload 白名单 API sendAudioInput()
  → Electron main 转发 AvatarShellUpstreamFrame{kind:'audio_input'}
  → DesktopUIController 写入 data/artifacts/asr/
  → AudioService.recognizeSpeech() → ASR lane → engines/audio → FunASR
  → AvatarShellDownstreamFrame{kind:'audio_transcript'} 回执 UI
  → DesktopUIController 将识别文本注入 PerceptionAppService
  → 后续与普通文字消息完全同链路
```

**address_mode 设计原则**：
- `address_mode` 是由 Adapter Cortex 归一化的语义字段，不是平台私有字段
- Cognition 认知核 看到的是"是否被明确呼唤"这一通用社会语义，不是 QQ @/wake_word 等平台细节
- 沉默不等于无感——情绪系统和记忆仍正常运行，只是不发送回复

### 4.2 频道注意力流（插件 → 本体）

```
[插件] ctx.ports.sceneAttention.reportSceneAttention(channelId, focused)
  ↓
[Application] ExtensionHostAppService.reportSceneAttention()
  发布 SceneAttentionChangedEvent
  ↓
[Foundation] EventBus
  ↓
[Domain] LifeClockManager 订阅 SceneAttentionChangedEvent
  查阅策略 → 更新频道状态 → 计算本体状态
  发布 OrganismAttentionChangedEvent
  ↓
[Domain] AttentionSessionManager 订阅
  调整防抖间隔（focused: 700ms / normal: 1400ms）
```

**关键约束**：频道被唤醒 ≠ 月见本体被唤醒。插件只上报频道状态，本体最终状态由 LifeClockManager 策略引擎决定。

## 五、依赖方向规则

| 层 | 允许依赖 | 绝对禁止 |
|---|---|---|
| `@protocol` | 无（零依赖） | 任何 kernel 内部路径 |
| `foundation/` | `@protocol` | `infrastructure/` `domain/` `application/` `host/` |
| `infrastructure/` | `@protocol` `foundation/` | `domain/` `application/` `host/` |
| `domain/` | `@protocol` `foundation/` | `infrastructure/` `application/` `host/` |
| `application/` | `@protocol` `foundation/` `infrastructure/` `domain/` | `host/` |
| `host/` | `@protocol` + Node 标准库 | `foundation/` `infrastructure/` `domain/` `application/` |
| `app.ts` | 所有层 | 无限制（Kernel 唯一入口） |
| `core/lifecycle/runtime/*` | 所有层 | 仅允许做 app-root delegate：组装、启动、停机、readiness gate；禁止承载业务判断 |

## 六、配置体系

配置文件位于 `configs/` 目录，由 `ConfigManager` 统一加载并 **ajv 校验**（阶段 P.4b
起取代原 Zod；schema 单一事实源在 `protocol/src/schemas/config/`，铁律 1）：

| 文件 | 用途 | 对应 schema |
|---|---|---|
| `system/identity.yaml` | 应用身份、路径、备份策略 | `schemas/config/AppConfig.schema.json` |
| `system/kernel.yaml` | IPC + Lifecycle + IngressGate | `schemas/config/{IPC,Lifecycle,IngressGate}Config.schema.json` |
| `system/surfaces.yaml` | Avatar Shell + Desktop Shell | `schemas/config/RendererRuntimeConfig.schema.json` |
| `system/skills.yaml` | Skill Plane、MCP Server、用户技能入口 | `schemas/config/SkillPlaneConfig.schema.json` |
| `system/memory.yaml` | 记忆运行时参数 | `schemas/config/MemoryRuntimeConfig.schema.json` |
| `system/observability.yaml` | 日志、级别、轮转 | `schemas/config/ObservabilityConfig.schema.json` |
| `cognition/persona.yaml` | 身份种子 + 安全边界 | `schemas/config/PersonaConfig.schema.json` |
| `cognition/inference.yaml` | 推理、注意力、上下文、多模态、embedding | `schemas/config/InferenceConfig.schema.json` |
| `cognition/providers.yaml` | LLM provider 路由 | `schemas/config/LLMConfig.schema.json` |
| `cognition/knowledge-base.json` | 知识库条目集合 | `schemas/config/KnowledgeBaseConfig.schema.json` |
| `extensions/active.yaml` | 启用扩展白名单 | （无 schema，列表） |
| `secrets/secrets.yaml` | API Key 等敏感信息（gitignored） | （独立处理，secrets 注入） |
| `extensions/<id>.yaml` | 各扩展自有配置（每扩展一份） | 由扩展自定义 Zod schema |

**格式原则**：本地客户端形态下，`configs/` 是配置事实源；Control Center 的设置界面只是这些文件的可视化编辑器。人维护的配置优先 YAML，大型机器条目可用 JSON；Kernel、Cognition、Renderer 与 Extension 消费的不是原始 YAML/JSON，而是 schema 校验后的冻结配置投影。跨进程 IPC、WebSocket、HTTP 与未来云端同步只传 JSON 结构。

未来无客户端 / 云端部署不应把 YAML 当运行时数据库。云端形态的事实源应是配置服务或数据库，YAML 仅保留为导入、导出、bootstrap 与开发模板；只要 schema 与 `GlobalConfig` 投影不变，上层业务不应感知底层事实源从本地文件换成远端服务。

**校验链路**：YAML parse → `normalizeSystemYamlNulls`（处理 `module_levels:` 空键
→ {}）→ ajv 分段校验（`useDefaults:true` 自动填默认值）→ 注入 Cognition 认知核
（`SELRENA_CONFIG` env，Pydantic 二次校验，运行时 `frozen=True`）。

**核心配置冻结**：人设注入后配置被冻结（`Object.freeze` + Pydantic `frozen=True`），
运行期间禁止修改，防止人格漂移。

## 七、启动与停机

### 启动顺序（LifecycleOrchestrator + App Runtime Delegates）

```
App.start()
  ↓
LifecycleOrchestrator.startPhase('foundation')
  FoundationRuntime
  ConfigManager.init()
  initLogger() / startMetrics() / startTracer()
  DBManager.init()
  ↓
LifecycleOrchestrator.startPhase('transport')
  KernelTransportRuntime
  IngressGateManager.init()                ← 只初始化，不放行
  IPCServer.start()
  ↓
LifecycleOrchestrator.startPhase('application')
  ApplicationRuntime
  SkillCatalogAppService
  Core / MCP Server / User SkillProvider.start()
  PerceptionAppService + ExtensionHostAppService
  ↓
LifecycleOrchestrator.startPhase('presentation')
  SurfaceRuntime
  ActionStreamManager.init()
  VisualCommandDispatcher.init()
  AvatarEngineController / AvatarShell sidecar / DesktopUIController
                                            ← 桌面壳与状态通道先亮起，但入站仍关闭
  ↓
LifecycleOrchestrator.startPhase('core-readiness', parallel)
  ├─ CognitionRuntime
  │   CognitionManager.start()             ← 启动 Python 子进程 + embedding 模型准备 + IPC 握手 + 配置/知识注入
  └─ CapabilityRuntime
      AudioService.initOptionalAudioProviders()
      AudioService.prepareRequiredAudioResources()
                                            ← TTS / ASR 并行完成检测、自动下载或明确不可用
  ↓
LifecycleOrchestrator.startPhase('extensions')
  ExtensionRuntime
  ExtensionManager.init()
  ExtensionManager.startAllExtensions()
  ↓
LifecycleOrchestrator.startPhase('organism')
  OrganismRuntime
  AttentionSessionManager.init(aiProxy, actionStream)
  LifeClockManager.init(aiProxy) → start()
  ↓
KernelTransportRuntime.openIngress()       ← required runtime 全部就绪后才接收入站
```

`app.ts` 只声明上述阶段计划；执行、耗时统计、启动报告和停机逆序由
`core/kernel/src/core/lifecycle/runtime/lifecycle-orchestrator.ts` 统一处理。阶段内部的跨层组装放在
`core/kernel/src/core/lifecycle/runtime/*`。这些 runtime 文件是 app-root delegate，不属于普通业务层，
允许跨层 import，但只能表达生命周期、依赖注入、readiness gate 与停机顺序。

`LifecycleOrchestrator` 是轻量编排器，不是动态依赖图框架：阶段顺序固定、依赖在代码中显式表达，
只有同阶段互不依赖的模块才允许 `parallel`。当前 `core-readiness` 并行启动 Cognition 与默认能力资源门，
因为二者都依赖基础设施和应用服务，但彼此不消费对方的启动结果。新增官方能力时优先新增 runtime module 或扩展现有 phase，
不得让普通业务模块反向依赖 orchestrator。

runtime module 返回统一的 `RuntimeReadinessSnapshot`：`runtime_id / owner / phase / state / blocking / summary`
是启动主线的正式就绪语言。`runtime.pretty.log` 只展示 `runtime_id:state` 这样的摘要，
完整路径、下载输出和进程细节进入 `runtime.log` 或 `data/observability/logs/processes/`。
这条契约让 Cognition、Audio、未来 Unity Avatar Shell、视觉和本地推理都能用同一套方式说明自己是否 ready、
是否降级、是否阻塞入站。

Avatar Shell 属于 presentation runtime：Kernel 总是先打开 `8082` 外壳网关，再按
`configs/system/surfaces.yaml -> avatar_shell.sidecar` 决定外部 Unity 身体是否由 Kernel 拉起。
当前默认 `launch_mode: managed`，标准构建产物位于
`data/packages/avatar-shell/windows/SelrenaAvatarShell.exe`，由 `pnpm avatar:build` 生成。
这是当前实现中的过渡路径，不是目标数据域边界：月见自身构建产物后续迁移到 `out/dev/avatar-shell/`，
安装时投影到 `resources/components/avatar-shell/`。历史目标安装结构见 [legacy 客户端安装形态](../legacy-guides/客户端安装形态.md)，迁移前本段继续如实记录代码现状。
`launch_mode: manual` 只保留为 Unity Editor 调试方式。无论哪种模式，Avatar Shell 不阻塞文字对话入站，但必须通过
`RuntimeReadinessSnapshot(runtime_id='avatar.shell')` 和 `shell_status` 明确暴露 ready / degraded 状态。
其中 `shell_hello` 只表示 Unity 进程和协议连接存在；只有 catalog 投影出的 Cubism prefab 与真实
`CubismAvatarModelDriver` 初始化完成后才允许发送 `shell_ready`。连接存在但模型 runtime 或 prefab 缺失时，
Kernel 投影为 degraded，不得用静态贴图预览冒充 Live2D ready。

桌面壳属于早期可见表面：它可以先显示连接、预热、缺资源等状态，但真实输入仍由 `IngressGateManager`
挡在 `openIngress()` 之前。这样用户不会在冷启动阶段面对空白桌面，也不会让 UI 在半启动态把消息注入认知链路。

模型准备属于启动编排的一部分，不属于第一次业务请求的副作用。默认启用且影响语义质量或交互闭环的资源
（例如 Cognition embedding、FunASR、当前默认 TTS）必须在对应 runtime 宣告 ready 前完成本地检测、自动下载或明确不可用。
大型可选能力（例如 GPT-SoVITS 身份声线、未来视觉、实验 TTS provider）可以懒加载或显式启用，但必须通过 readiness/status 明确告诉 UI 当前能力是否可用。
缺失资源的上游下载日志进入 `data/observability/logs/processes/`，Kernel 主时间线只保留资源门开始、完成、失败摘要。

### 启动日志边界

Kernel 是启动编排的中枢，也是主启动时间线的唯一叙述者。`runtime.pretty.log`
只展示适合人眼阅读的阶段、runtime module、耗时、状态与摘要告警；路径、完整启动计划、trace 细节等全量字段留在
`runtime.log`。Cognition 认知核内部的自述（如 `self_entity`、`kernel_bridge`、
`cognitive_loop` 的 info/debug）属于子进程时间线，写入 `data/observability/logs/processes/cognition.log`，
不再逐条镜像进 Kernel 主时间线。只有 Cognition 子进程的 warn/error 会被 Kernel 摘要转写为主日志事件。

这条边界避免“多个主体同时宣布启动成功”：Cognition 可以记录自己内部如何醒来，Kernel 只记录它作为中枢确认到的
readiness gate。开发者排查时先看 `runtime.pretty.log` 的主线，再按需进入 `processes/cognition.log`
查看认知核内部细节。

### Electron Desktop Surfaces 启动顺序

```
0. 开发期：Vite / tsc watch 可先启动，但 Electron 窗口必须等待 Kernel DesktopUIController(8083) ready
1. Electron main ready
2. DesktopShell 组装 TrayController + SurfaceRegistry + IPC bridge
3. 创建 Tray Resident                         ← 系统托盘成为常驻入口，使用 `assets/icons/selrena-tray.png`
4. 创建 Presence Surface                      ← 读取模型 registry；默认模型首选 Unity 时进入透明等待态；只有显式 Sprite / Web Live2D debug 后端才在 Electron 中绘制身体
5. 连接 Kernel DesktopUIController(8083)       ← 接收 thought / reply / emotion / shell_status / audio_status / audio_play
6. 用户点击托盘图标或菜单
7. 按需创建 / 显示 Control Center              ← 正常软件窗口：任务栏可见、frameless 自绘中文标题栏、可拖动、可最小化、不默认置顶；基础面板采用自然 light / dark 主题，以受控 accent token 提供强调，左侧导航优先承载状态 / 对话 / 配置，再进入语音 / 形象 / 记忆 / 扩展 / 诊断
```

窗口生命周期约定：Presence Surface 与 Control Center 的关闭按钮都只隐藏窗口；`window-all-closed` 不退出应用。真正退出必须走托盘菜单的「退出月见」，这样月见的“存在体”与“管理面板”不会被一个普通关窗动作误杀。退出不是 Electron 的局部 `app.quit()`：Desktop Shell 先通过 `shutdown_request` 请求 Kernel 关机，Kernel 关闭入站并严格逆序停止 runtime；`SurfaceRuntime` 先向 Unity 发送 `shutdown`，等待其自主退出，超时后才终止并回收受管进程树，随后关闭 Desktop UI WebSocket。Electron 观察到 Kernel 断连后才退出；开发期根 `pnpm dev` 进程组随之收束。Control Center 使用独立 renderer 入口，不通过 URL query 与 Presence 共享根页面。

开发期 `core/renderer/desktop/scripts/dev-electron.mjs` 只把 Vite 和 TypeScript watch 当作前端构建服务；Electron 窗口进程会等待 Kernel 8083 端口 ready 后再启动。Vite dev server 固定使用 `http://127.0.0.1:5173`，端口被占用时快速失败，不自动漂移。默认等待时间与模型资源准备同量级（20 分钟，可用 `SELRENA_KERNEL_READY_TIMEOUT_MS` 覆盖），避免首次 embedding / ASR / TTS 下载时 UI 先超时并反向杀掉 Kernel。

Unity / 未来 3D 身体属于 Avatar Shell runtime：它直接连接 Kernel 8082 并消费统一 AvatarShellFrame 协议，不再通过旧 `core/renderer/avatar` TS 桥进程。模型的首选后端由 Avatar catalog 决定；Electron Presence 是托盘存在体入口、静态 sprite 后端和显式 Web Live2D 调试面。目标后端为 Unity 时，Electron Presence 不渲染临时身体，而是切到透明穿透等待态，避免和 Unity 身体抢窗口、抢鼠标或闪烁切换。

当前正式外显由 DirectComposition Host 单独拥有窗口；Unity Player 只是隔离的 Render Worker。Worker 通过显式帧泵持续渲染 alpha surface，不依赖其容器窗口刷新，经过 GPU 导出坐标归一化后由渲染线程提交给 Host。Host 首帧 present 成功才算身体 ready；默认驻留与半身露出按平台 monitor work area 计算，不使用 Unity 分辨率推导桌面坐标。

Control Center 的形象页通过 Electron main 的受控诊断投影展示 Avatar Shell 状态：是否启用、启动模式、Unity 资产投影、受管构建产物、runtime 包目录与 Avatar 日志入口。Renderer 不直接读取 `configs/`、`data/` 或 Unity 工程文件；完整路径和打开动作由 preload 白名单 IPC 提供。

当前默认 Live2D 模型是 `youling`，采用现代 `model3.json/.moc3` 资产并以 `cubism4` 作为最低兼容代际。月见只接纳 Cubism4/5，二者共用 `cubism-unity` runtime 供应边界；模型清单声明格式，runtime catalog 声明本机官方 SDK 来源，Control Center 只展示诊断投影，Kernel 和 Desktop Surfaces 不写死角色或 Cubism SDK 小版本。

### 停机顺序（严格逆序）

```

`SurfaceRuntime` 内部的停机顺序也固定：先广播 Avatar Shell `shutdown` 帧，等待受管 Unity 子进程退出；若超过优雅期限，再终止该进程及其子树；之后才关闭 Avatar `8082` 网关、Desktop UI `8083` 网关与本地订阅。这样桌面壳的断连可以作为“身体已收束”的完成信号，不能反过来让 Electron 先退出并遗留 Unity。
OrganismRuntime
  → ExtensionRuntime
  → CapabilityRuntime
  → CognitionRuntime
  → SurfaceRuntime
  → ApplicationRuntime
  → KernelTransportRuntime
  → FoundationRuntime
  → EventBus / Logger close
```

### Python 认知核启动顺序（main.py）

```
1. DIContainer.init(config)                ← 装配 5 个 Provider + CognitiveLoop + Arousal
2. ExperienceRecorder.start()              ← 经历日志后台落盘（阶段 1，脊柱①）
3. start_metrics + start_tracer            ← 遥测三支柱写入器（阶段 3）
4. ArousalManager.start()                  ← 觉醒态机后台 tick（阶段 4）
5. CognitiveLoop.start()                   ← 认知循环九阶段后台 tick（阶段 5；5.9 默认开）
6. CognitionDatabase.connect() + 长期记忆/知识库加载
7. KernelBridge.start(bind_address)        ← ZMQ DEALER 接入 Kernel 内核
8. SelfEntity.wake_up()                    ← 月见醒来
```

逆序停机；CognitiveLoop 在 ArousalManager 之前停（避免读已关闭 arousal）。
