# 04 - 应用层（Application）

> 迁移材料：当前 Kernel 边界见 `docs/architecture/current/07-子系统当前视图/Kernel与Runtime.md` 与 `docs/architecture/implementation/Kernel与Runtime实现.md`。本页不再更新。

> 物理位置：`core/kernel/src/core/application/`  
> 定位：**系统编排与能力聚合**——唯一允许跨层组合的编排层

## 一、设计原则

- Application 层是**唯一可同时引用 Foundation、Infrastructure、Domain 的层级**
- 对外暴露 **Service（服务）** 和 **Capability（能力）** 两类组件
- Service 面向上层消费（Host / 外部 API），Capability 封装底层子系统（AI、音频、记忆等）
- Domain 层通过 `@protocol/ports` 定义的接口访问 Application 能力，由 `core/lifecycle/runtime/` 中的 app-root delegate 注入

## 三、服务详解（Services）

### 3.1 ExtensionHostAppService — 扩展宿主服务

**实现接口**：`IExtensionHostService`（定义于 `core/kernel/src/core/foundation/ports/extension-host.port.ts`）

**核心作用**：Host 层（ExtensionManager）获取一切系统能力的**唯一入口**。ExtensionManager 不直接 import Foundation 或其他 Application 模块，只通过此服务间接访问。

| 方法 | 委托目标 | 说明 |
|---|---|---|
| `getConfig()` | ConfigManager | 将 GlobalConfig 投影为精简的 IExtensionSystemConfig |
| `getRepoRoot()` | ConfigManager | 返回项目根路径 |
| `loadEnabledExtensions()` | ConfigManager | 读取启用扩展列表 |
| `createLogger(module)` | Logger | 工厂方式创建日志器 |
| `createStorage(extensionId)` | DBManager | 为扩展创建独立 K/V 存储实例 |
| `subscribeEvent(name, handler)` | EventBus | 封装事件订阅 |
| `publishExtensionEvent(...)` | EventBus | 发布扩展事件 |
| `publishDomainEvent(event)` | EventBus | 发布领域事件 |
| `injectPerception(event)` | PerceptionAppService | 注入感知事件 |
| `reportSceneAttention(...)` | EventBus | 发布 SceneAttentionChangedEvent |
| `registerAgent(extensionId, profile)` | SkillCatalogAppService | 将当前 SDK 的 SubAgentProfile 适配为 Extension Skill 并写入统一能力目录 |
| `createShortTermMemory(extensionId)` | ExtensionShortTermMemoryRepository | 创建扩展级短期记忆实例 |

### 3.1.1 Skill Plane — 技能平面

Skill Plane 是 Kernel 内部统一管理 agentic 能力的 Application 层设施。当前已落地最小骨架：

| 模块 | 说明 |
|---|---|
| `SkillCatalogAppService` | Application 层能力目录门面；上层查询能力清单、ExtensionHost 写入 Extension Skill 都应经过这里。 |
| `SkillProvider` | Core / Extension / MCP Server / User Provider 的统一生命周期接口，Provider 通过 `SkillRegistrationTarget` 写入目录。 |
| `SkillRegistry` | Core / Extension / MCP Server / User Skill 的底层统一注册表，并提供不暴露 handler 的 `SkillCatalogSnapshot` 能力目录快照。 |
| `SkillPolicyEngine` | 技能权限、风险、确认与审计策略入口；当前会拒绝 `contract_only` 技能与未接入确认通道的高风险调用。Skill 提供默认策略，Tool 可收紧覆盖。 |
| `SkillInvocationGateway` | 统一调用 Skill Tool / Resource / Prompt 的入口，调用前必须经过 `SkillPolicyEngine`。 |
| `providers/core/` | 月见自带 Minimal Universal Skills 的契约 Provider，当前注册 desktop / clipboard / notification / screen-context / confirmation 五类核心技能。 |
| `providers/mcp-server/` | 外部 MCP Server Provider；使用官方 MCP TypeScript SDK 托管 stdio / Streamable HTTP / WebSocket 连接，完成 initialize、tools/resources/prompts 枚举、目录刷新、调用与停止回收。 |
| `providers/user/` | 用户自定义 Skill Provider 的生命周期骨架；真实脚本配置、沙箱与审计待接入，不注册假技能。 |
| `extension-skill-provider.ts` | 将 `ctx.ports.agents.registerSubAgent()` 注册的 SubAgentProfile 转成 Extension Skill。 |

`DEFAULT_SKILL_PROVIDERS` 是 Kernel 启动时的 Provider 组装点，当前包括 Core、MCP Server、User 三个 Provider。Core Skill Provider 当前为 `runtime_status=contract_only`：它把月见自带能力的边界、名称、参数、风险与确认策略放入统一注册表，但不会伪装成可执行的桌面能力；真实平台服务接入前，调用网关会明确拒绝执行。MCP Server Provider 会读取 `SkillPlaneConfig.mcp_servers` 中 `enabled=true` 的连接目标，并在后台建立 stdio / Streamable HTTP / WebSocket MCP 会话；只有 initialize 与能力枚举成功才注册 `runtime_status=ready` 的 Skill。每个 Server 对应一个 Provider 归属的 Skill，远端断连、超时或刷新失败会撤销该 Skill 并留下 provider 降级状态，不产生假 handler。当前扩展 SDK 的外部入口仍叫 `registerSubAgent()`，这是兼容现有扩展作者的 API 面；Kernel 内部不再使用 `SubAgentRegistry` 作为事实源。

`SkillCatalogSnapshot` 是 planner、调试面板和未来 Control Center 能力页的共同发现入口。它只包含技能描述、provider、tools/resources/prompts 摘要、policy 与 metadata，不导出 tool handler 或扩展内部对象，避免把运行时执行细节泄露给上层消费方。

依赖方向上，Host 层和后续 planner 不应直接触碰 `SkillRegistry.instance`。需要读取目录或注册 Extension Skill 时走 `SkillCatalogAppService`；`SkillRegistry` 只保留为 Skill Plane 内部事实表。

### 3.2 PerceptionAppService — 感知应用服务

**核心作用**：所有感知事件的统一入口，协调入站防护、路由、转录、音频和注意力会话。

```
PerceptionEvent (来自插件)
  ↓
processIngress()
  ├── IngressGateManager.admit()   → 速率限制/熔断/就绪检查（自动透明）
  ├── resolveScene()               → SceneRoutingManager 解析场景
  ├── appendSceneTranscript()      → 持久化对话记录
  └── AttentionSessionManager.ingest() → 防抖+批处理+AI推理
  ├── complete(true)  ← 成功时上报
  └── complete(false) ← 失败时上报（驱动熔断器）
```

| 方法 | 说明 |
|---|---|
| `processIngress(event)` | 感知事件主入口 |
| `resolveScene(request)` | 场景路由解析 |
| `appendSceneTranscript(pluginId, entry)` | 追加对话转录 |
| `synthesizeSpeech(request)` | TTS 合成 |
| `recognizeSpeech(request)` | ASR 识别 |
| `getAudioStatus()` | 返回 ASR/TTS provider 状态快照，供 DesktopUIController 下发 `audio_status` |

## 四、能力详解（Capabilities）

### 4.1 AIProxy — AI 能力门面

**单例**，实现 `IAICapabilityPort` 接口。所有模块通过此门面与 Cognition 认知核通信。

| 方法 | 说明 |
|---|---|
| `isReady` | Cognition 认知核是否已就绪 |
| `sendPerceptionMessage(request, traceId?)` | 发送感知消息给 AI |
| `cancelPerception(request)` | 取消进行中的推理 |
| `requestAgentPlan(request)` | 请求代理规划 |
| `sendLifeHeartbeat(request)` | 发送生命心跳 |

内部委托给 `CognitionManager.instance`。

### 4.2 CognitionManager — Python 子进程管理

**单例**，负责 Cognition 认知核的完整生命周期。

**启动流程**：

```
start()
  ├── spawn() Python 子进程
  │   入口：core/cognition/.venv/{Scripts/python.exe (Win) | bin/python (Mac/Linux)}
  │        + src/selrena/main.py
  │   环境变量：SELRENA_IPC_BIND_ADDRESS, SELRENA_CONFIG, LOG_DIR
  ├── waitForReady()        → 轮询 IPC 连接状态
  ├── initAIConfig()        → 推送配置到 Python
  └── initKnowledge()       → 推送知识库到 Python
```

> 长期记忆同步（原 initMemory）已随蓝图阶段 2「持久化收归认知核」移除 ——
> 长期记忆直接写认知核本地 `cognition.db`，不再经 IPC 同步。

**日志解析**：支持 structlog JSON 和 Python stdlib logging 两种格式，自动转换为 TS Logger 输出。

### 4.3 ActionStreamManager — 动作流管理器

**单例**，实现 `IActionStreamPort` 接口。采用**纯信号模型**，仅发布流生命周期事件，不产生中间帧数据。渲染层（Live2D）根据 Started/Cancelled 信号驱动本地思考状态；Completed 目前只代表感知投递链路结束，不等同于 Cognition 已生成回复。

**流生命周期**：

```
startThinkingStream(sceneId, traceId)
  → 发布 ActionStreamStartedEvent
  → DesktopUIController 转译为 AvatarShellDownstreamFrame{kind:'thought', active:true}
  ↓
[等待 Cognition Act 通过 ActionCommand 推出真实回复]
  ↓
ChannelReplyEvent
  → DesktopUIController 先转译为 AvatarShellDownstreamFrame{kind:'thought', active:false}
  → 再发布 reply / emotion 下行帧；reply 携带完整 text 与可分条投递的 messages
```

**中断路径**：

```
cancelStream(sceneId, traceId, reason)
  → 发布 ActionStreamCancelledEvent
  → DesktopUIController 转译为 AvatarShellDownstreamFrame{kind:'thought', active:false}
```

> **设计决策**：早期版本使用 `setInterval` 每 80ms 发布 `ActionStreamChunkEvent`（含 gaze/motion/expression 帧），造成日志洪泛且无实际语义。现已移除全部中间帧机制，渲染层应在收到 `Started` 信号时启动本地动画循环，收到 `Completed`/`Cancelled` 信号时停止。

### 4.4 SceneRoutingManager — 场景路由

将消息来源映射为唯一的 `scene_id`：

```

// by_actor 策略额外追加

```

| 场景类型 | 路径 |
|---|---|
| `group_scene` | `data/state/extensions/transcripts/scenes/group/{groupId}.md` |

### 4.6 AudioService — 音频服务

TTS / ASR 的本体能力入口，不属于 Extension。当前落地边界为：
- `core/kernel/.../audio/` 负责调度、provider 顺序、失败日志与 trace；默认不制造静音伪结果。
- `engines/audio/` 已通过 stdio provider 接入 Kernel：TTS 默认路线为 Piper；ASR 默认路线为 FunASR；GPT-SoVITS v2（停云模型）保留为显式启用的身份声线，不再默认占用实时 TTS 主线。
- TTS 当前默认走 `official-audio:piper`；具体声线、artifact 相对路径与下载 URL 由 Audio Engine 的资源 catalog 声明，模型落入统一 `data/models/` 根。Windows SAPI 只作为工程兜底。设置 `SELRENA_TTS_PROVIDER=gpt-sovits-v2` 时 GSV 才作为首选 provider，设置 `SELRENA_ENABLE_GSV_TTS=1` 时只作为可选 provider 加入。
- 官方音频 provider 在 Kernel 内按能力拆成独立 lane：TTS lane 使用 `uv run --extra tts` 启动 `engines/audio` 并预热 Piper；ASR lane 使用 `uv run --extra asr` 启动 `engines/audio` 并预热 FunASR。GSV sidecar 仅在显式启用时启动，避免默认对话路径被重型克隆声线拖慢。
- 模型准备属于能力层资源门，不属于第一次业务请求的副作用。`CapabilityRuntime` 会调用 `AudioService.prepareRequiredAudioResources()`，等待默认启用的 TTS / ASR 完成本地检测、自动下载或明确不可用。Audio Engine 从 `resources.json` 解析所选 resource id，再准备对应 artifact；Piper / FunASR 适配器不保存具体声线文件名、远端仓库 ID 或下载 URL。`engines/audio` 的 stdout 只输出 JSON 协议响应，第三方库日志统一隔离到 stderr，再由 Kernel 写入 `data/observability/logs/processes/audio-tts.log` / `audio-asr.log`，避免下载进度或 warning 污染主启动时间线。
- GSV 声线模型按 Audio resource catalog 的 `modelDirectory` 映射到 `data/models/voice/gpt-sovits/<voice-id>/`，不进入 Git；当前具体声线只是 catalog 数据，不进入 provider 代码。
- GPT-SoVITS 以 sidecar HTTP API 形式接入，模型加载和显存生命周期不进入 Kernel 进程；`engines/audio` 负责探测和拉起 sidecar。
- `data/packages/gpt-sovits/` 是默认 GSV 上游包目录，可放 `api_v2.py`、独立 `.venv` 与上游配置，不进入 Git。
- `data/cache/audio/tts/` 是默认 TTS 缓存目录，按文本与 provider 路线复用已生成短句；`data/artifacts/tts/` 只用于显式输出或手动验证。
- `AudioService.prepareRequiredAudioResources()` 在 Kernel 的 `core-readiness` 阶段与 Cognition 并行执行，是默认语音能力的 readiness gate；完成后返回 TTS / ASR 的 `ready` 或 `unavailable` 摘要，并投影为统一 `RuntimeReadinessSnapshot`。TTS / ASR 失败会显示为 `degraded`，不阻断文字对话，但必须通过 `audio_status` 与启动报告暴露给 Control Center。
- `AudioService` 维护本体音频能力的缓存快照。Control Center 首次连接时只接收 `getCachedStatus()`，不会因 UI 打开而反向触发 Piper / FunASR 冷启动；能力层预热完成后由 `CapabilityRuntime` 主动广播 `AvatarShellDownstreamFrame{kind:'audio_status'}`。`getStatus()` 只在资源门允许后探测官方音频引擎 `health`，用于刷新 provider 顺序、首选 provider、不可用原因。
- `DesktopUIController` 收到桌面 `ChannelReplyEvent` 后，会先下发 `reply` 文字帧，再异步调用 `AudioService.synthesizeSpeech()` 生成 TTS，并以同一 `trace_id` 下发 `AvatarShellDownstreamFrame{kind:'audio_play'}`。音频是回复的外显媒介，不进入 `ChatMessage` 内容块。
- `DesktopUIController` 收到桌面 `AvatarShellUpstreamFrame{kind:'audio_input'}` 后，会把录音落到 `data/artifacts/asr/`，调用 `AudioService.recognizeSpeech()` 走 FunASR，并下发 `AvatarShellDownstreamFrame{kind:'audio_transcript'}` 作为识别回执；识别成功的文本会被注入 `PerceptionAppService`，后续与普通文字消息同链路。
- `VoiceRecorder` 在 renderer 只承担采集和音频清洗：浏览器回声消除 / 噪声抑制 / 自动增益，Web Audio 高通、低通、动态压缩，以及轻量 PCM 噪声门与归一化；模型识别仍只由 Kernel → AudioService → FunASR 执行。
- Control Center 的识别态带前端守护超时；Kernel 正常路径下成功或失败都必须返回 `audio_transcript`，避免 UI 无限停在“识别中”。
- Control Center 的配置页通过 renderer → preload → Electron main 白名单 IPC 读写受控配置投影：推理参数、注意力节奏、embedding、provider、人格锚点和桌面表面端口。renderer 不直接访问文件系统，不编辑 `configs/secrets/`，也不提供任意 YAML 编辑器；核心配置保存后以重启 / 下一次启动为完整生效边界。UI 风格遵循白 / 黑 / 青 / 蓝的 MD3/Monet 浅色桌面管理舱约束，见 `docs/guides/桌面渲染开发指南.md`。
- Control Center renderer 已拆成稳定的管理面板分层：`components/ControlCenter.tsx` 只做页面路由和 store 投影；`components/control-center/model.ts` 收口页面模型、状态标签、健康摘要和配置投影类型；`ui.tsx` 收口 Shell、中文标题栏、可调宽导航与通用控件；`pages.tsx` 承载总览 / 配置 / 语音 / 形象 / 记忆 / 扩展 / 诊断页面。后续新增页面或配置项必须进入这一结构，不再回到单文件巨型组件。
- Control Center 的形象页通过 `ui:get-avatar-shell-diagnostics` 消费 Electron main 聚合的 Avatar Shell 本机诊断投影，包括 sidecar 配置、Unity 资产投影、构建产物、按模型格式收集的 runtime 状态、runtime 包与日志入口。renderer 不直接读路径，不探测 Unity 进程，也不把构建细节写入 Zustand store。
- `audio_play` payload 默认携带 `file://` URI；小于 8 MiB 的 TTS 输出会同时内联 `audio_data`，让 Electron renderer 在 dev / packaged 两种模式下都能稳定播放。Control Center 文字回复会按自然短句触发渐进 TTS，先播放已完成短句，再继续生成后续短句。
- `native/` 是 C/C++ 性能底座，只承载 ABI / FFI / 热路径加速，不直接拥有模型编排；默认不启用。
- Edge-TTS / whisper.cpp 仅在 `SELRENA_ENABLE_AUDIO_FALLBACKS=1` 时作为排障备用 provider 加入；Windows SAPI 是工程兜底，不再是默认声带。后续可继续评估 sherpa-onnx Piper / streaming GSV，但默认主线必须先保持 Piper + FunASR 的稳定闭环。

> **记忆同步已移除**（月见架构蓝图 阶段 2）：长期/短期记忆的持久化已收归
> 认知核 `data/state/cognition/cognition.db`，内核不再承载记忆持久化。原 `MemorySyncManager`
> 与 `MemoryRepository` 已删除。详见 [阶段2-数据持久化设计](../../history/architecture-decisions/阶段2-数据持久化设计.md)。

### 4.8 AvatarEngineController — Avatar Shell - Unity 适配器

**单例**，管理外部 Unity Avatar Shell 的 WebSocket 接入（端口 8082，蓝图 §2.2）。

**职责**：
- 接收 Unity 客户端的 `shell_hello` / `shell_ready` / `pong` / `animation_complete` / `error` 等 `AvatarShellUpstreamFrame`
- 订阅 `VisualCommandDispatchEvent` / `ActionStreamCompletedEvent`，翻译为统一 `AvatarShellDownstreamFrame` 后下发到 Unity 端
- Unity 心跳超时 / 断开 / 连入时**事件驱动**地 publish `AvatarShellStatusChangedEvent`（阶段 8.2 起，
  取代 1500ms setInterval 轮询，消 smell E）
- 按 `configs/system/surfaces.yaml -> avatar_shell.sidecar` 决定外部 Unity Shell 是否由 Kernel 拉起：
  `manual` 等待手动启动，`managed` 通过 `AvatarShellProcessManager` 启动、记录进程日志并在停机时回收；停机先发送
  `shutdown` 帧给 Unity，等待优雅退出，超时才终止并回收 Windows 进程树
- 向 `SurfaceRuntime` 返回 `RuntimeReadinessSnapshot(runtime_id='avatar.shell')`，让主启动时间线能清楚表达
  Unity Shell 是已连接、等待手动启动、进程已启动等待连接，还是降级失败

**当前协议**：外部 Avatar Shell 只讲 `AvatarShellUpstreamFrame` / `AvatarShellDownstreamFrame`。
旧 `{type:'visual_command'|'audio_push'|'ping'}` 私有 wire format 已删除。`VisualCommand` 只作为
Kernel 内部过渡事件载荷存在，并在本控制器边界翻译成统一帧。

### 4.9 DesktopUIController — Desktop Surfaces 适配器

**单例**，管理 Electron Presence Surface 与 Control Center 的 WebSocket 接入（端口 8083）。它承载管理、对话、状态和轻量存在表面；正式复杂身体由 Unity Avatar Shell 承担。Desktop Surfaces 与 Avatar Shell 都属于本体渲染层，不是普通扩展，但二者不共享窗口所有权或模型生命周期。

**职责**：
- 收 Electron 上行帧（阶段 8.3 起按 `AvatarShellUpstreamFrame.kind` 分发）：
  * `chat_input` → 合成 `PerceptionEvent`（`source: 'desktop-ui:user'`）注入 `PerceptionAppService`
  * `audio_input` → 录音落盘 → `AudioService.recognizeSpeech()` → `audio_transcript` 回执 → 文本注入 `PerceptionAppService`
  * `shell_hello` → 日志记录 capabilities（8.7+ Avatar Shell Registry 用于路由）
  * `shutdown_request`（仅 `requested_by: desktop-shell`）→ 请求 App Root 执行全局有序停机；不是 Electron 自己退出
  * `heartbeat` / `ping` → 无副作用
- 订阅 `ChannelReplyEvent`，按 `target_channel.startsWith('desktop-ui')` 严格匹配（阶段 8.2 起取代脆性
  `trace_id.startsWith('ui_')` 路由；阶段 8.4 已删除旧前缀 fallback）
- 下行帧切 `AvatarShellDownstreamFrame`（阶段 8.3，消 smell B/C/H）：
  * `kind=thought` —— 月见思考态（ActionStreamStartedEvent 开启；ChannelReplyEvent 或 ActionStreamCancelledEvent 关闭）
  * `kind=reply` —— 月见回复正文（`reply.text` 为完整正文，`reply.messages` 为可投递切片；正文不承载情绪标签）
  * `kind=emotion` —— **独立**情绪帧（与 reply 分帧，消 smell H）
  * `kind=shell_status` —— shell 接入状态
  * `kind=audio_status` —— ASR/TTS 本体能力状态，Control Center 用于展示 Piper / FunASR / GPT-SoVITS / fallback 当前可用性；Electron main 会缓存最后一次状态，renderer 首屏通过白名单 IPC 读取缓存，避免订阅时序导致状态丢失。DesktopUIController 首连只发送 AudioService 缓存快照，不主动探测引擎进程。
  * `kind=audio_transcript` —— ASR 识别回执，供 Control Center 更新语音输入状态并展示用户语音文本
- 订阅 `AvatarShellStatusChangedEvent`（来自 AvatarEngineController），立即转发到 Electron 端
- 广播下行帧时通过 protocol runtime helper 记录 `AvatarShellFrameClass`：
  * `expression_flow` —— `thought/reply/emotion`
  * `embodiment_control` —— `audio_play/expression/motion/lip_sync/parameter/idle`
  * `shell_lifecycle` —— `shell_status/audio_status/load_scene/unload_scene/ping`

**8.4 收口状态**：
- ChannelReplyEvent 缺 `target_channel` 时不再回落 `trace_id.startsWith('ui_')` 前缀判定
- 上行帧只接受 `AvatarShellUpstreamFrame.kind`，不再尝试旧 `type` 字段
- `chat_input` 顶层 `text` 与 ChatOutput 内联 `emotion_snapshot` 仍保留兼容，待调用方全部切净后再删

**Renderer public 资产规则**：
- Presence / Control Center 的 HTML 入口与 renderer 代码都必须使用相对 public 资产路径，经
  `resolvePublicAssetUrl()` 解析：Vite dev 走 `/public/assets/...`，built Electron 走相对 HTML 的 `assets/...`。
- 禁止在 renderer 运行时直接 fetch 或引用 `/assets/...` 站点根路径；该写法只在 Vite dev server 下成立，
  built Electron 的 `file://` 模式会解析到磁盘根路径，导致 Live2D / Q版 sprite / Cubism Core 静默丢失。
- Avatar 渲染层按 Shell / Interaction / Backend / Concrete Renderer 分层：Electron Presence Surface 是本地轻量占位、托盘存在体入口和显式调试入口；正式复杂身体由外部 `core/renderer/avatar-unity/` Avatar Shell 接管。Avatar Backend 消费 `AvatarShellFrame` 投影；Electron 不再保留第二条 Web Live2D 身体渲染链路。
- `assets/avatar/models.json` 是内置角色唯一 catalog：`kind` 描述资产类型，`preferredBackend` 描述首选后端，`modelPath`、`emotionMapPath` 与 `interactionMotions` 描述模型自身资源和行为。Desktop 只按稳定 model id 解析条目；正式后端失败时只使用 catalog 的显式 `fallback`，不再内置某个角色作为静默兜底。
- Presence 加载模型清单时保持透明空 surface，不再先显示 Q版临时身体；目标 backend 是 Unity 时，Electron Presence 标记 `unity-pending` 并切到透明穿透等待态，不渲染静态占位。Unity Shell 在线后，Electron Presence 继续让位，不与 Unity 同时显示复杂身体。
- Electron main 维护 `PresenceInteractionPolicy`：正式 Unity 身体由 Composition Host 管理命中与窗口；静态 Sprite 降级才使用 `alpha-shape`，并将净化后的 alpha 几何写入 `setShape()`。默认只记录异常日志，设置 `SELRENA_SURFACE_DEBUG=1` 才输出 surface 细节。
- Presence 的形象模型与显示大小走同一份 `AvatarAppearanceSettings`：Control Center 通过 preload 白名单 IPC 写入，Electron main 持久化到 `data/state/renderer/avatar-appearance.json` 并广播给所有 renderer 窗口。Unity 的 `placement_id` 只驱动完整透明表面的驻留比例与停靠位置；模型始终以 `contain` 方式完整适配相机，不存在半身相机倍率或垂直焦点。显示大小改变完整表面尺寸与对应输出采样，Avatar Backend 不在固定窗口里二次放大模型。
- Unity Avatar Shell 位于 `core/renderer/avatar-unity/`，通过 Kernel `8082` 消费 `AvatarShellDownstreamFrame`、上报 `AvatarShellUpstreamFrame`。Unity 端按 `AvatarShellClient` / `SelrenaLive2DController` / `CubismAvatarModelDriver` 分层：协议连接、身体语义和具体模型 runtime 分离。`assets/avatar/models.json` 经同步脚本生成 StreamingAssets 元数据与 Resources 模型投影，Cubism SDK 生成 prefab，运行时只由 catalog `resourceKey` 加载。`shell_hello` 只证明进程与协议在线；`shell_ready` 由 Composition Host 首帧呈现事件经 `AvatarPresentationController` 单向传播到 `AvatarShellClient`，并且只在模型已初始化、hello 已发送和连接仍有效时发布。Unity Render Worker 使用隐藏窗口，Player 保留 framebuffer alpha 并运行于 Linear 色彩空间；Composition Host 是唯一正式可见窗口。Kernel 会把已连接但首帧尚未呈现单独投影为 degraded。Control Center 仍归属 Electron / React Desktop Surfaces：它是配置、状态与诊断工作台；Unity 只负责高性能身体渲染，不承载管理 UI。
- Unity Avatar Shell 现以 `AvatarBehaviorRuntime` 作为身体行为入口，以独立 `AvatarActionScheduler` 持有可保持动作的权威集合。动作请求使用显式 `trigger / activate / deactivate`，Scheduler 校验模型声明、`manualOnly`、前置依赖、互斥组与反向关闭依赖，再调用模型驱动；驱动会确认目标 expression / motion 实际存在且可执行，失败以 `rejected` 上报，不能把未生效命令记为已开启。每次处理后 Shell 上报完整 `avatar_action_state`。Kernel 只发布事件并投影最后快照，Control Center 不再靠“刚才点过哪个按钮”猜测状态；一次性动作仍显示为命令，可保持动作按类别显示为开关并明确开启、关闭、依赖或互斥原因。
- `actionsPath` 是模型私有的能力事实源，遵循 `assets/avatar/avatar-actions.schema.json`。`sync-unity-assets.mjs` 在生成 Unity 投影前校验版本、稳定 ID、目标类型、toggle 规则、依赖引用、重复项与依赖环；非法清单会阻断构建，而不是把错误推迟到用户点击控制面板时。
- 模型 `behaviorPath.baseline` 声明眨眼、呼吸、静态姿态与低频 idle，`behaviorPath.gaze` 声明注意力参数映射。`CubismBodyParameterMixer` 是头部、眼球与身体连续参数的唯一写入点：它在 Expression / Motion / blink / harmonic 之后、Physics 之前合并基线和注意力，因此头身变化会继续通过模型层级与 Physics 带动手臂、衣袖、衣发和挂件产生从属摆动。Runtime 只使用模型实际声明的 Parameter、Motion、Expression 与 Physics；未声明的独立手臂或姿态能力不会被猜测或伪造。鼠标只有发生有效位移时才短暂取得注意力通道，停止移动后经过保持与释放阶段平滑回到空闲基线。
- 当前行为主线已拆除旧 `AvatarBehaviorDirector`。`AvatarBehaviorRuntime` 是 Unity 内部组合根：`AvatarBehaviorEvaluator` 把语音脉冲与本地鼠标注意力采样为不可变 `AvatarBehaviorFrame`，`AvatarIdleMotionScheduler` 是作者 idle motion 的唯一生命周期入口，`AvatarActionScheduler` 处理离散动作与可保持表情状态；模型驱动每帧只通过 `ApplyBehaviorFrame()` 消费连续结果，不重新解释鼠标、口型或窗口输入。这样身体基线、视线和口型不再由多个控制器并列直接写入 Cubism。
- 通用的 `BehaviorRequest -> 通道租约仲裁 -> 分阶段 Blend Graph -> ResolvedAvatarFrame` 仍是蓝图中的后续扩展：当 Cognition、用户、场景和扩展需要并发覆盖同一身体通道时，再在现有 `AvatarBehaviorFrame` 边界之前引入租约与解析器。当前不为未出现的来源建立空队列或假状态机；不得恢复 Director，也不得让 UI、Cognition 或扩展直接写 Cubism 参数。
- Presence 与 Control Center 的公开显示倍率统一为 `50%~250%`，首次默认 `120%`。拖动滑块只更新本地预览并以有限频率发送，手势结束强制提交最终绝对倍率；Composition Host 按自然基准尺寸计算窗口，不从当前窗口尺寸反向累乘。
- Desktop 两类窗口均显式允许系统桌面捕获。发布配置使用 `productName=Selrena`、`Selrena.exe` 与 `com.glimmer.cradle.desktop` AppUserModelID；开发期仍由 `electron.exe` 承载，因此 NVIDIA 等第三方工具可能把它归类为 Chrome/Electron，这不是通过关闭 Chromium 安全策略解决的问题。
- 正式身体外显由 `AvatarCompositionHost` 承担：Unity Worker 使用显式帧泵把相机持续渲染到 alpha RenderTexture，不依赖其 OS 容器窗口是否可见；GPU 导出纹理统一 Unity / DXGI 的纵轴约定后，通过 `native/` 的通用 Composition Host C ABI 交给平台合成器。Windows 适配器使用同一 D3D11 device 将导出纹理复制到 DirectComposition 的 premultiplied-alpha swap chain；宿主 HWND 使用 `WS_EX_NOREDIRECTIONBITMAP`，视觉树是唯一画面源，resize 只更新窗口 bounds 与 visual transform，不暴露普通窗口黑底。Unity 项目启用 framebuffer alpha 与 Linear 色彩空间，Worker 容器窗口使用 `HIDEWINDOW` 完全隐藏，不作为第二个透明或黑底窗口留在桌面。至少一帧成功进入视觉树后 Host 才报告 ready。渲染纹理可按 `renderScale` 高于实际宿主尺寸生成，再由 DirectComposition 二维变换映射到窗口，专门抵抗 DPI 缩放与小尺寸驻留下的边缘发虚。`AvatarPresentationController` 只管理构图、位置状态和模型交互 hull；默认停靠由 Host 按真实 monitor work area、可见高度和边距计算，避免 Unity 分辨率与多显示器/DPI 坐标混用。透明区由 Host 返回 `HTTRANSPARENT` 继续交给桌面，模型 hull 内由原生窗口提供拖动。
- 拖动期间只移动 Electron main 持有的单一 Presence 窗口；Live2D renderer 不暂停、不恢复、不重算 layout，也不参与窗口位置计算。Electron main 以拖动起点和当前绝对屏幕坐标计算窗口位置，不在 renderer 累加 delta。`setShape()` 在 Windows 上会裁剪视觉，因此动态 Live2D 不使用 shape 裁剪；像素级透明交互交给 Unity / 原生身体层。拖动 IPC 是 fire-and-forget 事件，不走 request/response 式 invoke，避免高频 Promise 排队造成闪烁或拖动放大。
- `Live2DAvatar` 按模型条目声明的 `live2dVersion` 与 `runtimeScriptPath` 加载 Web 调试 runtime；组件不猜 Cubism core 文件名，也不按 model id 分支动作。
- `assets/avatar/models.json` 等 manifest 写相对 public 路径；进程内需要绝对路径时由各自边界统一解析。
