## 零、Canonical Data Layout（当前事实源）

`data/` 不是缓存杂物间，而是月见连续性与可观测性的物理投影。当前 canonical layout：

```text
data/
├── state/
│   ├── cognition/
│   │   ├── cognition.db          # Cognition 认知核主库：长期记忆 / 知识库 / 向量 / 记忆图谱
│   │   └── dead_letters.db       # Cognition 侧 DLQ：认知核事件失败审计
│   ├── kernel/
│   │   └── kernel.db             # Kernel 内核基础设施库：扩展宿主数据 / TS DLQ
│   ├── renderer/
│   │   └── avatar-appearance.json # 桌面外显本地状态：形象模型 / 显示大小
│   ├── experience/
│   │   ├── segments/             # Moment append-only 分段
│   │   ├── snapshots/            # 经历投影快照；坏快照隔离到 snapshots/corrupt/
│   │   └── archive/              # 冷归档
│   └── extensions/
│       ├── transcripts/          # 扩展侧用户可读转录 / 导出
│       └── storage/              # 后续文件型扩展存储
├── observability/
│   ├── logs/                     # runtime.log / runtime.pretty.log / runtime-error.log / processes/
│   ├── metrics/                  # metrics-*.jsonl
│   └── traces/                   # spans-*.jsonl
├── blobs/                        # 图片 / 音频 / 附件等大块二进制内容
├── models/                       # 用户下载或导入的模型权重
│   ├── embedding/                # Cognition embedding 模型，如 m3e-small
│   └── voice/                    # 语音模型，如 Piper 与 GPT-SoVITS 停云
├── packages/                     # 可重装外部运行包：GPT-SoVITS、FFmpeg、whisper.cpp 等
├── artifacts/                    # 可再生运行产物：临时音频、子进程输出
├── cache/                        # 可清理缓存
├── backup/                       # 迁移 / 升级 / 手动导出的安全快照
└── legacy/                       # 一次性迁移源；不可作为新写入目标
    └── cognition_core.db
```

规则：

- `data/state/cognition/` 只属于 Cognition；Kernel 不直接读写。
- `data/state/kernel/` 只属于 Kernel 基础设施；长期记忆不回流到这里。
- `data/state/renderer/` 保存桌面外显的用户本地状态，例如 `avatar-appearance.json`；它不是模型资产，不进入 `assets/`，也不是可随意丢弃的运行产物。
- `data/state/experience/` 是经历之流，一等公民，不塞进任何数据库目录。
- `data/observability/` 承载日志、metrics、traces 三支柱；metrics/traces 不再挂在 `logs/` 下面。
- `data/observability/logs/runtime.log` 是机器 JSONL 事实源；`runtime.pretty.log` 是人读摘要时间线，只保留 phase、runtime_module、耗时、状态与错误等主线字段；`processes/` 存放 Cognition、音频引擎等子进程内部日志，例如 `cognition.log`、`audio-tts.log`、`audio-asr.log`。
- `data/blobs/`、`data/models/` 是可能较大的本地资产；是否备份由用户策略决定。
- `data/artifacts/` 与 `data/cache/` 可删除、可重建；`data/packages/` 可重装；不得再出现仓库根 `runtime/` 或 `tmp/`。
- `data/legacy/` 只读迁移源；一旦迁移完成，不允许新代码继续写入。

## 一、为什么这一章存在

记忆和日志原本散落在 02-基础设施层、04-应用层、07-Cognition 认知核里，但它们有一个共同性质：

- **跨进程跨语言**——同一个数据结构在 Python 写一次、TS 写一次、协议层 schema 又写一次
- **横切所有业务层**——任何模块都用，但又不属于任何一层
- **代际数据**——长期记忆和日志归档要活得比单次进程长

把这两个模块抽出来放在 §08，是为了：

1. **强制单一事实源**——所有限额、类型、字段名只能在一个地方定义
2. **让"月见的连续性"在文档里可被指认**——记忆持续性是 OC 身份的物理保证，必须有专门一节
3. **trace_id 等横切观测能力的归宿**——之前没有专属位置

## 三、目标架构

### 3.1 核心设计原则

1. **协议层是唯一事实源**——所有跨语言枚举 / 数据形状从 `protocol/src/schemas/` 单点定义；`pnpm sync:contracts` 生成 TS 与 Python 两端产物。
2. **Cognition 是认知持久化所有者**——长期记忆、知识库、向量、记忆图谱写入 `data/state/cognition/cognition.db`；Kernel 不再承载长期记忆同步库。
3. **配置先于硬编码**——任何阈值、限额、衰减率必须从 `configs/` 对应运行域读取，硬编码的只是"在配置缺失时的兜底默认"。
4. **可观测性是基础设施**——trace_id / event_id 在协议层就被规定，Logger 自动注入，无需调用方手动传

### 3.2 记忆模块目标形态

#### 3.2.1 类型系统（解决 M1, M11）

**单一事实源在 `protocol/src/models/memory.ts`**：

```ts
export const LongTermMemoryType = {
  EPISODIC: 'episodic',         // 一次具体的交互事件
  PREFERENCE: 'preference',     // 用户的偏好/习惯（永久；不衰减但有 cap）
  FACT: 'fact',                 // 关于用户/世界的事实陈述
  MULTIMODAL: 'multimodal',     // 含图像/音频引用的复合记忆
} as const;

export type LongTermMemoryType = typeof LongTermMemoryType[keyof typeof LongTermMemoryType];
```

> **决策点**：以 Python 的 4 类为准。Python 是认知层，类型由它的语义需求驱动；TS 当前的 GENERAL / INTERACTION 是文档与代码不同步的产物，删掉。

当前 codegen 入口为 `protocol/build/{gen-ts.ts, gen-py.py}`，统一通过 `pnpm sync:contracts` 运行；旧 `scripts/sync_contracts.py` 已在阶段 P.9b 删除。

#### 3.2.2 Embedding 持久化（解决 M2）

**SQLite 端新增列**：

```sql
ALTER TABLE long_term_memory ADD COLUMN embedding BLOB DEFAULT NULL;
ALTER TABLE long_term_memory ADD COLUMN embedding_model TEXT DEFAULT NULL;
ALTER TABLE long_term_memory ADD COLUMN embedding_dim INTEGER DEFAULT NULL;
```

`embedding` 列以 Float32 little-endian 二进制存储；`embedding_model` 记录"是哪一个模型生成的"（比如 `m3e-small@v0.1`），换模型时自动失效旧向量。

**协议层加字段**：

```ts
// protocol/src/models/memory.ts
export interface LongTermMemoryRecord {
  // ...
  embedding?: Float32Array | null;
  embedding_model?: string | null;
}
```

**写入路径**：

```
Python LongTermMemory.add() →
  调用 EmbeddingEngine.encode() 生成向量（若可用） →
  随 MemorySyncEvent 一起发出 →
  MemoryRepository 持久化 embedding bytes
```

**读取路径**：

```
Kernel 启动 → MemoryRepository.loadAll() →
  通过 IPC 把 (id, content, embedding, embedding_model, ...) 全量推 Python →
  Python LongTermMemory.init_from_kernel(records) →
  按 embedding_model 字段判断：与当前模型匹配 → 直接复用；不匹配 → 入"待重算队列"
```

**模型迁移**：换 embedding 模型不需要全量阻塞重算；后台 task 慢慢补，每条记录写完再 sync 回去。

#### 3.2.3 Preference Memory 上限策略（解决 M3）

> 月见对用户偏好的记忆是身份连续性的核心，但不能无界。

##### 量级估算

按用户使用强度分三档建模（每档独立的偏好积累速率）：

| 档次 | 行为 | 偏好率 | 1 年 | 3 年 |
|---|---|---|---|---|
| 轻度 | 每周聊 2-3 次，纯娱乐 | ~30 chats / month × 1/30 | ~12 条 | ~36 条 |
| 中度 | 每天聊 2-5 次，AI 是日常一部分 | ~100 chats / month × 1/20 | ~60 条 | ~180 条 |
| 重度 | 多次/天，主动 "训练" 月见 | ~300 chats / month × 1/10 | ~360 条 | ~1080 条 |

> "偏好率"是从一段聊天里能提炼出独立 PREFERENCE 条目的概率。consolidator 配合语义去重（cosine ≥ 0.85 时视为同义、合并到现有 fragment 而非新建）后，实际增长率应再打 0.5-0.7 折扣。

##### 其它边界

- **prompt 注入预算**：偏好走"全量注入"（所有 PREFERENCE 在每个 prompt 都附带）。平均每条 30 token，500 条 ≈ 15k token——在现代 LLM 128k+ 上下文里仍属可接受范围
- **检索成本**：每条 384 维 float32 ≈ 1.5KB；500 条 ≈ 750KB；cosine 全量扫描 ~1ms，不是瓶颈
- **认知一致性**：超过 ~300 条独立微偏好，月见的"性格"会开始感觉散乱（互相冲突的细节增多）。这是软上限，不是硬限

##### 决策（默认值 = 500）

- `memory.preference.max_count: 500` 作为默认值
  - 覆盖中度用户 ~9 年；重度用户 ~1.5 年（带去重 0.6 折扣后是 ~16 年 / ~2.5 年）
  - prompt 预算 ~15k token 在所有主流模型可接受
  - 重度用户可在 `configs/system/memory.yaml` 调高至 2000；架构上限不设，由用户自行权衡
- 满了之后采取 **importance + LRU** 混合驱逐：
  - 选 importance 最低 10% 的子集，再按 `last_accessed_at` 最久未触达者驱逐
  - `last_accessed_at` 由 retrieval 路径更新（每次 `retrieve_relevant` 命中即刷新）
  - 命中的偏好提升 importance（小幅 +0.05，封顶 1.0）—— 用得多 → 更难驱逐
- 驱逐前发 `MemoryEvictionEvent`：
  - 默认行为：生成驱逐审计日志（含 fragment 全文）写入 `data/backup/memory-evictions/{date}.jsonl`
  - 扩展可订阅此事件做归档、外部备份、用户提醒（"月见快忘掉一些旧偏好了"）
  - **被驱逐的偏好不立即从 SQLite 物理删除**，先标记 `evicted_at`，30 天 grace period 内可恢复

**SQLite 端新列**：

```sql
ALTER TABLE long_term_memory ADD COLUMN last_accessed_at TEXT DEFAULT NULL;
ALTER TABLE long_term_memory ADD COLUMN access_count INTEGER DEFAULT 0;
ALTER TABLE long_term_memory ADD COLUMN evicted_at TEXT DEFAULT NULL;  -- 软删除时间戳，NULL = 活跃
```

#### 3.2.4 配置外部化（解决 M4）

`configs/system/memory.yaml` 承载 memory 运行时参数：

```yaml
memory:
  short_term:
    max_length: 20                # 单场景短期记忆条数上限
    importance_decay_per_turn: 0.05  # 每轮衰减（用于"重要短期"过滤）

  long_term:
    decay_rate_per_day: 0.02      # 每日衰减率（不应用于 PREFERENCE）
    weight_floor: 0.1             # 衰减下限
    retrieval_top_k: 5            # retrieve_relevant 默认返回数

  preference:
    max_count: 500                # PREFERENCE 类型上限
    eviction_policy: lru_importance  # lru_importance | strict_lru | manual

  consolidation:
    short_term_threshold: 3       # 短期记忆 ≥ 此值时触发整合
    schedule_interval_seconds: 300  # 定时检查所有场景是否需要整合
    max_parallel_per_run: 2       # 一次最多并发整合多少个场景

  jsonl:
    rotation_size_mb: 50          # 单 JSONL 文件超此大小开始 rotate
    keep_files: 5                 # 保留多少个历史 rotation 文件
    snapshot_recent_limit: 20     # 与 short_term.max_length 对齐
    snapshot_every_n_records: 20  # 每 N 条新记录生成一次 snapshot

  extension:
    short_term_max_per_scene: 200
```

Python 端通过 IPC 拿到这份配置（已有 `config_init` 通道）；TypeScript 端通过 ConfigManager。

#### 3.2.5 Consolidator 调度化（解决 M6）

新增 `MemoryConsolidationScheduler`（Python 端）：

- 由 `LifeClockManager` 在 `ambient` 心跳时触发
- 每次扫描所有 `_short_term_stores`，对达到阈值（`short_term_threshold`）的场景排队
- 用 `asyncio.Semaphore(max_parallel_per_run)` 限制并发
- 整合完成后发 `MemoryConsolidatedEvent`，附带摘要 + 提取的 fragment 列表

**关键约束**：consolidator 不再"按需"调用，而是一个守护循环。`MemoryConsolidator.consolidate(scene_id)` 仍保留作为强制触发入口（测试和 admin command 用）。

#### 3.2.6 JSONL Rotation（解决 M5, M10）

`MemorySyncManager` 写 JSONL 时检查文件大小：

```
data/state/extensions/transcripts/short_term/{scene_id}.jsonl              ← 用户可读导出（可选）
data/state/extensions/transcripts/short_term/{scene_id}.jsonl.1            ← 上一个 rotation
data/state/extensions/transcripts/short_term/{scene_id}.jsonl.2.gz         ← 更老的压缩
...
data/state/extensions/transcripts/short_term/{scene_id}.jsonl.{keep_files}.gz  ← 最老
```

> **路径收口（2026-06-03）**：记忆主库收归 `data/state/cognition/cognition.db`，不再以 `data/memory/` 作为主目录。用户可读转录 / 导出属于 `data/state/extensions/transcripts/`，迁移和删除前的安全快照属于 `data/backup/`。
>
> `data/artifacts/` 与 `data/cache/` 严格用于"丢了不心疼"的运行产物和缓存（subprocess TTS 输出、临时音频片段、avatar shell 状态文件等）；`data/packages/` 用于可重装的外部运行包。仓库根不再保留 `runtime/` 或 `tmp/`。

`snapshot_recent_limit` 与 `short_term.max_length` 同源（默认都是 20）；不再各写各的。

> **数据保护策略**：rotate 出去的 `.jsonl.N.gz` 永远只在用户主动 `pnpm memory:purge` 命令下才删除（待实现）。日常运行不会自动 GC 历史记忆——这是月见的"记忆档案"，不是日志。

### 3.3 日志模块目标形态

#### 3.3.1 trace_id 自动传播（解决 L1） —— **本次会话落地**

**Python 端**（`contextvars`）：

```python
# core/cognition/src/cognition_core.core/observability/trace_context.py
import contextvars
from typing import Optional

_trace_id_var: contextvars.ContextVar[Optional[str]] = contextvars.ContextVar(
    'selrena_trace_id', default=None
)

def get_current_trace_id() -> Optional[str]:
    return _trace_id_var.get()

def set_current_trace_id(trace_id: Optional[str]) -> contextvars.Token:
    return _trace_id_var.set(trace_id)

class TraceContext:
    """with 上下文管理器：进入时设置，退出时还原。"""
    def __init__(self, trace_id: str): self.trace_id = trace_id
    def __enter__(self): self._token = set_current_trace_id(self.trace_id); return self
    def __exit__(self, *exc): _trace_id_var.reset(self._token)
```

structlog 加一个 processor，自动从 contextvar 抓 trace_id 注入到所有日志记录。

**TypeScript 端**（`AsyncLocalStorage`）：

```ts
// core/kernel/src/core/foundation/logger/trace-context.ts
import { AsyncLocalStorage } from 'node:async_hooks';

interface TraceStore { trace_id: string; }
const _als = new AsyncLocalStorage<TraceStore>();

export function getCurrentTraceId(): string | undefined {
  return _als.getStore()?.trace_id;
}

export function withTrace<T>(trace_id: string, fn: () => T): T {
  return _als.run({ trace_id }, fn);
}
```

Winston format 加一个 inject step，从 ALS 读 trace_id 自动塞进 meta。

**入口点**：

- IPC 入站：`KernelIngressCortex` 收到一个 `PerceptionEvent` → 取出（或生成）trace_id → `withTrace(traceId, () => process(...))`
- 扩展事件订阅：`this.subscribe()` 自动 wrap handler 进 `withTrace(event.trace_id, ...)`
- AI 推理路径：Python `CognitiveLoop` 每拍 tick 开根 `TraceContext(new_trace_id())`；
  原 perception trace_id 从 PerceptionEntry 经 Appraise 写进 Moment + 经 Act 串进
  ActionCommand（下游 ChannelReplyEvent 路由键，阶段 7.5b-7 起为唯一推理路径）

**协议层约束**：所有 `*Event` 在 `protocol/src/events/` 必须含 `trace_id: string`。已有大部分；本次补全缺失的（详见 §四）。

> ⚠️ 这不是新增"feature"，是把已有的零散 trace_id 字段贯通起来：协议里已经有 trace_id，只是日志器不会自动读取它。

#### 3.3.2 UTF-8 强制（解决 L2）

```ts
new winston.transports.File({
  filename: ...,
  encoding: 'utf8',           // ← 显式
  // ...
});
```

Python 端已经显式 `encoding="utf-8"`，对齐。

#### 3.3.4 Async 异常捕获（解决 L4）

Python 端在 `cognition_core.main` 启动时：

```python
def _on_loop_exception(loop, context):
    exc = context.get('exception')
    msg = context.get('message')
    selrena_crash_logger.critical(
        '未捕获的 async 异常',
        message=msg,
        exception=str(exc) if exc else None,
        future=str(context.get('future')),
    )

asyncio.get_event_loop().set_exception_handler(_on_loop_exception)
```

TypeScript 端：`process.on('unhandledRejection', ...)` + `process.on('uncaughtException', ...)`。两者都已经在 `app.ts` 有占位但没接到 logger，本次补齐。

#### 3.3.5 Rotation / Level 配置外部化（解决 L5, L9）

见 §3.3.3 的 `observability` 配置块。

每模块 level override：getLogger(name) 创建子 logger 时读 `module_levels[name]`，找不到就用全局 level。

#### 3.3.6 时间戳统一（解决 L6, L7）

约定：

- 协议层定义 `Timestamp = string`（ISO 8601 + millisecond），格式 `2026-05-08T07:03:33.522Z`
- Python 把 microsecond 截断到 millisecond：`structlog.processors.TimeStamper(fmt="%Y-%m-%dT%H:%M:%S.%fZ")` + 自定义 processor 删掉后 3 位微秒
- TS 用 `winston.format.timestamp({ format: 'YYYY-MM-DDTHH:mm:ss.SSSZ' })`
- level：协议规定 `'debug' | 'info' | 'warn' | 'error' | 'critical'`，Python `warning` 在序列化时映射到 `warn`

#### 3.3.7 Meta key 命名约定（解决 L8）

**全部 snake_case**——理由：JSON 在 Python 自然 snake_case，logger 输出主要被 Python / shell tooling 消费；TS 调用方手动写 snake_case 也没成本。

写一份 `docs/architecture/current/09-日志字段表.md`（待补）登记保留字段：`trace_id`, `module`, `level`, `timestamp`, `source`, `error`, `stack`, `extension_id`, `scene_id`, `event_type`, ...

#### 3.3.8 异步刷盘（解决 L10，可选）

winston 已经异步；Python `RotatingFileHandler` 是同步 I/O，但当前频次（每秒数十条）远低于风险阈值。**本次不动**；标记为"未来优化"。

#### 3.3.9 Logger 测试隔离（解决 L11）

把 `logger.py` 的模块级初始化改成函数式：

```python
def initialize_logging(config: LoggingConfig) -> None: ...
```

由 `cognition_core.main` 在确定配置后调用一次。`get_logger()` 检查未初始化时给出可识别的错误。

测试通过 `pytest fixture` 在 setup 注入 mock config + tmpdir。

### 3.4 横切：可观测性管线

```
            ┌─────────────────────────────────────────┐
            │     Logger（双端，自动注入 trace_id）       │
            └─────────────────────────────────────────┘
                              │
                              ▼
            ┌─────────────────────────────────────────┐
            │  stdout JSONL（含 trace_id, module, ...）  │
            └─────────────────────────────────────────┘
                  │                              │
            Python 进程                       Kernel 进程
                  │                              │
                  └──→ Kernel IPC 收口 ──────────┘
                              │
                              ▼
            ┌─────────────────────────────────────────┐
            │  data/observability/logs/runtime.log       │
            │  + runtime.pretty.log                      │
            │  + runtime-error.log                       │
            │  （UTF-8，rotation，per-module level）       │
            └─────────────────────────────────────────┘
```

Kernel 主日志是中枢时间线。被 Kernel 监管的子进程（Cognition、engines/audio、GPT-SoVITS sidecar）
可以有自己的内部时间线；info/debug 默认进入 `logs/processes/` 或对应 sidecar 日志，模型下载进度、jieba/ModelScope/torch warning 等上游噪声不进入主线。warn/error 才摘要进入
`runtime.log`。这样既保留完整可追溯信息，又避免主启动日志被子进程自述打散。

未来可加：

- 指标导出（Prometheus）—— 不在本设计范围
- 链路追踪 UI（基于 trace_id 重建调用图）—— 单独提案

## 八、经历之流（Moment Stream）—— 月见架构蓝图 §4.1 / 脊柱①

> 本节记录 [月见架构蓝图](../../architecture/blueprint/月见架构蓝图.md) 迁移路线 **阶段 1 + 阶段 3.6**
> 的落地成果。经历之流是蓝图的脊柱① —— 一条不可变、因果优先的真相记录。
>
> **阶段 3.6 重构（2026-05-19）**：`ExperienceEvent` → `Moment`；`session_id`/
> `epoch_id` 从经历层与遥测层全部退役；遥测三层降为 `boot_id / trace_id / span_id`。

### 8.1 定位

经历之流由 **Moment** 组成 —— 月见经历到的一切（感知、思维、情绪、回复、沉默、反思）
作为不可变 Moment 追加，Moment 之间通过 `causation_ids` 编织因果网。
阶段 1 只建**记录基底 + 快照 + 回放工具**，认知核现有逻辑不动；由经历之流驱动的
状态投影、因果链的真正填充留待阶段 5（认知循环）。

### 8.2 模块构成（`core/cognition/src/selrena/experience/`）

| 文件 | 职责 |
|---|---|
| `events.py` | `Moment`（冻结）+ `MomentKind` 枚举 + `AffectSnapshot` |
| `log_writer.py` | append-only 分段写入器；异步单写者 + 批量 flush |
| `snapshot.py` | 状态快照读写，避免启动 fold 全量；原子写入，损坏快照自动隔离（不含 session/epoch） |
| `replay.py` | 读取 / 完整性校验 / fold / 按 trace 或 causation 回溯 |
| `recorder.py` | `ExperienceRecorder` —— 经历记录统一门面 |

### 8.3 Moment 结构（蓝图 §4.1）

| 字段 | 说明 |
|---|---|
| `seq` | 进程内单调递增序号（写入器赋值） |
| `moment_id` | UUIDv4 hex，Moment 唯一身份 |
| `occurred_at` | UTC 毫秒 ISO8601 时间戳 |
| `kind` | `MomentKind` 值：`perception`/`thought`/`emotion`/`reply`/`silence`/`reflection`/`arousal`（觉醒态转换，阶段 4） |
| `content` | 本刻的具体内容（按 `kind` 解释） |
| `causation_ids` | **因**：直接催生此 Moment 的上游 `moment_id` 列表（蓝图 §4.1） |
| `scene_id` | 场景 ID（不 sanitize） |
| `affect` | 当下情绪快照（valence/arousal/label） |
| `importance` | 重要性 [0,1]（Generative Agents 风格） |
| `trace_id` | 与 telemetry 关联的薄注解（可空） |
| `schema_version` | 当前 `2`（v1 → v2：因果优先重构） |

> `causation_ids` 当前允许为空 —— Moment 结构已就位，真正的因果链填充由
> 阶段 5 的认知循环驱动。

### 8.4 telemetry 三层 trace（蓝图 §6.2）

`trace_context.py` 提供 telemetry 侧 trace 三层（已不再混入经历语义）：

| 层 | 粒度 | 实现 |
|---|---|---|
| `boot_id` | 一次进程启动周期 | 进程级，启动时设定一次（参考 systemd `boot_id`） |
| `trace_id` | 跨层因果关联 | 协程级 contextvar（W3C TraceContext 对齐） |
| `span_id` | trace 内原子操作 | 协程级 contextvar；由 `tracer.py` 的 `span()` 管理 |

合成兜底：`boot_id` 已定 → `run-{boot_id}-{n}`；未定 → `synthetic-{module}-{n}`。

> 注意：trace_id / span_id **只活在 telemetry 层**，对齐 W3C / OTel 用于跨进程
> 互操作；月见的"经历"不再依赖 trace，而由 Moment + causation_ids 自洽表达。

#### 8.4.1 span（阶段 3.4 起）

`tracer.py`（Python）/ `tracer.ts`（TS）提供 `span(name)` 上下文管理器：进入
新建 + 设当前 `span_id`，退出记 `duration_ms`/`status`，落盘到
`data/observability/traces/spans-{proc}.jsonl`（结构对齐 OTel Span）。

- **Kernel 当前埋点**：`attention.flush` / `attention.ipc.perception_message`
- **认知核当前埋点**：`multimodal.route` / `memory.retrieve` / `llm.generate`
- **跨进程父 span**：IPC 入站读取信封的 `trace_id` + `span_id`，用
  `with_remote_parent_span`（Py）/ `withSpan`（TS）建为父上下文，子 span 自动
  挂到远端父之下，**span 树跨进程贯通**
- **字段表**详见 [日志字段表 §六](./09-日志字段表.md)

### 8.5 物理布局

```
data/state/experience/
├─ segments/seg-{seq:08d}.elog    分段 JSON Lines（Moment 流），不可变
└─ snapshots/
   ├─ snap-{seq:08d}.json         状态快照（不含 session/epoch）
   └─ corrupt/                    损坏快照隔离区；不参与恢复
```

> 路径选 `data/state/experience/` 而非 `data/artifacts/` —— 遵循[决策 #2](#五决策记录)：经历是真相，
> 不属于"丢了不心疼"的运行产物或缓存。

快照只是经历日志的检查点，不是真相源。`SnapshotStore.write()` 必须先写临时文件、
flush + fsync 后再用原子替换发布正式 `snap-*.json`；`load_latest()` 发现损坏快照时
将其移入 `snapshots/corrupt/` 并继续回退到上一份可用快照。

### 8.6 配置（可选块）

`GlobalAIConfig.experience`（`ExperienceConfig`，可选；内核未注入时用默认值）：
`enabled` / `segment_max_events` / `flush_interval_ms` / `flush_max_buffer` /
`snapshot_interval_s`。阶段 1 仅 Python 侧；经 protocol Schema + yaml 暴露为可调

### 8.7 开发者工具

#### 经历之流

`python scripts/experience.py {tail|show <trace_id>|causes <moment_id>|kinds|verify}`

- `tail [N]` —— 最近 N 个 Moment
- `show <trace_id>` —— 与 telemetry trace 关联的 Moment 序列
- `causes <moment_id>` —— 直接由它催生的下游 Moment（沿 `causation_ids` 走一跳）
- `kinds` —— 按 `MomentKind` 统计
- `verify` —— seq 单调连续性校验

#### 遥测（metrics / spans，阶段 3.5）

`python scripts/telemetry.py {metrics|spans} ...`

- `metrics tail [N]` / `metrics names` / `metrics grep <name>`
- `spans tail [N]` —— 最近 N 个 span
- `spans latest` —— 最近一个 trace 的 span 树（排查最新一次体感延迟；会显示
  `queue_wait_ms`、`batch_size`、`backend`、`model` 等关键属性）
- `spans tree <trace_id>` —— 把同一 trace 的 spans 还原为父子树（跨进程 span 一起）
- `spans slowest [N]` —— 耗时 Top N（默认 10）

#### 统一门面（Telemetry Facade）

`from cognition_core.core.observability import telemetry`（Python）/
`import { telemetry } from '.../foundation/logger/telemetry'`（TS）—— 一个
import 同时提供 `log` / `metric` / `span` + trace 上下文管理。现有调用方继续
按支柱 import 也无须改写。
