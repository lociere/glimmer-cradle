## 一、设计原则

- Foundation 是所有层的公共基础，任何层（Domain / Application / Host）直接 import 不产生越层
- 只依赖 `@glimmer-cradle/protocol`，不依赖 kernel 内其他任何层
- 每个模块都是**单例**或**工厂函数**，全生命周期内唯一

## 三、模块详解

### 3.1 ConfigManager（配置管理器）

**单例**，负责全项目配置的加载、校验、冻结、热重载。

**加载的配置文件**（与 [00 §六 配置体系](00-架构总览.md) 一致）：

| 文件 | 对应 schema |
|---|---|
| `configs/system/identity.yaml` | AppConfig |
| `configs/system/kernel.yaml` | IPCConfig + LifecycleConfig + IngressGateConfig |
| `configs/system/surfaces.yaml` | RendererRuntimeConfig |
| `configs/system/skills.yaml` | SkillPlaneConfig |
| `configs/system/memory.yaml` | MemoryRuntimeConfig |
| `configs/system/observability.yaml` | ObservabilityConfig |
| `configs/cognition/persona.yaml` | PersonaConfig |
| `configs/cognition/inference.yaml` | InferenceConfig |
| `configs/cognition/providers.yaml` | LLMConfig |
| `configs/cognition/knowledge-base.json` | KnowledgeBaseConfig |
| `configs/extensions/active.yaml` | 扩展启用白名单（无 schema） |
| `configs/secrets/secrets.yaml` | API Key 等敏感信息（gitignored） |
| `configs/extensions/<id>.yaml` | 各扩展自有配置（扩展自定义 Zod schema） |

**事实源与格式边界**：

- 本地客户端形态：`configs/` 是配置事实源；Control Center 设置页后续只负责读写这些结构化文件。
- 运行时形态：ConfigManager 输出的是 schema 校验后的冻结 `GlobalConfig`，调用方不读取原始 YAML/JSON，也不判断配置来自文件、UI 还是远端。
- 传输形态：跨进程与网络边界只传 JSON 结构，不把 YAML 放进 IPC、WebSocket 或 HTTP。
- 云端形态：未来无客户端部署应把事实源替换为配置服务或数据库，YAML 降级为导入、导出、bootstrap 与开发模板。

**关键方法**：

```typescript
class ConfigManager {
  static get instance(): ConfigManager;
  async init(): Promise<void>;             // YAML 加载 + ajv 校验 + 密钥注入
  getConfig(): GlobalConfig;               // 获取冻结后的配置快照
  freezeCoreConfig(): void;                // 深度冻结，禁止运行时修改
  loadKnowledgeBaseConfig(): Promise<KnowledgeBaseConfig>;  // 单独加载 knowledge-base.json
  loadEnabledExtensions(): Promise<string[]>;               // 读取启用扩展列表
}
```

**校验链路**（阶段 P.4b 起取代 Zod）：

```
YAML parse
  ↓
normalizeSystemYamlNulls(data)       ← protocol/runtime/yaml-normalizers
  ↓
validateConfig('AppConfig', data)
validateConfig('IPCConfig', data.ipc)
... 各 system 子块逐段                ← protocol/runtime/validator（ajv + useDefaults）
  ↓
Object.freeze(config)
```

ajv `useDefaults: true` 自动填 schema 中的 `default` 值；schema 单一事实源在
`protocol/src/schemas/config/`（铁律 1）。详见 [01-协议层 §四 运行时 helper](01-协议层.md)。

**配置冻结机制**：`init()` 完成后调用 `Object.freeze()` 深度冻结核心配置对象，
任何运行时修改尝试将抛出异常。Python 侧 Pydantic 模型也带 `frozen=True`
（由 codegen post-process 注入，仅 config/ 子目录）。

### 3.2 EventBus（事件总线）

**单例**，内核模块间解耦通信的唯一方式。

**特性**：
- 异步非阻塞发布
- 按 `event_type` 字段分发
- 全链路 `trace_id` 透传
- `StateSyncEvent` 等高频事件自动静默日志

**API**：

```typescript
class EventBus {
  static get instance(): EventBus;
  subscribe(eventType: string, handler: (event) => Promise<void>): void;
  unsubscribe(eventType: string, handler): void;
  publish(event: DomainEvent): Promise<void>;  // 等待所有 handler 完成
  shutdown(): Promise<void>;
}
```

**事件发布语义**：`publish()` 返回 Promise，会**等待所有订阅者的 handler 执行完成**后才 resolve。这确保了事件处理的可靠性，但也意味着 handler 不应执行耗时操作（应异步排队）。

### 3.3 Logger（日志器）

基于 **Winston**，支持控制台彩色输出 + JSON 文件输出。

**输出目标**：
- 控制台：彩色 + 模块标签格式 `HH:mm:ss.SSS [level] [module] 消息 {meta}`
- 主日志文件：JSON 格式，10MB × 5 轮转
- 错误专属文件：JSON 格式，5MB × 3 轮转

**使用方式**：

```typescript
import { getLogger } from '../../foundation/logger/logger';
const logger = getLogger('module-name');
logger.info('消息', { key: 'value' });
```

`initLogger(appConfig)` 在 `app.ts` 启动时调用一次，配置日志级别和输出路径。

### 3.4 DBManager（数据库管理器）

**单例**，管理 SQLite 连接（使用 `better-sqlite3`）。

**表结构**：
- `long_term_memory` — 长期记忆持久化
- `short_term_memory` — 短期记忆（由 Cognition 认知核同步）
- `extension_storage` — 扩展 K/V 存储（按 `extension_id` 命名空间隔离）
- `extension_short_term_memory` — 扩展级短期记忆（按 `extension_id` + `scene_id` 隔离，每场景自动淘汰至 200 条）

启动时仍会从历史 `plugin_storage` / `plugin_short_term_memory` 表执行一次迁移导入；迁移逻辑只存在于 DB 初始化边界，不作为当前架构事实继续对外暴露。

**仓库模式**：

| 仓库类 | 职责 |
|---|---|
| `MemoryRepository` | 长期记忆 CRUD（关键词搜索、时间范围查询） |
| `ExtensionStorageRepository` | 扩展沙箱 K/V 存储（每个扩展独立命名空间） |
| `ExtensionShortTermMemoryRepository` | 扩展短期记忆（实现 `ShortTermMemoryPort`，按 extension_id 隔离，自动淘汰） |

### 3.5 IngressGateManager（入站防护管理器）

**单例**，为整个系统提供输入层面的基础设施级防护。与内容过滤、注意力管理完全无关。

**三重防护机制**：

| 机制 | 职责 | 触发条件 |
|---|---|---|
| 系统就绪守卫 | 系统未就绪时拒绝一切输入 | AI 进程未启动或正在关闭 |
| 来源速率限制 | 滑动窗口限制单来源请求频率 | 同一 sourceId 在窗口内超额 |
| 全局熔断器 | 连续失败达阈值后临时拒绝所有输入 | AI 推理连续失败 N 次 |

**透明接入**：由 `PerceptionAppService.processIngress()` 自动调用，所有插件的感知输入自动获得防护，无需显式调用。

**配置来源**：`configs/system/kernel.yaml` 的 `ingress` 节：

```yaml
ingress:
  rate_limit_per_source: 30          # 单来源每窗口最大请求数
  rate_limit_window_ms: 60000        # 滑动窗口时长（ms）
  max_concurrent_requests: 10        # 全局最大并发处理请求数
  circuit_breaker_threshold: 5       # 连续失败几次后触发熔断
  circuit_breaker_recovery_ms: 30000 # 熔断恢复等待时间（ms）
```

**API**：

```typescript
class IngressGateManager {
  static get instance(): IngressGateManager;
  init(config: IngressGateConfig): void;    // 由 app.ts 在配置加载后调用
  setSystemReady(ready: boolean): void;     // 由 app.ts 在启动/关闭时调用
  admit(sourceId: string): IngressGateResult; // 评估+消费配额
  complete(success: boolean): void;          // 上报请求完成（驱动熔断器）
  stop(): void;                             // 清理定时器
}
```

**生命周期**：
1. `app.ts` 配置加载后调用 `init(config)`
2. 所有模块启动完成后调用 `setSystemReady(true)` 开放入站
3. 应用停止时调用 `setSystemReady(false)` + `stop()` 拒绝新输入并清理

## 五、Infrastructure 层说明

`core/kernel/src/core/infrastructure/` 目前**仅保留** `ipc-broker/` 一个子目录：

```
infrastructure/
└── ipc-broker/
    └── ipc-server.ts     ← ZeroMQ Router + Publisher 双套接字
```

### IPCServer

**单例**，负责 Kernel 内核与 Cognition 认知核的双向通信：

- **Router 套接字**（`tcp://127.0.0.1:8765`）：处理请求-响应模式
- **Publisher 套接字**（`tcp://127.0.0.1:8766`）：事件广播

**内置消息处理器**：
- `LIFE_HEARTBEAT` — Cognition 认知核 心跳
- `LOG` — Python 日志转发到 Winston
- `ACTION_COMMAND` — Cognition 认知核自主输出；reply 会被规范化为 `ChannelReplyPayload`
- `STATE_SYNC` — Python 状态同步

**请求超时**：每个待处理请求都有超时计时器，超时自动清理并报错。
