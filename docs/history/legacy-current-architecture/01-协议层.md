## 一、设计原则（三铁律）

详见 [CLAUDE.md](../../../CLAUDE.md) 与 [阶段P 设计](../../history/architecture-decisions/阶段P-Protocol契约层重构.md)。

### 铁律 1：跨语言必走 schema
**凡有 ≥ 2 个语言层消费的类型** → 权威定义在 `protocol/src/schemas/`（JSON Schema），
由 codegen 多端产出。**手写镜像禁止**，**没有例外**。

### 铁律 2：TS-only 设施不进 protocol
EventBus 类、TS interface ports、Exception 实现类等只有 TS 看见的设施住在
`core/kernel/src/core/foundation/` 各子目录，不混进 protocol。

### 铁律 3：扩展开发面独立成包
`@glimmer-cradle/extension-sdk` 与 `@glimmer-cradle/protocol` 平级（P.6 拆出）。
扩展作者 vs Python 认知核受众不同，版本独立。

### 2.1 包入口边界

`@glimmer-cradle/protocol` 是 workspace 包，不是源码别名。`package.json` 的运行时入口必须指向
`dist/index.js`，类型入口指向 `dist/index.d.ts`：

```json
{
  "main": "dist/index.js",
  "types": "dist/index.d.ts"
}
```

原因：Kernel dev 由 `ts-node-dev` 执行时可以处理 TypeScript，但 Electron main / 生产构建
是 CommonJS 运行时，不能直接 `require("protocol/src/index.ts")`。因此任何消费方启动前都必须确保
protocol 已构建；`kernel dev/start/build/typecheck`、`desktop-surfaces dev:electron/start/build/typecheck` 和
root `dev:all/build:all` 已通过 lifecycle script 先执行 `pnpm --filter @glimmer-cradle/protocol run build`。

## 四、运行时 helper（`runtime/`）

阶段 P.9c 起从 kernel 上提到 protocol 公共面，任何 TS 包共用。

### 4.1 `validateConfig(name, data)` —— ajv 校验

```ts
import { validateConfig } from '@glimmer-cradle/protocol';

const result = validateConfig('PersonaConfig', personaData);
if (!result.ok) {
  throw new Error(result.errors.join('; '));
}
// data 已被 ajv 原地填默认值
```

特性：
- `useDefaults: true` 自动填 schema 的 `default` 值
- 所有 15 份 config schema 一次性 addSchema —— $id 索引便于 `$ref` 解析
- ValidateFunction 缓存（首次编译，重复校验零开销）
- `additionalProperties` 由各 schema 自身决定（`removeAdditional: false`）

### 4.2 `normalizeSystemYamlNulls(obj)` —— YAML null 归一化

YAML 中 `module_levels:` 空键被 parse 成 `null`，但 schema 期望 object。
本函数原地修正（阶段 P.4a-4 调研发现的唯一 transform 用例）。

### 4.3 `AvatarShellFrame` 语义分类

`AvatarShellDownstreamFrame` 是 Kernel 发给月见可见身体的统一语义帧。它不是“UI 事件”，也不是 Live2D 私有命令；当前由 Unity Avatar Shell 消费，并允许显式 placeholder 在正式 Shell 不可用时按同一语义降级。

协议层用 `AvatarShellFrameClass` 将下行帧归纳为三类：

| class | 中文 | kind | 含义 |
|---|---|---|---|
| `expression_flow` | 表达流 | `thought` / `reply` / `emotion` | 月见正在思考、说话、呈现情绪 |
| `embodiment_control` | 身体控制 | `audio_play` / `expression` / `motion` / `lip_sync` / `parameter` / `presentation` / `idle` | 驱动身体表现、声音、动作、参数与桌面呈现 |
| `shell_lifecycle` | 外壳生命周期 | `shell_status` / `audio_status` / `audio_transcript` / `load_scene` / `unload_scene` / `ping` | 外部身体、音频能力、语音识别回执、场景和连接状态 |

TS 侧统一从 protocol 导入 helper：

```ts
import {
  getAvatarShellFrameClass,
  isAvatarShellFrameKind,
} from '@glimmer-cradle/protocol';
```

设计约束：分类只描述“月见如何显现”，不要用 Electron / Live2D / Unity 等实现名来分组。实现层可以各自解释帧，但不能发明私有分类。

`AvatarShellUpstreamFrame{kind:'audio_input'}` 是桌面端语音输入的唯一上行契约。renderer 只采集麦克风并编码为 WAV，Kernel 负责落盘、调用 ASR、下发 `audio_transcript` 回执，并在识别成功后把文本注入 `PerceptionAppService`。这样语音输入、文字输入和未来 Unity / 其他 Avatar Shell 都共享同一条感知入口。

`AvatarShellUpstreamFrame{kind:'avatar_presentation'}` 是 Control Center 对身体呈现的请求：可携带 `placement_id`、`display_scale` 与 `reset_placement`。`placement_id` 只选择完整透明身体表面的桌面驻留预设，不允许改变相机焦点或模型比例。Kernel 在 `DesktopUIController` 过滤请求后，将其转为 `AvatarShellDownstreamFrame{kind:'presentation'}` 广播给已 ready 的 Avatar Shell。请求与下行载荷虽然当前字段相近，仍保持 `AvatarPresentationRequest` / `AvatarPresentationPayload` 两种语义，避免未来把 UI 意图、权限或保存行为泄漏进身体执行协议。

### 4.4 `normalizeReplyMessages(text, messages?)` —— 回复消息切片归一化

回复链路采用 `ReplyTurn → ReplyMessage[] → ChatMessage 投影` 的结构：

| 层 | 契约 | 含义 |
|---|---|---|
| Cognition → Kernel | `ActionCommand.payload.text` | 一轮回复的完整规范正文，用于记忆、日志、审计和 fallback |
| Cognition → Kernel | `ActionCommand.payload.messages` | 可选的回复切片；代码块、分句、多平台消息可分开投递 |
| Kernel EventBus | `ChannelReplyPayload` | Kernel 将 ActionCommand 规范化后的通道回复事件，字段为 `trace_id / text / messages / emotion_state / target_channel` |
| Kernel → Avatar Shell | `AvatarShellDownstreamFrame.reply.text/messages` | 桌面与未来 Unity 共同消费的回复帧 |
| Renderer | `ChatMessage` | UI 气泡投影，不是跨层事实源 |

`normalizeReplyMessages()` 由 protocol runtime 提供：当 `messages` 缺失时，会按换行、句末标点和代码块边界把 `text`
转为稳定的消息切片；当 `messages` 已存在时，只做清洗、排序和空内容过滤。这样月见可以把“完整回复”与“分条投递”
分离，避免 renderer、NapCat 或 extension-sdk 各自发明一套消息类。

## 五、IPC 消息协议（schema-driven）

阶段 P.3b 起所有 IPC payload 走 schema codegen。消息类型 enum 在
`schemas/enums/IPCMessageType.schema.json`。

| 消息类型 | 方向 | 用途 |
|---|---|---|
| `CONFIG_INIT` | TS → Python | 初始化配置注入（仅 persona 块，其余通过 SELRENA_CONFIG env 注入） |
| `KNOWLEDGE_INIT` | TS → Python | 知识库初始化（schema: KnowledgeInitPayload） |
| `PERCEPTION_MESSAGE` | TS → Python | 感知消息（schema: PerceptionEvent） |
| `PERCEPTION_CANCEL` | TS → Python | 取消正在进行的生成 |
| `LIFE_HEARTBEAT` | 双向 | 生命心跳（schema: LifeHeartbeatPayload） |
| `AGENT_PLAN` | TS → Python | Agent 规划（schema: AgentPlanPayload；只传 `SkillCatalogSnapshot` 投影的 ready `skill_id + tool_name` 工具） |
| `AGENT_SYNTHESIS` | TS → Python | Agent 工具结果合成（schema: AgentSynthesisPayload） |
| `CONSOLIDATE_MEMORY` | TS → Python | 闲时记忆固化 |
| `LOG` | Python → TS | Python 日志转发 |
| `STATE_SYNC` | Python → TS | 状态同步（含 ArousalSnapshot） |
| `SUCCESS_RESPONSE` / `ERROR_RESPONSE` | 双向 | RPC 响应 |

> `MEMORY_INIT` / `MEMORY_SYNC` / `SHORT_TERM_MEMORY_SYNC` 等记忆同步消息已随蓝图
> 阶段 2「持久化收归认知核」**移除**（长期记忆改为认知核本地 `cognition.db` 直接写）。

## 九、铁律（开发时检查）

1. **不在 schema 之外手写镜像类型**——任何"TS interface 长得像某个 schema"都是 smell
2. **不 import kernel 路径** —— protocol/src/ 中的任何文件不得出现 `../kernel/`
3. **runtime/ 只放跨包公用的 helper** —— 业务逻辑放消费方，不要污染 protocol
4. **改 schema 立即跑 `pnpm sync:contracts`** —— CI drift 检查会卡，本地少绕一次
