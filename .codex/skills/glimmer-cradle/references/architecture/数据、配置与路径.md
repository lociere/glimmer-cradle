# 数据、配置与路径

适用任务：`configs/`、`data/`、资产、模型、缓存、备份、路径 resolver、打包路径或迁移。

## 规则

- 普通配置在 `configs/system/`、`configs/characters/<character-id>/`、`configs/extensions/`。
- 密钥只在 `configs/secrets/` 或环境变量。
- 用户状态在 `data/state/`；模型在 `data/models/`；缓存材料在 `data/cache/`；任务工作材料在 `data/work/`；可观测数据在 `data/observability/`；第三方包在 `data/packages/`；备份在 `data/backups/`；短期协调在 RunRoot。
- `assets/` 是只读默认资产源，不写用户状态。
- 开发路径和打包路径都必须通过 resolver 或 catalog，不在业务代码拼接源码相对路径。

## 实施顺序

1. 判断数据是否不可再生。
2. 确定 owner 和写入边界。
3. 需要迁移时先备份，再写 migration。
4. 更新 resolver、ignore、打包、Reference 和 Guide。
5. 搜索旧路径并删除旧写入。

## 禁止项

Renderer 直接读写 `configs/`/`data/`；模型下载到源码或安装目录；把缓存当记忆；在文档/示例写密钥；运行时继续写 legacy。

验证空态、旧版本、损坏数据、重复迁移、打包版路径和恢复路径。

## 配置变更检查

新配置要有 Schema/normalizer、默认来源、owner、是否需要重启、UI 投影和非法值错误。删除配置要迁移旧键并搜索旧默认值。密钥只显示存在/缺失/来源，不显示内容。

## 打包检查

安装目录默认只读。打包版必须写 user-data/data 域，不能因为开发路径存在而通过。Avatar/Audio/native 的构建投影不等于源码事实源，私人模型和本地缓存不得进包。

## 恢复语义

涉及不可再生状态的迁移必须能说明如何恢复：停止哪些 runtime、复制哪些备份、重新运行哪些检查、如何判断恢复成功。只写“删除缓存重启”不能用于 Cognition DB、用户模型或扩展状态。

## 何时更新 Reference

只要新增或移动路径、配置键、默认值、密钥来源、模型目录、缓存目录、打包投影，就要同步 `docs/reference/`。路径事实不应只藏在脚本、代码注释或对话记录里。

路径相关问题优先用 resolver、catalog 或 config schema 解决，不要在调用点散落 `if dev/prod`。

恢复和迁移涉及用户数据时，不要自动删除原始材料；先备份，再迁移，再提供人工恢复线索。
