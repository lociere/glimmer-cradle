# 可观测性与诊断

适用任务：日志、trace、metrics、DLQ、process log、性能、排障和错误分类。

## 规则

日志说明离散事件；trace/span 说明一次操作的关键路径；metrics 说明趋势；DLQ 保存无法安全处理但需要追查的失败。四者不能互相代替。

入口必须创建或继承 trace，并跨 IPC、WebSocket、stdio、Engine、MCP、Extension、renderer 投影延续。子进程 stdout/stderr 写 process log；主日志写摘要和路径。日志、DLQ 和 trace 不记录密钥、token、完整 provider key 或隐私大 payload。

## 诊断顺序

1. 从用户症状定位入口和 trace。
2. 沿 span 找失败阶段。
3. 看对应 runtime/process log。
4. 看 DLQ 是否有结构化失败。
5. 修复后重放同类输入，确认 trace 完整且 DLQ 不新增。

## 性能

性能问题先建立基线，再按 span 拆分模型、网络、资源准备、编排、渲染和 IO。不要跳过 ready、权限或校验来换取速度。高频 Avatar/鼠标/逐帧参数默认不进主日志。

变更字段或格式时同步 `reference/observability.md` 和排障 Guide。

## 字段检查

日志至少能定位 timestamp、level、module、event_type、trace_id、runtime/provider/extension、error_code 和 message。不是每条日志都要全量字段，但同名字段语义必须一致。

## DLQ 检查

DLQ 项应包含失败阶段、来源、trace、脱敏摘要和建议动作。修复后要确认同类输入不再进入 DLQ。不要把 DLQ 当垃圾箱，也不要在没有调查前批量删除。

## 交付检查

交付排障或性能修复时，说明原症状、定位到的阶段、修改点、验证输入和新日志/trace 证据。若没有拿到 trace，要说明原因，并至少给出下一次复现如何采集。

## 何时新增指标

如果问题需要比较前后趋势，例如启动耗时、TTS 延迟、ASR 耗时、MCP timeout、Avatar 首帧，就应新增或使用 metrics/span，而不是靠单次日志判断。指标要有 owner 和采样策略。

排障结论必须能回指具体 trace、日志片段、DLQ 项或复现步骤；没有证据的猜测只能标为假设。

新增日志时优先结构化字段，不要只写自然语言长句；这样后续才能按 runtime、provider、skill、trace 聚合。

如果日志字段被多个 runtime 使用，必须保持同名同义，避免排障脚本按字段聚合时得到错误结论。

同一问题重复出现时，应优先补指标或错误分类，而不是继续增加相同自然语言日志。
