# 架构边界与决策

适用任务：模块归属、进程边界、依赖方向、重构、ADR、蓝图或 Current 修改。

## 判断方法

- 用户可见语义、人格、情绪、记忆、推理由 Cognition 拥有。
- 生命周期、Ingress、路由、权限、状态投影由 Kernel 拥有。
- 窗口、托盘、preload、Control Center、Presence 由 Desktop 拥有。
- 复杂身体、动作、口型、透明命中、DPI 由 Avatar/Native 拥有。
- 模型执行器和媒体能力由官方 Engine 拥有。
- 可安装生态能力由 Extension/MCP/User Provider 拥有。
- 跨语言/进程结构由 Protocol 拥有。

## 决策顺序

1. 先读 Blueprint，确认长期审美和不变量。
2. 读 Current，确认当前真实结构。
3. 读 Implementation，确认代码入口和现有接线。
4. 设计唯一 owner、事实源、边界契约、ready/degraded、失败语义和删除对象。
5. 长期取舍写 ADR；目标形态改变写 Blueprint；当前结构改变写 Current。

## 禁止项

不要让 UI、平台 Adapter、Extension 或 Engine 承担领域判断；不要用 roadmap 代替架构设计；不要让两个进程共同写同一个事实源；不要保留只能靠历史解释的模块。

交付时说明新设计如何保持 Glimmer Cradle 的“认知核、中枢监督树、身体、能力生态、记忆脊柱”完整性，并区分平台层与当前角色 Selrena。

## 需要 ADR 的情况

长期改变进程边界、持久化 owner、安全/权限模型、公开 SDK、发布形态或蓝图不变量时，需要 ADR。普通实现选择不一定需要 ADR，但必须在 Implementation 或 Guide 中说清入口和验证。

## 决策反例

为了“快”把 provider key 传给 renderer；为了“方便”让 Extension import Kernel service；为了“演示”把 Unity 连接当 ready；为了“兼容”长期保留旧协议。这些都不是局部实现问题，而是架构边界错误。

## 验证设计

架构决策必须能落到验证：边界用 import/调用链验证，owner 用写入路径验证，readiness 用失败注入验证，权限用拒绝路径验证，数据域用打包版路径验证。不能验证的架构声明需要拆小或重新定义。

## 何时停止

如果新边界会改变用户数据归属、密钥处理、外部平台副作用或用户可见行为，但用户没有授权该方向，先报告设计选项。除此之外，按最小可验证切片推进，不把“需要更好设计”当成停工理由。

完成后检查 Skill、AGENTS、docs 入口是否仍指向旧结构；架构迁移不完整会让后续 agent 继续读错事实。
