# 协议与跨边界契约

适用任务：Schema、IPC、WebSocket、Avatar frame、config schema、Extension SDK、Engine 协议或跨语言结构。

## 实施顺序

1. 定位 `protocol/src/schemas/`、生成物、生产者、消费者、测试和文档。
2. 修改权威 Schema 或公开 SDK/Port，不修改生成物。
3. 明确 required/optional、默认值、unknown、错误 code、trace 和兼容策略。
4. 运行 `pnpm sync:contracts`。
5. 按生产者、映射层、消费者、UI/日志投影顺序实现。
6. 删除旧字段、旧消息、手写镜像和无期限 fallback。

## 判断

两个语言或进程都需要理解的字段进入 Protocol；单进程内部对象不进入 Protocol；SDK 暴露给扩展的结构进入 Extension SDK；Engine 内部模型参数不泄漏到 Kernel/Renderer；Unity/Cubism 对象不进入 Avatar frame。

## 验证

覆盖成功、缺字段、未知枚举、非法组合、错误 code、降级和旧字段搜索。交付时说明是否已运行 `pnpm sync:contracts`，以及 TS/Python 消费端是否验证。

禁止在 renderer、extension、Python 或 Kernel 本地手写“临时镜像类型”。

## 交付检查

- 生成物是否由命令生成而非手改。
- 所有 producer/consumer 是否同一语义。
- 错误 code 和 trace 是否延续。
- UI 是否只消费投影，没有把 view model 写回协议。
- 旧字段是否已搜索删除，若暂留是否有删除条件。

如果变更影响公开 SDK 或 Extension manifest，还要同步 `reference/extension-sdk.md` 和扩展开发指南。

## Avatar/Audio 特别注意

Avatar frame 不携带 Cubism 参数名、Unity 对象或本机窗口句柄；只携带稳定身体语义。Audio engine 协议不混入第三方库 stdout 文本；stdout 保持 JSON，日志进 process log。协议层要保护这些边界。

## 何时不进 Protocol

纯 UI 布局状态、单进程缓存、provider SDK 私有响应、Unity 内部求值类型、Python repository row、测试 fixture helper 不应进入 Protocol。只有跨边界稳定语义才进入 Schema。

如果只是内部重构，不要为了“统一”制造公共协议；公共契约一旦发布就会增加迁移成本。

对外可见的错误语义也属于契约，不能只改 message；稳定 code、降级状态和文档必须一起更新。
