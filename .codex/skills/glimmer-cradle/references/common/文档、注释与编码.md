# 文档、注释与编码

适用任务：写或修改文档、注释、术语、命名、编码风格和交付说明。

## 文档归属

按 `docs/文档维护规范.md` 选择唯一权威页：

- Blueprint：Glimmer Cradle 的解释、架构审美、长期不变量；角色身份事实归 Character Profile。
- Current：当前结构、边界、运行关系。
- Implementation：代码入口、组装、状态机、调试链路。
- Reference：字段、配置、路径、日志、SDK、打包映射。
- Guide：可执行步骤、失败恢复、验证、同步文档。
- Roadmap：未完成成果、范围、风险、验收门。
- ADR：长期取舍。
- History：退出当前事实源的证据。

非索引页必须有范围、事实依据、维护触发；Guide 还要有适用场景、前置条件、步骤、失败与恢复、验证和需要同步的文档。

## 注释

注释只解释 WHY、跨边界契约、不变量、生命周期/并发/平台限制、可删除条件。不要翻译代码，不写无 owner/期限的 TODO，不把旧设计留在注释里当兼容理由。

## 编码

中文正文、注释和文档为 UTF-8 无 BOM；代码标识符、协议字段、配置键、事件类型、扩展 ID 和文件名沿用既有英文命名。密钥、token、provider key 不进入文档、示例、Skill、日志或 agent profile。

新增或批量改写 `.ts`、`.tsx`、`.py`、`.yaml`、`.json`、`.md` 时必须保持无 BOM。PowerShell 写文件时避免会写入 BOM 的旧式 `Set-Content -Encoding UTF8` 行为；需要批量改写时运行 `pnpm check:encoding`，并使用项目 `.editorconfig`。

改动后搜索旧路径、旧术语、冲突陈述和断链；文档写“已落地”前必须核对代码、Schema、配置或测试。

## 文档质量门

Reference 必须能查字段/owner/变更规则/验证；Guide 必须能照着做并处理失败；Roadmap 必须有成果、范围、非范围、风险、验收门；Implementation 必须指向真实代码入口和链路。只有链接列表、口号、原则摘要的页面不是完成文档。

## 迁移旧文档

旧文档不能原样搬运。先拆分信息：审美进 Blueprint，当前结构进 Current，代码链路进 Implementation，精确表进 Reference，步骤进 Guide，未完成承诺进 Roadmap，历史证据进 History。迁移后搜索旧路径并更新 Skill 路由。

## 命名一致性

文档正文使用中文解释，代码名保持既有英文。Glimmer Cradle 的正式中文名是“微光摇篮”，上下文明确时文档和注释可以用项目内简称“摇篮”；代码标识符、包名、命令、环境变量、Schema、IPC、配置键和文件名仍使用稳定英文命名。不要把 Protocol、Kernel、Cognition、Extension、Skill Plane、Avatar、Presence、Engine、readiness 等术语翻译成多个版本。新增术语若跨文档使用，先加入 Current 术语表。

新增或重命名代码标识符时必须先读取 `docs/guides/development/命名规范.md`，并判断 owner 与语义：产品/企划用 Glimmer Cradle，默认角色用 Selrena，进程承载用 Host，第三方能力包用 SDK，协议适配用 Adapter，状态快照用 State 或 Snapshot，行为入口用 Controller，核心业务包用 Core。`Runtime` 只用于生命周期监督、语言/平台固定术语或第三方正式名称，不作为泛用后缀。

## 何时删除文档

如果一个页面只能重复其他权威页，删除或并入权威页；如果它记录的是历史过程，移入 History；如果它仍被 Skill 或 README 引用，先改路由再删除。不要让空壳页留在活跃目录里制造“看似完整”的错觉。

交付文档改动时，要说明迁移了哪些旧事实、哪些旧材料只保留为历史、当前开发应从哪个入口进入。
