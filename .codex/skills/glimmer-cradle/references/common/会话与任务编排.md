# 会话与任务编排

适用任务：所有总控、执行、审查、发布和生产运维会话。目标是在不牺牲工程质量的前提下，减少重复读取、重复推理、重复验证和并发写入冲突。

## 角色边界

| 角色 | 负责 | 不负责 |
|---|---|---|
| 总控会话 | 明确目标与禁止项、选择执行会话、维护里程碑状态、处理真实决策、最终验收 | 持续编码、长时间等待、完整日志分析、重复测试、直接发布或生产操作 |
| 执行会话 | 读取事实源、实现、维护验证账本、完成当前验收门、给出结构化结果 | 扩大里程碑、修改其他执行者范围、替总控决定产品取舍 |
| 审查会话 | 基于固定代码状态检查缺陷、风险、架构边界和验证缺口 | 与实现会话并行修改同一工作树 |
| 发布或运维会话 | 对固定 commit/制品执行一次发布、部署、回滚或恢复矩阵 | 边发布边改代码、与其他会话共同操作同一环境 |

小型任务不需要人为拆成多个会话；由一个执行会话端到端完成通常最高效。只有边界独立、输入输出稳定且不会写同一事实源的工作才并行。

## 下发契约

下发任务必须包含：

1. 单一用户可观察目标和明确不做的范围。
2. 必读事实源、目标目录、允许修改的边界和禁止依赖。
3. 当前 branch、commit、工作树状态以及是否允许提交。
4. 验收门、必要验证和不得重复运行的既有成功证据。
5. 外部环境、凭据、生产操作 owner 与停止条件。
6. 推荐模型和推理等级；按复杂度选择，不固定最高或最低。

执行者发现目标需要跨越另一子系统、仓库、生产环境或里程碑时，应先报告边界变化，不得自行扩大范围。

## 单一负责人

- 一个工作树同一时间只有一个会话写入。并行只用于只读调查，或使用独立 worktree 且文件所有权清晰的实现。
- 一个 Release、服务器、数据库迁移或扩展发布过程同一时间只有一个操作 owner。
- 总控不得在执行会话工作期间“顺手修复”同一范围。需要接管时，先停止原执行者、确认进程与工作树状态，再显式移交。
- 长任务由启动它的会话负责等待、终止和清理。其他会话只读取紧凑状态，不重复启动。

## 验证账本

每个执行任务维护以下最小记录：

| 字段 | 内容 |
|---|---|
| 状态 | branch、commit 或未提交 diff 标识 |
| 验证 | 已运行命令及目标范围 |
| 结果 | passed、failed、blocked |
| 证据 | 失败摘要、报告路径、Release run 或生产观察 |
| 失效条件 | 哪些代码或环境变化会要求重跑 |

验证采用逐级门禁：

1. 受影响单测、lint、schema 或语法检查。
2. 受影响包的 typecheck/build。
3. 相关子系统集成、Playwright 或生命周期测试。
4. 固定 commit 的全仓门禁。
5. 固定制品的一次 Release 和生产验收。

相同代码、依赖和环境上的 `passed` 结果可直接复用。只因另一个会话“不放心”而重跑不是有效理由。失败后先读取最小失败摘要并修复根因；未经相关改动，不重复运行同一昂贵命令。

## 输出与上下文预算

- 搜索先限定目录、文件类型和关键词；工具调用设置合理的输出上限。
- 日志先看退出码、失败步骤和最后相关片段，再按需要扩展。不得默认读取完整 CI、完整线程或整库搜索结果。
- 总控接收结构化状态：完成项、改动文件、验证账本、阻塞、风险和下一门；不复述执行过程。
- 状态无变化时不轮询、不向用户重复播报。等待长任务使用有界 wait 或紧凑快照。
- 复杂任务按里程碑拆分，但不要把一个紧密实现拆成多个互相重新读取上下文的微会话。

## 停止与切换

出现以下任一情况，先保护工作状态，再明确写出 `建议现在切换到新会话`：

- 当前目标完成，下一步进入新里程碑、子系统或仓库；
- 发生上下文压缩，关键决策难以可靠追溯；
- 同一问题连续两次理解偏差、返工或无效调试；
- 大量日志、截图、diff 或跨仓输出淹没有效上下文；
- 总控开始承担持续实现或长时间测试；
- 精简交接的成本明显低于继续重复读取。

未经用户同意不自动创建、归档或切换会话。切换前不得执行 `reset`、`clean`、无关提交或终止必要命令。

## 交接摘要

交接必须包含：

1. 当前目标与明确范围。
2. 已确认架构决策和禁止事项。
3. 当前实现、branch、工作树与提交状态。
4. 验证账本及结果。
5. 真实阻断、剩余风险和下一验收门。
6. 建议的新会话模型与推理等级。
7. 可直接下发的新会话提示词。

## 效率反模式

- 总控和执行会话同时写代码或操作生产。
- 多个会话为同一问题各自读取整库并给出平行方案。
- 单测失败时反复跑全仓、构建 Release 或部署验证。
- 没有固定 commit 就开始发布，发布失败后在生产环境现场改代码。
- 为节省单次调用选择明显不足的模型，导致多轮返工。
- 默认使用最高推理等级处理机械修改。
- 把“更多检查”当成严谨，却没有验证账本、风险映射和停止条件。
