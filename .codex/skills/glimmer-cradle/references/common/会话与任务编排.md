# 会话与任务编排

适用任务：所有总控、执行、审查、发布和生产运维会话。目标是让每个会话只有一个角色、每个里程碑只有一条决策链、每项独占资源只有一个 owner。

## 1. 角色模型

`session_role` 只能取以下一个值；同一会话不得中途换角色。需要换角色时创建新会话并重新下发契约。

| `session_role` | 负责 | 禁止 |
|---|---|---|
| `[总控]` | 锁定范围与禁止项、指定 owner、维护里程碑状态、处理跨边界决策、消费结构化结果并最终验收 | 写代码、运行长时间测试、分析完整日志、直接发布或操作生产 |
| `[执行]` | 读取事实源、在授权边界内实现、维护验证账本、完成当前验收门并回报 | 自称总控、扩大里程碑、替总控决定产品或架构取舍、改其他执行者范围 |
| `[审查]` | 基于固定 commit/diff 做只读审查，报告缺陷、风险和验证缺口 | 修改被审查工作树、边审边修、接管实现或发布 |
| `[发布/运维]` | 对固定 commit/制品执行一次获授权的发布、部署、回滚或恢复矩阵 | 边发布边改代码、扩大目标环境、与其他会话共同操作同一发布流程或环境 |

小型任务可由一个 `[执行]` 会话端到端完成，无需人为增加总控层。只允许边界独立、输入输出稳定且不写同一事实源的任务并行。

在支持编辑会话标题的界面中，建议使用 `[总控]`、`[执行]`、`[审查]`、`[发布/运维]` 前缀；标题只是识别辅助，不代替 `session_role` 契约，也不假设所有工具都有标题能力。

## 2. 决策 owner 与资源 owner

- 同一里程碑最多一个决策 owner；有总控时由当前 `[总控]` 担任。执行、审查、发布/运维会话只能在下发契约内做局部技术判断。
- 创建执行、审查或发布/运维会话不等于更换总控；除非用户明确更换或总控完成换届，原总控默认继续任职。
- 一个工作树同一时间只有一个写入 owner。并行实现必须使用独立 worktree，且文件边界和集成顺序明确。
- 一个 Release、服务器、数据库迁移、生产环境或扩展发布过程同一时间只有一个操作 owner。
- 已有 active owner 时，新总控先接收其会话 ID、资源、状态和回报路径；新总控与新执行者都不得写入同一工作树或操作同一外部环境。
- 需要接管独占资源时，先要求原 owner 停止，确认命令、进程、工作树和临时资源状态，再显式移交；不得靠新会话直接覆盖。
- 长任务由启动它的 owner 等待、终止和清理。其他会话只消费紧凑快照，不重复启动。

## 3. 创建角色会话与更换总控

不要单独说“切换会话”，也不得先给通用切换提示、再在下一句补充角色。创建新角色和结束当前会话必须使用不同的强类型动作：

- `建议创建执行会话`：当前总控保留，向唯一执行 owner 下发新实现或验证任务。
- `建议创建审查会话`：当前总控保留，对固定代码状态进行独立只读审查。
- `建议创建发布/运维会话`：当前总控保留，向唯一环境 owner 下发固定制品操作。
- `建议现在切换到新会话（更换总控会话）`：当前总控发生上下文健康问题或用户明确要求换届；这是决策权交接，不是实现任务下发。
- 其他角色需要替换时使用 `建议现在切换到新会话（更换执行会话）` 等包含精确切换短语和目标角色的措辞，并先保护、停止和移交其独占资源。

里程碑完成、进入实现阶段或进入发布阶段，本身不要求更换总控。通常由原总控保留决策权，并创建相应角色会话。只有以下情况才建议更换总控：

- 总控发生上下文压缩，关键决策已难以可靠追溯；
- 总控连续两次出现同一理解偏差、返工或无效调试；
- 大量日志、截图、diff 或跨仓输出已淹没总控所需的决策上下文；
- 总控持续承担实现、长时间测试、发布或生产操作，角色边界已经失守；
- 一份精简总控交接明显比继续重复读取更可靠；
- 用户明确要求更换总控。

未经用户同意不自动创建、归档或更换会话。建议前先保护当前工作状态，不执行 `reset`、`clean`、无关提交，也不中断必要命令。

## 4. 新会话启动契约

每个新会话的首条提示必须显式包含：

1. `session_role`：四类角色之一。
2. `parent_controller`：下发或交接来源的总控会话 ID；没有总控时写 `none`。
3. 单一用户可观察目标和明确禁止项。
4. 必读事实源、允许修改边界和禁止依赖。
5. 当前 branch、commit、工作树状态和提交权限。
6. `exclusive_resources`：工作树、Release、服务器、数据库或生产环境及其现有 owner。
7. `authorized_actions`：允许写入、提交、测试、外部操作的精确范围。
8. `validation_ledger`：可复用的既有结果、必跑门禁、证据和失效条件。
9. `stop_conditions`：越界、凭据缺失、owner 冲突、需要产品取舍或验收门失败时如何停止。
10. `report_contract`：回报对象、结构、频率和完成定义。
11. 推荐模型与推理等级；按复杂度选择，不默认最高或最低。

新会话必须先复述 `session_role`、目标、禁止项、独占资源和 owner，再采取任何写入或外部操作。字段缺失时先做安全的只读核对；不得假定自己是总控、写入 owner 或生产 owner。

执行者发现目标需要跨越另一子系统、仓库、环境或里程碑时，按 `report_contract` 报告边界变化并停止扩张。

## 5. 总控换届

总控换届是显式的决策权转移：

1. 旧总控冻结新决策，整理交接摘要和所有 active 会话 ID/owner。
2. 新总控读取交接，确认角色边界、未决决策、active owner 和回报路径。
3. 新总控明确回复：自己不直接写代码、运行长时间测试、发布或操作生产。
4. 用户或既定授权确认接任后，新总控成为唯一决策 owner。
5. 旧总控停止下发、变更范围和最终验收；只转发换届前已在途的必要结果，不再形成第二决策链。

总控换届不暂停仍安全运行的执行任务，也不改变它们的资源 owner。新总控不得让新执行者碰触 active owner 的工作树或环境。

### 可复制：总控换届提示词

```text
session_role: "[总控]"
parent_controller: "<旧总控会话 ID>"

你是 Glimmer Cradle 的新总控会话。单一目标：<当前里程碑与用户可观察结果>。

禁止项：
- 不直接写代码；
- 不运行或等待长时间测试；
- 不执行发布或生产操作；
- 不重复执行验证账本中仍有效的成功检查。

必读事实源：<AGENTS.md、项目 Skill、canonical references、必要 docs>
当前状态：branch=<branch>；commit=<commit>；worktree=<状态>
exclusive_resources：
- <工作树/服务器/Release/环境>：owner=<会话 ID>；状态=<active/idle/待移交>
active_sessions：
- <角色、会话 ID、目标、owner、回报路径>

authorized_actions：
- 锁定范围、下发任务、处理跨边界决策、汇总状态和最终验收；
- 只读检查限于执行结果所需的最小证据。

validation_ledger：
- <代码状态；命令；结果；证据；失效条件>

stop_conditions：
- 发现 owner 冲突、交接缺口或需要用户产品取舍时停止下发并报告；
- 不得自行接管 active 资源。

report_contract：
- 向用户回报范围、owner、里程碑状态、真实阻断、风险和下一验收门；
- 不复述完整执行日志。

推荐模型/推理：<选择及理由>

启动后先回复：
“已确认 session_role=[总控]。我是当前里程碑的决策 owner；我不直接写代码、运行长时间测试、发布或操作生产。已接收的 active 会话及资源 owner：<列表>。”
```

## 6. 执行任务下发

创建执行会话时，总控继续任职。执行者只取得明确资源和动作的授权，不取得里程碑决策权。

### 可复制：执行下发提示词

```text
session_role: "[执行]"
parent_controller: "<当前总控会话 ID>"

你是本里程碑的唯一执行会话，不是总控会话。
单一用户可观察目标：<一个可验收结果>

禁止项：
- 不扩大里程碑或修改其他执行者范围；
- 不替总控决定产品或跨边界架构取舍；
- 不触碰未授权工作树、Release、服务器或生产环境。

必读事实源：<AGENTS.md、项目 Skill、canonical references、必要 docs>
允许修改：<目录/文件>
禁止修改/依赖：<目录/系统/依赖>
当前状态：branch=<branch>；commit=<commit>；worktree=<状态>；提交权限=<允许/禁止>

exclusive_resources：
- <工作树/服务器/环境>：owner=<本会话 ID 或待分配>；
- 已有 active owner：<会话 ID、资源；无则写 none>。

authorized_actions：
- <允许的写入、命令、测试、提交或外部操作>。

validation_ledger：
- 可复用：<代码与环境状态；命令；passed 证据；失效条件>
- 必须运行：<按风险排序的门禁>
- 禁止重复：<仍有效的昂贵检查>

stop_conditions：
- 目标越界、owner 冲突、需要用户取舍、凭据缺失或前置门失败时停止并回报；
- 长任务由本会话负责等待、终止和清理。

report_contract：
- 回报给 <总控会话 ID>；
- 格式为完成项、改动文件、验证账本、阻断、风险、下一门、commit/工作树状态；
- 完成定义：<明确验收门>。

推荐模型/推理：<选择及理由>

启动后先回复：
“已确认 session_role=[执行]。我不是总控会话，不扩大里程碑；我只操作 <exclusive_resources>，并向 <parent_controller> 回报。”
```

## 7. 审查任务下发

审查必须固定代码状态且默认只读；发现问题后回报总控，不得边审边修。

### 可复制：审查下发提示词

```text
session_role: "[审查]"
parent_controller: "<当前总控会话 ID>"

你是本里程碑的独立审查会话，不是总控或执行会话。
单一目标：审查固定状态 <commit/diff/制品> 的 <风险范围>。

禁止项：
- 不修改任何工作树、代码、文档、Issue 或 PR；
- 不运行发布或生产操作；
- 不把建议修法当成已完成实现；
- 不扩大到未授权子系统。

必读事实源：<AGENTS.md、项目 Skill、审查 reference、必要 docs>
fixed_state：branch=<branch>；commit=<完整 SHA>；diff=<范围>
exclusive_resources：
- 被审查工作树：owner=<执行会话 ID>；本会话只读；
- active owner：<会话 ID、资源；无则写 none>。

authorized_actions：
- 读取固定代码、目标测试和必要证据；
- 运行获授权的短时只读检查；不得重跑仍有效的昂贵门禁。

validation_ledger：
- 可复用：<命令、结果、证据、失效条件>
- 审查重点：<边界、契约、生命周期、安全、验证缺口>

stop_conditions：
- fixed_state 变化、需要写入、缺少关键事实或发现 owner 冲突时停止并回报。

report_contract：
- 回报给 <总控会话 ID>；
- 先按严重度列问题，每项包含文件/链路、证据、风险和建议修法；
- 再列未发现问题、验证缺口和剩余风险；
- 无问题时明确写“未发现阻塞问题”，不得编造问题。

推荐模型/推理：<选择及理由>

启动后先回复：
“已确认 session_role=[审查]。我不是总控或执行会话；我只读审查 <fixed_state>，不修改被审查工作树，并向 <parent_controller> 回报。”
```

## 8. 发布/运维任务下发

发布/运维必须固定 commit 与制品、目标环境、回滚门和唯一操作 owner。没有明确授权不得执行。

### 可复制：发布/运维下发提示词

```text
session_role: "[发布/运维]"
parent_controller: "<当前总控会话 ID>"

你是本次发布/运维操作的唯一环境 owner，不是总控或实现会话。
单一目标：对 <固定 commit/制品> 在 <目标环境> 完成 <发布/部署/回滚/恢复>。

禁止项：
- 不修改代码或边发布边修复；
- 不操作未列出的环境、数据库、域名或制品；
- 不重复创建 Release、部署或迁移碰运气；
- 不输出密钥、token、主机地址或签名 URL query。

必读事实源：<AGENTS.md、项目 Skill、发布/运维指南、回滚手册>
fixed_state：commit=<完整 SHA>；artifact=<版本、摘要、digest>；environment=<目标>
exclusive_resources：
- <Release/服务器/数据库/环境>：owner=<本会话 ID>；
- 其他 active owner：<会话 ID、资源；无则写 none>。

authorized_actions：
- <精确列出允许的发布、备份、部署、验证、回滚和清理动作>；
- 凭据只使用既有安全来源，不读取或回报其内容。

validation_ledger：
- 发布前：<固定 commit/制品与既有通过门禁>
- 环境前置：<备份、容量、权限、健康状态>
- 发布后：<readiness、版本、关键链路、日志/指标>
- 回滚门：<触发条件、恢复步骤、恢复成功判据>

stop_conditions：
- commit/制品不固定、备份失败、owner 冲突、凭据缺失、前置门失败或环境偏离时立即停止；
- 失败后读取最小摘要并按回滚门行动，不现场改代码。

report_contract：
- 回报给 <总控会话 ID>；
- 包含时间、固定状态、执行动作、验证证据、回滚状态、环境与临时资源清理；
- 长任务由本会话唯一负责等待、终止和清理。

推荐模型/推理：<选择及理由>

启动后先回复：
“已确认 session_role=[发布/运维]。我是 <目标环境> 的唯一操作 owner；我只操作固定的 <commit/制品>，不修改代码，并向 <parent_controller> 回报。”
```

## 9. 验证账本

每个任务维护以下最小记录：

| 字段 | 内容 |
|---|---|
| 状态 | branch、commit 或未提交 diff 标识 |
| 验证 | 已运行命令及目标范围 |
| 结果 | passed、failed、blocked |
| 证据 | 失败摘要、报告路径、Release run 或生产观察 |
| 失效条件 | 哪些代码、依赖或环境变化要求重跑 |

验证采用逐级门禁：

1. 受影响单测、lint、schema 或语法检查。
2. 受影响包的 typecheck/build。
3. 相关子系统集成、Playwright 或生命周期测试。
4. 固定 commit 的全仓门禁。
5. 固定制品的一次 Release 和生产验收。

相同代码、依赖和环境上的 `passed` 结果可直接复用。另一个会话“不放心”不是重跑理由。失败后先读取最小失败摘要并修复根因；未经相关改动不重复运行同一昂贵命令。

## 10. 状态、输出与交接

- 搜索先限定目录、文件类型和关键词；日志先看退出码、失败步骤和最后相关片段，再按需要扩展。
- 总控只接收结构化状态：完成项、改动文件、验证账本、阻断、风险、下一门和资源状态；不重复读取完整日志或实现。
- 状态无变化时不轮询、不重复播报。等待长任务使用有界 wait 或紧凑快照。
- 复杂任务按里程碑拆分，但不要把紧密实现拆成多个互相重读上下文的微会话。

任何角色交接摘要必须包含：

1. 当前 `session_role`、接任角色、`parent_controller` 和回报路径。
2. 当前目标、明确范围、架构决策和禁止事项。
3. branch、commit、工作树、未提交 diff 和提交状态。
4. active 会话 ID、独占资源、当前 owner、命令和临时资源状态。
5. 验证账本、真实阻断、剩余风险和下一验收门。
6. 推荐模型与推理等级。
7. 对应角色的可直接复制提示词。

## 11. 反模式

- 用模糊的“切换会话”同时指代创建执行者和更换总控。
- 一个会话自称多个角色，或执行者在完成实现后自行接管总控验收。
- 总控和执行会话同时写代码，或多个执行者写同一工作树。
- 新总控忽略仍在运行的执行会话，重新下发第二 writer。
- 多个会话为同一问题各自读取整库、重跑相同验证并给出平行决策。
- 没有固定 commit 就开始发布，发布失败后在生产环境现场改代码。
