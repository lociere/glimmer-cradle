# 测试与交付

适用任务：选择验证范围、运行测试、交付结果或说明未验证风险。不要用“类型通过”代替功能完成。

## 最低基线

- TypeScript：`pnpm typecheck`、`pnpm build`
- Cognition：`cd core/cognition; uv run pytest -q`
- Audio：`cd engines/audio; uv run pytest -q`
- UI：相关 desktop typecheck/build、Playwright、必要时 Electron 实机
- Protocol：`pnpm sync:contracts` 后再运行两端验证

## 风险到验证

| 风险 | 必须补充 |
|---|---|
| runtime/lifecycle | 启动、ready、degraded、failed、restart、stop、子进程回收 |
| Schema | 成功、缺字段、未知枚举、非法组合、旧字段搜索 |
| 数据迁移 | 备份、旧版本、损坏样本、重复迁移、回滚 |
| Avatar | 透明、首帧、DPI、多显示器、拖动、动作、口型、打包版 |
| Audio | health、warmup、synthesize/recognize、timeout、process log |
| Extension/MCP | 注册、拒权、确认缺失、调用、断连、撤销、dispose |
| 可观测性 | trace、日志、metrics、DLQ、脱敏 |

## 交付说明

交付必须包含：完成行为、影响边界、已运行验证及结果、未运行验证及原因、真实剩余风险、已同步文档。若环境缺失导致无法验证，给出具体缺失项和可复现命令。

禁止通过放宽测试、吞错、改成 warn、保留假 fallback 或跳过 ready 检查来让验证“通过”。

## 未验证时怎么写

未验证不是失败，但必须具体。例如：“未运行 Avatar 打包版验证，因为本机没有 Unity batchmode/目标 runtime；已运行 desktop typecheck，剩余风险是打包版 `host_ready` 和 native composition 未实机确认。”不要写“未测试”四个字就结束。

## 常见错误

- 只跑 `pnpm typecheck` 就声称功能完成。
- 改 Cognition 但没有跑 `uv run pytest -q`，也不说明原因。
- 改 Avatar 但只看 renderer 页面，不验证 Unity/Native。
- 改 Schema 但忘记 `pnpm sync:contracts`。
- 日志里有错误或 DLQ 但交付说明不提。

## 文档同步验证

如果验证改变了事实，例如发现某 runtime 只能 degraded、某配置默认值不同、某打包路径已迁移，必须同步 Reference 或 Implementation。测试结果不能只留在对话里，否则下一次开发仍会读到旧事实。
