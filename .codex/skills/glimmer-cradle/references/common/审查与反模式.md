# 审查与反模式

适用任务：代码审查、设计自查、文档自查、发现架构异味时判断是否需要重构。

## 审查顺序

1. 事实源是否唯一：字段、配置、路径、状态、catalog、readiness 是否各有 owner。
2. 边界是否正确：Kernel/Cognition/Renderer/Extension/Engine 是否各守职责。
3. 契约是否稳定：跨进程/语言是否走 Schema、Port 或 SDK。
4. 生命周期是否完整：start、ready、degraded、failed、restart、stop、dispose 是否覆盖。
5. 失败是否可解释：错误 code、trace、日志、DLQ、UI 状态是否一致。
6. 旧路径是否删除：没有永久兼容壳、影子 bridge、双份字段。
7. 物理形态是否同步：目录结构、文件名、配置键、默认模板和加载链路是否真实采用新架构。
8. 验证是否匹配风险：不是只跑 typecheck。

## 常见反模式

- Renderer 直接读配置、数据、日志或 Unity 工程文件。
- Kernel 写人格判断、回复语义或记忆解释。
- Cognition 访问平台 IO、窗口、剪贴板或外部平台原始 payload。
- Extension import Kernel 内部 service 或绕过 SDK Port。
- Engine 在第一次业务请求才下载/准备 required 模型。
- `host_hello`、进程存在或 socket connected 被当作 ready。
- `contributes.skills` 有目录项但 handler 不存在，却仍可执行。
- 旧字段 optional 双轨长期存在，没有删除条件。
- 架构概念改名后仍继续使用旧目录、旧配置文件或旧入口作为运行时主线。
- 文档只写链接和原则，不给事实、步骤或验证。

发现反模式时，优先修边界和事实源；如果属于目标形态变化，先更新 Blueprint/ADR。

## 审查输出

审查结论要按严重度和证据写：阻塞问题、建议问题、可后续处理。每条问题最好包含文件/链路、违反的边界、用户可见风险和建议修法。不要只写“这里不优雅”；要说明它造成了双事实源、假 ready、泄露权限、无法诊断或维护成本。

## 可接受的临时状态

临时 fallback、双轨或兼容桥只有在迁移窗口内可接受，且必须有 owner、删除条件、验证方式和文档记录。没有删除条件的兼容就是新架构的一部分，应重新设计。

## 文档审查重点

文档也会产生反模式：Reference 写成散文、Guide 只有原则、Roadmap 夹带已完成事实、Implementation 不指向代码、Skill reference 复制过期路径。审查文档时按质量门判断，不以“有一个文件存在”视为完成。

## 何时升级为架构问题

如果问题需要改变 owner、进程边界、公开契约、数据域、权限模型或发布形态，不要只在当前 PR 打补丁；应要求设计更新、ADR 或里程碑收口。局部修复不能掩盖结构性问题。
