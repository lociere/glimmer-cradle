# 开发工作流

适用任务：任意实现、调试、文档、审查或交付任务。目标是先确定事实源和边界，再动代码；不凭记忆猜当前实现。

## 顺序

1. 检查 `git status --short`，保留用户已有改动。
2. 读取 `docs/README.md` 与 `docs/guides/开发手册.md`，按任务进入 Blueprint、Current、Implementation、Reference 或 Guide。
3. 用 `rg` 找入口、owner、生产者、消费者、配置、测试和旧术语。
4. 写清：用户可观察目标、唯一事实源、生命周期 owner、跨边界契约、成功/失败/降级语义、验证矩阵。
5. 单层改动遵循既有抽象；跨层改动按 Schema/Port、生产者、映射、消费者、投影顺序。
6. 删除被替代的旧入口、旧字段、旧 bridge、旧配置和无期限 fallback。
7. 架构升级必须检查物理形态：目录结构、文件名、配置键、默认模板、加载链路、测试和文档是否已同步到目标架构。
8. 运行风险匹配验证，交付时区分已验证、未验证、剩余风险。

## 判断

- 改字段/事件：先读 `reference/protocol.md` 和 `implementation/Protocol契约层实现.md`。
- 改配置/路径：先读 `reference/configuration.md` 或 `reference/data-layout.md`。
- 改 runtime：先读 `current/03`、`current/05` 和 `implementation/Kernel与Runtime实现.md`。
- 改 UI/Avatar：先读 `implementation/Desktop与Avatar实现.md` 和 `reference/ui-design-tokens.md`。
- 改 Extension/Skill：先读 `reference/extension-sdk.md` 和 `implementation/Extension与SkillPlane实现.md`。

架构不变量改变先更新 Blueprint 或 ADR；实现落地更新 Current/Implementation；字段和路径更新 Reference；操作变化更新 Guide；未完成承诺进入 Roadmap。

## 交付前检查

- 有没有把用户原有改动覆盖掉。
- 有没有把临时兼容、旧入口、旧字段留在主线。
- 有没有留下旧目录或旧配置文件，只靠文档解释成新架构。
- 有没有只改实现不改唯一事实源文档。
- 有没有跳过实际风险对应的验证。
- 有没有把密钥、个人路径、provider key、私人模型写进输出。

如果任务只是分析或审查，不要擅自修改代码；但可以做只读检查并给出证据。若任务要求实现，默认推进到可验证结果，而不是停在计划。

## 阻塞处理

真正阻塞只包括：缺少用户必须决定的产品取舍、缺少无法替代的外部凭据/工具、或同一事实源连续核对后仍无法判断。阻塞时报告已查证内容、阻塞点、需要的输入和可继续的最小下一步。不要因为任务范围大就停在“建议规划”。
